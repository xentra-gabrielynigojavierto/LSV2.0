# STORY ID
LS-REPORTS-08-002 — Server-Derived Actor Identity Enforcement

# OBJECTIVE
Ensure all actor/user attribution is derived from authenticated context and cannot be spoofed by client input.

---

# REQUIRED IMPLEMENTATION

# A. Identify All Actor Fields

Search for usage of:
- RequestedByUserId
- CreatedByUserId
- UpdatedByUserId
- PublishedByUserId
- any similar fields

Across:
- DTOs
- Services
- Audit calls
- Execution
- Export
- Scheduling
- Views
- Templates

---

# B. Actor Context Source

Use:

ICurrentTenantContext.UserId

(already introduced in LS-REPORTS-08-001)

---

# C. Service Layer Enforcement

For ALL services:

Replace patterns like:
var actorId = request.RequestedByUserId;

With:
var actorId = _currentTenantContext.UserId;

---

## If UserId is NULL:
- block operation
- return 403 Forbidden or 401 Unauthorized (match existing pattern)

DO NOT fallback to:
- request values
- hardcoded "system"
- null

---

# D. Affected Services

Update actor handling in:

- ReportExecutionService
- ReportExportService
- ReportScheduleService
- TenantReportViewService
- Template management services
- Assignment services
- Override services

---

# E. DTO Handling Strategy

## Keep DTO fields (DO NOT remove yet)
- maintain backward compatibility

## But:
- IGNORE all client-supplied actor fields
- DO NOT use them anywhere in logic

Optional:
- log if client sends mismatched values (debug only)

---

# F. Audit Integration

Ensure ALL audit events use:

actorUserId = _currentTenantContext.UserId

NOT:
- request DTO values

---

# G. Scheduling Special Case

For scheduled runs:

Choose and apply consistently:

## Recommended:
- use schedule.CreatedByUserId as business actor
- OR use current user for "Run Now"
- system-triggered runs must still be attributable

DO NOT lose attribution.

---

# H. Execution & Export

Ensure:
- execution requests ignore RequestedByUserId
- export requests ignore RequestedByUserId
- actor always comes from context
