OBJECTIVE
==================================================

Implement the full CareConnect user workflow UI on top of the stabilized backend (LSCC-001, LSCC-002, LSCC-002-01).

Enable end-to-end usage:
Provider discovery → Referral → Acceptance → Appointment booking → Tracking

Do NOT redesign backend logic.
Do NOT introduce new domain models.
Do NOT bypass API contracts.

This is a UI + integration layer implementation.

==================================================
SCOPE (STRICT)
==================================================

You are implementing:

A. Dashboard (role-aware)
B. Provider discovery (map + list integration)
C. Provider detail (decision-ready)
D. Referral creation flow
E. Referral detail + timeline
F. Provider referral inbox (accept/decline)
G. Appointment booking flow
H. Appointment detail view

Do NOT:
- add analytics
- add messaging/chat
- add billing/lien integrations
- redesign backend APIs
- create new core entities

==================================================
ARCHITECTURE RULES (MANDATORY)
==================================================

1. MUST use existing API contracts (LSCC-001/002)
2. MUST respect backend authorization and scoping
3. Separate:
   - UI components
   - API service layer
   - state management
4. No hardcoded statuses — use backend enums
5. All flows must handle:
   - loading states
   - empty states
   - error states
6. Do NOT duplicate business logic in frontend
7. Prefer reusable UI components

==================================================

Backend is COMPLETE and provides:

- Provider search APIs
- Referral APIs (scoped + secured)
- Appointment APIs (scoped + secured)
- Provider availability endpoint:
  GET /api/providers/{id}/availability
- Capability-based authorization
- Org-scoped data access

Frontend already has partial components:
- provider pages
- referral pages
- appointment components
- map components

These must be reused and completed — not rewritten from scratch.

==================================================
REQUIREMENTS
==================================================

--------------------------------------------------
A. DASHBOARD (ROLE-AWARE)
--------------------------------------------------

Create dashboard page:

/careconnect/dashboard

Behavior:

Law Firm (Referrer):
- active referrals list
- upcoming appointments
- quick link: Find Providers

Provider (Receiver):
- incoming referrals (pending)
- today’s appointments
- quick link: Referral Inbox

Requirements:
- role detection based on capabilities or role flags
- minimal but functional UI
- must not be blank

Fallback:
- empty state messaging for no data

--------------------------------------------------
B. PROVIDER DISCOVERY (MAP + LIST)
--------------------------------------------------

Enhance provider discovery:

/careconnect/providers

Requirements:
1. Map + list must use SAME query source
2. Filters:
   - name
   - location
   - specialty (if available)
3. Map interaction:
   - clicking marker selects provider
   - sync selection with list

Behavior:
- selection persists across UI interaction
- avoid duplicate fetches

--------------------------------------------------
C. PROVIDER DETAIL
--------------------------------------------------

Route:
/careconnect/providers/{id}

Must include:

- name
- specialties/services
- locations
- availability preview (using API)

Availability preview:
- call availability endpoint
- show limited upcoming slots (e.g., next 5)

Primary CTA:
- "Refer Patient"

Fallback:
- no availability → show empty state (not error)

--------------------------------------------------
D. REFERRAL CREATION
--------------------------------------------------

Entry:
- from provider detail

UI:
- modal or dedicated page

Fields:
- patient name
- basic info (minimal required)
- notes

Behavior:
- POST /api/referrals
- on success → redirect to referral detail

Fallback:
- validation errors inline
- API failure handled gracefully

--------------------------------------------------
E. REFERRAL DETAIL + TIMELINE
--------------------------------------------------

Route:
/careconnect/referrals/{id}

Sections:

1. Header
- patient
- provider
- status

2. Timeline (required)
- status changes
- existing notes (read-only for now)

3. Actions (role-based)

Referrer:
- "Book Appointment"

Provider:
- Accept
- Decline

Requirements:
- reflect real backend status
- no frontend state overrides

Fallback:
- 404 → redirect or show access error

--------------------------------------------------
F. PROVIDER REFERRAL INBOX
--------------------------------------------------

Route:
/careconnect/referrals (provider view)

Requirements:
- list incoming referrals
- show:
  - patient
  - referring org
  - status

Actions:
- Accept
- Decline

Behavior:
- PATCH /api/referrals/{id}/status

UI updates immediately after action

Fallback:
- API failure → revert UI state

--------------------------------------------------
G. APPOINTMENT BOOKING FLOW
--------------------------------------------------

Entry:
- from referral detail

Steps:

1. Fetch availability:
   GET /api/providers/{id}/availability

2. Display slots

3. Select slot

4. Confirm booking

Behavior:
- POST /api/appointments
- update referral status → Scheduled

Fallback:
- slot unavailable → refetch + retry
- no slots → show empty state

--------------------------------------------------
H. APPOINTMENT DETAIL
--------------------------------------------------

Route:
/careconnect/appointments/{id}

Display:
- date/time
- provider/location
- status

Actions:
- cancel
- reschedule (if endpoint exists)

Requirements:
- use backend status
- reflect real-time updates

--------------------------------------------------
STATE MANAGEMENT RULES
--------------------------------------------------

- Use centralized API service layer
- Avoid duplicating fetch logic across components
- Cache where reasonable, but do NOT override backend truth

--------------------------------------------------
ERROR HANDLING
--------------------------------------------------

Every page must handle:

1. Loading
