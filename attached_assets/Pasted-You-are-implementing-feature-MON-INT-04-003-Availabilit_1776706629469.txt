STEP 3 — DEFINE THE AVAILABILITY BAR DESIGN

Design an external-facing availability bar visualization.

The timeline should:
- show historical availability for each component
- use simple color-coded bars
- be easy to scan on desktop and acceptable on mobile

Recommended default:
- 24 hourly bars for the last 24h
OR
- daily bars for a longer window if the current API shape supports it better

Choose the most credible version based on actual available data, and document why.

Color mapping must be explicit:
- Healthy / Operational → green
- Degraded → amber/yellow
- Down / Outage → red
- Unknown / insufficient data → gray

Each component row should still show:
- component name
- current status
- availability bar strip
- optional compact uptime % if safely derivable from existing rollup data

Do NOT add charts yet.
Do NOT add hover-only interactions as the only way to understand the bars.
Public users should get value without needing hidden UI.

Document:
- chosen time window
- bar count
- color mapping
- any labels / legend
- why the design is appropriate for a public status page
