Validation Tasks:

STEP 1 — Build Validation
Run and document results for:
- dotnet build Notifications.Infrastructure
- dotnet build Notifications.Api
- Control Center TypeScript/build validation using the existing project command

If any build fails:
- Fix the smallest safe issue.
- Re-run validation.
- Document the error and fix.

STEP 2 — Adaptive Routing Functional Validation
Validate:
- adaptive_quality selects the highest quality provider when sufficient quality snapshots exist.
- adaptive_quality falls back to priority when quality data is missing or below minimum threshold.
- adaptive_balanced considers provider quality and cost where both are available.
- adaptive_balanced falls back to hybrid/priority when data is insufficient.
- adaptive_regional uses country-specific provider quality when inferred country data exists.
- adaptive_regional falls back to adaptive_quality/priority when country cannot be inferred.
- excluded providers are never selected.
- MaxEstimatedCostPerMessage is respected.
- RequireHealthyProvider is respected if health data exists.

STEP 3 — Regional Inference Validation
Validate:
- +1 maps according to documented behavior.
- +44 maps to GB.
- +52 maps to MX.
- +61 maps to AU.
- +91 maps to IN.
- +33 maps to FR.
- +49 maps to DE.
- unknown or malformed phone numbers return null safely.
- raw phone numbers are not persisted.

STEP 4 — Existing Routing Regression Validation
Validate existing modes remain unchanged:
- priority
- cost_optimized
- health_optimized
- hybrid
- regional

Confirm:
- Twilio default behavior remains intact.
- Vonage support remains intact.
- unsupported provider behavior remains safe.
- no duplicate sends are introduced.
- contact/preference enforcement still occurs before routing.

STEP 5 — Optimization API Validation
Validate:
- GET /v1/admin/sms/routing/quality
- GET /v1/admin/sms/routing/quality/trends
- GET /v1/admin/sms/routing/latency
- GET /v1/admin/sms/routing/regional
- GET /v1/admin/sms/routing/optimization

Confirm:
- endpoints require PlatformAdmin/admin authorization.
- responses do not expose credentials, SettingsJson, CredentialsJson, raw phone numbers, recipient JSON, or raw provider payloads.
- filters work where implemented:
  - tenantId
  - provider
  - providerConfigId
  - providerOwnershipMode
  - countryCode
  - region
  - from
  - to

STEP 6 — Control Center UI Validation
Validate:
- SMS Routing page still loads.
- Optimization tab renders.
- Provider quality data renders.
- Regional analytics data renders.
- Latency/optimization sections render.
- Empty states render safely.
- Error states render safely.
- No credentials, raw phones, webhook URLs, or raw payloads are displayed.

STEP 7 — Security Validation
Search/inspect implementation to confirm:
- RecipientPhoneForInferenceOnly is transient only.
- Raw phone numbers are not stored in:
  - SmsProviderQualitySnapshot
  - SmsRoutingDecision
  - SmsRoutingPolicy
  - optimization API DTOs
  - Control Center optimization UI
- CredentialsJson, SettingsJson, auth tokens, webhook URLs, and raw provider payloads are not returned by optimization APIs.
- Logs do not print raw recipient phones or provider secrets.

- webhook reconciliation

- tenant runtime provider resolution
- activity APIs
