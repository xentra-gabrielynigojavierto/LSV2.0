# LS-COR-AUT-007 — Enforcement Completion + Hardening

## Objective

Finalize LS-COR-AUT by completing **authorization enforcement across all services** and performing **production hardening** of the access-control system.

This feature ensures:

* all APIs enforce correct product and role-based authorization
* no endpoints rely on weak or generic authentication (`AuthenticatedUser`)
* legacy data is fully removed
* role model boundaries are enforced in code
* system is ready for production-scale usage

This is the final step before **full production readiness**.

---

### Identity

* JWT issuance remains unchanged
* Claims structure must remain:

  * `role` (system roles)
  * `product_roles` (PRODUCT:Role)

### Synq Audit

* Must remain intact
* No mutation path should lose audit coverage

### Compatibility

* LS-COR-AUT-005 UI must remain fully functional
* LS-COR-AUT-001 filters must remain unchanged in behavior

## In Scope

---

## 1. Enforcement Completion Across Services

### Problem

Many endpoints still use:

```csharp
[Authorize(Policy = "AuthenticatedUser")]
```

This is insufficient.

---

### Required Action

Replace with proper policies:

#### Product-level access

```csharp
[Authorize(Policy = "RequireProductAccess:SYNQ_FUND")]
```

#### Role-level access

```csharp
[Authorize(Policy = "RequireProductRole:SYNQ_FUND:SYNQFUND_REFERRER")]
```

---

### Target Services

Apply to ALL relevant endpoints in:

* Fund Service
* CareConnect Service
* Any other product services

---

### Example

Replace:

```csharp
[Authorize(Policy = "AuthenticatedUser")]
```

With:

```csharp
[Authorize(Policy = "RequireProductRole:SYNQ_FUND:SYNQFUND_REFERRER")]
```

---

### Validation Rules

* Every endpoint must require:

  * product access OR
  * specific product role

* No endpoint should rely solely on:

  * authenticated user
  * system role unless explicitly required

## 3. Enforce Role Model Boundaries

### Problem

`ScopedRoleAssignment` still supports `PRODUCT` scope.

---

### Required Action

Enforce:

* `ScopedRoleAssignment`:

  * ONLY supports GLOBAL scope

* Block or validate:

```csharp
ScopeType != GLOBAL → reject
```

---

### Implementation Options

* validation in service layer
* DB constraint (preferred if feasible)
* guard clauses in assignment logic

---

## 4. Effective Access Hardening

### Add Caching (Optional but Recommended)

Implement short-lived caching:

* cache key:

  * `tenantId:userId:accessVersion`

* TTL:

  * 2–5 minutes

---

### Cache Invalidation

* triggered automatically via AccessVersion change
* no manual eviction required
