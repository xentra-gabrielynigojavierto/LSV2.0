# LS-ID-TNT-007 — Invite-Based User Creation

## BUILD INSTRUCTIONS (STRICTLY FOLLOW)

You are implementing the next incremental tenant identity feature on top of:
- LS-ID-TNT-001 — Tenant Users List Stabilization
- LS-ID-TNT-002 — Add User Flow
- LS-ID-TNT-003 — User Actions (Edit + Status Control)
- LS-ID-TNT-004 — Password Reset + Last Admin Protection
- LS-ID-TNT-005 — Backend Security Hardening
- LS-ID-TNT-006 — Email Delivery Integration for Password Reset

This feature MUST:
- Be incremental (NO rewrites)
- Preserve all existing user-management behavior unless changes are required to support secure invite onboarding
- Reuse existing token, email, and notifications patterns where appropriate
- Keep backend, API, UI, and email-delivery concerns clearly separated
- Include explicit fallback behavior
- NOT introduce placeholder logic
- Validate the actual current codebase before implementing
- Prefer secure activation by invited user over admin-handled passwords
- Preserve production security (no raw invite token exposure in production)
- Preserve non-production usability

## FEATURE OBJECTIVE

Replace the current admin-supplied password onboarding model with a secure invite-based user creation flow so tenant admins invite users by email and invited users set their own password through a secure activation link.

## STEP 2 — DESIGN THE INVITE MODEL

Based on actual findings, design the safest invite model.

You must determine and document:

- whether invites should use:
  - a new invite token model, OR
  - the existing password reset token model with a distinct purpose/state, OR
  - an extension of the current reset-confirm flow
- whether newly invited users should be:
  - inactive until activation
  - active but unable to authenticate without a password
  - represented by a separate pending/invited state if supported
- whether invite acceptance should:
  - reuse the reset-password page/confirm handler
  - use a dedicated accept-invite page/handler
- whether resend-invite is safe to include now

Preferred design principles:
- minimize new concepts if existing reset-token architecture can be safely reused
- invited users must not have a usable password before accepting
- invited users must not be able to authenticate normally before activation
