# PART 1 — CUTOVER PLAN (HIGHEST COMPLEXITY)

## Goal

Design and partially implement a safe cutover from:

* legacy audit storage / endpoints / flows
  to
* canonical `AuditEventRecords` + `/internal/audit/*` + `/audit/*`

## What to analyze

Inspect and identify:

* legacy `AuditEvents` model and endpoints
* canonical `AuditEventRecords`
* old ingest/query endpoints
* new internal ingest endpoints
* all currently active code paths that still use legacy audit behavior
* any admin or UI components still pointed at legacy audit data

## Deliverables required

Create a real cutover plan that includes:

### 1. Current-state inventory

Identify:

* legacy write paths
* legacy read paths
* canonical write paths
* canonical read paths
* services still not integrated

### 2. Cutover model

Define phases such as:

* Phase 0: coexistence
* Phase 1: producer onboarding
* Phase 2: dual-read or canonical-read switch
* Phase 3: legacy freeze
* Phase 4: legacy retirement

### 3. Implementation changes

Where practical, implement safe cutover helpers such as:

* config flag(s) for canonical vs legacy read preference
* adapter layer(s)
* clear service selection logic
* warnings/logs if legacy endpoints are still used
* documentation for deprecated paths

### 4. Legacy safety rules

Make sure:

* legacy writes are not silently lost
* canonical writes are not mixed incorrectly
* read paths are explicit and inspectable
* current UI screens are not broken

## Strong recommendation

If it is safe, add a configuration-driven read preference such as:

* `AuditReadMode = Legacy | Canonical | Hybrid`
  or equivalent, and use it only where it improves safety and observability.

Do NOT fake a full migration if historical data migration is not implemented.
Document clearly what still requires data migration or dual-write planning.
