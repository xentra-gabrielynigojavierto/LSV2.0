# STORY ID
**LS-REPORTS-01-002 — Template Management API**

---

# OBJECTIVE
Expose the first Template Management API layer so control-center workflows can create, update, version, publish, and retrieve `ReportTemplate` records through stable REST endpoints.

This story must preserve:
- clean architecture
- product scoping
- organization-type scoping
- template version governance
- portability

---

# REQUIRED IMPLEMENTATION

## A. API Route Surface
Add versioned REST endpoints under:

- `/api/v1/templates`

### Required endpoints

#### 1. Create template
- `POST /api/v1/templates`

#### 2. Update template
- `PUT /api/v1/templates/{templateId}`

#### 3. Get template by id
- `GET /api/v1/templates/{templateId}`

#### 4. List templates by product and org type
- `GET /api/v1/templates?productCode={code}&organizationType={type}`

#### 5. Create template version
- `POST /api/v1/templates/{templateId}/versions`

#### 6. Get latest version
- `GET /api/v1/templates/{templateId}/versions/latest`

#### 7. Get published version
- `GET /api/v1/templates/{templateId}/versions/published`

#### 8. Publish version
- `POST /api/v1/templates/{templateId}/versions/{versionNumber}/publish`

Optional only if low-risk and clean:
- `GET /api/v1/templates/{templateId}/versions`

## F. Versioning Rules
Implement these rules in the service layer.

### 1. Template creation
- creates a `ReportTemplate` record only
- does NOT automatically create a version unless there is a compelling reason and it is clearly documented
- preferred model: template and version are separate actions

### 2. Version creation
- version numbers increment sequentially per template
- latest version is highest version number

### 3. Publish version
- publishing a target version sets that version as published
- any previously published version for the same template must be unset
- there must be at most one published version per template
- operation should be idempotent or return a clear success result if already published

---

## G. Validation Rules
Implement basic validation.

### Template validation
- `Code` required on create
- `Name` required
- `ProductCode` required
- `OrganizationType` required

### Version validation
- `TemplateBody` required
- `OutputFormat` required
- template must exist before version creation

### Publish validation
- template must exist
- version must exist
- invalid publish attempts should return clear results

Use sensible 400/404/409 behavior.

---

## H. Error Handling
Use consistent API responses and status codes.

### Recommended behavior
- `400` invalid request
- `404` template/version not found
- `409` duplicate code or conflicting action
- `500` unexpected server failure

Keep this lightweight and consistent.
Do not overbuild a full enterprise exception framework unless something minimal already exists.
