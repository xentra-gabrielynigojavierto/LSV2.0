## Work items

### Phase A — make OrganizationType operational

Current problem:
- Organization has OrganizationTypeId, but Organization.Create() and validation still rely on orgType string.

Tasks:
- inspect Organization aggregate, commands, handlers, DTOs, validators, admin endpoints, and any org creation/update flows
- add support for OrganizationTypeId as the canonical field in create/update paths
- keep OrgType string only as transitional compatibility/read-model support if needed
- when OrganizationTypeId is provided, resolve/set OrgType string from the catalog for backward compatibility
- if only OrgType string is provided, resolve OrganizationTypeId from OrganizationTypes
- centralize this mapping logic so create/update paths do not drift

Important:
- do not break existing request payloads
- support both old and new payload styles if practical

### Phase B — fully activate DB-backed product eligibility

Current problem:
- AuthService has logic to check OrgTypeRules first
- but repository loading is incomplete, so fallback to ProductRole.EligibleOrgType still dominates

Tasks:
- inspect UserRepository, AuthService, and all queries used during login/auth context loading
- eager load:
  - org memberships
  - organization
  - organization.OrganizationTypeRef
  - organization products
  - product roles
  - OrgTypeRules
  - OrgTypeRules.OrganizationType
- make sure AuthService truly evaluates ProductOrganizationTypeRule first
- preserve fallback to EligibleOrgType only for compatibility if OrgTypeRules are absent
- add tests or at least validation logic to prove the DB-backed rule path is now active

Also:
- inspect ScopedRoleAssignment usage
- start resolving roles from ScopedRoleAssignment where appropriate
- keep UserRoleAssignment fallback if full cutover is too risky
- document exact resolution order in the report

### Phase C — activate CareConnect relationship persistence

Current problem:
- Referral and Appointment now have OrganizationRelationshipId
- but service logic does not yet populate it

Tasks:
- inspect ReferralService and AppointmentService
- when creating a referral:
  - if ReferringOrganizationId and ReceivingOrganizationId are present
  - attempt to find a matching active OrganizationRelationship in Identity
  - if found, set Referral.OrganizationRelationshipId
  - if not found, preserve current behavior and allow null
- when creating an appointment from a referral:
  - copy OrganizationRelationshipId from the referral
- update any repository/configuration/DTO mapping needed
- preserve current APIs and existing workflows

Important:
- do not require OrganizationRelationshipId for old data
- null must remain valid

### Phase D — improve provider/facility identity alignment

Current problem:
- Provider.OrganizationId and Facility.OrganizationId exist but are not yet consistently used

Tasks:
- inspect provider create/update/import flows
- where a provider already corresponds to an Identity organization, populate OrganizationId
- where a facility belongs to a provider org, populate Facility.OrganizationId
- do not redesign everything into ProviderProfile yet unless clearly low-risk
- keep this as a transitional identity alignment step

### Phase E — add minimal control-center/frontend compatibility

Current problem:
- backend admin endpoints exist
- frontend types/pages were not updated

Tasks:
- inspect:
  - apps/control-center/src/types/index.ts
  - apps/control-center/src/lib/control-center-api.ts
  - relevant admin pages/routes/components
- add API client methods and TS types for:
  - organization types
  - relationship types
  - organization relationships
  - product org-type rules
  - product relationship-type rules
- if practical, add minimal admin listing pages
- if full UI is too much, at least wire the client/types so future pages can use them cleanly
- document what was done vs deferred

### Phase F — reduce legacy architecture dependency

Tasks:
- identify where ProductRole.EligibleOrgType is still read
- identify where UserRoleAssignment is still the active source of truth
- reduce direct usage where low-risk
- keep backward compatibility, but add TODO/deprecation comments where needed
- do not remove schema yet unless fully safe
- document what remains to retire later
