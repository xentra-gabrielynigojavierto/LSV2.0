Build the next feature phase for the Coach Daxie coach platform using the new **shared coach implementation layer**.

Do not create disconnected page-level implementations.
Everything in this phase must be built from the shared:

* coach feature spec
* types/schema
* constants
* mock data
* selectors/helpers
* parity checklist

## Objective

Implement the first major coach workflow that fully uses the shared foundation across both:

* coach web dashboard
* coach mobile app

This phase should build:

1. Plans
2. Plan Detail
3. Plan Builder
4. Templates
5. Template Detail
6. Template Assignment Flow

The goal is to prove that web and mobile can stay aligned while still using platform-appropriate UI.

---

## Core Build Rule

All of these features must be created from the shared coach layer.
Do not hardcode separate structures for web and mobile.

Use shared:

* entities
* field names
* labels
* statuses
* actions
* tabs
* workflow states

Web and mobile can render differently, but the capabilities must match.

---

## Features to Build

# 1. Plans

Build the coach Plans experience on both web and mobile.

### Required capabilities

* list active plans
* search plans
* filter plans
* sort plans
* open plan detail
* duplicate plan
* archive plan
* edit plan

### Shared fields per plan

At minimum include:

* id
* plan name
* assigned athlete
* goal type
* duration
* current week
* status
* adherence/compliance
* last updated

### Web

Create:

* `/coach/plans`
* table or structured multi-column plan list
* filters and search in a toolbar
* quick actions inline

### Mobile

Create:

* `CoachPlansScreen`
* stacked cards
* filter sheet/drawer
* search input
* plan row/card tap to open detail
* touch-friendly action menu

---

# 2. Plan Detail

Build plan detail on both web and mobile.

### Required capabilities

* open a specific plan
* show athlete attached to the plan
* show current goal
* show week selector
* show weekly schedule
* open workout details
* jump to assigned athlete
* assign a template to the plan

### Shared states

* loading
* empty plan
* populated
* unsaved changes
* save success
* validation error
* archived plan if relevant

### Web

Create:

* `/coach/plans/[id]`
* wide layout with header and weekly schedule
* side panel or modal for workout editing

### Mobile

Create:

* `CoachPlanDetailScreen`
* stacked weekly view
* day cards
* bottom sheet or full-screen workout editor

# 3. Plan Builder

This is the most important feature in this phase.

Build a full coach-facing plan editing workflow across web and mobile.

### Workout types

Use shared constants for:

* easy run
* intervals
* tempo
* long run
* recovery
* swim
* bike
* strength
* rest

### Required actions

* add workout
* edit workout
* remove workout
* duplicate workout
* move workout to another day
* copy day
* adjust weekly volume %
* save draft
* save changes
* revert unsaved changes if feasible
* assign template to plan

### Workout fields

At minimum:

* workout type
* title
* duration or distance
* intensity / zone
* notes / instructions

### Web UX

* weekly planner view
* more visible editing controls
* inline edit or side panel where appropriate

### Mobile UX

* day-based editing flow
* bottom sheet or full-screen editor
* mobile-first touch interactions
* keep all core actions available

Do not remove functionality on mobile.
Adapt it.

## Shared Layer Requirements

For every feature in this phase, wire it into:

* shared types

* shared selectors/helpers

## Cross-Linking Requirements

Add strong linking between features.

Required flows:

* Athlete Detail → Open active plan
* Athlete Detail → Assign template
* Plans → Open assigned athlete
* Plans → Assign template
* Templates → Assign to athlete/plan
* Dashboard alerts if relevant → open plan needing attention

These flows must work on both web and mobile.
