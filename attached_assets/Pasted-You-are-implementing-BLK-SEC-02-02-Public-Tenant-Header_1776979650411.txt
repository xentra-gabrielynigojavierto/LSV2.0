OBJECTIVE

Eliminate blind trust in `X-Tenant-Id` for public CareConnect routes.

At the end of this block:

- public CareConnect endpoints no longer rely on an unprotected upstream trust assumption
- requests cannot spoof another tenant by directly supplying `X-Tenant-Id`
- the trusted gateway/BFF boundary is explicitly enforced in code and/or gateway configuration
- production-grade tenant isolation is achieved for public routes

--------------------------------------------------

CURRENT KNOWN GAP

BLK-SEC-02-01 confirmed that public routes are secure ONLY under the assumption that:

- the Next.js BFF resolves tenant from host/subdomain
- the BFF sets `X-Tenant-Id`
- direct public callers cannot reach CareConnect or inject that header

That assumption is not enforced strongly enough yet.

This block must close that gap.

PART B — IMPLEMENT TRUST BOUNDARY ENFORCEMENT

--------------------------------------------------

Implement ONE of the following, preferring the strongest approach that fits the repo architecture:

### Option A (preferred if gateway/BFF path supports it cleanly)
Harden the proxy/gateway boundary so:

- inbound client-supplied `X-Tenant-Id` is stripped
- `X-Tenant-Id` is injected only after server-side tenant resolution
- public requests reaching CareConnect cannot carry arbitrary caller-supplied tenant IDs

### Option B (required if direct service hardening is needed in addition)
Harden CareConnect public endpoints so they reject untrusted direct `X-Tenant-Id` usage unless the request is clearly from the trusted server layer.

Acceptable patterns include:
- trusted internal header pair / shared secret for BFF-originated public proxy calls
- internal-only route topology validation
- explicit gateway-origin marker validated server-side

RULES:
- do NOT rely purely on “network isolation” as the only control
- do NOT accept arbitrary `X-Tenant-Id` from direct callers
- do NOT break legitimate public flows
