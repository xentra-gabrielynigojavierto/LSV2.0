# LS-ID-TNT-022-002 — Insights UI Permission Gating

## FEATURE OBJECTIVE

Implement permission-aware UI gating across the Insights product so the existing Insights screens now reflect the permission catalog introduced in LS-ID-TNT-022-001.

This feature should wire real Insights UI actions to the new permission codes using the same shared patterns already established in:
- LS-ID-TNT-015
- LS-ID-TNT-015-004
- LS-ID-TNT-015-001

This is a **UI gating feature**, not a backend-enforcement rewrite. Backend enforcement for Insights APIs remains a follow-on item after the UI is wired. The permission-code foundation already exists. :contentReference[oaicite:0]{index=0}

---

## WHY THIS FEATURE COMES NEXT

LS-ID-TNT-022-001 completed the missing authorization foundation for SynqInsights:
- `SYNQ_INSIGHTS` product seeded
- 7 Insights capabilities defined
- frontend `PermissionCodes.Insights.*` added
- default role-capability assignments established. :contentReference[oaicite:1]{index=1}

The current remaining gap is that Insights screens still do not consume those permissions in the UI.

This feature closes that gap by making Insights screens permission-aware in the same way CareConnect and Fund already are.

## STEP 5 — APPLY HIDE / DISABLE / READ-ONLY RULES CONSISTENTLY

Use the platform rules consistently:

### Hide
Hide actions when visibility adds no value and would just clutter or confuse.

### Disabled with tooltip
Use disabled-with-tooltip when the action is meaningful to discoverability and workflow understanding.

### Read-only
Use read-only mode for builder/detail screens where the user may view but not modify.

### ForbiddenBanner
Use when the user can access the surface/page contextually but none of the relevant actions are permitted.

Requirements:
- choose the correct rule per surface
- document the rule used per covered action
- keep ordering stable
- do not create inconsistent treatment for similar Insights actions
