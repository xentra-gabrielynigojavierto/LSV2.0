Feature ID: LS-NOTIF-SMS-011

Title:
SMS External Escalation and Incident Notification Integrations

Objective:
Add operational escalation and incident-notification integrations inside Notification Service so critical SMS operational alerts can notify platform operators through configured external operational channels such as email, Microsoft Teams, Slack, PagerDuty, Opsgenie, and internal platform notifications.

Critical Architectural Constraint:
Although this feature will be implemented inside the LegalSynq monorepo, all SMS alert escalation policy evaluation, escalation persistence, retry handling, and external incident notification integration logic must remain inside the Notification Service boundary.

Requirements:
- Do not build Control Center UI in this feature.
- Do not add SMS escalation logic to unrelated services.
- Do not create shared cross-service escalation abstractions unless an existing Notification Service shared/internal module already exists.
- Do not modify other services except where strictly necessary for build/config compatibility.
- Notification Service must own:
  - SMS alert escalation policy configuration
  - SMS alert escalation routing
  - SMS alert escalation delivery adapters
  - SMS alert escalation persistence/history
  - SMS alert escalation retry handling
  - SMS alert escalation admin APIs
- Control Center should consume Notification Service escalation APIs in a future UI feature.
- Preserve Notification Service as the system of record for SMS operational alerts and SMS alert escalation outcomes.

Implementation Requirements:
1. Add SMS alert escalation policy support inside Notification Service.
2. Escalate eligible SMS operational alerts based on severity, alert type, tenant, provider, and configured policy rules.
3. Support at least internal platform notification and email escalation if existing Notification Service delivery patterns support them cleanly.
4. Add Microsoft Teams and Slack webhook escalation adapters if straightforward within existing infrastructure patterns.
5. Treat PagerDuty and Opsgenie as optional unless existing webhook/client patterns make them low-risk to implement.
6. Persist escalation attempts and outcomes.
7. Link escalation records to the originating SmsOperationalAlert.
8. Add deduplication/cooldown to prevent escalation spam.
9. Add retry handling for transient escalation delivery failures.
10. Add admin APIs to list escalation history, summarize escalation outcomes, manage policies, and manually retry failed escalations.
11. Preserve existing SMS alert evaluation behavior from LS-NOTIF-SMS-010.
12. Preserve existing outbound SMS send, retry/dead-letter, webhook, compliance, tenant provider resolution, reconciliation, activity APIs, dashboard APIs, and Control Center dashboard behavior.
13. Do not add or expand conversational inbound SMS behavior.

Critical Constraints:
- Reuse existing SmsOperationalAlert, audit, worker, endpoint, authorization, migration, notification delivery, HTTP client, and repository patterns where appropriate.
- Do NOT introduce large rewrites.
- Keep implementation incremental and production-oriented.
- Escalation evaluation must not trigger SMS sends unless explicitly using the existing Notification Service as an internal notification channel and doing so does not create recursive SMS alert loops.
- Avoid alert escalation through SMS by default to prevent recursion and incident noise.
- External escalation failures must not crash Notification Service.
- Escalation delivery must not expose provider credentials, CredentialsJson, SettingsJson, auth tokens, raw provider payloads, or full phone numbers.
- Policy targets such as webhook URLs or email destinations must be treated as sensitive operational configuration and never returned in full unless existing admin patterns explicitly allow it.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 8 — Add Escalation Service
Add service to evaluate and execute escalation for alerts.

Suggested interface:
ISmsAlertEscalationService

Suggested methods:
- EscalateAlertAsync(Guid alertId, CancellationToken ct)
- EscalateAlertAsync(SmsOperationalAlert alert, CancellationToken ct)
- RetryEscalationAsync(Guid escalationId, string requestedBy, CancellationToken ct)
- ProcessPendingRetriesAsync(int limit, CancellationToken ct)

Expected behavior:
1. Load alert.
2. Find enabled matching policies.
3. For each policy:
   - build safe payload
   - compute payload hash
   - check recent duplicate/cooldown
   - create pending escalation record
   - send via adapter
   - update status sent/failed/suppressed
   - schedule retry if retryable and policy allows
4. Audit escalation lifecycle event if audit patterns support it.

Requirements:
- External delivery failures must not crash alert evaluation.
- No matching policy should result in a safe no-op/log.
- Dedup/cooldown must prevent repeated spam.
- Retry scheduling must be bounded by MaxRetryCount.
- Do not trigger SMS sends.
- Do not expose secrets.

- Escalation failure must not fail alert creation/update.
