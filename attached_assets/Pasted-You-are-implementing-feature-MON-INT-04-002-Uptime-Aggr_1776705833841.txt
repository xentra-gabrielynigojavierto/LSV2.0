You are implementing feature MON-INT-04-002 — Uptime Aggregation Engine.

OBJECTIVE
Add a durable uptime aggregation layer to the Monitoring Service so the platform can compute accurate 24h / 7d / 30d / 90d uptime metrics and historical availability buckets for each monitored component without recalculating from raw check history on every request.

This is the data foundation for uptime percentages, availability bars, and future response-time trend views.

ARCHITECTURE RULE (NON-NEGOTIABLE)
- Monitoring Service = source of truth
- Raw check history remains canonical
- Uptime metrics must be derived from Monitoring Service persistence, not invented in Control Center
- Manual alert resolution must NOT affect uptime metrics
- Do NOT fabricate historical uptime
- Do NOT compute public uptime from active alerts; compute it from check history

MANDATORY EXECUTION RULES
- Create the report file FIRST at: /analysis/MON-INT-04-002-report.md
- Update the report incrementally after each meaningful step
- Do NOT wait until the end to write the report
- Do NOT introduce fake rollups
- Do NOT change existing read endpoints unless required and documented
- Prefer minimal, reversible changes
- Keep raw facts, rollups, and read APIs clearly separated

CURRENT CONTEXT
- Monitoring Service already stores:
  - monitored entities
  - check result history
  - current status projection
  - alerts
- Public status page now exists, but does NOT yet show uptime percentages or availability history
- The next requirement is a real aggregation engine for uptime windows

GOAL OF THIS FEATURE
By the end of this feature:
- Monitoring Service can derive uptime from persisted check history
- rollup persistence exists for efficient reads
- read APIs exist for uptime windows and bucketed history
- the model is suitable for later public-page uptime percentages and availability bars
- no fake or alert-derived uptime is introduced
