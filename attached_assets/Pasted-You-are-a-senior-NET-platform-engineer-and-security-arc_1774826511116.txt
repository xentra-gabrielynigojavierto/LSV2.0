You are a senior .NET platform engineer and security architect.

Your task is to implement **ClamAV-based malware scanning as an internal capability of the .NET Documents Service**, while keeping the scanning logic modular, secure, and production-appropriate.

## Primary Goal

Add production-ready malware scanning to the **.NET Documents Service** so that scanning is treated as part of the Documents Service platform, but implemented as a **separate internal scanning module / background workflow**, not mixed directly into controller logic.

This must preserve clean architecture and keep the service maintainable.

---

## Architectural Direction

The Documents Service should remain the single product/service boundary.

Inside that service boundary, implement:

* Documents API
* Documents application logic
* Documents persistence
* Documents scan-status enforcement
* Documents internal background scan processing
* ClamAV integration via provider abstraction

### Important design rule

ClamAV is part of the Documents Service solution, but it should be integrated through infrastructure + background processing, not directly embedded into route handlers.

---

## Desired Production Model

Use this model:

1. User uploads a document
2. Documents Service stores the file in a **quarantine location**
3. Document/version record is created with:

   * `scanStatus = Pending`
4. A scan job is queued or internally scheduled
5. Internal background worker processes the job
6. Worker calls `ClamAvFileScannerProvider`
7. Worker updates document/version scan result:

   * `Clean`
   * `Infected`
   * `Failed`
8. Access APIs must deny access unless scan policy allows it
9. If clean, file is promoted to clean storage or marked safe for access
10. Audit all critical scan events

---

Implement this for the **.NET Documents Service** only.

Do NOT modify the Node.js service.

Do NOT collapse clean architecture layers.

## Implementation Requirements

### 1. Internal scan workflow

Implement an async internal workflow inside the .NET Documents Service.

Preferred approach:

* use a background worker (`IHostedService` / `BackgroundService`)
* queue scan jobs in-process initially, but structure code so it can later be replaced with Redis/SQS/etc.

Required behavior:

* upload does NOT do full inline production scan
* uploaded files should enter `Pending`
* background worker scans them
* access remains blocked until allowed by scan policy

### 2. ClamAV provider

Implement:

* `ClamAvFileScannerProvider`

Requirements:

* connect to ClamAV daemon (`clamd`) over TCP
* configurable host/port
* safe timeout handling
* return structured scan result:

  * status
  * threats
  * duration
  * engine info if available

Also keep:

* `NullScannerProvider`
* `MockScannerProvider`

### 3. Quarantine model

Implement a quarantine-safe storage model.

At minimum:

* newly uploaded files must not be treated as clean immediately
* either:

  * store under a quarantine prefix/path
  * or mark with a quarantine state while awaiting scan

Preferred:

* quarantine storage key / prefix
* clean storage key / prefix

If promotion/move is too large for one pass, document and implement the safest minimal version.

### 4. Scan status enforcement

Ensure access endpoints enforce scan policy.

At minimum:

* `Infected` → always blocked
* `Pending` → blocked in production-safe mode
* `Failed` → blocked in production-safe mode
* `Clean` → allowed
* `Skipped` → only allowed when intentionally configured

Make behavior config-driven where appropriate.

### 5. Audit events

Audit all scan lifecycle events, including:

* scan requested
* scan started
* scan completed clean
* scan failed
* infected file detected
* access denied due to scan status

### 6. Configuration

Add config for:

* ClamAV host/port
* scanner provider selection
* require clean scan for access
* scan worker polling/queue settings
* quarantine prefix/path
* clean prefix/path if applicable
* scan timeout

### 7. Swagger / API documentation impact

If any API behavior changes due to async scanning, document it in analysis:

* upload response behavior
* pending scan semantics
* access denied while pending

You do not need to rewrite all OpenAPI docs now, but document the impact.
