OBJECTIVE

Enable authorized administrative users to safely intervene in workflow execution from Control Center by adding controlled admin actions for Retry, Force Complete, and Cancel, while preserving Flow as the single execution authority, maintaining full auditability, enforcing tenant isolation, and preventing any direct DB manipulation or uncontrolled override paths.

This phase converts Control Center from observe-only to controlled intervention, but only for explicitly authorized admin users and only through the Flow service.

SCOPE INCLUDES

- admin-only Flow endpoints for:
  - retry workflow
  - force complete workflow
  - cancel workflow
- read-only Control Center workflow detail drawer enhancement with an Admin Actions section
- reason-required confirmation flows for each admin action
- audit logging for all admin actions
- authorization and visibility enforcement
- documentation/reporting

DO NOT BUILD

- bulk actions
- automatic retries
- async/outbox processing
- tenant-user access to admin actions
- workflow definition editing
- workflow template administration
- tenant-portal admin actions
- direct DB manipulation outside Flow
- broad Control Center redesign

CRITICAL REQUIREMENTS

You MUST:
- keep Flow as the single execution authority
- expose admin actions only through Flow admin endpoints
- require an explicit reason for every admin action
- record every admin action in the Audit service
- preserve tenant isolation and cross-tenant non-leakage
- ensure admin actions are intentional, confirmable, and clearly separated from normal workflow actions
- keep the Control Center drawer read-heavy and intervention-light
- avoid creating any alternate execution path that bypasses the workflow engine/governed state model

IMPLEMENTATION REQUIREMENTS

1. BACKEND ADMIN ENDPOINTS

Add admin-only Flow endpoints:

- POST /api/v1/admin/workflow-instances/{id}/retry
- POST /api/v1/admin/workflow-instances/{id}/force-complete
- POST /api/v1/admin/workflow-instances/{id}/cancel

Requirements:
- endpoints must be under admin scope
- endpoints must be protected by admin authorization policy
- endpoints must be tenant-safe and non-leaky
- unauthorized or out-of-scope access should not expose cross-tenant existence
- request body must require a non-empty reason
- actor identity must come from auth context, not user input
- return a structured response including:
  - workflowInstanceId
  - action
  - previousStatus
  - newStatus
  - timestamp
  - performedBy

If existing admin controller structure is appropriate, extend it. Otherwise, add a focused admin actions controller.

2. ADMIN ACTION RULES

Implement these actions safely:

A. Retry
Allowed when:
- workflow status is Failed
- or workflow is Active and has an actionable error condition if existing model supports that safely

Behavior:
- re-enter engine-managed execution path
- do not silently rewind workflow state
- do not bypass engine validation

B. Force Complete
Allowed when:
- workflow status is Active or Pending
- workflow is not already terminal

Behavior:
- mark workflow Completed
- set completion timestamp
- capture admin override reason
- ensure audit trail is explicit that this was a forced admin completion

C. Cancel
Allowed when:
- workflow status is Active or Pending
- workflow is not already terminal

Behavior:
- mark workflow Cancelled
- prevent further progression
- capture admin reason
- ensure audit trail is explicit that this was an admin cancellation

Document the exact allowed-state matrix you implement.

3. STATE / CONCURRENCY SAFETY

Requirements:
- admin actions must run inside Flow transaction boundaries
- admin actions must respect the current workflow state
- stale or conflicting transitions must fail safely
- if current engine/state logic can surface:
  - concurrent_state_change
  - instance_not_active
  - expected_step_mismatch
  continue using structured error semantics
- do not directly mutate workflow rows from Control Center
- do not create a second state machine outside Flow

4. AUDIT REQUIREMENTS

This is mandatory.

For every admin action, create an audit record using the existing Audit service integration.

Minimum audit fields:
- workflowInstanceId
- tenantId
- productKey
- action (retry / force_complete / cancel)
- performedBy
- timestamp
- reason
- previousStatus
- newStatus
- source = ControlCenterAdminAction (or similar explicit marker)

Requirements:
- audit must be written even when the action is an override
- do not rely only on application logs
- if the current audit adapter has fallback behavior, document how failures are handled
- ensure audit content is explicit that the action was administrative, not normal workflow progression

5. CONTROL CENTER UI

Extend the existing E9.2 workflow detail drawer.

Requirements:
- add a clearly separated “Admin Actions” section
- visible only to authorized admin users
- include actions:
  - Retry
  - Force Complete
  - Cancel
- each action must open a confirmation UI
- each confirmation UI must require a reason before submission
- action controls must be visually distinct from read-only workflow details
- if an action is not allowed in the current workflow state, hide or disable it with a clear explanation if feasible
- drawer remains otherwise read-only

Recommended interaction:
- button click opens small modal or inline confirmation area
- reason textarea/input required
- confirm / cancel buttons
- success refreshes drawer and list state
- error remains visible in drawer without collapsing it

6. UI STATE / ERROR HANDLING

Requirements:
- mutation buttons show loading/submitting state
- on success:
  - drawer refreshes
  - list can refresh or revalidate if feasible
  - updated status is visible immediately
- on failure:
  - keep drawer open
  - show compact user-friendly error
  - do not lose current workflow context
- map common structured backend codes to concise operator-facing messages
- do not expose raw technical stack traces in the UI

7. AUTHORIZATION / VISIBILITY MODEL

Requirements:
- initial UI visibility should be PlatformAdmin only
- ordinary product users must never see admin actions
- if the backend supports TenantAdmin in future, document whether UI remains PlatformAdmin-only for now
- backend remains authoritative even if the UI hides actions
- cross-tenant admin action attempts must not leak existence

Recommended initial policy:
- PlatformAdmin only in Control Center UI
- backend may allow PlatformOrTenantAdmin only if tenant scoping is correctly enforced and documented, otherwise keep it PlatformAdmin-only for now

8. API / BFF BOUNDARY

- if the existing Control Center server API client pattern can support admin actions, extend it consistently
- keep mutations centralized in the admin/API layer rather than embedding raw fetch logic in UI components

SUCCESS CONDITION

E10.1 is successful when:
- PlatformAdmin users can safely retry, force-complete, and cancel eligible workflows from the Control Center detail drawer
- every action is reasoned, audited, and traceable
- tenant isolation and non-leakage are preserved
- no alternate execution path or direct DB mutation is introduced

FAILURE CONDITION

E10.1 fails if:
- admin actions bypass Flow execution authority
- actions are unaudited or allow empty reasons
- cross-tenant visibility/isolation is weakened
- the Control Center becomes an unsafe execution surface
- existing workflow behavior regresses or state consistency is weakened
