Feature ID: LS-NOTIF-SMS-009

Title:
Control Center SMS Dashboard UI Integration

Objective:
Add the Control Center SMS Dashboard UI that consumes the Notification Service dashboard APIs from LS-NOTIF-SMS-008, allowing platform admins to monitor SMS delivery activity, reconciliation health, tenant activity, provider usage, and failure trends without duplicating SMS aggregation logic outside Notification Service.

Critical Architectural Constraint:
Notification Service remains the source of truth for SMS delivery, reconciliation, provider attribution, and dashboard aggregation. Control Center must only consume Notification Service APIs and render the dashboard UI.

Requirements:
- Build UI inside the existing Control Center/admin portal application only.
- Do not add SMS dashboard aggregation logic to Control Center.
- Do not add SMS delivery, reconciliation, provider lookup, or compliance logic to Control Center.
- Do not modify Notification Service unless strictly necessary for API client compatibility.
- Control Center must consume the existing Notification Service endpoints from LS-NOTIF-SMS-008:
  - GET /v1/admin/sms/dashboard/summary
  - GET /v1/admin/sms/dashboard/trends
  - GET /v1/admin/sms/dashboard/failures
  - GET /v1/admin/sms/dashboard/tenants
  - GET /v1/admin/sms/dashboard/providers
- Dashboard UI must be read-only.
- Dashboard UI must not trigger SMS sends, retries, reconciliation, provider calls, or config changes.
- Do not expose provider credentials, CredentialsJson, SettingsJson, auth tokens, raw provider payloads, or full phone numbers.
- Preserve existing Control Center functionality and routing.

Implementation Requirements:
1. Add a PlatformAdmin-only SMS Dashboard page in Control Center.
2. Add API client methods for the Notification Service SMS dashboard endpoints.
3. Add dashboard filters for date range, tenant, provider, provider ownership mode, status, and trend bucket.
4. Render high-level SMS KPI cards.
5. Render delivery/reconciliation trend chart.
6. Render failure category/error breakdown table.
7. Render tenant activity breakdown table.
8. Render provider/provider-config breakdown table.
9. Handle loading, empty, and error states cleanly.
10. Preserve existing authentication/authorization behavior.
11. Do not duplicate aggregation logic already owned by Notification Service.

Critical Constraints:
- Reuse existing Control Center architecture, routing, layout, API client, auth, role guard, table, card, and chart patterns where available.
- Do NOT introduce a large UI rewrite.
- Keep implementation incremental and production-oriented.
- Do NOT create fake/mock dashboard data unless the existing app already uses a development mock pattern.
- Do NOT hardcode provider secrets or tenant-sensitive values.
- Do NOT expose full phone numbers.
- Use backend-provided aggregate DTOs directly where practical.
- If tenant name enrichment already exists in Control Center, use it; otherwise display tenantId and document tenant-name enrichment as a future improvement.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.
