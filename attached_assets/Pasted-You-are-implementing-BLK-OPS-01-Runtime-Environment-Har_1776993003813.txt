OBJECTIVE

Make the platform safer to run in production by enforcing environment, secret, and runtime configuration correctness.

At the end of this block:

- required production secrets/configs are validated at startup
- insecure placeholder values fail fast outside Development
- gateway/BFF/service trust-boundary config is consistent
- runtime misconfiguration is visible and diagnosable
- local development remains usable without weakening production behavior

SCOPE

This block includes:

- production environment validation audit
- secret/config fail-fast enforcement
- placeholder/default value detection
- trust-boundary secret validation
- gateway route/runtime config validation
- service dependency URL validation
- production readiness checklist generation

This block does NOT include:

- secret rotation implementation
- infrastructure provisioning
- cloud deployment scripts
- new monitoring dashboards
- new business features

PART B — CENTRALIZE CONFIG VALIDATION WHERE PRACTICAL

--------------------------------------------------

If the repo already has config validation patterns, reuse them.

Otherwise, add minimal startup validation helpers.

Preferred direction:

- create lightweight validation helper(s), for example:
  - RuntimeConfigValidation.RequireNonEmpty(...)
  - RuntimeConfigValidation.RequireNotPlaceholder(...)
  - RuntimeConfigValidation.RequireAbsoluteUrl(...)
  - RuntimeConfigValidation.RequireConnectionString(...)

Rules:
- keep it simple
- avoid overbuilt framework
- use in services where it reduces duplicate code
- validation should throw clear InvalidOperationException at startup

PART C — FAIL-FAST PRODUCTION SECRET VALIDATION

--------------------------------------------------

Ensure non-Development environments fail fast when required secrets are:

- missing
- empty
- whitespace
- known placeholder values such as:
  - REPLACE_VIA_SECRET
  - CHANGE_ME
  - dev-only
  - localhost-only production URL where invalid

At minimum validate:

Gateway:
- Jwt:SigningKey
- PublicTrustBoundary:InternalRequestSecret

CareConnect:
- Jwt:SigningKey
- TenantService:ProvisioningToken
- IdentityService:ProvisioningToken
- PublicTrustBoundary:InternalRequestSecret
- ConnectionStrings default / CareConnectDb if applicable

Tenant:
- Jwt:SigningKey if used
- TenantService:ProvisioningSecret
- DB connection string

Identity:
- Jwt:SigningKey
- TenantService:ProvisioningSecret
- NotificationsService settings needed for production user flows
- DB connection string

Web/BFF:
- INTERNAL_REQUEST_SECRET must be documented as required for production
- GATEWAY_URL must be configured outside local dev

Do NOT commit real secrets.

--------------------------------------------------

PART D — TRUST BOUNDARY CONFIG CONSISTENCY

--------------------------------------------------

Validate consistency between:

- Gateway `PublicTrustBoundary:InternalRequestSecret`
- CareConnect `PublicTrustBoundary:InternalRequestSecret`
- Web/BFF `INTERNAL_REQUEST_SECRET`

Requirements:
- document that all three must match
- fail fast where services can validate local presence
- do not attempt to compare across services at startup unless existing infrastructure supports it
- add clear error messages if missing/placeholder

--------------------------------------------------

PART E — SERVICE DEPENDENCY URL VALIDATION

--------------------------------------------------

Validate required service URLs in production:

CareConnect:
- TenantService.BaseUrl
- IdentityService.BaseUrl
- Audit service URL if used

Web/BFF:
- GATEWAY_URL

Rules:
- must be absolute URL
- must not be empty
- must not default to localhost in production unless explicitly allowed
- error messages should identify config key
