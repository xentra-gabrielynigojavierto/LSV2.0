### 1. Password Reset Token Returned in HTTP Response Body

**File:** `apps/services/identity/Identity.Api/Endpoints/AuthEndpoints.cs`
**OWASP:** A07:2021 – Identification and Authentication Failures

```csharp
// ❌ Raw token exposed in API response
return Results.Ok(new
{
    message = "If an account exists with that email, a password reset link has been generated.",
    resetToken = rawToken,  // ← 32-byte secret returned in plain HTTP response
});
```

`POST /api/auth/forgot-password` generates a cryptographically secure (256-bit) token but returns it as `resetToken` in the HTTP response body instead of emailing it. A comment in the code acknowledges this is a temporary state: *"the raw token will be emailed instead"*.

**Impact:** Any server log, intercepting proxy, CDN edge log, or frontend code that reads this response can silently hijack any user account's password reset flow. This completely negates the security intent of the reset mechanism.

**Fix:** Remove `resetToken` from the response immediately. Deliver it **only** via transactional email. The `message` field is the correct and sufficient response.

```csharp
// ✅ Return only the neutral message — token delivered out-of-band via email
return Results.Ok(new
{
    message = "If an account exists with that email, a password reset link has been generated.",
});
```

### 2. Hardcoded Internal Service Token Fallback

**File:** `apps/services/careconnect/CareConnect.Api/Endpoints/InternalProvisionEndpoints.cs`
**OWASP:** A02:2021 – Cryptographic Failures / A05:2021 – Security Misconfiguration

```csharp
// ❌ Hardcoded credential used as fallback if config key is missing
private const string ExpectedToken = "legalsynq-internal-service-2024";

var configToken = httpContext.RequestServices
    .GetService<IConfiguration>()?["InternalServiceToken"] ?? ExpectedToken; // fallback!
```

If `InternalServiceToken` is absent from config (e.g., a misconfigured deployment), the well-known hardcoded string `"legalsynq-internal-service-2024"` becomes the effective secret. Any attacker who reads this source code can call `POST /internal/provision-provider` on any deployment.

**Impact:** Unauthorized provider provisioning in any tenant, data integrity compromise.

**Fix:** Remove the fallback entirely. Fail fast on missing configuration, the same pattern used for `Jwt:SigningKey`:

```csharp
// ✅ Fail fast — no fallback to a known hardcoded secret
var configToken = httpContext.RequestServices
    .GetService<IConfiguration>()?["InternalServiceToken"]
    ?? throw new InvalidOperationException("InternalServiceToken is not configured.");
```

### 3. Symmetric JWT Signing Key Shared Across All Services (HMAC-SHA256)

**Files:** `Identity.Infrastructure/Services/JwtTokenService.cs`, `Gateway.Api/Program.cs`, `Liens.Api/appsettings.json`, `Comms.Api/appsettings.json`
**OWASP:** A02:2021 – Cryptographic Failures

All services share the same symmetric HMAC-SHA256 signing key. With a symmetric algorithm, any service that can **verify** a token can also **forge** one. If any downstream service (liens, comms, careconnect) is compromised, an attacker can issue arbitrary JWTs with any user identity, tenant, or roles.

**Impact:** Lateral movement — a compromise in any one service is a full platform compromise.

**Fix:** Migrate to RS256 (asymmetric). The identity service holds the RSA **private** key and signs; all other services hold only the **public** key and can verify but cannot forge.

```csharp
// ✅ JwtTokenService — sign with RSA private key
var rsa = RSA.Create();
rsa.ImportFromPem(File.ReadAllText(privateKeyPath));
var key = new RsaSecurityKey(rsa);
var credentials = new SigningCredentials(key, SecurityAlgorithms.RsaSha256);

// ✅ Gateway / downstream services — verify with RSA public key only
var rsa = RSA.Create();
rsa.ImportFromPem(File.ReadAllText(publicKeyPath));
IssuerSigningKey = new RsaSecurityKey(rsa)
```

### 4. No Rate Limiting on Authentication Endpoints

**Files:** `apps/gateway/Gateway.Api/Program.cs`, `apps/services/identity/Identity.Api/Program.cs`
**OWASP:** A07:2021 – Identification and Authentication Failures

Neither the identity service nor the gateway registers `AddRateLimiter`. The login, `forgot-password`, `accept-invite`, and `password-reset/confirm` endpoints have no IP-level or account-level throttle. Account lockout exists at the domain layer, but:
- `POST /api/auth/forgot-password` can be called in a tight loop to exhaust token windows for any user
- `POST /api/auth/accept-invite` and `POST /api/auth/password-reset/confirm` can be brute-forced (no attempt ceiling despite 256 bits of entropy)
- `POST /api/auth/login` is protected only by account lockout, not by IP-based throttling

**Fix:** Register rate limiting at the gateway for all anonymous auth routes:

```csharp
// ✅ Gateway Program.cs
builder.Services.AddRateLimiter(opts =>
{
    opts.AddFixedWindowLimiter("auth", o =>
    {
        o.Window              = TimeSpan.FromMinutes(1);
        o.PermitLimit         = 10;
        o.QueueProcessingOrder = QueueProcessingOrder.OldestFirst;
        o.QueueLimit           = 0;
    });
    opts.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
});

// Apply to auth routes in YARP config: "RateLimiterPolicy": "auth"
```

### 5. `dangerouslySetInnerHTML` Without Sanitization (XSS)

**Files:**
- `apps/control-center/src/components/notifications/branded-preview-modal.tsx` (line 147)
- `apps/control-center/src/components/notifications/global-template-version-form.tsx` (line 152)
- `apps/control-center/src/components/notifications/template-preview-modal.tsx` (line 119)

**OWASP:** A03:2021 – Injection (XSS)

```tsx
// ❌ Unsanitized HTML from backend injected into DOM — DOMPurify not installed or used
<div dangerouslySetInnerHTML={{ __html: preview.body }} />
<div dangerouslySetInnerHTML={{ __html: bodyHtml }} />
<div dangerouslySetInnerHTML={{ __html: result.bodyHtml }} />
```

These three components render notification template HTML from the backend without any sanitization. A malicious or compromised template (e.g., injected via the notifications service) could render `<script>` tags in a control-center admin session. No instance of `DOMPurify` exists anywhere in either frontend app.

**Fix:** Install `dompurify` and wrap all three instances:

```bash
pnpm add dompurify @types/dompurify --filter control-center
```

```tsx
// ✅ Sanitize before injection
import DOMPurify from 'dompurify';

<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(preview.body) }} />
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(bodyHtml) }} />
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(result.bodyHtml) }} />
```

### 6. Audit Query/Export Routes Exposed as Anonymous at Gateway

**File:** `apps/gateway/Gateway.Api/appsettings.json`
**OWASP:** A01:2021 – Broken Access Control

```json
"audit-service-query": {
  "ClusterId": "audit-cluster",
  "AuthorizationPolicy": "Anonymous",
  "Order": 130,
  "Match": { "Path": "/audit-service/audit/{**catch-all}" }
},
"audit-service-export": {
  "ClusterId": "audit-cluster",
  "AuthorizationPolicy": "Anonymous",
  "Order": 131,
  "Match": { "Path": "/audit-service/export/{**catch-all}" }
}
```

Audit logs contain identity events, access history, security events, and PII. These routes are explicitly `"Anonymous"` at the gateway, bypassing the gateway's default `RequireAuthorization()`. The audit service's own `QueryAuth:Mode` defaults to `"None"`, which grants `PlatformAdmin` scope to all callers.

**Impact:** Any unauthenticated user can query or export the full audit log, including security events and PII.

**Fix:**
1. Remove `"AuthorizationPolicy": "Anonymous"` from both audit routes in `appsettings.json`.
2. Set `QueryAuth:Mode=Bearer` in the audit service's production configuration.

### 7. PII (Email Address) Logged in Plaintext

**File:** `apps/web/src/app/api/auth/login/route.ts` (line 45)
**OWASP:** A02:2021 – Cryptographic Failures / GDPR / HIPAA

```typescript
// ❌ Email (PII) written to server logs in plaintext — visible to anyone with log access
console.log(`[login] host=${rawHost}, subdomain=${rawSubdomain}, explicitTenant=${explicitTenantCode}, resolvedTenantCode=${tenantCode}, email=${email}`);
```

Similar issues appear in `forgot-password/route.ts` (line 60) and `reset-password/route.ts` (line 41) which log the full identity response body.

**Fix:** Redact or hash emails before logging:

```typescript
// ✅ Redact PII — keep enough context for debugging without exposing full email
const redactedEmail = email.replace(/(.{2}).+@/, '$1***@');
console.log(`[login] host=${rawHost}, subdomain=${rawSubdomain}, email=${redactedEmail}`);
```

### 8. `AllowedHosts: "*"` — Host Header Injection Risk

**File:** `apps/services/identity/Identity.Api/appsettings.json` (line 31)
**OWASP:** A05:2021 – Security Misconfiguration

```json
"AllowedHosts": "*"
```

Wildcard host filtering disables ASP.NET Core's built-in host header validation. This opens the door to host header injection attacks — a classic vector for poisoning password reset links (the reset URL can be constructed from the unvalidated `Host` header), cache poisoning, and SSRF.

**Fix:** Restrict to known hostnames:

```json
"AllowedHosts": "api.legalsynq.com;identity.internal;localhost"
```

### 9. Missing Security Headers on Identity Service and Gateway

**Files:** `apps/services/identity/Identity.Api/Program.cs`, `apps/gateway/Gateway.Api/Program.cs`
**OWASP:** A05:2021 – Security Misconfiguration

Only the audit service sets `X-Content-Type-Options` and `X-Frame-Options`. Neither the identity service nor the gateway sets any security headers, leaving responses without standard browser protections.

**Fix:** Add a middleware pipeline step to both `Program.cs` files:

```csharp
// ✅ Add before app.UseAuthentication()
app.Use(async (ctx, next) =>
{
    ctx.Response.Headers["X-Content-Type-Options"] = "nosniff";
    ctx.Response.Headers["X-Frame-Options"]        = "DENY";
    ctx.Response.Headers["X-XSS-Protection"]       = "0"; // rely on CSP, not legacy header
    ctx.Response.Headers["Referrer-Policy"]        = "strict-origin-when-cross-origin";
    await next();
});
```

### 10. All Tenant Codes Logged on Tenant-Not-Found in `forgot-password`

**File:** `apps/services/identity/Identity.Api/Endpoints/AuthEndpoints.cs` (line 633)
**OWASP:** A01:2021 – Broken Access Control / Information Disclosure

```csharp
// ❌ Full tenant code list written to logs on every failed lookup — tenant codes are sensitive
var allTenants = await db.Tenants.Select(t => new { t.Code, t.IsActive }).ToListAsync(ct);
logger.LogInformation("[forgot-password] Available tenants: {Tenants}",
    string.Join(", ", allTenants.Select(t => $"{t.Code}(active={t.IsActive})")));
```

In a multi-tenant SaaS platform, tenant codes are sensitive organizational identifiers. This debug statement issues a full `SELECT` on every failed lookup and writes all tenant codes to the log stream.

**Fix:** Remove these two lines entirely. The `LogWarning("[forgot-password] Tenant not found...")` on the line above is sufficient for diagnosing the failure.

### 11. JWT Expiry Set to 480 Minutes in Development Config

**File:** `apps/services/identity/Identity.Api/appsettings.Development.json`

`ExpiryMinutes: 480` (8 hours) is configured for development. Not a production risk if dev/prod configs are properly separated via environment variables, but developer machines with stolen sessions have a very long exploitation window.

**Recommendation:** Consider reducing to 60 minutes even for development, or document explicitly that this must never appear in non-Development environments.

## What's Done Well ✅

- **bcrypt with work factor 12** used consistently for password hashing (`BcryptPasswordHasher.cs`) — correct and strong.
- **HttpOnly cookies** used for session tokens — the raw JWT is never returned to browser JavaScript.
- **CSRF protection** — `SameSite: Strict` in production on the session cookie is correct.
- **Session version invalidation** — `session_version` and `access_version` embedded in JWT and validated server-side, enabling immediate invalidation after password change.
- **JWT signature verification** — all services use `ValidateIssuerSigningKey = true`, `ValidateLifetime = true`, and `ClockSkew = TimeSpan.Zero`. No `jwt.decode()` without verification.
- **IDOR prevention** — tenant-scoped queries consistently filter by `tenantId` from the validated JWT claim, not from user-supplied request body.
- **Token hashing** — password reset and invitation tokens are stored as SHA-256 hashes, not in plain text.
- **Comprehensive audit logging** — security events (login, logout, password change, invite accept, etc.) all emitted with actor, tenant, and IP context.
- **EF Core ORM** — parameterized queries used throughout; no raw SQL string interpolation visible in application code.
- **Input validation** — `string.IsNullOrWhiteSpace` guards and regex-validated `Guid.TryParse` used consistently at endpoint boundaries.
