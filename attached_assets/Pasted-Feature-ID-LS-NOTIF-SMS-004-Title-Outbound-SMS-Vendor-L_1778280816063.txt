Feature ID: LS-NOTIF-SMS-004

Title:
Outbound SMS Vendor Log Reconciliation

Objective:
Add outbound SMS vendor-log reconciliation to Notification Service so the service can query Twilio, and later other SMS providers, for outbound message delivery status/logs and reconcile local notification attempts when webhooks are delayed, missed, incomplete, or unavailable.

Critical Architectural Constraint:
Although this feature will be implemented inside the LegalSynq monorepo, all outbound SMS vendor-log reconciliation work must remain isolated within the Notification Service boundary.

Requirements:
- Do not add SMS reconciliation logic to unrelated services.
- Do not create shared cross-service SMS abstractions unless an existing Notification Service shared/internal module already exists.
- Do not modify other services except where strictly necessary for build/config compatibility.
- Notification Service must own:
  - SMS vendor status lookup abstraction
  - Twilio outbound message status lookup
  - SMS reconciliation orchestration
  - SMS reconciliation audit records
  - SMS stale/pending attempt discovery
  - SMS reconciliation API/job if implemented
- Other services should continue interacting with Notification Service through existing notification APIs/events/contracts where possible.
- Preserve Notification Service as the only SMS delivery, tracking, and reconciliation boundary.

Implementation Requirements:
1. Extend the SMS provider abstraction to support outbound vendor message status lookup by provider message ID.
2. Implement Twilio outbound message status retrieval using Twilio Message SID lookup.
3. Normalize vendor status/log data into internal SMS delivery states.
4. Reconcile local NotificationAttempt / notification delivery state when vendor status differs from local state.
5. Add manual reconciliation for a specific outbound SMS attempt or provider message ID if consistent with existing API patterns.
6. Add batch reconciliation for stale/pending outbound SMS attempts if existing worker/job patterns support it.
7. Audit reconciliation attempts, updates, no-op outcomes, and failures.
8. Preserve existing outbound SMS send behavior from LS-NOTIF-SMS-001.
9. Preserve SMS preference/compliance enforcement from LS-NOTIF-SMS-002 and LS-NOTIF-SMS-003.
10. Do not add or expand conversational inbound SMS behavior.

Critical Constraints:
- Reuse existing Notification Service architecture patterns.
- Reuse existing provider adapter, provider routing, NotificationAttempt, webhook normalization, audit, worker, and repository patterns where appropriate.
- Do NOT introduce large rewrites.
- Keep implementation incremental and production-oriented.
- Do NOT hardcode Twilio-specific logic into provider-agnostic reconciliation orchestration.
- Provider-specific vendor lookup may remain in TwilioAdapter only.
- Do NOT expose Twilio credentials or tenant provider secrets.
- Vendor lookup failures must not crash Notification Service.
- Reconciliation must not duplicate sends.
- Reconciliation must not retry or resend messages unless an existing explicit retry workflow already does so.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 2 — Extend SMS Provider Status Lookup Abstraction
Extend the SMS provider abstraction with a provider-agnostic outbound message status lookup method.

Suggested C# shape:

public interface ISmsProviderAdapter
{
    Task<SmsSendResult> SendAsync(
        SmsSendPayload payload,
        CancellationToken cancellationToken = default);

    Task<SmsMessageStatusResult> GetMessageStatusAsync(
        string providerMessageId,
        CancellationToken cancellationToken = default);
}

If changing the existing interface would affect non-SMS code or cause excessive disruption, create a small internal capability interface such as:

public interface ISmsProviderStatusLookup
{
    Task<SmsMessageStatusResult> GetMessageStatusAsync(
        string providerMessageId,
        CancellationToken cancellationToken = default);
}

Requirements:
- Keep abstraction local to Notification Service.
- Do not require unrelated providers/channels to implement SMS lookup.
- Result must normalize vendor response shape.
- Result must safely represent lookup failures.
- Missing/blank providerMessageId must return a structured failure or be skipped by caller.
- Do not expose credentials or raw sensitive provider payloads.

Suggested result model:

public sealed class SmsMessageStatusResult

STEP 3 — Implement Twilio Message Status Lookup
Implement outbound status lookup inside TwilioAdapter.

Expected behavior:
- Use the stored/configured Twilio Account SID and Auth Token.
- Query Twilio Message resource by Message SID / providerMessageId.
- Capture Twilio fields where available:
  - sid
  - status
  - error_code
  - error_message
  - date_created
  - date_sent
  - date_updated
  - to/from if safe and already handled by masking conventions
- Normalize Twilio status into internal status.
- Use existing HTTP/client timeout and error classification patterns where possible.
- Do not log credentials.
- Do not log full phone numbers unless existing service conventions explicitly allow it.

Recommended status mapping:

Twilio queued      -> queued
Twilio accepted    -> queued
Twilio scheduled   -> queued
Twilio sending     -> processing
Twilio sent        -> sent
Twilio delivered   -> delivered
Twilio undelivered -> failed
Twilio failed      -> failed
Twilio canceled    -> failed

Failure handling:
- 401/403 -> non-retryable auth/config failure
- 404 -> non-retryable message_not_found
- 429 -> retryable provider_rate_limited
- 5xx/network/timeout -> retryable provider_unavailable
- Unknown errors -> retryable unexpected_provider_error unless clearly invalid input

STEP 4 — Add Reconciliation Service
Create or extend a Notification Service-local service to reconcile outbound SMS attempts.

Suggested service:
ISmsReconciliationService

Suggested methods:
- ReconcileByAttemptIdAsync(Guid attemptId, CancellationToken ct)
- ReconcileByProviderMessageIdAsync(string providerMessageId, CancellationToken ct)
- ReconcileStalePendingAsync(int limit, TimeSpan olderThan, CancellationToken ct)

Responsibilities:
- Load matching NotificationAttempt by attempt ID or provider message ID.
- Confirm attempt/channel/provider is SMS/Twilio or supported SMS provider.
- Skip attempts without provider message ID.
- Call provider status lookup.
- Compare vendor normalized status with local attempt/notification status.
- Update local attempt and notification state only when appropriate.
- Never send or resend SMS as part of reconciliation.
- Return a structured reconciliation result.

Suggested result model:

public sealed class SmsReconciliationResult
{
    public bool Success { get; init; }
    public bool Updated { get; init; }
    public string Outcome { get; init; } = "";
    public Guid? NotificationId { get; init; }
    public Guid? AttemptId { get; init; }
    public string? Provider { get; init; }
    public string? ProviderMessageId { get; init; }
    public string? PreviousStatus { get; init; }
    public string? VendorStatus { get; init; }
    public string? NewStatus { get; init; }
    public string? ErrorCode { get; init; }
    public string? ErrorMessage { get; init; }
    public bool Retryable { get; init; }
}

Expected outcomes:
- updated
- no_change
- skipped_missing_provider_message_id
- skipped_not_sms
- skipped_unsupported_provider
- vendor_lookup_failed
- attempt_not_found
- provider_message_not_found

STEP 6 — Update Local Attempt / Notification Status Safely
When vendor status differs from local status:

Requirements:
- Update NotificationAttempt delivery/status fields using existing conventions.
- Update parent Notification status only where existing webhook reconciliation logic already does this or equivalent rules are clear.
- Do not regress a terminal local state to a non-terminal vendor state unless existing status precedence rules allow it.
- Do not mark delivered notifications as failed based on stale/ambiguous vendor states without clear provider evidence.
- Preserve webhook reconciliation behavior.
- Add timestamp/metadata fields where existing models support them.
- If a `DeliveryIssue` record should be created for vendor-confirmed failure, reuse existing patterns.
- If status precedence rules are unclear, implement conservative updates and document limitations.

Recommended status precedence:
- delivered is terminal success
- failed/undelivered/canceled is terminal failure when vendor confirms it
- sent can move to delivered or failed
- queued/processing can move forward to sent/delivered/failed
- blocked/dead-letter should not be changed by reconciliation unless existing patterns explicitly allow it
