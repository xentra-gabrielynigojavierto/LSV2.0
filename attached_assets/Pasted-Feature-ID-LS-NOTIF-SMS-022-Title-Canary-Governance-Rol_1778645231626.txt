Feature ID: LS-NOTIF-SMS-022

Title:
Canary Governance Rollout, Tenant Segmentation, and Staged Governance Deployment

Objective:
Extend the governance release-management platform with canary rollout orchestration, tenant segmentation, staged governance deployment, progressive activation strategies, rollback thresholds, deployment cohorts, pause/resume controls, and progressive governance promotion visibility.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- governance rollout plans
- rollout stage orchestration
- tenant cohort targeting
- canary activation
- staged rollout progression
- rollback threshold evaluation
- rollout auditability
- rollout analytics APIs

Control Center must only consume Notification Service APIs and render/administer rollout visibility. Do not duplicate rollout progression, cohort selection, threshold evaluation, or rollback orchestration logic in Control Center.

Requirements:
- Implement governance rollout orchestration inside Notification Service.
- Preserve LS-NOTIF-SMS-017 governance policies.
- Preserve LS-NOTIF-SMS-018 template governance.
- Preserve LS-NOTIF-SMS-019 dynamic rules.
- Preserve LS-NOTIF-SMS-020 versioning, rollback, import/export, and analytics.
- Preserve LS-NOTIF-SMS-021 release management.
- Preserve LS-NOTIF-SMS-021-HARDENING concurrency, approval, locking, audit, retry/backoff, and integrity behavior.
- Rollouts must extend release management, not replace it.
- Rollout failures must not corrupt active governance state.
- Rollout orchestration must not block delivery pipeline execution.
- Do not store raw phone numbers in rollout/cohort/analytics tables.
- Do not expose provider credentials, webhook URLs, SettingsJson, CredentialsJson, auth tokens, raw provider payloads, or recipient phone numbers.
- Keep Control Center as admin/visibility UX only.
- Preserve all LS-NOTIF-SMS-001 through LS-NOTIF-SMS-021-HARDENING behavior.

Implementation Requirements:
1. Add governance rollout plan persistence.
2. Add rollout stage persistence.
3. Add tenant cohort persistence.
4. Add rollout audit event persistence if existing release audit events are not sufficient.
5. Add rollout orchestration service.
6. Add rollout threshold evaluator.
7. Add rollout worker for scheduled/progressive stage advancement.
8. Add rollout analytics APIs.
9. Add start, pause, resume, rollback, and complete rollout flows.
10. Integrate rollouts with LS-NOTIF-SMS-021 release package activation.
11. Ensure canary/staged activation respects release activation locks.
12. Ensure rollback uses existing LS-NOTIF-SMS-020/021 rollback semantics where practical.
13. Add Control Center rollout management UI if feasible.
14. Ensure all failures degrade safely.

Critical Constraints:
- Reuse existing SmsGovernanceReleasePackage, SmsGovernanceReleaseItem, SmsGovernanceReleaseAuditEvent, SmsGovernanceReleaseService, SmsGovernanceReleaseIntegrityService, SmsGovernanceReleaseActivationWorker, SmsGovernanceVersioningService, SmsGovernanceRuleEngine, SmsGovernanceDecision, SmsTemplateGovernanceDecision, SmsGovernanceRuleMatchMetric, admin endpoint, EF/migration, audit/logging, worker, and Control Center governance UI patterns where appropriate.

STEP 9 — Add Rollout Worker
Add optional worker for progressive rollout.

Suggested worker:
SmsGovernanceRolloutWorker

Behavior:
- Poll active rollouts.
- Evaluate current stage health.
- If threshold breach:
  - pause or rollback based on config/threshold action.
- If stage duration elapsed and health acceptable:
  - advance to next stage.
- If final stage completed:
  - complete rollout.
- Bounded batch size.
- Uses IServiceScopeFactory.
- Respects CancellationToken.
- Logs safely.
- Disabled by default.

Requirements:
- Worker failures must not crash app.
- No delivery pipeline blocking.
- No external API calls.
- No raw phones/secrets in logs.
- Worker/manual actions must coordinate with release locks where needed.
