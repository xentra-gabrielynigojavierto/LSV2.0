You are implementing BLK-CC-01 — CareConnect Onboarding Switch (Phase 1) for the LegalSynq platform.

CRITICAL (NON-NEGOTIABLE):

1. Create the report FIRST:
   /analysis/BLK-CC-01-report.md

2. Update the report incrementally after each major step.

3. The report MUST be saved in the ROOT /analysis folder.

4. This block switches CareConnect provider onboarding away from Identity-owned tenant creation
   and toward the new architecture:

   CareConnect → Tenant service (tenant creation / code validation / product activation)
               → Identity service (user ↔ tenant membership / roles)

5. This is PHASE 1 of the CareConnect onboarding switch.
   The goal is to rewire the service calls and restore the broken flow caused by BLK-ID-01.

6. DO NOT:
   - reintroduce tenant creation into Identity
   - modify provider registry model
   - modify referral core logic
   - redesign provider lifecycle
   - remove backward-compatible internal provisioning behavior from Identity

--------------------------------------------------

OBJECTIVE

Refactor CareConnect onboarding so that:

- tenant code availability checks call Tenant service
- tenant creation / self-provisioning calls Tenant service
- Identity is used only for tenant membership / role assignment
- provider lifecycle remains:
    URL → COMMON_PORTAL → TENANT

At the end of this block:

- CareConnect no longer calls retired Identity tenant endpoints
- provider onboarding works again
- provider transitions to TENANT stage through:
    Tenant service → Identity membership APIs → CareConnect provider update

--------------------------------------------------

CONTEXT

Current known state:

- BLK-TS-01 created Tenant code-check + tenant provision APIs
- BLK-TS-02 added Tenant-owned provisioning state + product activation
- BLK-ID-01 retired Identity tenant creation and tenant code-check endpoints (410 Gone)
- BLK-ID-02 added Identity membership APIs:
    - POST /api/internal/users/assign-tenant
    - POST /api/internal/users/assign-roles

CareConnect onboarding is expected to still be pointing at the retired Identity endpoints.
This block must replace that.

PART B — REPLACE TENANT CODE CHECK

--------------------------------------------------

Replace old call:

- GET /api/admin/tenants/check-code    (Identity, retired)

With Tenant service call:

- GET /api/v1/tenants/check-code?code={code}

Requirements:
- preserve normalization behavior
- preserve code-availability UX in onboarding page
- preserve conflict/error handling semantics where practical

If CareConnect currently has a typed client for Identity, add or adapt a typed client for Tenant service instead of misusing Identity clients.

--------------------------------------------------

PART C — REPLACE SELF-PROVISION / TENANT CREATION

--------------------------------------------------

Replace old call:

- POST /api/admin/tenants/self-provision   (Identity, retired)

With Tenant service call:

- POST /api/v1/tenants/provision

The provisioning flow must now become:

1. CareConnect validates provider is eligible for onboarding
   - provider exists
   - AccessStage == COMMON_PORTAL
   - IdentityUserId exists

2. CareConnect calls Tenant service to provision tenant
   Input should include at least:
   - tenantName
   - tenantCode

3. CareConnect receives:
   - tenantId
   - tenantCode
   - subdomain

4. CareConnect then calls Identity membership API to assign the existing user to the new tenant
   - POST /api/internal/users/assign-tenant
   - include roles as needed (e.g. TenantAdmin)

5. CareConnect updates provider:
   - provider.MarkTenantProvisioned(tenantId)

6. CareConnect returns onboarding success result to frontend
   - tenantId
   - subdomain
   - portalUrl

--------------------------------------------------

PART D — IDENTITY MEMBERSHIP INTEGRATION

--------------------------------------------------

Use Identity only for membership / role assignment.

Required behavior:
- DO NOT create a new Identity user
- reuse existing provider.IdentityUserId
- assign the user to the new tenant
- assign required tenant role(s)
- preserve login behavior at new tenant subdomain

Use the new BLK-ID-02 APIs.
Do NOT bypass them with direct DB assumptions from CareConnect.

Refactor service clients cleanly.

Make the split explicit in code so future maintainers can see:

--------------------------------------------------
