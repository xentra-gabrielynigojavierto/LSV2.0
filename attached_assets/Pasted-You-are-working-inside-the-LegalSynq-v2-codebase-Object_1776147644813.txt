- Preserve existing APIs as much as practical
- Keep authorization in the API/filter layer
- Keep business logic in the application layer
- Keep persistence logic in repositories
- Do NOT introduce cross-service DB joins

- Do NOT expose direct status mutation or sale-finalization operations in this feature

- `createdAtUtc`
- `updatedAtUtc`

### List/Search response
Use the existing paginated result pattern already established in Case/Lien APIs.

### Create request DTO
Include only client-supplied fields:
- `lienId`
- `offerAmount`
- `notes` (optional)
- `expiresAtUtc` (optional)

Do NOT allow caller-supplied:
- `buyerOrgId`
- `sellerOrgId`
- `status`
- audit fields
- tenant id / acting user id

### No update request DTO for generic mutation
Do NOT create a generic update DTO that allows direct status mutation.
This feature is for creation + retrieval only, not accept/reject workflow execution.

STEP 3 — ADD APPLICATION SERVICE FOR LIENOFFER OPERATIONS
--------------------------------------------------

Implement the application-layer orchestration needed for:

- create offer
- get offer by id
- list/search offers
- list offers by lien id

Use the existing repository layer:
- `ILienOfferRepository`
- `ILienRepository` (to validate target lien and determine seller org)
- related repositories only if actually needed

Keep endpoint handlers thin.
Do NOT put repository orchestration directly in endpoint methods if the Case/Lien pattern uses application services.

Expected behaviors:

### CreateOfferAsync
- validate lien exists
- validate lien belongs to tenant
- validate lien is in an offerable state (e.g., Offered / appropriate marketplace state)
- validate lien is not sold / settled / cancelled
- validate offer amount > 0
- derive buyer org from request context
- derive seller org from lien
- prevent obviously invalid self-offers if seller org == buyer org, if appropriate to current business rules
- optionally enforce one active offer per buyer per lien if that aligns with donor behavior or sound marketplace design
- create offer through domain factory with status = Pending
- persist via repository

### GetByIdAsync
- tenant-scoped retrieval
- return DTO only

### SearchAsync
Support filters such as:
- `lienId`
- `status`
- `buyerOrgId`
- `sellerOrgId`
- `page`
- `pageSize`

Use request context to determine safe defaults where helpful.
Do NOT require client to send buyer org for their own offers if the context already provides it.

### GetByLienIdAsync
- tenant-scoped retrieval of all offers for a lien
- useful for seller review UI and lien detail views
