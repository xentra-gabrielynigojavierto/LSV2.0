OBJECTIVE

Enable WorkflowTask to support lifecycle transitions (Open → InProgress → Completed / Cancelled) so tasks become actionable and trackable work units.

This phase must NOT change workflow execution behavior and must keep task lifecycle independent from workflow transitions.

SCOPE INCLUDES

- lifecycle service (start, complete, cancel)
- transition validation rules
- timestamp updates
- persistence updates
- concurrency-safe updates
- documentation/reporting

DO NOT BUILD

- workflow progression on task completion
- task reassignment
- task APIs for external consumers (beyond minimal internal usage)
- notifications
- SLA for tasks
- UI

CRITICAL REQUIREMENTS

You MUST:
- enforce valid transitions strictly
- prevent invalid transitions
- update timestamps consistently
- keep lifecycle independent from workflow execution
- avoid touching outbox (E10.2) and SLA (E10.3)
- ensure concurrency safety
- avoid modifying existing task creation/assignment behavior

CURRENT CONTEXT

Already completed:
- E11.1 task entity
- E11.2 task creation
- E11.3 task assignment
- workflow engine / outbox / SLA all stable

This phase activates task lifecycle behavior.

IMPLEMENTATION REQUIREMENTS

1. LIFECYCLE SERVICE

Create a service, for example:

- IWorkflowTaskLifecycleService

Methods:
- StartTask(taskId, ...)
- CompleteTask(taskId, ...)
- CancelTask(taskId, ...)

Requirements:
- service must be stateless
- must use DbContext
- must participate in existing unit-of-work
- must not call SaveChanges internally if system uses external unit-of-work pattern (align with existing patterns)

2. TRANSITION RULES

Allowed transitions:

- Open → InProgress
- Open → Cancelled
- InProgress → Completed
- InProgress → Cancelled

Disallowed transitions (must fail):

- Completed → anything
- Cancelled → anything
- Open → Completed
- InProgress → Open

Requirements:
- enforce rules explicitly
- invalid transitions must throw or return controlled failure
- document behavior

3. TIMESTAMP HANDLING

On Start:
- set StartedAtUtc (if not already set)
- update UpdatedAtUtc

On Complete:
- set CompletedAtUtc
- update UpdatedAtUtc

On Cancel:
- set CancelledAtUtc
- update UpdatedAtUtc

Requirements:
- use UTC timestamps
- align with WorkflowTask.EnsureValid() invariants from E11.1
- do not allow inconsistent states (e.g. Completed without CompletedAtUtc)

4. CONCURRENCY HANDLING

Ensure safe updates under concurrent access.

Acceptable approaches:
- optimistic concurrency (RowVersion / concurrency token)
- or safe update pattern with state check in WHERE clause

Requirements:
- only one transition succeeds
- conflicting updates fail cleanly
- no silent overwrites

Document chosen approach.

5. VALIDATION RULES

Before any transition:

- task must exist
- task must be accessible (respect tenant scoping)
- current status must allow requested transition

Requirements:
- fail fast on invalid state
- do not partially update task

6. ASSIGNMENT INTERACTION

Requirements:
- lifecycle transitions must NOT modify AssignedUserId / AssignedRole / AssignedOrgId
- assignment remains unchanged across lifecycle

7. NO WORKFLOW MUTATION

CRITICAL:

Lifecycle service must NOT:
- change WorkflowInstance
- advance workflow
- trigger workflow transitions

This will be handled later (E11.7).

8. LOGGING (LIGHT)

Requirements:
- log transitions:
  - taskId
  - previous status
  - new status
- do not over-log
- avoid sensitive data

9. NON-INTERFERENCE

You MUST confirm:

- workflow engine unchanged
- task creation (E11.2) unchanged
- assignment (E11.3) unchanged
- outbox unchanged
- SLA unchanged

VALIDATION REQUIREMENTS

- InProgress → Completed works

- invalid transitions are blocked
