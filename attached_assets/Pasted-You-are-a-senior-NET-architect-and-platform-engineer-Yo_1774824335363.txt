You are a senior .NET architect and platform engineer.

Your task is to create a **new parallel .NET Core implementation** of the current Documents Service, while **preserving the existing Node.js/TypeScript service unchanged**.

## Primary Goal

Build a **.NET Core version of the Documents Service** that mirrors the current Node.js service architecture and behavior, but lives as a separate service in the repository.

This is a **parallel implementation**, not a replacement.

### Important constraints

* Do NOT remove or rewrite the current Node.js service
* Do NOT break existing Node.js builds
* Do NOT rename or repurpose the current service
* The .NET version must be created as a **new service** alongside the existing implementation
* Preserve the current API behavior and security model as closely as practical
* Use the current Node.js implementation as the source of truth for behavior

---

## Target Outcome

Create a new standalone service, for example:

* `apps/services/documents-dotnet/`

or equivalent repo-appropriate location.

This new service should:

* run independently
* expose equivalent endpoints
* follow the same business rules
* use a clean .NET Core architecture
* be ready for incremental parity validation against the Node.js service

---

## Technology Requirements

Use:

* **.NET 8 / ASP.NET Core**
* C#
* Swagger / OpenAPI
* dependency injection
* structured logging
* clean architecture style

Recommended packages / patterns:

* ASP.NET Core Web API

## Behavior Parity Requirements

Match the Node.js service as closely as possible for:

### Authentication

* Bearer JWT
* tenant-aware principal extraction
* role-based access control
* default deny

### Tenant isolation

* enforce tenant scoping in repository/service flows
* preserve the same multi-tenant security model

### Malware scanning

* preserve scan-status-aware behavior
* allow pluggable scanner provider
* block access when scan rules require it

### Access mediation

* preserve token-based access flow
* keep one-time-use token design where implemented
* do not expose storage keys to clients

### Rate limiting

* scaffold or implement equivalent rate limiting behavior
* document if exact parity is deferred

### Audit logging

* preserve audit-oriented design and core events
* log critical document lifecycle and access events

## Database Requirement

IMPORTANT:
Before implementing, inspect the current Node.js service and determine the actual schema and database assumptions.

Then do the following:

1. document whether the current implementation is actually PostgreSQL-specific, MySQL-specific, or mixed
2. choose a realistic strategy for the .NET version:

   * either match the existing real DB behavior
   * or explicitly isolate the DB layer behind repositories and document assumptions

Do NOT blindly claim MySQL compatibility if the current schema is PostgreSQL-specific.

Generate a clear report on this.
