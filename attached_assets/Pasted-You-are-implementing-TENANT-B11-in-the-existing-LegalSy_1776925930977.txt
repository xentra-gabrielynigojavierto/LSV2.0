FEATURE ID
TENANT-B11 — Complete Tenant Write Ownership (Controlled)
================================================================

OBJECTIVE
Move tenant-management write ownership further from Identity into the Tenant service in a controlled, reversible way, while preserving Identity’s role for authentication, admin-user creation, role assignment, provisioning, and session enforcement.

This block must:
1. move safe tenant-management writes to the Tenant service,
2. expose richer Tenant admin read APIs so Control Center can stop depending on Identity for tenant-management views,
3. introduce controlled Tenant → Identity orchestration only where Identity still owns auth/provisioning concerns,
4. close known sync gaps from prior blocks,
5. preserve rollback capability and backward compatibility.

This block must NOT:
- delete Identity tenant tables/columns,
- remove Identity fallback/compatibility endpoints,
- move auth/session enforcement out of Identity,
- force a big-bang CreateTenant rewrite without preserving provisioning dependencies.

SCOPE
================================================================

A) Expand Tenant admin read APIs so Control Center can use Tenant as the main tenant-management source

Implement richer Tenant admin read endpoints so Control Center no longer needs Identity for basic tenant-management screens.

Target endpoints in Tenant service:
- GET /api/v1/admin/tenants
- GET /api/v1/admin/tenants/{id}
- optionally:
  - GET /api/v1/admin/tenants/by-code/{code}
  - GET /api/v1/admin/tenants/{id}/summary

The response must include enough data for tenant-management UI and BFFs, at minimum:
- tenant core profile
- branding basics
- domains
- product entitlements
- capabilities summary or counts
- settings summary or selected key settings
- migration/cutover indicators only if it fits naturally

IMPORTANT:
- If session timeout still truly belongs to Identity auth policy, do NOT duplicate ownership.
- Instead, either:
  1. expose it as compatibility/read-through data from Identity in a clearly marked compatibility field, OR
  2. omit it from Tenant admin response and document why.
- Prefer a clean aggregated admin DTO rather than leaking internal entities one by one.

The goal is to enable Control Center tenant-admin views to read primarily from Tenant.

B) Transition safe tenant-management write APIs to Tenant service

Move safe tenant-management writes into Tenant service where Tenant is already the true owner.

Candidate writes to move in this block:
- tenant profile update
- branding update (already mostly Tenant-owned)
- domain management (already Tenant-owned)
- product entitlements
- capabilities
- tenant settings
- possibly tenant activation/suspension status updates where this does not break auth semantics

Expose/complete Tenant admin endpoints as needed, for example:
- PUT /api/v1/admin/tenants/{id}
- PATCH /api/v1/admin/tenants/{id}/status
- existing admin branding/domain/settings/product endpoints as appropriate

Rules:
- Tenant becomes the primary write owner for tenant master data
- writes must update Tenant persistence first
- if Identity still needs mirrored state for auth/session/login gating, use controlled sync/orchestration
- do not move session-timeout enforcement ownership out of Identity in this block unless you can prove it is safe and already modeled cleanly

C) Introduce controlled Tenant → Identity orchestration where Identity still owns downstream concerns

This is the reverse-direction counterpart to earlier Identity → Tenant sync.

Implement a controlled adapter/orchestration path from Tenant service to Identity for the cases where Identity still must do work.

Examples:
- Create tenant with admin user
- update auth-relevant status fields if Identity must mirror them
- provision admin access/roles if the originating write now starts in Tenant
- optionally read-through for session settings if needed

Recommended architecture:
- Tenant-side interface such as:
  - IIdentityProvisioningAdapter
  - IIdentityTenantCompatAdapter
- Infrastructure HTTP implementation in Tenant service
- Feature-flagged and non-breaking
- explicit timeout handling
- strict vs non-strict behavior if needed

Rules:
- keep the service boundary clean
- no direct DB access from Tenant to Identity DB
- use HTTP/internal service APIs or existing admin endpoints
- failures must be contained and logged
- choose strict vs non-strict carefully and default to safe behavior

D) Controlled CreateTenant ownership transition

This is the highest-risk part of B11.
Do NOT blindly remove CreateTenant from Identity.

Implement one of these safe patterns:

Pattern A (recommended):
- Introduce a new Tenant service admin endpoint for orchestrated tenant creation, e.g.:
  - POST /api/v1/admin/tenants
- Tenant service persists the canonical tenant record
- Tenant service then calls Identity adapter to:
  - create admin user / tenant admin context
  - assign roles
  - provision auth-related scaffolding
  - trigger existing provisioning hooks as needed
- If downstream Identity provisioning fails:
  - choose explicit behavior:
    - rollback Tenant create if safe, OR
    - mark tenant as Pending / ProvisioningFailed and return a controlled error/result
- Document chosen transactional model clearly

Pattern B:
- Keep CreateTenant endpoint in Identity, but make Tenant the explicit primary owner by moving tenant persistence responsibility first into Tenant and making Identity act as orchestrator only.

Choose the pattern that best fits current code structure with minimum risk.
The goal is:
- new tenant master record ownership is in Tenant,
- Identity still performs auth/provisioning steps,
- no big-bang breakage.

E) Fix known sync gaps from B10

You MUST fix the pre-existing sync gaps noted in B10, especially:
- Identity ClearTenantLogo / ClearTenantLogoWhite must trigger sync/update behavior so Tenant does not lag when fallback endpoints are still used
- ensure all remaining Identity-side tenant write operations that affect Tenant-owned fields either:
  - sync correctly, or

Document all write paths audited and their sync status.

G) Compatibility and rollback

Identity compatibility endpoints may remain, but they should be:

- safe for rollback

Do NOT:

This block is about completing ownership transfer, not destructive retirement.
