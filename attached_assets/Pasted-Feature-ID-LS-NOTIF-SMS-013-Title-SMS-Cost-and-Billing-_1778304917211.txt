Feature ID: LS-NOTIF-SMS-013

Title:
SMS Cost and Billing Analytics

Objective:
Add SMS cost tracking, provider spend visibility, tenant usage allocation, and operational billing analytics inside Notification Service and Control Center so platform operators can analyze SMS spend, provider efficiency, tenant attribution, retry/failure spend, and usage trends.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- SMS delivery execution metadata
- tenant/provider attribution
- provider message identifiers
- SMS reconciliation state
- SMS cost metadata
- SMS cost aggregation APIs

Control Center must only consume Notification Service APIs and render cost analytics UI. Do not duplicate cost aggregation logic in Control Center.

Requirements:
- Implement SMS cost persistence and aggregation inside Notification Service.
- Implement Control Center cost analytics UI as a consumer of Notification Service APIs only.
- Do not introduce a real invoicing engine.
- Do not add payment processing.
- Do not add tax calculation.
- Do not add external accounting integration.
- Do not expose provider credentials, webhook URLs, CredentialsJson, SettingsJson, auth tokens, raw provider payloads, or full phone numbers.
- Preserve existing outbound SMS delivery, reconciliation, dashboard, alerting, escalation, and incident-management behavior.
- Keep implementation incremental and production-oriented.

Implementation Requirements:
1. Persist safe SMS cost metadata on SMS attempts or a Notification Service-local cost table.
2. Attribute SMS costs by tenant, provider, providerConfigId, providerOwnershipMode, status, and date.
3. Support estimated cost initially where provider actual billing data is unavailable.
4. Support actual/reconciled cost if provider lookup returns safe billing/cost fields.
5. Add admin cost analytics APIs inside Notification Service.
6. Add Control Center SMS Cost Analytics UI.
7. Support summary, trend, provider, tenant, ownership-mode, failure-cost, and retry-cost analytics.
8. Support CSV/export-friendly reporting endpoint or export-ready JSON endpoint.
9. Preserve all existing SMS operational APIs and UI behavior.
10. Keep all cost analytics read-only except explicit cost metadata backfill/update operations if absolutely necessary.

Critical Constraints:
- Reuse existing NotificationAttempt, ProviderConfigId, ProviderOwnershipMode, TenantId, provider, status, reconciliation, dashboard, activity, endpoint, authorization, migration, and Control Center UI patterns where appropriate.
- Do NOT introduce large rewrites.
- Do NOT store provider credentials or raw provider payloads in cost fields.
- Do NOT expose full phone numbers.
- Do NOT call external payment/billing processors.
- Do NOT perform FX conversion in this feature.
- Use a single currency per cost record, preferably ISO currency code such as USD.
- If provider actual cost is unavailable, store/use estimated cost and clearly mark costSource = estimated.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 2 — Define SMS Cost Persistence Model
Choose the least disruptive persistence model.

Preferred option:
Add cost fields to NotificationAttempt if it already owns SMS provider execution metadata.

Suggested fields:
- EstimatedCostAmount decimal? nullable
- ActualCostAmount decimal? nullable
- CostCurrency string? nullable, default USD where configured
- CostSource string? nullable
- CostRecordedAt DateTime? nullable
- ProviderBillingReference string? nullable

Alternative option:
Create SmsAttemptCost table if adding fields to NotificationAttempt would be inappropriate.

Suggested cost source values:
- estimated
- provider_reconciled
- manual
- unavailable

Requirements:
- Cost metadata must be scoped to SMS attempts only.
- Do not store provider credentials.
- Do not store raw provider billing payloads.
- Do not store phone numbers.
- Use decimal for monetary values.
- Use ISO currency codes.
- Existing attempts must remain valid.
- Default values must not imply actual spend when unknown.

STEP 3 — Add Cost Estimation Configuration
Add configuration for baseline SMS cost estimation.

Suggested config:
SmsCostAnalytics:
- Enabled bool default true
- DefaultCurrency string default USD
- DefaultEstimatedOutboundSmsCost decimal default 0.00 or configurable
- TwilioEstimatedOutboundSmsCost decimal nullable
- FailedMessageCostPolicy string default count_estimated_when_provider_accepted
- RetryCostPolicy string default per_attempt

Environment variable examples:
- SMS_COST_DEFAULT_CURRENCY=USD
- SMS_COST_DEFAULT_ESTIMATED_OUTBOUND_COST=0.0000
- SMS_COST_TWILIO_ESTIMATED_OUTBOUND_COST=0.0075

Requirements:
- Safe defaults.
- If cost is unknown, use 0.00 or null with CostSource = unavailable; do not invent actual cost.
- Estimated values must be configurable.
- Documentation must clearly state estimated cost is not invoice-grade billing.

STEP 4 — Populate Cost Metadata During SMS Send
Update outbound SMS send attempt handling so successful provider acceptance can record estimated cost metadata.

Requirements:
- Cost should be recorded only for SMS attempts.
- Use provider-specific configured estimate where available.
- If no estimate configured, set CostSource = unavailable or EstimatedCostAmount = null.
- Preserve existing send/retry/dead-letter behavior.
- Do not affect delivery success/failure semantics.
- Do not block send if cost recording fails unless existing repository patterns require it.
- Do not store recipient phone or provider secrets.

STEP 5 — Update Reconciliation to Capture Actual Cost if Available
Review Twilio Message lookup response from vendor reconciliation.

Requirements:
- If Twilio lookup provides safe billing price/currency fields and current adapter parses them, map to ActualCostAmount / CostCurrency / CostSource = provider_reconciled.
- If Twilio billing price is unavailable or not parsed, do not fake actual cost.
- Preserve existing reconciliation behavior.
- Provider lookup failures must not impact cost data except maybe CostSource remains estimated/unavailable.
- Do not store raw vendor payloads.
- Do not store provider credentials.

Note:
If actual provider billing fields are not available in current Twilio response parsing, document actual cost as a known gap and keep estimated-cost analytics only.
