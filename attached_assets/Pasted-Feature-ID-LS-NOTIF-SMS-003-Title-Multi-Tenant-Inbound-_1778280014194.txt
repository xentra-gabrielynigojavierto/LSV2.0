Feature ID: LS-NOTIF-SMS-003

Title:
Multi-Tenant Inbound SMS Resolution + Compliance Hardening

Objective:
Harden SMS compliance for multi-tenant production use by resolving inbound Twilio SMS webhooks to the correct tenant/provider configuration, preserving immutable SMS preference history, and improving operational compliance traceability inside Notification Service.

Critical Architectural Constraint:
Although this feature will be implemented inside the LegalSynq monorepo, all inbound SMS tenant resolution, compliance hardening, and SMS preference history work must remain isolated within the Notification Service boundary.

Requirements:
- Do not add SMS compliance logic to unrelated services.
- Do not create shared cross-service SMS abstractions unless an existing Notification Service shared/internal module already exists.
- Do not modify other services except where strictly necessary for build/config compatibility.
- Notification Service must own:
  - inbound SMS tenant/provider resolution
  - SMS preference history
  - unresolved inbound webhook handling
  - SMS compliance audit enrichment
  - SMS preference history APIs if implemented
- Other services should continue interacting with Notification Service through existing notification APIs/events/contracts where possible.
- Preserve Notification Service as the only delivery execution and compliance enforcement boundary for SMS.

Implementation Requirements:
1. Resolve inbound Twilio SMS webhooks to tenant/provider configuration using the inbound Twilio `To` number.
2. Store STOP/START/HELP processing under the resolved tenant when possible.
3. Add immutable SMS preference history records for all preference-impacting events.
4. Preserve the current-state SMS preference behavior implemented in LS-NOTIF-SMS-002.
5. Safely handle unresolved inbound numbers without crashing webhook ingestion.
6. Enrich audit records with tenant/provider context where available.
7. Preserve existing outbound Twilio delivery-status webhook behavior.
8. Preserve compatibility with LS-NOTIF-SMS-001 and LS-NOTIF-SMS-002.

Critical Constraints:
- Reuse existing Notification Service architecture patterns.
- Reuse existing tenant provider configuration, webhook ingestion, SMS preference, audit, and repository patterns where appropriate.
- Do NOT introduce large rewrites.
- Keep implementation incremental and production-oriented.
- Do NOT hardcode tenant resolution logic outside Notification Service.
- Do NOT expose Twilio credentials or tenant provider secrets.
- All SMS compliance logic must remain provider-agnostic except Twilio-specific inbound parsing/verification.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 2 — Add Inbound SMS Tenant/Provider Resolution
Implement a Notification Service-local lookup that resolves inbound Twilio SMS webhooks by the Twilio `To` number.

Expected behavior:
- Twilio inbound webhook includes:
  - `To` = platform or tenant Twilio number that received the inbound message
  - `From` = recipient/contact phone number
  - `Body` = inbound SMS text
  - `MessageSid` = provider message ID
- Lookup should attempt to resolve:
  - tenantId
  - provider name, expected `twilio`
  - provider config id where available
  - inbound/to phone number
- Prefer using existing TenantProviderConfig or provider configuration records if available.
- If no suitable repository/method exists, add a minimal Notification Service-local repository method for lookup by provider + phone/from number.
- Do not expose or return provider credentials.
- Normalize phone numbers consistently before comparison where existing normalization exists.

Required lookup semantics:
- Match provider = `twilio`
- Match configured sender/from number or equivalent field to inbound webhook `To`
- Support tenant-owned provider configurations if already modeled
- Support platform/default provider config if already modeled
- Return null/empty result if unresolved

STEP 3 — Handle Unresolved Inbound Webhooks Safely
If inbound Twilio `To` number cannot be resolved to a tenant/provider configuration:

Requirements:
- Do not crash webhook ingestion.
- Do not create tenant-scoped current-state SMS preference with incorrect tenant.
- Record a structured log and/or audit event for unresolved inbound SMS.
- Do not log full phone numbers if existing masking convention requires masking.
- Preserve Twilio webhook response behavior so Twilio does not retry unnecessarily unless the architecture intentionally returns failure.
- Include safe metadata:
  - provider = twilio
  - masked from phone
  - masked to phone
  - provider message id
  - keyword classification if available
  - reason = unresolved_inbound_sms_tenant

Suggested audit/log event:
- sms.inbound.unresolved_tenant

STEP 5 — Update SMS Preference Service to Write History
Extend `ISmsPreferenceService` and implementation so every preference-related action writes history.

Required cases:
- Inbound STOP-family keyword:
  - current state becomes `opted_out`
  - history record appended with previous state and new state
- Inbound START-family keyword:
  - current state becomes `opted_in`
  - history record appended with previous state and new state
- Inbound HELP keyword:
  - current state should not change
  - history record appended with `NewState = help_requested`
- Manual API update:
  - current state changes to requested state
  - history record appended with previous state and new state
- Unknown/unresolved inbound keyword:
  - do not create incorrect tenant-scoped current state
  - append unresolved history only if a safe nullable-tenant pattern is supported; otherwise structured log/audit is acceptable

Requirements:
- History write should be best-effort only if existing service patterns treat audit/history as best-effort; otherwise it should be part of the same transaction as preference update.
- Avoid partial current-state update without history if transaction patterns are available.
- Include provider metadata for inbound events where available.
- Include actor/user metadata for manual updates where available.

STEP 6 — Update Twilio Inbound Keyword Processing
Modify Twilio inbound handling so tenant resolution occurs before SMS preference processing.

Expected flow:
1. Verify Twilio webhook signature as existing behavior requires.
2. Detect inbound message using `Direction` field.
3. Extract:
   - From
   - To
   - Body
   - MessageSid
4. Classify keyword using existing LS-NOTIF-SMS-002 keyword classification.
5. Resolve tenant/provider using `To`.
6. If resolved:
   - process keyword under resolved tenant
   - include provider/providerConfigId/inboundToNumber metadata
   - audit using resolved tenant
7. If unresolved:
   - do not mutate tenant-scoped current preference
   - log/audit unresolved inbound event safely
8. Preserve outbound delivery status handling unchanged.

Inbound detection should continue supporting:
- Direction = inbound
- Direction = inbound-api
