==================================================
CORE FLOWS TO TEST
==================================================

You must test these flows end to end as far as the current repo/environment allows.

--------------------------------------------------
FLOW 1 — SYSTEM STARTUP / BOOTSTRAP
--------------------------------------------------

Validate:
- app builds successfully
- backend services start
- frontend starts
- key routes load
- migrations / schema state do not block startup
- obvious environment/config issues are resolved or clearly documented

Check:
- no fatal startup exceptions
- APIs reachable
- frontend can talk to backend
- auth/session behavior works in local/dev environment

--------------------------------------------------
FLOW 2 — ADMIN LOGIN / ACCESS
--------------------------------------------------

Validate:
- admin can log in
- admin can access CareConnect admin routes
- non-admin access is blocked where expected

Key routes:
- /careconnect/admin/dashboard
- /careconnect/admin/providers/provisioning
- /careconnect/admin/providers/blocked
- /careconnect/admin/referrals
- /careconnect/admin/performance

--------------------------------------------------
FLOW 3 — LAW FIRM REFERRAL CREATION
--------------------------------------------------

Validate:
- law firm user can search/select provider
- law firm can create a referral
- referral is saved with canonical initial state
- referral appears in law firm view
- no appointment dependency is required

Check specifically:
- status starts as New
- selected provider is attached correctly
- referral list/detail reflects created referral

--------------------------------------------------
FLOW 4 — PROVIDER EMAIL / LINK ENTRY
--------------------------------------------------

Validate:
- provider notification flow is wired correctly
- referral email link format is generated correctly
- token-based referral entry route works
- invalid token handling works
- login redirect occurs correctly

Check:
- /referrals/view?token=...
- returnTo behavior
- invalid / expired / revoked token flow
- no public referral detail exposure

If actual email sending cannot be fully exercised in local/dev, validate via:
- generated notification records
- rendered URLs
- email service invocation paths
- logs / test doubles / local mail capture if available

--------------------------------------------------
FLOW 5 — AUTHENTICATED PROVIDER REFERRAL ACCESS
--------------------------------------------------

Validate:
- properly provisioned provider can log in
- provider can access referral detail
- provider sees Accept action only when valid
- blocked provider sees admin-contact blocking page
- referral details are hidden from blocked providers

Test both:
1. correctly provisioned provider
2. authenticated but unprovisioned/misconfigured provider

--------------------------------------------------
FLOW 6 — PROVIDER ACCEPTANCE
--------------------------------------------------

Validate:
- provider can accept referral only through authenticated path
- status changes from New → Accepted
- duplicate acceptance is blocked
- wrong provider cannot accept
- law firm sees updated Accepted status after refresh

Check:
- referral detail action visibility
- backend enforcement
- audit/history entry
- repeat click behavior

--------------------------------------------------
FLOW 7 — ACCEPTANCE NOTIFICATIONS
--------------------------------------------------

Validate:
- law firm acceptance notification is triggered
- client acceptance notification is triggered when client email exists
- missing client email does not block acceptance
- notification duplication does not happen on repeated accepts

If real SMTP is unavailable, verify by:
- notification records
- logs
- mocked delivery
- retry queue / send pipeline behavior

--------------------------------------------------
FLOW 8 — BLOCKED PROVIDER FLOW
--------------------------------------------------

Validate:
- blocked provider attempt is logged
- provider sees “contact your administrator” style blocking page
- no referral data leaks
- blocked provider appears in admin blocked queue
- remediation link points to provisioning page

--------------------------------------------------
FLOW 9 — ADMIN PROVISIONING WORKFLOW
--------------------------------------------------

Validate:
- admin can inspect provider readiness
- admin can provision CareConnect receiver access
- provisioning is idempotent
- readiness recheck updates correctly
- provider can retry login/access after provisioning

Check:
- missing role case
- already-ready case
- invalid target handling where feasible

--------------------------------------------------
FLOW 10 — ADMIN OPERATIONAL VISIBILITY
--------------------------------------------------

Validate:
- dashboard metrics load
- blocked provider queue loads
- referral queue loads
- filters / sorting work at basic level
- remediation navigation works

Check:
- empty states
- recent data visibility
- queue pages do not crash on partial data

--------------------------------------------------
FLOW 11 — PERFORMANCE METRICS
--------------------------------------------------

Validate:
- performance page loads
- summary metrics render
- aging distribution renders
- provider responsiveness table renders
- empty state works
- no divide-by-zero / NaN issues

--------------------------------------------------
FLOW 12 — REGRESSION PASS
--------------------------------------------------

Re-check that recent hardening remains true:
- no unauthenticated referral acceptance
- no cross-account leakage
- blocked providers cannot see referral details
- accepted state remains canonical
- admin-only routes remain protected

OUTPUT REPORT REQUIREMENTS
==================================================

Generate a report at:

2. Environment / Startup Readiness

- [PASS] Admin login works
- [PASS] Law firm can create referral

6. Open Defects / Blockers

7. Regression Check

- what remains uncertain

8. Test Data / Accounts Used
- what user roles/data were used
- how to reproduce the scenarios

9. Final Readiness Verdict
Choose one:
- READY FOR MANUAL TESTING
- READY WITH MINOR KNOWN ISSUES
- NOT READY

10. Recommended Next Step
- smallest, most practical next action after this validation pass
