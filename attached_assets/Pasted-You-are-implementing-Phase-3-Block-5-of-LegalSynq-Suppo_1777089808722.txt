FEATURE NAME:
Multi-Mode Support

Report file:
 /analysis/SUP-TNT-04-report.md

PRIMARY OBJECTIVE:
Introduce tenant-level Support mode configuration so each tenant can operate Support in one or both supported modes:
1. Internal/platform support only
2. Tenant-to-client customer support

This must allow customer-facing Support capabilities to be enabled or disabled per tenant without breaking existing internal/admin/agent Support workflows.

IMPORTANT CONTEXT:
Completed:
- SUP-TNT-01 — External Customer model
- SUP-TNT-02 — Ticket ownership + visibility
- SUP-TNT-02-01 — migration default correction
- SUP-TNT-03 — Customer RBAC Layer
- SUP-INT-07 — Customer Support Portal UI

Current customer support capability:
- Customer-safe endpoints exist:
  - GET  /support/api/customer/tickets
  - GET  /support/api/customer/tickets/{id}
  - POST /support/api/customer/tickets/{id}/comments
- Customer UI exists but fails closed until customer token issuance/login exists.
- Customer access requires JWT role ExternalCustomer, tenant_id, and external_customer_id.

CRITICAL RULES:
1. DO NOT rewrite Support.
2. DO NOT break existing internal/admin/agent ticket flows.
3. DO NOT weaken JWT/RBAC.
4. DO NOT weaken tenant isolation.
5. DO NOT inject X-Tenant-Id in production.
6. DO NOT add customer login/token issuance in this block.
7. DO NOT add Comms/email reply/SLA/Task logic.
8. DO NOT add queue/assignment/status controls to customer UI.
9. DO NOT store file bytes in Support DB.
10. Preserve standalone deployability.
11. Support mode must be tenant-scoped.
12. Customer-facing endpoints must fail closed when tenant support mode is disabled.

STEP 3 — SUPPORT MODE MODEL:
Introduce tenant-level support mode.

Preferred simple enum:
SupportTenantMode:
- InternalOnly
- TenantCustomerSupport

Meaning:
- InternalOnly:
  - Internal/admin/agent Support works
  - Customer endpoints disabled for that tenant
  - Customer UI should show unavailable/access denied
- TenantCustomerSupport:
  - Internal/admin/agent Support works
  - Customer endpoints enabled for that tenant
  - Customer-visible ticket access allowed through customer RBAC

Important:
- This must be tenant-scoped.
- Default must be InternalOnly for safety.
- Do not enable customer support globally by default.

STEP 4 — STORAGE STRATEGY:
Implement the smallest safe storage model.

Preferred option:
Create a tenant-scoped table:

support_tenant_settings

Fields:
- tenant_id string PK or unique
- support_mode string required default 'InternalOnly'
- customer_portal_enabled bool required default false
- created_at datetime UTC
- updated_at datetime UTC nullable

Rules:
- tenant_id is the isolation boundary
- no relation to Identity tenant table required
- no cross-service FK
- one row per tenant
- if no row exists, effective mode = InternalOnly

Alternative:
If an existing tenant settings/config table exists in Support, extend it instead. Do not create duplicate settings tables.

STEP 5 — SETTINGS SERVICE:
Create service:

ISupportTenantSettingsService

Required methods:
- GetEffectiveSettingsAsync(tenantId)
  - if no row exists, return default InternalOnly/customerPortalEnabled=false
- SetSupportModeAsync(tenantId, mode, customerPortalEnabled, actorUserId)
  - internal/admin use only
- IsCustomerSupportEnabledAsync(tenantId)
  - true only if mode = TenantCustomerSupport AND customerPortalEnabled = true

Do not expose public customer mutation access.

STEP 7 — CUSTOMER ENDPOINT ENFORCEMENT:
Update customer endpoints so they check tenant support mode before serving data.

For all:
- GET /support/api/customer/tickets
- GET /support/api/customer/tickets/{id}
- POST /support/api/customer/tickets/{id}/comments

Required behavior:
- If customer support mode disabled:
  - return 403 or 404 consistently
  - prefer 403 if authenticated customer but tenant feature disabled
  - do not leak ticket existence
- If enabled:
  - proceed with existing customer RBAC:
    tenantId + externalCustomerId + CustomerVisible

Do not alter internal/admin endpoints.
