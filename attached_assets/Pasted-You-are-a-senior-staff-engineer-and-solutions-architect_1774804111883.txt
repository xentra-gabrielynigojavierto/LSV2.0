You are a senior staff engineer and solutions architect. Your job is to upgrade the existing LegalSynq DMM TypeScript service into an enterprise-grade, **independent, portable, cloud-agnostic, highly secure, and compliance-ready Docs Service**.

---

## Primary Goal

Transform the current DMM service into a **standalone Docs Service** that:

* runs independently of other LegalSynq services
* can be deployed in isolation (containerized)
* is reusable across different systems
* is cloud-provider agnostic (AWS, GCP, others)
* enforces **enterprise-grade security and compliance standards**
* is suitable for **legal + medical document handling (HIPAA-aligned readiness)**

---

## Critical Design Constraints

---

### 1. Service Independence

The Docs Service must:

* NOT depend directly on:

  * TPP or any LegalSynq service
  * shared databases
* function with only:

  * its own database
  * its own storage provider
  * its own auth validation
* use adapters for ALL external integrations

---

### 2. Cloud Provider Agnosticism (MANDATORY)

The service must NOT be tightly coupled to AWS.

Support:

* AWS (S3)
* GCP (Cloud Storage)
* future providers

#### Required abstractions:

* `StorageProvider`
* `AuthProvider`
* `SecretsProvider`
* `FileScannerProvider` (future)
* `AuditProvider` (optional)

#### Implement:

* `S3StorageProvider`
* `LocalStorageProvider`
* scaffold `GCSStorageProvider`

#### Rules:

* NO AWS/GCP SDK usage outside infrastructure layer
* provider selected via config:

  * STORAGE_PROVIDER
  * AUTH_PROVIDER
  * SECRETS_PROVIDER

---

### 3. Enterprise Security & Compliance (MANDATORY)

This service handles **sensitive legal and medical data**.

You MUST design for:

#### Data Protection

* encryption at rest (provider-level support assumed, but enforced via config)
* encryption in transit (HTTPS only assumptions)
* NO public file access
* all files must be private
* access via short-lived signed URLs only

#### Authentication & Authorization

* replace Basic Auth + MD5 with secure JWT validation
* support:

  * tenantId
  * productId
  * userId
  * roles
* enforce:

  * RBAC (role-based)
  * ABAC (tenant + document scope)
* default DENY access model

#### Input & File Security

* validate:

  * MIME types
  * file size limits
  * extension vs MIME mismatch
* reject dangerous file types if needed
* prepare hook for malware scanning (`FileScannerProvider`)

#### Audit & Traceability (CRITICAL)

* ALL actions must be auditable:

  * document upload
  * version upload
  * metadata update
  * status changes
  * deletions
  * access attempts (success + denied)
  * signed URL generation

#### Compliance Features (HIPAA-aligned readiness)

* audit trail (immutable-friendly design)
* document access logging
* tenant isolation
* soft delete + retention fields
* support:

  * legal hold (future)
  * retention policies (future)
* no sensitive data in logs

#### Secrets Management

* NO hardcoded secrets
* support:

  * env-based config
  * pluggable secrets provider
