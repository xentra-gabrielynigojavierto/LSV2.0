You are implementing LS-LIENS-UI-002 for the LegalSynq Liens v2 migration.

Feature ID: LS-LIENS-UI-002
Feature Title: Wire Cases UI to Real APIs with Enforced API Service Layer Pattern

Objective:
Wire the SynqLiens Cases UI to real backend APIs and enforce a strict frontend API service layer pattern. This feature must preserve the existing UI baseline established in LS-LIENS-UI-01 while replacing mock/store-driven backend data behavior with real API-backed integration. This is the reference implementation pattern for later Liens modules.

Critical architecture rules:
1. Preserve the existing v2 top nav and side nav.
2. Preserve the current Cases UI layout and screen flow unless a minimal change is required for real integration.
3. Use a strict layered frontend data-access pattern:
   - API client layer
   - Cases service layer
   - DTO-to-UI mapper/adapter layer
   - UI state layer
   - Presentation layer
4. No direct backend HTTP calls in page or presentation components.
5. No significant response mapping logic inside page/components.
6. No production mock/static/local fake repository behavior may remain in the Cases production flow.
7. If Zustand or an equivalent store is used, restrict it to UI state only unless an existing platform-standard async state pattern requires a small exception. Document any exception explicitly.
8. Reuse existing v2 auth, interceptor, routing, shell, and platform conventions.
9. Preserve donor SynqLiens behavior and API contracts as much as practical.
10. Do not invent backend APIs that do not already exist. If an expected endpoint is missing, document the gap and complete the rest.

Mandatory frontend layering pattern:

Layer 1 — API client layer
- raw HTTP only
- endpoint paths
- query param serialization if consistent with repo pattern
- typed request/response DTOs
- no UI model shaping
- no page logic

Layer 2 — Cases service layer
- the only approved data-access entry point for Cases UI
- exposes business-facing methods such as:
  - getCases(query)
  - getCase(caseId)
  - getCaseSummary() if supported
  - getCaseFiltersMetadata() if supported
  - updateCaseStatus(...) only if backend already supports it
  - other case actions only if already supported
- isolates backend contract peculiarities from components

Layer 3 — Mapper / adapter layer
- converts backend DTOs to UI-facing typed models
- normalizes null/optional fields
- maps statuses/enums/display strings where appropriate
- no JSX/template mapping duplication

Layer 4 — UI state layer
- only UI state, such as:
  - search text
  - filters
  - current page
  - sort state
  - selected case id
  - drawer open state
  - loading/error flags
- must not act as the persistent source of truth for backend data

Layer 5 — Presentation layer
- receives typed UI models
- renders only
- no direct HTTP
- no ad hoc response parsing
