STEP 5 — EXTEND ENUMS / RULE DEFINITIONS
--------------------------------------------------

Add or extend rules for operational workflow.

Required rules:

Queue rules:
- every conversation may have zero or one current queue assignment in this block
- new inbound external email may route to default queue if no queue exists and that is the chosen design
- inactive queue cannot be assigned
- tenant isolation must apply

Assignment rules:
- assignment is to a single user at a time in this block
- assigned user must belong to the same tenant context as the conversation, if that validation is currently possible within service boundaries
- reassignment preserves audit trail
- unassigned conversations can still exist in queues
- only authorized internal users can assign/reassign

Operational status rules:
Choose a clean operational status model, for example:
- OPEN
- PENDING_INTERNAL
- PENDING_EXTERNAL
- RESOLVED
- CLOSED

You may reuse existing conversation status if that is clean enough, or introduce a separate operational status only if necessary.
Document the choice clearly.

SLA rules:
- priority determines due times
- first response SLA should start when a conversation enters an active inbound/open state
- resolution SLA should start when a conversation is created or when first actionable inbound message arrives, depending on chosen design
- internal notes should not count as external response
- outbound SHARED_EXTERNAL response may satisfy first response SLA
- conversation resolution may satisfy resolution SLA
- breached flags must update deterministically
- exact SLA durations can be config constants for now if no shared config service exists

Recommended default SLA durations:
- LOW: first response 24h, resolution 120h
- NORMAL: first response 8h, resolution 72h
- HIGH: first response 4h, resolution 24h
- URGENT: first response 1h, resolution 8h

Use config/constants if possible rather than hardcoding in many places.
