# STORY ID
**LS-REPORTS-00-003 — Audit Integration**

---

# OBJECTIVE
Integrate the Reports Service with the **v2 shared Audit service** so report-related actions are recorded through the platform’s real audit pathway instead of mock/local-only hooks.

This story must:
- replace mock-only audit behavior with a real shared-service integration path
- standardize audit event payloads
- propagate tenant/user/correlation context consistently
- keep audit calls non-blocking so primary report operations are not interrupted

---

# REQUIRED IMPLEMENTATION

## A. Review Current Audit Usage
First inspect the current Reports Service audit touchpoints and identify all existing audit hook calls.

At minimum, audit coverage should include:
- template created
- template updated
- template version created
- template version published
- template assignment created
- template assignment updated
- tenant override created
- tenant override updated
- tenant override deactivated
- report execution started
- report execution completed
- report execution failed

Document in the report:
- where audit calls existed before
- what changed in this story

---

## B. Standard Audit Event Model
Create a typed, reusable audit payload model.

### Recommended DTO
- `AuditEventDto`

### Suggested fields
- `EventType`
- `OccurredAtUtc`
- `TenantId`
- `ProductCode`
- `EntityType`
- `EntityId`
- `ActorUserId`
- `CorrelationId`
- `RequestId`
- `Outcome`
- `Metadata` or `MetadataJson`

You may also introduce supporting DTOs or typed metadata helpers if useful.

### Rules
- keep the model platform-agnostic
- do not leak controller-specific or EF-specific details into the audit DTO
- centralize the payload shape instead of building inconsistent anonymous objects in services

---

## C. Shared Audit Service Adapter / Client
Implement a real shared Audit service integration path.

### Preferred structure
- refine `IAuditAdapter` if needed
- add a real implementation such as `SharedAuditAdapter`
- preserve `MockAuditAdapter` as fallback

### Real adapter behavior
- send typed audit events to the shared Audit service
- use a typed HTTP client or the platform-standard transport pattern if already established elsewhere
- keep transport implementation isolated in Infrastructure

### Important
Do NOT hardwire business services to HTTP calls directly.

---

## D. Configuration
Add configuration needed for the real audit integration.

### Suggested config
- `AuditService:Enabled`
- `AuditService:BaseUrl`
- `AuditService:TimeoutSeconds`
- `AuditService:EndpointPath`
- auth placeholder if needed for future
- any mock/real mode toggle if clean

Document:
- default behavior when config is missing
- how mock fallback works

---

## E. Audit Event Builder / Mapping
Centralize audit event construction so service code stays consistent.

### Acceptable approaches
- `AuditEventFactory`
- `AuditEventBuilder`
- reusable helper methods inside a focused mapping component

### Requirement
Business services should not all build raw event payloads differently.
There must be a shared mapping approach.

---

## F. Context Propagation
Ensure audit events consistently include:
- tenant context when applicable
- actor/user context
- request/correlation context
- product context where applicable

Use the existing request/context models already introduced in prior stories.

If some context is unavailable in a given flow, document the fallback behavior.

---

## G. Non-Blocking Dispatch
Audit failures must not fail the primary operation.

### Required behavior
- audit calls wrapped safely
- warning/error logged locally on failure
- primary action still succeeds

Do NOT introduce hard failures for audit dispatch in this story.

---

## H. DI / Mode Selection
Update DI so:
- real shared audit adapter is used when configured/enabled
- mock audit adapter is used when not configured
- readiness can distinguish the state

---

## I. Readiness Behavior
Update readiness behavior for audit adapter state.

`/api/v1/ready` should reflect:
- `ok` when real audit integration is configured and healthy
- `mock` when fallback mode is used
- `fail` only when real audit mode is expected but unavailable

If current readiness probes need refinement, do it carefully and document the result.

---

## J. Validation and Error Handling
Validate at minimum:
- required audit event fields populated
- event type non-empty
- correlation/request identifiers passed when available
- real adapter handles service failure gracefully
- mock fallback works unchanged

---

## K. Preserve Existing APIs
Existing APIs must continue to work with audit integration active or in mock mode.
