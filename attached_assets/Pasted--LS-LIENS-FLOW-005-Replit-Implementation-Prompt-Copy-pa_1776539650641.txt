==================================================
OBJECTIVE
==================================================

Enhance the existing Synq Liens workflow configuration by adding a Workflow Transition Engine.

The system currently supports:
- workflow definition
- workflow stages
- task templates
- task automation
- stage-aware behavior

It does NOT yet support controlled stage-to-stage transitions.

Implement:
1. Workflow transition definitions
2. Tenant/admin configuration of allowed transitions
3. Runtime transition validation
4. UI changes so stage moves are governed by allowed paths
5. Audit logging for transition configuration changes

This feature must upgrade the workflow from:
- stage list only
to
- stage list + controlled movement

This is NOT:
- a BPM engine
- SLA enforcement
- role-based stage gating
- approval routing
- dependency orchestration

==================================================
MANDATORY ARCHITECTURE RULES
==================================================

Use v2 shared services and existing Synq Liens architecture patterns.

Use:
- existing workflow config governance model
- existing audit service integration
- existing repository / DTO / service / endpoint patterns

Do NOT:
- introduce a heavy rules engine
- merge transitions into a generic config blob
- redesign the workflow domain entirely
- add approval logic
- add required-task gating
- add SLA/timer conditions
- add role-based transition restrictions yet

Keep this feature focused on:
- allowed transitions only
- clear runtime validation
- understandable admin configuration

==================================================
STEP 1 — ANALYZE THE EXISTING CODEBASE
==================================================

Before implementing:

Inspect and document:
- existing FLOW-001 workflow config entity/service/API
- existing workflow stage entity and ordering model
- tenant-side workflow settings page
- control-center workflow settings page
- any runtime stage movement logic currently used by tasks/cases/liens
- any frontend stage selection components currently allowing unrestricted moves
- existing versioning / lastUpdated / updateSource workflow governance pattern
- existing audit integration for workflow updates
- existing case/lien/task workflow stage usage

Document findings in the report before coding.

Clarify in the report:
- where transitions need to be enforced first
- whether task stage movement, case stage movement, and lien stage movement all currently exist
- what runtime paths can actually be validated in this phase

==================================================
STEP 2 — IMPLEMENT WORKFLOW TRANSITION DOMAIN
==================================================

Create a new workflow transition model tied to workflow configuration.

--------------------------------------------------
2.1 Entity / Model
--------------------------------------------------

Create a workflow transition entity with at minimum:

- id
- workflowId
- fromStageId
- toStageId
- isActive
- sortOrder
- createdAt
- updatedAt

Use existing naming conventions and entity patterns.

Recommended name:
- LienWorkflowTransition
or equivalent consistent with current workflow naming

--------------------------------------------------
2.2 Rules
--------------------------------------------------

A transition means:
- from stage A
- to stage B
- movement is allowed if transition is active

Phase 1 rules:
- backward moves are allowed only if explicitly configured
- no self-transition unless codebase requires it and it is justified
- from/to stages must belong to the same workflow
- duplicate identical transitions must not be allowed

--------------------------------------------------
2.3 EF / Persistence
--------------------------------------------------

Implement:
- entity configuration
- foreign keys to workflow config / workflow stages
- indexes for efficient lookup

Recommended indexes:
- (workflowId, fromStageId)
- uniqueness protection or service-level guard for (workflowId, fromStageId, toStageId)

--------------------------------------------------
2.4 Default Initialization Strategy
--------------------------------------------------

For existing workflows, implement a safe default.

Recommended behavior:
- when a workflow has stages but no transitions yet, auto-generate linear transitions based on stage order

Example:
1 -> 2
2 -> 3
3 -> 4
4 -> 5

Do this in the cleanest way supported by the codebase:
- migration seed if appropriate
OR
- service-level initialization on first workflow load/save

Document clearly which strategy you choose.

==================================================
STEP 3 — EXTEND WORKFLOW CONFIGURATION SERVICE
==================================================

Extend the existing workflow configuration domain/service to include transitions.

--------------------------------------------------
3.1 Service Behavior
--------------------------------------------------

Tenant/admin must be able to:
- list transitions
- create transitions
- update transitions if needed
- remove or deactivate transitions
- save workflow config including transitions

Recommended approach:
- treat transitions as part of workflow configuration editing
- preserve existing versioning / lastUpdated / updateSource behavior

--------------------------------------------------
3.2 Governance
--------------------------------------------------

Transition changes must:
- update workflow version
- update lastUpdatedAt
- update lastUpdatedBy
- update lastUpdatedSource

The same underlying workflow record/configuration must be editable from:
- Synq Liens Product Settings
- Control Center

Do not create a separate duplicated transition store for admin vs tenant surfaces.

==================================================
STEP 4 — IMPLEMENT RUNTIME TRANSITION VALIDATION
==================================================

Introduce a reusable validation layer for workflow transitions.

--------------------------------------------------
4.1 Validation Service
--------------------------------------------------

Create a service/component that can answer:

- IsTransitionAllowed(workflowId, fromStageId, toStageId)
- GetAllowedNextStages(workflowId, fromStageId)

This must be reusable by:
- task stage movement
- case stage movement
- lien stage movement
- future automation logic

--------------------------------------------------
4.2 Runtime Enforcement
--------------------------------------------------

Wherever stage changes currently happen in the codebase, add validation.

At minimum, enforce in the runtime paths that already exist and are active in Synq Liens.

Examples may include:
- task stage/status updates if workflow stage is used there
- case workflow advancement
- lien workflow advancement

If some stage runtime paths do not yet exist in a clean way, enforce where possible now and document the boundary clearly.

--------------------------------------------------
4.3 Error Behavior
--------------------------------------------------

If transition is invalid:
- reject request
- return clear validation error

Use messages like:
- "Transition from In Progress to To Do is not allowed"
- "This workflow stage change is not permitted"

Do not use vague error messages.

==================================================
STEP 5 — API CHANGES
==================================================

Implement the necessary API changes to manage transitions and support runtime validation.

--------------------------------------------------
5.1 Workflow Config APIs
--------------------------------------------------

Extend existing workflow config endpoints or add clearly related transition endpoints.

Support at minimum:
- get workflow with stages + transitions
- add transition
- remove/deactivate transition
- save workflow transition configuration

Prefer integrating transitions into the existing workflow config API surface unless the codebase structure clearly favors dedicated transition routes.

--------------------------------------------------
5.2 Runtime APIs
--------------------------------------------------

If frontend needs allowed next stages for a runtime move UI, add a helper endpoint only if truly needed.

Examples:
- GET allowed next stages from current stage
- validate transition endpoint

Prefer not to add unnecessary endpoints if existing workflow config payload is sufficient.

==================================================
STEP 6 — FRONTEND WORKFLOW SETTINGS UI
==================================================

Enhance the existing workflow settings page.

Current problem:
- tenant can define stages but not transition behavior

Implement a new section for transition configuration.

--------------------------------------------------
6.1 Recommended UX
--------------------------------------------------

Use a simple per-stage allowed-destinations model.

For each stage show:
- stage name
- allowed next stages
- multi-select or checkbox list of destination stages

Recommended over matrix because:
- easier for tenant admins
- easier to implement cleanly
- less visually dense

--------------------------------------------------
6.2 Tenant Side
--------------------------------------------------

Update:
- Synq Liens → Product Settings → Workflow

Add:
- transition configuration section
- save changes support
- version / updated metadata display preserved

--------------------------------------------------
6.3 Control Center Side
--------------------------------------------------

Mirror the same capability in:
- Control Center → Tenant → Synq Liens → Workflow

Use the same underlying workflow config APIs/data.

--------------------------------------------------
6.4 UX Constraints
--------------------------------------------------

Keep UI understandable.
Do not build a drag-and-drop graph designer.
Do not build a BPM diagramming tool.

==================================================
STEP 7 — FRONTEND RUNTIME MOVEMENT UX
==================================================

Where users can currently move between stages/statuses, improve the UI to reflect allowed moves.

--------------------------------------------------
7.1 Preferred Behavior
--------------------------------------------------

If practical:
- only show valid target stages in dropdown/select controls

If existing UI structure makes that too invasive:
- keep current UI but allow backend to block invalid transitions
- document the current UX limitation

--------------------------------------------------
7.2 Error Handling
--------------------------------------------------

If invalid move is attempted:
- show clear error toast/message
- do not silently fail
