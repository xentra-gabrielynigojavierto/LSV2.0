==================================================
EXISTING CONTEXT
==================================================

LSCC-005 already implemented:

- referral creation via provider card / search flow
- referral save in Pending-equivalent state (`New` canonical)
- provider notification email with secure HMAC token link
- pending-provider public accept page
- active-tenant provider deep link through login + returnTo
- acceptance confirmation emails
- 30-day referral activity dashboard cards
- notification records already being created for email events
- token validation already exists
- duplicate acceptance is functionally prevented by status checks

Current gaps:
- email failures are logged but not visible/actionable in product UI
- no resend capability
- tokens cannot be revoked once sent
- repeated/invalid token use is not surfaced as an audit signal
- invalid-token UX is basic

Treat LSCC-005 as the baseline and build on it.

==================================================
REQUIREMENTS
==================================================

--------------------------------------------------
A. EMAIL DELIVERY TRACKING
--------------------------------------------------

Track email delivery state for referral-related emails.

Use existing Notification-related structures if they already support this well enough.
If not, extend them minimally.

Minimum per-email tracking fields/behavior needed:
- notification type
- delivery status
- attempt count
- last attempt timestamp
- error message or failure summary when applicable

Delivery status values should support:
- Pending
- Sent
- Failed

Behavior:
1. when an email-triggering action occurs, create/update the notification record in Pending
2. attempt the send
3. on success, mark Sent
4. on failure, mark Failed and store useful failure context

Email-triggering actions covered in this feature:
- new referral notification to provider
- referral accepted confirmation to provider
- referral accepted confirmation to referrer

Rules:
- referral creation or acceptance must still succeed even if email fails
- email failure must never be silent
- avoid broad notification-system redesign

--------------------------------------------------
B. RESEND EMAIL CAPABILITY
--------------------------------------------------

Add a minimal resend flow for referral-related emails.

Preferred scope:
- resend the provider notification email for a referral from the authenticated referral-detail experience
- optionally expose for admin if easy within current architecture

Minimum endpoint shape (adapt to repo conventions if needed):
- POST /api/referrals/{id}/resend-email

Rules:
- authenticated and role-restricted (referrer/admin by existing conventions)
- resend must use the correct current template + secure token strategy
- resend attempts must update delivery tracking
- repeated rapid clicks should not create unsafe spam behavior
  - simple in-flight button disable or basic debounce is sufficient
- if the referral is already accepted, either:
  1. block resend of the original provider notification and explain why, OR
  2. clearly label and send the appropriate acceptance-related email
Choose the minimal consistent behavior and document it.

Fallback behavior:
- failed resend must surface visible error to the user
- successful resend must surface visible success

--------------------------------------------------
C. TOKEN REVOCATION / INVALIDATION
--------------------------------------------------

Implement a deterministic way to invalidate previously sent referral view tokens.

Preferred approach:
- token versioning on the referral (or equivalent minimal per-referral invalidation mechanism)

Example intent only:
- Referral.TokenVersion
- token format includes version
- validation rejects mismatched version

Rules:
- revoking tokens must invalidate all previously issued view tokens for that referral
- newly generated links after revocation must work
- do NOT rely solely on expiry for hardening
- do NOT expose raw token version semantics in user-facing UI

Minimum endpoint shape (adapt if needed):
- POST /api/referrals/{id}/revoke-token

Access:
- authenticated and role-restricted (referrer/admin by current conventions)

Fallback behavior:
- revoked/expired/invalid token routes to the invalid-token page without leaking existence

--------------------------------------------------
D. DUPLICATE / REPLAY HARDENING
--------------------------------------------------

Existing LSCC-005 behavior already prevents a second acceptance if status is no longer Pending/New.
Strengthen this behavior operationally.

Requirements:
1. repeated accept attempts using the same valid token after acceptance must fail safely
2. repeated invalid token access attempts should be recorded as audit events if feasible
3. repeated valid token view attempts before acceptance are acceptable, but must remain read-only until the accept action is invoked
4. do not double-send acceptance confirmation emails

Use the minimal approach that works with the current design.

--------------------------------------------------
E. REFERRAL AUDIT EVENT VISIBILITY
--------------------------------------------------

Create a lightweight audit trail for key referral flow events.

Prefer reusing existing event/notification/history structures if appropriate.
If a new lightweight audit record is necessary, keep it minimal.

Minimum events to capture if feasible:
- ReferralCreated
- ProviderNotificationEmailSent
- ProviderNotificationEmailFailed
- ReferralViewedViaToken (optional but desirable)
- ReferralAccepted
- TokenInvalid
- TokenRevoked
- ReferralEmailResent

Rules:
- do not overbuild a general audit platform
- if some events are easier to record via the existing notification model + referral history, use that and document the mapping

UI requirement:
- show enough audit visibility on referral detail to make email state and hardening actions understandable
- this does NOT need to become a full timeline redesign

--------------------------------------------------
F. MINIMAL UI STATUS INDICATORS + ACTIONS
--------------------------------------------------

Add minimal UI to the existing referral detail experience for authenticated users.

At minimum include:
1. email delivery status indicator
   Examples:
   - Email: Sent
   - Email: Failed
   - Email: Pending

2. resend email action when appropriate

3. revoke referral link action when appropriate

4. clear success/error feedback via existing toast patterns if available

Rules:
- keep UI minimal and operational
- do not clutter the page
- do not redesign the referral detail screen

Recommended placement:
- near existing referral actions or in a compact "Referral Delivery" / "Referral Access" card

--------------------------------------------------
G. INVALID TOKEN PAGE IMPROVEMENT
--------------------------------------------------

Improve the invalid token experience at the public accept flow.

Current behavior is basic.
It should clearly communicate:
- the referral link is invalid, expired, or no longer active
- the user may need a new link from the referring law firm or provider contact path

Requirements:
- safe wording
- no confirmation that a specific referral exists
- usable UI state, not a raw dead end

Optional:
- include a short generic support/contact instruction if the current app has an appropriate contact pattern

--------------------------------------------------
H. DASHBOARD / ANALYTICS IMPACT
--------------------------------------------------

No analytics expansion is required in this feature.

However:
- do not break LSCC-005 basic referral activity cards
- if email status affects dashboard queries or data shape, maintain backward compatibility

--------------------------------------------------
I. TESTING (MINIMUM)
--------------------------------------------------

Add focused tests for:

1. email notification status transitions
   - Pending → Sent
   - Pending → Failed

2. resend flow
   - resend triggers a new attempt
   - tracking updates correctly
   - duplicate click handling is safe enough for the chosen implementation

3. token revocation
   - old token invalid after revoke
   - newly issued token valid

4. duplicate/replay hardening
   - second acceptance does not change state again
   - acceptance emails are not re-sent on replay

5. invalid token handling
   - malformed/expired/revoked token rejected safely

6. audit/event recording
   - key events are written or mapped correctly

7. UI/API integration for status display and resend/revoke actions
   - if full UI tests are impractical, test service/API integration layers and document the limitation

Pre-existing unrelated test failures should not be counted as regressions, but document them if still present.
