# LS-FLOW-MERGE-P3 — Product Consumption & Workflow Mapping

## Objective

Convert Flow from a platform-integrated standalone service into a **product-consumable workflow capability** for LegalSynq products by wiring application-level event emission, introducing product-aware authorization and capability enforcement, enabling workflow templates/mappings for target products, and cleaning up legacy tenant/data artifacts without breaking Flow’s service boundary.

Phase 3 is where Flow becomes operationally useful to product teams while still remaining a shared platform service.

---

## Scope

Phase 3 covers:

* application-service event emission inside Flow CRUD/action paths
* product-aware authorization and capability enforcement
* product workflow template/mapping foundation
* initial integration targets for LegalSynq products
* cleanup of legacy `"default"` tenant artifacts
* tenant normalization/data hygiene where needed
* product-facing API usage model for Flow
* minimal frontend/product integration needed to validate real consumption
* documentation updates for product-consumption patterns

Phase 3 does **not** cover:

* DB consolidation
* direct DB access from other services into Flow
* full enterprise event bus
* broad UI redesign
* full embedded UX across every product
* major workflow engine redesign
* deep monorepo/frontend workspace consolidation unless trivial
* advanced analytics/reporting layer
* generalized BPM suite expansion beyond current LegalSynq needs

# Requirements

## 1. Wire Real Event Emission in Application Services

Phase 2 established the event abstraction and adapter seams. Phase 3 must now emit real events from actual workflow/task operations.

### Requirements

* inject and call `IFlowEventDispatcher` from the relevant application/service-layer methods
* emit events from real action paths such as:

  * workflow created
  * workflow updated
  * workflow state changed
  * task assigned
  * task updated
  * task completed
  * workflow completed
* ensure emitted events contain:

  * workflow/task identifiers
  * tenant id
  * user id where available
  * relevant business context needed by downstream audit/notifications
* preserve transactional safety as much as practical
* ensure event failures do not corrupt core workflow persistence behavior unless the existing pattern explicitly requires hard failure

### Notes

* do not build a distributed event bus
* do not over-generalize event taxonomy beyond current Flow needs

---

## 2. Activate Audit and Notifications Through Real Flow Events

Audit and Notifications seams are already present. They now need meaningful runtime usage.

### Requirements

* connect emitted Flow events to Audit adapter delivery
* connect appropriate emitted Flow events to Notifications adapter delivery
* confirm at least these trigger paths work:

  * task assignment → notification
  * workflow completion → notification
  * workflow/task creation/update/completion → audit
* retain safe fallback behavior if downstream services are unavailable
* document which actions emit which downstream effects

### Important

* keep Notifications as system notification behavior only
* do not build threaded/two-way communications here

---

## 3. Product Capability Model for Flow Consumption

Flow must now be consumable by products without turning into product-specific code soup.

### Requirements

* introduce a product/capability-aware authorization model for Flow access
* align with LegalSynq product/capability conventions already present in the platform where feasible
* define how Flow access is granted for product contexts such as:

  * SynqLien
  * CareConnect
  * SynqFund
* distinguish authentication from authorization:

  * authenticated user is not automatically allowed to perform all workflow operations
* support platform roles and tenant roles where appropriate
* use claims/policies/capability checks rather than hardcoded product hacks in controller actions

### Examples

* user may be authenticated but only allowed to access Flow for products enabled to their tenant/org
* user may be allowed to view workflow instances but not create templates
* platform admin may have broader access than tenant-scoped users

### Non-goal

* do not redesign platform-wide authorization architecture

---

## 4. Product-to-Flow Mapping Foundation

Flow now needs a structured way for products to consume it.

### Requirements

* define and implement a mapping model that allows a product to:

  * register or reference workflow templates
  * instantiate workflows in response to product business operations
  * correlate Flow workflow instances with product records
* establish a consistent pattern for correlation keys and references
* avoid direct DB joins across services
* use API-first patterns and explicit references

### Minimum target mapping model

For each product integration, support:

* product key
* tenant id
* source entity type
* source entity id
* workflow template id or workflow type
* workflow instance id
* status/correlation metadata

### Notes

* this can be stored in Flow, in product service, or split via explicit API contracts depending on safest architecture
* whichever choice is made, document ownership clearly

---

## 5. Initial Product Integrations

Phase 3 should prove real consumption with initial product-aligned implementations.

### Required initial targets

Implement the maximum safe subset for these:

#### A. SynqLien

Support workflow consumption for one or more lien/case lifecycle scenarios, such as:

* case onboarding workflow
* lien intake/review workflow
* settlement/resolution workflow

#### B. CareConnect

Support workflow consumption for one or more referral/care coordination scenarios, such as:

* referral intake workflow
* provider follow-up workflow
* treatment coordination workflow

#### C. SynqFund

Support workflow consumption for one or more funding process scenarios, such as:

* funding intake workflow
* review/approval workflow
* payout readiness workflow

### Rules

* do not try to fully solve every product process in this phase
* choose the smallest real, useful subset that proves the architecture
* keep mappings/configuration extensible

---

## 6. Product-Facing API and Invocation Pattern

Products need a safe and consistent way to call Flow.

### Requirements

* define and implement product-facing Flow API usage patterns for:

  * create workflow from product context
  * query workflow(s) by correlated product entity
  * progress workflow/task under authorized user context
* ensure APIs enforce tenant and authorization rules
* avoid exposing raw internal-only operations unnecessarily
* add/adjust API endpoints only where required for product consumption

### Important

* preserve existing Flow API surface where possible
* do not perform broad contract rewrite unless required for product-consumption viability

---

## 7. Legacy Tenant/Data Cleanup

Phase 2 made `"default"` tenant artifacts unreachable. Phase 3 should now clean them up safely.

### Requirements

* inspect existing Flow data for legacy `"default"` tenant rows introduced before strict tenant enforcement
* implement safe cleanup and/or migration strategy
* normalize tenant identifiers if needed to align with platform conventions
* document:

  * what data was found
  * what was migrated
  * what was removed
  * what remains deferred

### Rules

* no destructive cleanup without clear logic
* prefer idempotent migration scripts or controlled update routines
* do not break valid tenant-scoped data

---

## 8. Product-Aware Frontend / UI Validation

Phase 3 should validate that Flow can be consumed meaningfully, but without broad UI redesign.

### Requirements

* implement the minimum safe UI changes needed to validate product consumption
* this may include:

  * launching/viewing workflows from product context
  * displaying correlated workflow status for a product record
  * progressing tasks from a product-aligned screen or Flow UI
* keep UI changes narrow and validation-oriented
* do not launch a broad redesign of product portals

### Notes

* standalone Flow frontend may remain the primary UI for now if product embedding is not yet safe
* however, Phase 3 should demonstrate real product-linked usage, not only isolated Flow screens

---

## 9. Documentation Updates

Flow docs must now describe product consumption patterns.

### Requirements

Update docs under `/apps/services/flow/docs` to cover:

* application-level event emission model
* audit/notification trigger behavior
* product authorization model
* product-to-flow mapping model
* initial integrated product scenarios
* API usage patterns for products
* legacy tenant cleanup decisions
* explicit deferrals for later phases

Suggested docs:

* `README.md`
* `architecture.md`
* `merge-phase-3-notes.md`
* optional product integration note if helpful

---

## 10. Preserve Service Boundary

Even with product consumption enabled, Flow must remain a bounded service.

### Requirements

* keep independent DB ownership
* keep API-first integration
* do not let product services read/write Flow tables directly
* do not move workflow business logic into product services except for product-side orchestration
* keep product integrations explicit and understandable

This remains mandatory.

---

## 11. Explicit Deferrals

Document clearly that the following remain out of scope for Phase 3:

* full event bus/platform streaming
* complete end-to-end workflow coverage for every LegalSynq product
* deep workflow analytics/reporting
* major UI redesign across products
* full frontend workspace unification
* DB consolidation across platform services
* advanced BPM/rules-engine expansion beyond current Flow scope
* generalized external partner integrations unless directly required by the chosen product-consumption subset

# Success Condition

Phase 3 is successful when:

* Flow emits real runtime events from application paths
* audit and notifications are meaningfully triggered
* Flow authorization becomes product/capability-aware
* at least a minimal real workflow consumption path is implemented for target LegalSynq product scenarios
* product-to-flow correlation is explicit and safe
* legacy tenant artifacts are cleaned up or safely migrated
* Flow remains independently bounded and ready for broader product adoption

---

# Failure Condition

Phase 3 fails if:

* product integrations create direct DB coupling
* authorization remains too generic for real product usage
* events exist only as abstractions and are not emitted from real runtime paths
* the work drifts into large UI redesign or overbroad process implementation
* tenant/data cleanup is unsafe or incomplete without documentation
* Flow becomes entangled into product-specific code that breaks its shared-service role
