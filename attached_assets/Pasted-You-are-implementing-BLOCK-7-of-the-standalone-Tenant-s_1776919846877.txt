A) Make Tenant the primary runtime read source under configuration

Update the read-source behavior so the system can safely run in:
- Identity
- HybridFallback
- Tenant

But in this block, Tenant should become the intended primary mode for validated environments.

Requirements:

- ensure both branding and tenant resolution can run Tenant-first
- preserve rollback path back to Identity

Recommended behavior:
- `Identity` = fully legacy path
- `HybridFallback` = Tenant first, Identity fallback
- `Tenant` = Tenant-only with safe platform-default behavior if tenant not found
- if you introduce separate branding/resolution source flags, keep them coherent and document them clearly

The goal is that after Block 7, operators can switch production to:
- `HybridFallback` first for validation
- then `Tenant` when ready
without code changes

B) Finalize gateway/backend resolution transition

Update the gateway/backend resolution consumption path so runtime tenant resolution can actually use Tenant service as the primary source.

Requirements:
- centralize runtime resolution source selection
- if current gateway still relies on Identity-specific resolution logic, refactor to go through the configured source
- in Tenant or HybridFallback modes:
  - call Tenant public/internal resolution endpoints
  - use Identity only as fallback in HybridFallback
- in Identity mode:
  - preserve current behavior

Rules:
- no duplicated business logic if Tenant service already exposes the needed contract
- maintain backwards-compatible routing behavior
- add explicit logging for:
  - source used
  - fallback triggered
  - unresolved host/code outcomes

Expected resolution paths to support:
- by host
- by subdomain
- by code

C) Finalize web/public branding transition

Complete the branding/bootstrap read transition in the web app.

Requirements:
- keep `/api/tenant-branding` as the central BFF route
- make it production-ready for Tenant-first use
- if necessary, add explicit timeout handling for Tenant service reads
- preserve fallback to Identity in HybridFallback
- keep client-side components source-agnostic

In this block also address remaining branding consistency gaps:
- logo public route behavior (`/api/branding/logo/public` or equivalent current route)
- favicon/public asset lookup if currently still bound to Identity assumptions
- ensure branding metadata and logo lookup paths are aligned to the same source strategy where practical

Rules:
- do not redesign frontend UX
- do not scatter migration logic into components
- keep source selection centralized in server routes/providers

D) Implement real dual-write hook points in Identity (feature-flagged, disabled by default)

This is the biggest new behavior in Block 7.

Use the scaffolding from Block 5 and implement the actual hook points from Identity tenant write operations into Tenant service.

Requirements:
- identify the authoritative tenant create/update code paths in Identity
- call a real `ITenantSyncAdapter` implementation from those paths when enabled
- dual-write must be:
  - explicitly feature-flagged
  - contained in try/catch
  - non-breaking to Identity runtime if Tenant sync fails (unless a strict mode is explicitly enabled)
- default behavior must remain OFF

Recommended config flags:
- Features:TenantDualWriteEnabled = false (default)
- optional:
  - Features:TenantDualWriteStrictMode = false
  - if false: log and continue on sync failure
  - if true: fail the originating operation (only for controlled environments)

Sync scope for this block:
- tenant core/profile updates
- branding basics that Identity still owns/changes
- subdomain compatibility fields
- do not overreach into unsupported areas unless clearly safe

Recommended implementation options:
- real `TenantSyncAdapter` in Identity or shared integration layer calling Tenant service internal/admin endpoints
- or internal secure sync endpoint in Tenant service if cleaner

Choose the safest approach that best fits the current codebase.

Rules:
- no hidden always-on coupling
- retries optional; simple reliable logging acceptable for this block
- if you add an internal sync endpoint, protect it and keep it non-public
