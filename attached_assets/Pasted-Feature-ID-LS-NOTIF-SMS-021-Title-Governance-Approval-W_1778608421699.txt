Feature ID: LS-NOTIF-SMS-021

Title:
Governance Approval Workflow, Multi-Stage Change Control, and Governance Release Management

Objective:
Extend the SMS governance lifecycle platform with enterprise-grade approval workflows, staged governance promotion, governance release packages, scheduled rollout activation, approval chains, release auditability, and controlled governance deployment management.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- governance approval workflows
- governance release packages
- approval request persistence
- approval decision persistence
- release promotion state
- scheduled release activation
- release auditability
- release activation/rollback orchestration
- governance release APIs

Control Center must only consume Notification Service APIs and render/administer governance release UX. Do not duplicate approval, release-state transition, scheduling, or activation logic in Control Center.

Requirements:
- Implement governance approval and release management inside Notification Service.
- Preserve LS-NOTIF-SMS-017 governance policies.
- Preserve LS-NOTIF-SMS-018 template governance.
- Preserve LS-NOTIF-SMS-019 dynamic rules.
- Preserve LS-NOTIF-SMS-020 versioning, import/export, rollback, and analytics.
- Release management must extend, not replace, existing governance lifecycle behavior.
- Governance changes should support approval before activation when configured.
- Release activation must be explicit, auditable, and safe.
- Scheduled activation must not block the delivery pipeline.
- Failed release activation must degrade safely.
- Do not store raw phone numbers in release/approval/audit tables.
- Do not expose provider credentials, webhook URLs, SettingsJson, CredentialsJson, auth tokens, raw provider payloads, or recipient phone numbers.
- Keep Control Center as admin/visibility UX only.
- Preserve all LS-NOTIF-SMS-001 through LS-NOTIF-SMS-020 behavior.

Implementation Requirements:
1. Add governance release package persistence.
2. Add release item persistence for grouped governance changes.
3. Add approval request persistence.
4. Add approval decision persistence.
5. Add release audit event persistence.
6. Add release state machine.
7. Add multi-stage approval workflow support.
8. Add release submit-review, approve, reject, schedule, activate, and archive flows.
9. Add scheduled activation worker if safe.
10. Add integration with LS-NOTIF-SMS-020 versioning/rollback where appropriate.
11. Add admin APIs for release packages, approvals, scheduling, activation, and audit trail.
12. Add Control Center governance release management UI if feasible.
13. Ensure all failures degrade safely.
14. Ensure delivery-time governance evaluation remains fast and stable.

Critical Constraints:
- Reuse existing SmsGovernanceRulePack, SmsGovernanceRule, SmsComplianceProfile, SmsGovernanceRuleVersion, SmsGovernanceRulePackVersion, SmsGovernanceVersioningService, SmsGovernanceImportService, SmsGovernanceAnalyticsService, SmsGovernanceRuleEngine, SmsGovernanceRuleResolver, SmsGovernancePolicy, SmsTemplateGovernanceService, admin endpoint, EF/migration, audit/logging, worker, and Control Center governance UI patterns where appropriate.
- Do NOT rewrite the delivery pipeline.
- Do NOT rewrite the dynamic rule engine.
- Do NOT replace LS-017, LS-018, LS-019, or LS-020 behavior.
- Do NOT implement AI/ML approval recommendations.
- Do NOT call external ticketing, workflow, approval, analytics, compliance, moderation, telecom, or fraud APIs.
- Do NOT persist raw phone numbers in release/approval/audit tables.
- Do NOT expose credentials or secrets.
- Do NOT add conversational inbound SMS.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 10 — Implement Release Activation
Implement activation behavior.

Initial activation semantics:
- Mark release package as active.
- For release items referencing rule packs/rules/profiles:
  - ensure referenced entities are enabled/active if actionType = activate.
  - optionally disable/supersede prior active release packages in the same tenant/scope.
- Do not mutate unrelated governance entities unless explicitly in release items.
- Create audit events.
- Create snapshots via LS-NOTIF-SMS-020 versioning where activation changes rule/rule-pack state.

Requirements:
- Activation must be transactional where practical.
- Failed activation should mark release activation_failed and leave existing active governance intact.
- Activation must not impact delivery pipeline if failure occurs.
- Activation must not send SMS.
- Activation must not call external APIs.

STEP 11 — Add Scheduled Activation Worker
Add optional worker for scheduled releases.

Suggested worker:
SmsGovernanceReleaseActivationWorker

Behavior:
- Reads scheduled releases where ReleaseState = scheduled and ScheduledActivationAt <= now.
- Calls release service ActivateAsync.
- Uses existing BackgroundService pattern.
- Uses IServiceScopeFactory.
- Bounded batch size.
- Logs safely.
- Respects cancellation.
- Disabled by default.

Requirements:
- No delivery pipeline blocking.
- Failure marks release activation_failed or logs according to service behavior.
- No external API calls.
- No secrets in logs.
