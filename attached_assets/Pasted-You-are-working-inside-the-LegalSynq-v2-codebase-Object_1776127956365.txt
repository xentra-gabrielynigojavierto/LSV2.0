You are working inside the LegalSynq v2 codebase.

Objective:
Add a lightweight capability abstraction layer on top of the existing Liens permission model so future Liens workflows can reason in terms of business capabilities instead of raw permission strings.

This is a refinement of LS-LIENS-02-002. It does NOT replace the existing permission model. It layers on top of it.

NON-NEGOTIABLE ARCHITECTURE RULE:
All cross-cutting concerns must use v2 services only:
- Identity
- Documents
- Notifications
- Audit

For this feature specifically:
- Reuse the existing Identity + permission model already implemented
- Do NOT introduce v1 roles, permissions, or session logic
- Do NOT replace permission-based enforcement
- Do NOT hardcode provider behavior inside Liens domain entities

DATABASE / OWNERSHIP RULE:
- Do not introduce any new Liens-owned persistence tables for this feature
- Avoid Identity schema changes unless absolutely necessary
- If any persistence change is required, it must follow the owning service’s existing prefix convention

--------------------------------------------------

Inspect the current implementation from LS-LIENS-02-002, including:

- `LiensPermissions`

--------------------------------------------------
STEP 2 — DEFINE LIENS BUSINESS CAPABILITIES
--------------------------------------------------

Define stable, business-facing capability identifiers for Liens.

At minimum:

- `LIENS_SELL`
- `LIENS_MANAGE_INTERNAL`

These are NOT replacements for permission codes.
They are higher-level capability names derived from the permission model.

Add a clear constants class or equivalent, for example:
- `LiensCapabilities.Sell`
- `LiensCapabilities.ManageInternal`

Follow existing codebase style.

--------------------------------------------------
STEP 3 — CENTRALIZE CAPABILITY DERIVATION
--------------------------------------------------

Implement a single, centralized mapping from permissions → capabilities.

At minimum, determine capability membership using the existing Liens permission codes.

Example intent:

### Capability: LIENS_SELL
Should evaluate true when the current context has the permission pattern required for seller-side marketplace behavior, such as:
- create lien
- offer/list lien
- read own liens

### Capability: LIENS_MANAGE_INTERNAL
Should evaluate true when the current context has the permission pattern required for internal lien management behavior, such as:
- read held liens
- service liens
- settle liens

Use the current permission set as the source of truth.
Do NOT invent a new security model.

IMPORTANT:
- keep this logic centralized
- do not scatter permission combinations across endpoints
- document the exact mapping clearly

--------------------------------------------------
STEP 4 — EXPOSE A CLEAN CONSUMPTION API
--------------------------------------------------

Add a clean way for future Liens features to check capabilities.

Examples:
- `context.HasCapability(LiensCapabilities.Sell)`
- `context.HasCapability(LiensCapabilities.ManageInternal)`

Implement this using the lightest approach consistent with current platform patterns:
- extension methods
- helper class
- adapter around `ICurrentRequestContext`

Requirements:
- future code should not need to inspect raw permission strings
- the abstraction should be easy to read and hard to misuse

Do NOT overbuild a generic policy engine unless the current codebase already has that pattern.

--------------------------------------------------
STEP 5 — KEEP PERMISSIONS AS THE ENFORCEMENT LAYER
--------------------------------------------------

The existing permission-based authorization must remain intact.

Do NOT:
- remove `RequirePermission`
- replace endpoint guards with capability checks
- weaken the current auth design

Capabilities are for:
- business logic readability
- workflow branching
- future UI/navigation gating
- provider operating mode interpretation

Permissions remain the enforcement mechanism.
