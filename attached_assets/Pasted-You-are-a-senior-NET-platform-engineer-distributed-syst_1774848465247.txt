### 1. Redis circuit breaker / resilience

Add a resilience layer around critical Redis operations so Redis degradation does not create uncontrolled failure storms.

## Goal

Protect the service when Redis is slow, degraded, or unavailable.

## Requirements

Implement Redis resilience around critical operations used by:

* Redis-backed scan queue
* Redis-backed notification/event publishing
* any other Redis-backed state if present

Preferred approach:

* Polly circuit breaker and/or resilience wrapper
* keep implementation infrastructure-layer only

### Behavior requirements

* repeated Redis failures should open the circuit
* when circuit is open:

  * fail quickly
  * avoid hammering Redis
  * expose circuit state via logs and metrics
* preserve safe behavior:

  * queue writes should fail explicitly
  * worker dequeue/read logic should degrade safely
  * notifications should fail but not corrupt scan state

### Metrics

Add metrics such as:

* `docs_redis_circuit_state`
* `docs_redis_circuit_open_total`
* `docs_redis_circuit_short_circuit_total`

### Health

Include circuit-open state meaningfully in readiness / health detail

### Important

Do not hide failures. Fast, explicit degradation is preferred.

---

### 2. Durable scan completion event delivery

Current Redis Pub/Sub delivery is at-most-once and ephemeral.

Upgrade the event pipeline so scan completion delivery has a **durable option**.

## Goal

Provide a reliable foundation for downstream consumers (CareConnect, portals, internal automation).

## Requirements

Add a durable scan completion publisher.

Preferred approach:

* Redis Streams-based event publisher

Alternative acceptable:

* outbox/persistent event table if done cleanly

### If using Redis Streams

Implement:

* publish completion event via `XADD`
* configurable stream key, e.g.:

  * `documents:scan:completed`
* support replay by downstream consumers
* keep existing publisher abstraction
* preserve existing `log` and `none` providers
* you may keep Pub/Sub as an option, but add a durable provider and make it clear which is recommended for production

### Delivery requirements

* scan completion event must be emitted for terminal outcomes:

  * `CLEAN`
  * `INFECTED`
  * `FAILED`
* event publishing failure must not corrupt document state
* event delivery failure must be measurable and logged
* document state persistence remains primary

### Metrics

Add metrics such as:

* `docs_scan_completion_stream_publish_total`
* `docs_scan_completion_stream_publish_failures_total`

### Configuration

Example:

```json
{
  "Notifications": {
    "ScanCompletion": {
      "Provider": "redis-stream",
      "Redis": {
        "Channel": "documents.scan.completed",
        "StreamKey": "documents:scan:completed"
      }
    }
  }
}
```

Document clearly:

* Pub/Sub = best-effort
* Stream = durable/replayable
* recommended production default

Do NOT introduce high-cardinality labels in Prometheus metrics.
Correlation IDs should be:

* in logs
* in event payloads
* optionally in audit payloads

## Implementation Guidance

### Preserve architecture

Keep boundaries clean:

* Domain → event contract / interfaces only
* Application → orchestration only
* Infrastructure → Redis resilience, durable publisher, health/metrics
* API/host → health endpoints, wiring, worker

### Backward compatibility

* preserve existing upload/scan/access API behavior
* existing notification providers can remain
* add durable provider rather than replacing all providers unless clearly safe

### Security

* do not include sensitive file contents in events
* preserve tenant-aware identifiers only
* preserve fail-closed scan behavior

### Reliability principle

* document state persistence is the source of truth
* event publishing is secondary but should be durable when configured that way
