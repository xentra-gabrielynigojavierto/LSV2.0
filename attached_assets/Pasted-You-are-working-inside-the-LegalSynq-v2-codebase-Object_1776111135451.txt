STEP 1A — DONOR BEHAVIOR & API PRESERVATION (CRITICAL)
--------------------------------------------------

Before defining entities:

1. Review v1 SynqLiens concepts at a high level:
   - Case structure
   - Contact types and roles
   - Facility structure
   - Lookup/status usage

2. Identify:
   - key fields that drive behavior
   - fields exposed in APIs
   - fields that influence workflows (status, type, references)

3. Apply these rules:

### Behavior Preservation
- Preserve meaning and purpose of core fields
- Ensure entities can support the same workflows as v1
- Do NOT remove critical business fields

### API Compatibility Preservation
- Prefer field names that map cleanly to existing APIs
- Avoid renaming important fields unnecessarily
- Design entities so future APIs can match v1 with minimal transformation

### v2 Alignment
- Replace `programId` → `TenantId` / `OrgId`
- Normalize naming where needed
- Keep structure clean and consistent with v2

4. If you change/remove fields:
- You MUST document and justify it in the report

STEP 3 — MODEL CLEAN V2 CANONICAL ENTITIES
--------------------------------------------------

Design entities with:

### Case
Support:
- TenantId, OrgId
- CaseNumber / ReferenceNumber
- ExternalReference (preserve compatibility)
- Title / DisplayName
- Status (lookup-driven)
- OpenedDate / ClosedDate
- future linkage to contacts, facilities, liens, tasks, settlements

### Contact
Support:
- TenantId, OrgId
- ContactType (LawFirm, Provider, Funder, etc.)
- FirstName, LastName, DisplayName
- Email, Phone
- IsActive

### Facility
Support:
- TenantId, OrgId
- Name
- Code / ExternalReference
- Address (value object preferred)
- Phone, Email
- IsActive

### LookupValue
Support:
- Category (e.g., CaseStatus, LienStatus)
- Code
- Name
- Description
- SortOrder
- IsActive
- tenant/global scoping decision
