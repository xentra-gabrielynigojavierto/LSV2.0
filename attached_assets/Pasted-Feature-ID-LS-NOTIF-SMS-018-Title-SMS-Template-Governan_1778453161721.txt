Feature ID: LS-NOTIF-SMS-018

Title:
SMS Template Governance, Content Classification, and Delivery Compliance Enforcement

Objective:
Add enterprise-grade SMS content governance including approved template management, template versioning, tenant-level template governance, content classification, prohibited-content enforcement, variable validation, and delivery compliance controls to ensure outbound SMS payloads are policy-safe before delivery execution.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- SMS template governance
- template approval lifecycle
- template versioning
- content classification
- prohibited-content enforcement
- variable validation
- compliance-safe delivery evaluation
- template auditability
- template governance APIs

Control Center must only consume Notification Service APIs and render/administer template governance visibility. Do not duplicate validation, classification, governance, or enforcement logic in Control Center.

Requirements:
- Implement template governance inside Notification Service.
- Governance evaluation must occur BEFORE provider execution.
- Preserve existing routing/orchestration/governance/suppression behavior.
- Preserve all existing compliance/preference enforcement.
- Preserve all existing retry/dead-letter semantics.
- Preserve all LS-NOTIF-SMS-001 through LS-NOTIF-SMS-017 behavior.
- Do not expose provider credentials, webhook URLs, SettingsJson, CredentialsJson, auth tokens, raw provider payloads, or sensitive recipient metadata.
- Do not persist raw phone numbers in template governance tables.
- Keep Control Center as admin/visibility UX only.

Implementation Requirements:
1. Add SMS template registry persistence.
2. Add template versioning.
3. Add template approval lifecycle.
4. Add content classification rules.
5. Add prohibited-content enforcement.
6. Add variable/token validation.
7. Add delivery compliance evaluation.
8. Add template audit history.
9. Add governance decision persistence for content violations.
10. Add admin APIs for template governance.
11. Add template analytics/audit APIs.
12. Add Control Center template governance UI if feasible.
13. Ensure compliance evaluation degrades safely if classification fails.

Critical Constraints:
- Reuse existing Notification, NotificationAttempt, governance pipeline, recipient intelligence, routing engine, audit/logging, admin endpoint, migration, and Control Center patterns where appropriate.
- Do NOT rewrite the Notification Service delivery pipeline.
- Do NOT bypass existing governance or recipient suppression logic.
- Do NOT implement AI/LLM content moderation.
- Do NOT call external compliance/moderation APIs.
- Do NOT persist raw phone numbers in governance tables.
- Do NOT expose credentials or secrets.
- Do NOT add conversational inbound SMS.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.
