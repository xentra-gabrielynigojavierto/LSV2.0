SCOPE
================================================================

A) Introduce a configurable read-source strategy

Create an explicit read-source abstraction for tenant-related runtime reads.

The platform currently reads tenant data from Identity in places such as:
- branding bootstrap
- tenant resolution
- login/public branding behavior
- possibly gateway tenant lookup

In this block, introduce a controlled read-source selector with at minimum:
- Identity
- Tenant
- HybridFallback

Recommended config:
- Features:TenantReadSource = Identity | Tenant | HybridFallback
or equivalent

Behavior:
- Identity = current legacy behavior
- Tenant = use Tenant service only
- HybridFallback = try Tenant first, then fall back to Identity when missing/unavailable or when configured rules say so

Rules:
- default to the safest non-breaking mode for current environment unless the repo/setup clearly supports immediate tenant-first reads
- all runtime consumers should use this abstraction instead of directly choosing their own source
- keep the implementation centralized and observable

B) Gateway / backend consumer transition

Update the gateway-side or backend resolution consumption path so tenant resolution can use Tenant service public/internal resolution endpoints in a controlled way.

Target behaviors:
- when read source = Identity, preserve current behavior
- when read source = Tenant, resolve using Tenant service endpoints
- when read source = HybridFallback:
  1. try Tenant resolution first
  2. if not found / unavailable / invalid result, fall back to Identity
  3. record which source was used

Use the existing Tenant resolution endpoints from Block 3 where possible.
Do not create redundant resolution logic in the gateway if the Tenant service already provides it cleanly.

Preferred consumer paths to update:
- host/subdomain resolution in gateway
- any internal bootstrap route that determines tenant context from domain/code

Rules:
- do not break current gateway routing
- do not remove Identity-based resolution yet
- ensure a misconfigured Tenant service does not take down the platform

C) Web / public branding consumer transition

Update the Next.js / frontend-facing tenant branding/bootstrap path so public branding can be read from Tenant service in a controlled way.

Expected current/related behavior:
- login/public pages bootstrap tenant branding
- a server route/provider likely calls Identity branding endpoints today

In this block:
- introduce a branding read abstraction or route-layer selector
- support:
  - Identity branding source
  - Tenant branding source
  - Hybrid fallback
- if Tenant branding endpoint returns missing/incomplete branding, optionally fall back to Identity or platform default depending on configured mode

Consumers likely include:
- login page branding bootstrap
- public logo / tenant display name / primary color resolution
- any tenant bootstrap provider used before authentication

Rules:
- do not redesign frontend UX
- keep stack aligned:
  - Next.js
  - TypeScript
  - Tailwind
  - existing provider/route conventions
- minimal UI changes only if needed to support the source transition cleanly

D) Fallback rules and resilience

Define and implement explicit fallback behavior.

Recommended fallback policy in HybridFallback:
1. Try Tenant service
2. If Tenant returns:
   - 404 / not found
   - missing required fields
   - timeout / transport failure
   - invalid payload
   then fall back to Identity
3. If Identity also fails, use platform default branding or unresolved-tenant behavior per existing app conventions

Define required fields for “usable tenant/branding result”.
At minimum for branding/bootstrap:
- tenantId or code
- displayName
- usable logo ref OR safe fallback
- usable primaryColor OR safe default

At minimum for tenant resolution:
- tenantId
- code
- displayName
- status compatible with runtime expectations

Record the source used in logs and, where practical, in non-sensitive diagnostics.

E) Read-path observability

Add observability so operators can see what source is being used.

At minimum add:
- structured logs indicating:
  - source attempted
  - source used
  - fallback triggered
  - reason for fallback
- optional lightweight diagnostic endpoint if it fits existing patterns, for example:
  - GET /info or a migration/read-source diagnostics payload enhancement
- do NOT expose sensitive internals publicly

If appropriate, add metrics counters or simple in-memory counters/log summaries for:
- reads from Tenant
- reads from Identity
- fallbacks
- Tenant read failures

3) Web transition

- centralize read-source selection in those server routes/providers

- timeout handling

Admin/internal endpoints remain protected.
Do not leak internal source/fallback diagnostics through public routes.
