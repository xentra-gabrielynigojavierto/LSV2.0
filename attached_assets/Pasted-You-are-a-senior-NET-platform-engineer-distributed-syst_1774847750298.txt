## Requirements

### A. Redis health / readiness

Add Redis-aware health and readiness support.

Required:

* Redis health check
* surface Redis status in `/health/ready`
* if Redis-backed queue is configured and Redis is unavailable:

  * readiness should reflect this appropriately
* if Redis is optional in local/dev mode:

Suggested endpoint behavior:

* `/health` → liveness (service process + DB)

### B. Startup validation

At startup, validate Redis requirements clearly.

Examples:

* if `ScanWorker:QueueProvider=redis` and Redis config is missing → fail startup

Prefer explicit, safe behavior.

Add structured logs for:

* startup connection

### D. Production configuration guidance

Document guidance for:

* standalone

You do not need to implement every HA topology deeply, but the service should be **ready to be configured correctly** and the documentation/report should explain supported patterns and limitations.

### 2. Scan completion notification model

Add a clean, extensible notification/event model for completed scans.

## Goal

Reduce client polling burden and create an internal integration foundation for products like CareConnect or future platform workflows.

## Preferred approach

Implement an **internal event foundation** first, with optional outbound delivery if feasible.

### Minimum required capability

When a scan finishes with:

* `CLEAN`
* `INFECTED`
* `FAILED`

the system should emit a **scan completion event** in a structured way.

This can be implemented as one of:

* a persisted internal event table
* an internal event publisher abstraction
* a Redis pub/sub or stream event
* a lightweight webhook dispatcher
* or a layered combination of these

Prefer maintainability and extensibility over cleverness.

---

## Notification requirements

### A. Internal event model

Create a structured event that captures at minimum:

* documentId
* tenantId
* versionId if applicable
* scanStatus
* occurredAt
* correlationId if available
* attempt count if relevant
* source/service name

Suggested event name:

* `DocumentScanCompleted`

### B. Delivery abstraction

Create an abstraction such as:

* `IScanCompletionPublisher`
* `IScanCompletionDispatcher`

Provide at least one concrete implementation.

Recommended options:

* log-only publisher (dev)
* internal persistence publisher
* Redis pub/sub publisher
* webhook publisher

If outbound delivery is too large for this phase, implement:

* internal event persistence + clean abstraction + documentation for extension

### C. Trigger point

Emit the event from the background worker after final scan resolution:

* `CLEAN`
* `INFECTED`
* `FAILED`

Do NOT emit on intermediate retry attempts unless clearly marked as retry events.

### D. Reliability

The notification model should not break the scan pipeline if delivery fails.

Requirements:

* scan result persistence remains primary
* notification delivery failure should:

  * be logged
  * be measurable
  * not corrupt document state
* if using persistence, preserve eventual delivery possibility

### E. Metrics

Add metrics for notification behavior.

Suggested:

* `docs_scan_completion_events_emitted_total`
* `docs_scan_completion_delivery_failures_total`
* `docs_scan_completion_delivery_success_total`

### F. Documentation

Clearly document:

* what emits the event
* event schema
* delivery guarantees
* whether it is at-most-once / at-least-once / best-effort
* how clients/products should use it
