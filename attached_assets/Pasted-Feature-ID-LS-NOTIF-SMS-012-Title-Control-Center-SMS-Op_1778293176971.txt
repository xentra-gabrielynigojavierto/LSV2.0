Feature ID: LS-NOTIF-SMS-012

Title:
Control Center SMS Operational Alert and Escalation Management UI

Objective:
Add a full operational incident-management UI inside Control Center for SMS operational alerts, escalation history, escalation policy management, retry handling, and incident lifecycle visibility using the Notification Service APIs from LS-NOTIF-SMS-010 and LS-NOTIF-SMS-011.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- SMS operational alerts
- escalation policies
- escalation execution
- escalation persistence
- escalation retry handling
- incident operational metadata

Control Center must only consume Notification Service APIs and render/administer the operational incident-management UI.

Requirements:
- Build UI only inside the existing Control Center/admin portal application.
- Do not add alert generation or escalation logic to Control Center.
- Do not duplicate escalation evaluation logic in Control Center.
- Do not modify Notification Service except where strictly necessary for API client compatibility.
- Control Center must consume Notification Service APIs from LS-NOTIF-SMS-010 and LS-NOTIF-SMS-011.
- UI must remain operationally safe and primarily read-only except explicit management actions:
  - resolve alert
  - retry escalation
  - create/update/disable policy
- Do not expose:
  - provider credentials
  - webhook URLs
  - CredentialsJson
  - SettingsJson
  - auth tokens
  - raw provider payloads
  - raw phone numbers
- Preserve existing Control Center dashboard functionality from LS-NOTIF-SMS-009.

Implementation Requirements:
1. Add PlatformAdmin-only SMS Incident Management navigation and routes.
2. Add alert management UI.
3. Add escalation history UI.
4. Add escalation policy management UI.
5. Add alert detail views.
6. Add escalation detail views.
7. Add alert resolve actions.
8. Add escalation retry actions.
9. Add policy create/update/disable flows.
10. Add filtering for alerts/escalations/policies.
11. Add loading/error/empty states.
12. Preserve existing Control Center functionality and security behavior.

Critical Constraints:
- Reuse existing Control Center architecture, routing, API client, auth, role guards, modal, form, table, card, drawer/panel, and notification/toast patterns where available.
- Do NOT introduce large UI rewrites.
- Keep implementation incremental and production-oriented.
- Use Notification Service APIs directly.
- Do NOT hardcode operational configuration.
- Do NOT expose sensitive target values such as webhook URLs in full.
- If Notification Service masks target values, preserve masking.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.
