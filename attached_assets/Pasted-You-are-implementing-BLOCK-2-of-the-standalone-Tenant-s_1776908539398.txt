SCOPE
================================================================

A) Expand the canonical Tenant profile model

Extend the Tenant entity and/or related profile structures to support at minimum:

Company / profile fields:
- LegalName
- DisplayName (already present; preserve)
- CompanyDescription or Description
- WebsiteUrl
- TimeZone
- Locale or DefaultLocale
- SupportEmail
- SupportPhone

Address fields:
- AddressLine1
- AddressLine2
- City
- StateOrProvince
- PostalCode
- CountryCode

Rules:
- Keep the model portable and generic
- Do not use LegalSynq-only terminology
- Use sensible nullability for migration-safe rollout
- Preserve existing Id / Code / Status / Subdomain / logo refs
- Do not introduce user/access concerns into Tenant

B) Tenant lifecycle model

Refine the tenant lifecycle/status model so it is future-safe.

Current Block 1 status model was intentionally simple. In this block:
- formalize an enum or equivalent lifecycle model suitable for future migration
- minimum statuses should support at least:
  - Active
  - Inactive
  - Pending
  - Suspended
- keep implementation migration-safe
- if Identity currently only maps boolean active/inactive, keep mapping compatibility documented
- do NOT implement provisioning workflow orchestration yet

C) Branding canonical model

Create a dedicated branding model owned by Tenant service.

At minimum support:
- BrandName (optional override if different from display name)
- LogoDocumentId
- LogoWhiteDocumentId
- FaviconDocumentId (nullable, can be scaffolded even if not yet used by frontend)
- PrimaryColor
- SecondaryColor
- AccentColor
- TextColor
- BackgroundColor
- WebsiteUrl override (nullable if same as tenant profile)
- SupportEmail override (nullable if same as tenant profile)
- SupportPhone override (nullable if same as tenant profile)

Rules:
- Tenant service owns branding metadata
- Do NOT store blobs/files directly
- Store document/asset references only
- Keep branding generic and reusable across systems
- Keep theme values validation-safe (hex, string length, etc.)
- If you decide some optional fields should be deferred, document them clearly in the report, but implement at least:
  - LogoDocumentId
  - LogoWhiteDocumentId
  - PrimaryColor

D) Persistence / schema

Add the necessary EF Core entities/configurations/migrations in tenant_db for:
- expanded tenant profile fields
- branding table or embedded owned model (choose the pattern most consistent with repo/service conventions)

Preferred approach:
- separate tenant_branding-style table or clean one-to-one branding entity
because this keeps the domain cleaner for future expansion

Be careful with naming consistency:
- verify the current service table naming convention established in Block 1
- keep naming consistent now to avoid painful later migrations

E) API contract

Complete the Tenant profile CRUD/read APIs and add branding APIs.

Profile APIs:
- GET /api/v1/tenants/{tenantId}
- GET /api/v1/tenants/by-code/{code}
- GET /api/v1/tenants
- POST /api/v1/tenants
- PUT /api/v1/tenants/{tenantId}

Branding APIs:
- GET /api/v1/tenants/{tenantId}/branding
- PUT /api/v1/tenants/{tenantId}/branding

Public branding endpoint:
- GET /api/v1/public/branding/by-code/{code}
AND/OR
- GET /api/v1/public/branding/by-subdomain/{subdomain}

Choose the endpoint shape that best matches the current gateway/web usage patterns, but:
- it must be unauthenticated/public
- it must be safe/read-only
- it must return enough information for login/public branding consumers
- it must not expose sensitive tenant internals

Branding response should include, at minimum:
- tenantId
- code
- displayName
- brandName
- logoDocumentId
- logoWhiteDocumentId
- primaryColor
- faviconDocumentId if available

F) Validation and domain rules

Add validation for:
- required display/code values
- code uniqueness
- optional subdomain uniqueness if applicable
- email format where used
- website URL format where used
- branding color format where used (at minimum accept valid hex values if color is supplied)

G) Consumer readiness

Design the branding/public contract so it is easy for the existing Next.js app to consume later.
Do not modify the frontend in this block unless absolutely necessary.
This block is backend-first and consumer-ready.
