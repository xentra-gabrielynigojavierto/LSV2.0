You are working inside the LegalSynq v2 codebase.

Objective:
Implement the BillOfSale document retrieval APIs so the Liens service can securely expose access to BOS PDFs that are stored in the v2 Documents service.

This feature should let Liens act as the business access gate for BOS documents while Documents remains the owner of storage and file delivery.

NON-NEGOTIABLE ARCHITECTURE RULE:
All document storage and binary ownership MUST remain in the v2 Documents service.
- Liens must NOT store files
- Liens must NOT create document tables
- Liens must ONLY store and use DocumentId references

CRITICAL IMPLEMENTATION RULES:
- Preserve existing Liens business authorization boundaries
- Do NOT bypass Liens by exposing raw Documents access directly
- Do NOT introduce cross-service DB joins
- Keep endpoint handlers thin
- Keep Documents retrieval behind an abstraction/service
- Do NOT add generic document APIs for all entities in this feature

--------------------------------------------------

Inspect how the current v2 Documents service exposes document retrieval.

Look for:
- download/view endpoints
- response/content streaming behavior
- auth requirements
- whether services proxy document bytes or return a redirect/location
- how MIME type / filename are returned
- any existing integration pattern already used by another service

Follow the existing platform pattern if one exists.
If no clear pattern exists, default to a Liens-side proxy download approach.

--------------------------------------------------

Implement the preferred endpoint:

- extract tenant from request context
- load BOS by id via application layer
- ensure BOS exists within tenant
- ensure `DocumentId` is present
- retrieve document from Documents service
- return file response safely

Keep the endpoint thin and consistent with existing Liens API patterns.
