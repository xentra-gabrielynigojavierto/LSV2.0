## A. FIX DURABILITY FIRST

### A1. Replace InMemory canonical storage with durable dev-safe storage

Current problem:
- audit service still uses `Database:Provider = "InMemory"`
- all canonical events disappear on restart

Do:
- change the audit service development/runtime default away from InMemory
- use the safest local durable option already supported by the codebase:
  - prefer SQLite if already feasible with current EF structure
  - otherwise wire MySQL/PostgreSQL cleanly if that is simpler in this repo
- keep development friction low, but persistence must survive service restart

Requirements:
- update config classes if needed
- update appsettings files
- update startup docs / README
- verify migrations still work
- make sure health/detail endpoints reflect real DB mode

Do not:
- remove InMemory support entirely if tests still depend on it
- break the existing test project

Implementation note:
- if tests require InMemory, keep it only in test factory / test overrides, not as the service’s default runtime path

## G. IMPROVE HASH-CHAIN SAFETY

### G1. Reduce concurrent chain fork risk

Current problem:
- concurrent ingest in same chain can fork `PreviousHash`

Implement the safest practical improvement without redesigning the whole service.

Preferred approaches:
1. per-chain DB locking in the append path
2. transactional compare/retry approach
3. serialized append within `(TenantId, SourceSystem)` boundary

- keep throughput reasonable
- do not introduce a dangerous global lock
- document remaining limitations
- add tests for concurrent same-chain ingest if feasible

- implement the best available mitigation
- document clearly what remains
