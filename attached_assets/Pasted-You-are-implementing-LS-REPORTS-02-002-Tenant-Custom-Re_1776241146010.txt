# STORY ID
**LS-REPORTS-02-002 — Tenant Custom Report Override (Inheritance Model)**

---

# OBJECTIVE
Introduce a tenant override model that allows a tenant to derive a tenant-specific report configuration from a published `ReportTemplate` without changing the global template itself.

This story must establish the inheritance foundation so a tenant can:
- inherit a globally managed template
- override selected configurable aspects
- keep a clear link to the source template
- preserve global governance while allowing tenant-specific flexibility

This is the first step toward:
- tenant custom report variants
- tier-based customization
- future report builder UX
- controlled report personalization by tenant and product

---

# REQUIRED IMPLEMENTATION

## A. Tenant Override Domain / Persistence Model
Add a persistence entity for tenant overrides.

### 1. TenantReportOverride
Suggested fields:
- `Id`
- `TenantId`
- `ReportTemplateId`
- `BaseTemplateVersionNumber`
- `NameOverride` (nullable)
- `DescriptionOverride` (nullable)
- `LayoutConfigJson` (nullable)
- `ColumnConfigJson` (nullable)
- `FilterConfigJson` (nullable)
- `FormulaConfigJson` (nullable)
- `HeaderConfigJson` (nullable)
- `FooterConfigJson` (nullable)

- `RequiredFeatureCode` (nullable)
- `MinimumTierCode` (nullable)
- `CreatedAtUtc`
- `CreatedByUserId`

- `UpdatedByUserId`

### Rules
- This entity stores tenant-specific overrides while preserving inheritance to the base template
- It must link to a base template and base template version number
- At this stage, JSON override fields are acceptable storage for future builder/editor features

## F. Override Rules
Implement at minimum:

### Rule 1 — Assignment dependency
Override is allowed only when:
- template exists
- template has a published version
- template is assigned to the tenant (globally or tenant-targeted)

### Rule 2 — One active override
A tenant may have at most one active override per template.

### Rule 3 — Base version anchoring
When override is created, capture the current published version number as `BaseTemplateVersionNumber`.

### Rule 4 — Effective resolution
When resolving effective tenant report:
1. resolve base template
2. resolve published version
3. resolve tenant override if present
4. overlay override fields on top of base metadata/version-derived output
5. return a coherent effective response

### Rule 5 — Soft deactivate
Deleting an override should be soft-deactivate if possible rather than physical delete.

## J. Validation and Error Handling
Implement at minimum:

### Override creation
- `TenantId` required
- template must exist
- published version must exist
- tenant must have valid assignment to template
- only one active override per tenant + template
- `CreatedByUserId` required

### Override update
- override must exist
- `UpdatedByUserId` required
- assignment/template validity must still hold

### Effective resolution
- `TenantId` required
- template must exist
- template must be assigned to tenant
- published version must exist

### Conflict behavior
- duplicate active override should return `409`

### Status code guidance
- `400` invalid request
- `404` template/override not found
- `409` duplicate/conflict
- `500` unexpected failure

Use lightweight, consistent responses.
