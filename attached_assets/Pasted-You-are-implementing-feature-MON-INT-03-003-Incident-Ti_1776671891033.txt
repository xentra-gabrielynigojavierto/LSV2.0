You are implementing feature MON-INT-03-003 — Incident Timeline / History View.

OBJECTIVE
Add an Incident Timeline / History View to the Control Center so operators can see the historical sequence of alerts and state changes for a component, including when alerts were triggered and resolved.

This introduces traceability and completes the incident lifecycle experience.

ARCHITECTURE RULE (NON-NEGOTIABLE)
- Monitoring Service = source of truth
- Control Center = read-only consumer
- Timeline data must come from Monitoring Service
- Do NOT fabricate history in Control Center
- Do NOT bypass Monitoring Service
- Do NOT break existing read endpoints

MANDATORY EXECUTION RULES
- Create the report FIRST at: /analysis/MON-INT-03-003-report.md
- Update the report incrementally after each meaningful step
- Do NOT wait until the end to write the report
- Prefer minimal, reversible changes
- Do NOT redesign the monitoring page
- Maintain compatibility with MONITORING_SOURCE=local and service
- If local mode cannot support timeline, degrade gracefully

STEP 3 — DESIGN TIMELINE API

Create a read endpoint:

GET /monitoring/alerts/history

Query params:
- entityName OR entityId (prefer entityId if available)
- limit (default 10, max 50)

Behavior:
- return most recent alerts for that entity
- include both active and resolved alerts
- order by TriggeredAtUtc DESC

Response model:

[
  {
    alertId,
    entityName,
    severity,
    message,
    triggeredAtUtc,
    resolvedAtUtc,
    isActive
  }
]

Rules:
- do NOT change existing endpoints
- do NOT break summary/status endpoints
- must be read-only
- must support service mode consumption
