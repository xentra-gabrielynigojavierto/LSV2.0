# LS-ID-TNT-017-002 — User Access Audit

## FEATURE OBJECTIVE

Implement a comprehensive **User Access Audit** layer that captures and exposes security-relevant access events across the platform, including:
- successful access to protected products/pages
- denied access attempts
- permission-denied API actions
- login / logout / session invalidation events where applicable
- audit-friendly access traces tied to actor, tenant, session, and target resource

This feature extends the audit system beyond **permission-change events** into **permission-usage and access events**, enabling:
- security monitoring
- compliance reporting
- anomaly investigation
- operational debugging of “who accessed what, when, and whether it was allowed”

This is primarily an **audit capture + query readiness feature**. It does NOT require full analytics or anomaly detection UI.

---

## WHY THIS FEATURE COMES NEXT

Current state:
- permission model is complete
- backend enforcement is complete
- UI gating is complete
- permission-change audit is complete
- audit viewer is complete

What is missing:
- we know **who changed access**
- but we do NOT yet fully know **who used access**, **who was denied**, and **what resources were accessed**

This feature closes that governance/compliance gap.

---

## HARD SCOPE BOUNDARY (DO NOT VIOLATE)

DO NOT IMPLEMENT:
- behavioral analytics dashboards
- anomaly detection engines
- SIEM integrations
- external monitoring platform connectors
- non-security business events (case edits, financial activity, etc.) unless they are already part of access enforcement flow
- broad redesign of the existing audit pipeline
- speculative capture points not grounded in actual code
- full export/reporting UI

ONLY IMPLEMENT:
- canonical audit capture for user access events
- denied-access and permission-denied event capture
- session/auth access events where appropriate and already supported by current architecture
- consistent event schema
- integration with the centralized audit pipeline already in place
- report-first implementation and honest documentation of coverage
