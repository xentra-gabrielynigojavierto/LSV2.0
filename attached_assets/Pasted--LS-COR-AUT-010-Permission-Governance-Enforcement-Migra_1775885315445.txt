# LS-COR-AUT-010 — Permission Governance + Enforcement Migration

## Objective

Evolve LS-COR-AUT from **permission-enabled** to **fully governed and permission-driven authorization (PBAC)** by introducing:

* explicit permission catalog (source-of-truth)
* role → permission governance
* permission lifecycle management
* enforcement migration from roles → permissions
* audit coverage for permission changes

At completion, the system must support:

```text
User → Roles → Permissions → Enforcement (Primary)
User → Roles → product_roles → Enforcement (Legacy fallback)
```

This feature establishes **permissions as the primary enforcement model**, while maintaining safe backward compatibility.

---

## Architectural Principles

### Controlled Evolution

* Do NOT break existing role-based enforcement
* Introduce governance before enforcing migration

### Single Source of Truth

* Permissions must exist as **first-class entities**, not derived-only artifacts

### Governance First

* Define, manage, and audit permissions before making them primary

---

## In Scope

---

## 1. Permission Catalog (Source-of-Truth)

### Create Entity

`Permission`

Fields:

* `Id`
* `PermissionCode` (string, unique)
* `Name`
* `Description`
* `ProductCode`
* `Category` (e.g., Referral, Billing, Admin)
* `IsActive`
* `CreatedAtUtc`
* `UpdatedAtUtc`
* `CreatedBy`
* `UpdatedBy`

---

### Naming Convention (FINALIZE)

```text
{PRODUCT}.{domain}:{action}
```

Examples:

```text
SYNQ_CARECONNECT.referral:create
SYNQ_CARECONNECT.referral:view
SYNQ_FUND.application:approve
```

---

### Rules

* ProductCode must be valid
* PermissionCode must be globally unique
* Must follow naming convention

---

## 2. Role → Permission Mapping (Governed)

### Update / Formalize Entity

`RolePermission` (or `RoleCapabilityAssignment` if already exists)

Fields:

* `Id`
* `RoleCode`
* `PermissionCode`
* `ProductCode`
* `IsActive`
* `CreatedAtUtc`
* `UpdatedAtUtc`

---

### Required Enhancements

* enforce product consistency
* prevent duplicate mappings
* support activation/deactivation (soft state)

---

## 3. Permission Resolution Refactor

### Update `EffectiveAccessService`

Replace implicit resolution with:

```text
UserRoles
→ RolePermission (catalog-driven)
→ Permission (catalog validation)
→ Effective Permissions
```

---

### Required Behavior

* only include permissions where:

  * RolePermission.IsActive = true
  * Permission.IsActive = true
* enforce product match
* preserve provenance tracking (Direct / Group)

---

## 4. Permission Lifecycle Management APIs

### Endpoints

#### Permission Catalog

* `GET /api/admin/permissions`
* `POST /api/admin/permissions`
* `PATCH /api/admin/permissions/{id}`
* `DELETE /api/admin/permissions/{id}` (soft delete)

#### Role Mapping

* `GET /api/admin/roles/{roleCode}/permissions`
* `POST /api/admin/roles/{roleCode}/permissions`
* `DELETE /api/admin/roles/{roleCode}/permissions/{permissionCode}`

---

### Rules

* admin-only (PlatformAdmin / TenantAdmin)
* enforce product alignment
* validate permission existence

---

## 6. Enforcement Migration (Controlled)

### Phase 1 — Dual Enforcement

Allow both:

```csharp
.RequireProductRole(...)
.RequirePermission(...)
```

---

### Phase 2 — Migration Targets

Identify endpoints and migrate:

```csharp
.RequireProductRole("SYNQ_FUND", "FUNDER")
```

→

```csharp
.RequirePermission("SYNQ_FUND.application:approve")
```

---

### Phase 3 — Deprecation Flag

Introduce config:

```text
EnableRoleFallback = true | false
```

* when false:

  * role-based enforcement disabled for selected endpoints

---

## 8. Validation Rules

### Permission

* must follow naming convention
* must belong to valid product

### Role Mapping

* role and permission must match product
* duplicate mappings rejected

### Enforcement

* permission must exist AND be active
* inactive permission must NOT grant access
