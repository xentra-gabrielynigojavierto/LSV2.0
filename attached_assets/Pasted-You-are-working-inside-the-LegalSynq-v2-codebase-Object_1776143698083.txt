You are working inside the LegalSynq v2 codebase.

Objective:
Implement the Sale Finalization Application Service for the Liens microservice so the core SynqLiens sale workflow can execute safely and consistently:

Accept Offer → Create BillOfSale → Update Lien → Reject competing offers

This is the first true multi-entity business workflow in Liens. It must coordinate repositories correctly and execute transactionally.

NON-NEGOTIABLE ARCHITECTURE RULE:
All cross-cutting concerns must use v2 services only:
- Identity
- Documents
- Notifications
- Audit

CRITICAL IMPLEMENTATION RULES:
- Do NOT put orchestration logic in repositories
- Do NOT put persistence logic in domain entities
- Do NOT add API endpoints in this feature
- Do NOT add cross-service joins
- Do NOT implement notifications/documents/audit publishing yet
- Preserve donor SynqLiens behavior and API compatibility as much as practical

--------------------------------------------------

--------------------------------------------------

Inspect how existing v2 services structure application services and workflow orchestration.

Look at relevant services such as:
- CareConnect

- any current Liens application services if present

Identify:

Follow the existing v2 pattern as closely as possible.

--------------------------------------------------
STEP 2 — CREATE THE SALE FINALIZATION SERVICE
--------------------------------------------------

Create an application service in the appropriate Liens Application location.

Suggested shape:
- `ILienSaleService`
- `LienSaleService`

The service should implement a method equivalent to:

`AcceptOfferAsync(Guid tenantId, Guid lienOfferId, Guid actingUserId, CancellationToken ct)`

You may choose a slightly different signature if the existing platform pattern strongly suggests a better one, but the method must support:
- tenant-scoped execution
- target offer id
- acting user id
- cancellation token

--------------------------------------------------
STEP 3 — LOAD AND VALIDATE THE TARGET OFFER
--------------------------------------------------

Using the repository layer:

1. Load the offer by `lienOfferId`
2. Validate:
   - it exists
   - it belongs to the specified tenant
   - it is in a valid actionable state
   - it has not expired
   - it has not already been accepted/rejected/withdrawn in a terminal way

Use the existing domain methods and status model where appropriate.
Do NOT duplicate domain status logic unnecessarily, but do enforce workflow preconditions at the application layer.

--------------------------------------------------
STEP 4 — LOAD AND VALIDATE THE LIEN
--------------------------------------------------

Load the lien associated with the offer.

Validate:
- lien exists
- lien belongs to the same tenant
- lien is in a state where accepting an offer is allowed
- lien has not already been finalized through another sale path
- lien does not already have a BillOfSale for the accepted sale path

If the workflow encounters an already-finalized condition, fail safely and clearly.

--------------------------------------------------
STEP 5 — BEGIN A TRANSACTION
--------------------------------------------------

Use the scoped `LiensDbContext` transaction mechanism consistent with the current v2 pattern.

All of the following steps must occur in a single transaction:
- offer acceptance
- competing offer rejection
- lien update
- BillOfSale creation

If anything fails, the whole operation must roll back.

Do NOT fake transactional behavior at the repository level.
Use the real DbContext transaction.

--------------------------------------------------
STEP 6 — CREATE THE BILL OF SALE
--------------------------------------------------

Create the BillOfSale using the domain factory.

Use the accepted offer + lien information to populate:
- tenant
- lien id
- lien offer id
- seller org
- buyer org
- purchase amount
- original lien amount
- bill of sale number / reference according to current domain/factory expectations
- acting user context where applicable

Persist it through the repository layer.

Do NOT generate documents in this feature.
Do NOT call the Documents service here.

--------------------------------------------------
STEP 7 — ACCEPT THE TARGET OFFER
--------------------------------------------------

Update the target offer using the domain model:
- status → Accepted
- response timestamps/notes as applicable

Persist it through the repository layer.

Make sure this does not conflict with the BillOfSale creation order or transaction semantics.

--------------------------------------------------
STEP 8 — REJECT COMPETING OFFERS
--------------------------------------------------

Load all other offers for the same lien.

For every competing offer that is still actionable:
- reject it
- stamp the relevant response metadata
- persist the update

Be careful not to:
- reject the accepted offer
- mutate already terminal offers incorrectly
- introduce partial state changes outside the transaction

This is a required part of the sale finalization workflow.

--------------------------------------------------
STEP 9 — UPDATE THE LIEN
--------------------------------------------------

Update the lien through the domain model to reflect sale completion.

At minimum:
- move to the sold state (or equivalent approved status)
- update holder/ownership fields appropriately
- record purchase/sale-related data needed by the existing domain model

Persist it through the repository layer.

Use the current Lien entity’s methods rather than directly mutating fields.

--------------------------------------------------
STEP 10 — COMMIT TRANSACTION
--------------------------------------------------

Commit only after:
- BillOfSale has been created
- target offer accepted
- competing offers rejected
- lien updated

If anything fails before commit:
- roll back
- return/throw a clear failure

--------------------------------------------------
STEP 11 — RETURN A STRUCTURED RESULT
--------------------------------------------------

Return a clean application-layer result model.

Do NOT return EF entities directly.

At minimum, return:
- accepted offer id
- lien id
- bill of sale id
- final lien status
- accepted offer status

Use an application-layer result/response model consistent with the codebase style.

--------------------------------------------------
STEP 12 — HANDLE IDEMPOTENCY / SAFE RE-RUNS
--------------------------------------------------

Design the workflow so it fails safely if re-run or double-invoked.

At minimum address:
- offer already accepted
- BillOfSale already exists for the same finalized sale
- lien already sold

You may either:
- return an existing result safely
OR
- throw a clear application error

Choose the approach that best matches existing v2 service style and document it clearly.
