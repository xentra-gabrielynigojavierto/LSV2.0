TASK-DB-ANALYSIS-01 — Workflow vs Task Table Ownership Analysis

OBJECTIVE:
Analyze the current Flow/Liens/Task database schemas and produce a precise ownership decision for workflow-related tables and models: which should move to the Task service, which should remain in Flow, and which should only be referenced by Task without being migrated.

CONTEXT / DECISION RULE:
Use this rule consistently:

- If the data answers: "what work should a person do and how is that work managed?" → it belongs in Task
- If the data answers: "where is the process instance in its orchestration lifecycle?" → it belongs in Flow

Task owns:
- task stages
- task-stage transitions
- task governance settings
- task templates
- task generation rules
- task reminders
- task notes/history
- task linkage to workflow instance / workflow step

Flow owns:
- workflow definitions
- workflow step graph / orchestration model
- workflow instance runtime
- workflow advancement/completion state
- process orchestration internals

IMPORTANT:
- Do NOT implement any schema changes in this prompt
- Do NOT migrate any data in this prompt
- This is an analysis + recommendation prompt only
- MUST create the report FIRST and update it incrementally
- MUST inspect the actual codebase and database schema definitions before making conclusions
- NO assumptions without code/schema evidence
