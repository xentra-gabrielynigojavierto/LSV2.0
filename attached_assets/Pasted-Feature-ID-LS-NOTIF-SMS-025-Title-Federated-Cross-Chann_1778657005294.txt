Feature ID: LS-NOTIF-SMS-025

Title:
Federated Cross-Channel Governance Enforcement Engines and Unified Policy Execution Runtime

Objective:
Operationalize active governance enforcement across Email, Push, Webhook, and future notification channels by introducing federated governance execution engines, a unified policy execution runtime, channel-specific enforcement adapters, shared governance execution telemetry, and cross-channel simulation integrated with the LS-NOTIF-SMS-024 federation topology.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- unified governance execution runtime
- channel-specific governance enforcement engines
- cross-channel effective rule execution
- governance execution telemetry
- governance execution auditability
- cross-channel governance simulation
- runtime governance APIs

Control Center must only consume Notification Service APIs and render/administer governance runtime visibility. Do not duplicate runtime rule evaluation, topology resolution, channel enforcement, simulation, or telemetry logic in Control Center.

Requirements:
- Implement unified governance execution runtime inside Notification Service.
- Preserve existing SMS governance behavior from LS-NOTIF-SMS-017 through LS-NOTIF-SMS-024.
- Preserve SMS-only behavior for all existing tenants.
- Add active governance enforcement for Email, Push, and Webhook where the current delivery pipeline supports integration.
- Use LS-NOTIF-SMS-024 governance topology resolution as the source of effective rules.
- Governance execution must be channel-aware and tenant-aware.
- Existing SMS governance must remain backward compatible and must not be rewritten.
- Non-SMS enforcement must degrade safely if channel payload context is insufficient.
- Runtime governance failures must fail open unless explicitly configured otherwise.
- Governance execution must not block unrelated notification channels.
- Do not store raw phone numbers, message bodies, credentials, webhook URLs, SettingsJson, CredentialsJson, auth tokens, or provider payloads in runtime telemetry/audit tables.
- Keep Control Center as admin/visibility UX only.
- Preserve all LS-NOTIF-SMS-001 through LS-NOTIF-SMS-024 behavior.

Implementation Requirements:
1. Add unified governance execution runtime abstraction.
2. Add channel enforcement engine abstraction.
3. Add Email governance enforcement engine.

12. Ensure topology resolver from LS-NOTIF-SMS-024 is used for effective rule resolution.
13. Add Control Center governance runtime dashboard if feasible.
14. Ensure all failures degrade safely.

STEP 3 — Define Unified Governance Execution Runtime
Add runtime orchestrator.

Suggested interface:
IGovernanceExecutionRuntime

Suggested methods:
- Task<GovernanceExecutionResult> EvaluateAsync(GovernanceExecutionContext context, CancellationToken ct)
- Task<GovernanceSimulationResult> SimulateAsync(GovernanceSimulationRequest request, CancellationToken ct)
- Task<IReadOnlyList<GovernanceChannelRuntimeStatus>> GetChannelRuntimeStatusAsync(CancellationToken ct)

Runtime behavior:
1. Normalize channel type.
2. Resolve topology using IGovernanceTopologyResolver.
3. Select registered IGovernanceChannelEnforcementEngine for the channel.
4. Execute channel-specific enforcement against effective rules.
5. Persist safe telemetry/audit.
6. Return decision result.
7. Fail open when configured.

Requirements:
- No external APIs.
- No delivery sends.
- No provider calls.
- No raw payload persistence.
- Failure in one channel engine must not affect others.
- Missing channel engine should fail open with reason insufficient_context or no_engine_registered.
- Must not alter existing SMS behavior unless explicitly using compatibility adapter.
