You are implementing feature MON-INT-04-004 — Response Time Charts.

OBJECTIVE
Add response time (latency) charts to the monitoring experience so operators can visualize recent latency trends per monitored component using real data from the uptime aggregation engine.

This is a presentation-layer feature built on top of MON-INT-04-002 (Uptime Aggregation) and MON-INT-04-003 (Availability Bars). It must use real Monitoring Service data only and must not introduce fabricated metrics.

ARCHITECTURE RULE (NON-NEGOTIABLE)
- Monitoring Service = source of truth
- Control Center = read-only consumer
- Charts must use uptime/history data (avgLatencyMs, maxLatencyMs)
- Do NOT compute latency from current status snapshots
- Do NOT modify aggregation formulas unless absolutely required
- Do NOT expose internal identifiers or admin controls on public pages
- Prefer internal (authenticated) monitoring page for charts

MANDATORY EXECUTION RULES
- Create the report FIRST at: /analysis/MON-INT-04-004-report.md
- Update the report incrementally after each step
- Do NOT wait until the end to write the report
- Keep implementation minimal and reversible
- Do NOT redesign entire monitoring UI
- Do NOT introduce heavy dependencies without justification

CURRENT CONTEXT
- Monitoring summary, alerts, resolve, timeline = complete
- Public status page = complete
- Availability bars = complete
- Uptime aggregation provides latency metrics
- Charts are the final visualization layer

GOAL OF THIS FEATURE
- Display latency trends per component
- Use real hourly data from uptime history
- Improve operator insight (internal UX)
- Maintain public safety boundaries
