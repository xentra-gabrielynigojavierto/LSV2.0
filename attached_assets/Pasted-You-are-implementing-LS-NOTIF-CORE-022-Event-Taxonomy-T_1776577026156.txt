OBJECTIVE
================================================================================

Standardize the semantic layer of the Notifications platform by defining and
implementing a canonical taxonomy for event keys and template keys.

The goal is to ensure all producers use consistent event naming and template
catalog conventions so the Notifications service is semantically stable,
queryable, and reusable across platforms.

================================================================================
EVENT TAXONOMY REQUIREMENTS
================================================================================

Define a canonical eventKey convention using lowercase dot-separated names.

Preferred format:
   <domain>.<entity>.<action>
or
   <domain>.<entity>.<action>.<qualifier>

Examples:
- lien.offer.submitted
- lien.offer.approved
- referral.invite.sent
- referral.accepted
- report.delivery.requested
- password.reset.requested
- task.assignment.created

Rules:
- lowercase only
- dot-separated
- stable meaning
- not channel-specific
- not template-specific
- not UI-label based

================================================================================
TEMPLATE TAXONOMY REQUIREMENTS
================================================================================

Define a canonical templateKey convention using lowercase kebab-case names.

Preferred format:
   <event-or-purpose>-<channel>
or
   <event-or-purpose>-<variant>-<channel>

Examples:
- lien-offer-submitted-email
- referral-invite-email
- report-delivery-email
- password-reset-email
- task-assignment-email

Rules:
- lowercase only
- kebab-case
- explicit channel suffix
- presentation-oriented
- separate from eventKey meaning

================================================================================
EVENT-TO-TEMPLATE MAPPING RULES
================================================================================

Implement or document the standard that:
- producers always send eventKey
- templateKey is optional and used only for explicit approved variants/overrides
- Notifications may resolve default template by eventKey + channel
- invalid or deprecated event/template combinations should be logged, aliased, or rejected depending on current system capabilities
