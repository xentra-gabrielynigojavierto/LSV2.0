### 1. Signature freshness monitoring

Add visibility into the freshness of ClamAV virus definitions.

## Goal

Detect stale definitions before malware detection quality silently degrades.

## Requirements

Implement the best safe practical version possible.

You may use one or more of these approaches, depending on what is practical in the current environment:

* inspect ClamAV database metadata files if accessible
* query ClamAV or its host environment for update timestamps
* surface a documented approximation if exact data is not available

### Must expose:

#### Metric

* `docs_clamav_signature_age_seconds`

#### Health detail

Include in readiness / health response something like:

* last signature update time
* age in seconds
* whether stale threshold is exceeded

#### Config

Add config such as:

```json
{
  "Scanner": {
    "ClamAv": {
      "SignatureMaxAgeHours": 24
    }
  }
}
```

### Behavior

* if definitions are stale beyond threshold:

  * health should be `Degraded`
  * log a warning
* do NOT block uploads solely because signatures are stale
* do NOT mark scans as failed just because freshness is stale
* this is an observability and operational safety feature

### Important

If exact signature freshness cannot be determined reliably from the current deployment model, implement the best practical method and document limitations clearly.

---

### 2. Large file handling policy

Implement a strict, explicit file-size policy aligned with the scanning architecture.

## Goal

Prevent files from entering the system when they exceed safe or supported scanning limits.

## Risks to address

* ClamAV stream limits
* long-running scan timeouts
* memory/throughput pressure
* inconsistent upload vs scan behavior

## Requirements

Introduce clear, config-driven limits:

```json
{
  "Documents": {
    "MaxUploadSizeMb": 25
  },
  "Scanner": {
    "ClamAv": {
      "MaxScannableFileSizeMb": 25
    }
  }
}
```

### Required behavior

#### API upload layer

* reject files larger than `Documents:MaxUploadSizeMb`
* return a clear machine-readable error
* do this as early as practical

#### Scan compatibility layer

* if file size exceeds `MaxScannableFileSizeMb`, reject before enqueueing scan
* do NOT allow a file that cannot be safely scanned to proceed as if valid
* do NOT let a too-large file remain indefinitely in `PENDING`

### Error behavior

Add/extend a clear error such as:

* `FILE_TOO_LARGE`
* or `FILE_EXCEEDS_SCAN_LIMIT`

Use appropriate status code:

* 413 recommended for upload limit breach
* 422 acceptable for scan-policy rejection if clearly distinguished

### Metrics

Expose metrics for:

* upload rejections due to file size
* scan-limit rejections due to scanner constraints

Suggested metrics:

* `docs_upload_file_too_large_total`
* `docs_scan_size_rejected_total`

### Logging

Log size-based rejection events at warning level with:

* correlation ID
* file size
* configured limit
* endpoint/context
  But do NOT log sensitive content.

---

### 3. Configuration validation

Add startup validation so misconfiguration is caught early.

Examples:

* `MaxScannableFileSizeMb` should not exceed what your ClamAV deployment safely supports
* `Documents:MaxUploadSizeMb` should not exceed `MaxScannableFileSizeMb` unless explicitly supported by an alternate policy
* invalid or contradictory configuration should:

  * fail startup
  * or emit a clear warning if intentionally allowed

Prefer safe startup validation.
