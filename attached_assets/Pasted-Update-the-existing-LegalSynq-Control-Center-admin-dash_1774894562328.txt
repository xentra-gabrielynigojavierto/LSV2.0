### 4. Mock pages for unsupported or incomplete features

For any nav item/page that does NOT yet have working backend support, create a clean mock page.

Rules for mock pages:
- keep the same layout shell as real pages
- include a visible badge near the page title: MOCKUP
- include 1–2 realistic sections/cards/tables that preview what the page will eventually show
- include placeholder filter controls if appropriate
- include realistic empty data examples
- do NOT fake actual CRUD behavior
- buttons can be disabled or no-op if clearly labeled

Examples of likely mock candidates if backend is not ready:
- Tenants management
- Tenant domains
- Product catalog admin
- Settings pages
- audit/ops pages that are not yet wired

---

### 5. Add status indicators in the UI

For each page or nav item, make the implementation state obvious.

Recommended states:
- LIVE → fully wired to backend
- IN PROGRESS → partially wired or mixed live/mock
- MOCKUP → no real backend wiring yet

Use subtle badges/tags that fit the existing design system.
Do not make the UI look unfinished or messy.
