==================================================
OBJECTIVE
==================================================

Enhance the existing LS-LIENS-FLOW-001 My Tasks capability by introducing Contextual Task Intelligence.

This feature must make task creation faster, more structured, and more workflow-aware without introducing full workflow automation.

Build:

1. Contextual Task Templates
2. Workflow-stage-aware task creation
3. Default assignment suggestions based on role
4. Pre-filled due date intelligence
5. Admin-managed task templates
6. Shared governance for templates across:
   - Tenant-side Synq Liens Product Settings
   - Control Center admin side

This is NOT a workflow engine.
This is NOT event-driven automation yet.
This is NOT SLA enforcement.

==================================================
MANDATORY ARCHITECTURE RULES
==================================================

Use v2 shared services only. Do not use v1 equivalents.

Mandatory shared services:
- Identity
- Audit
- Notifications only if existing task flows need it
- Documents are not needed here unless already reused by an existing pattern

Preserve Synq Liens donor behavior and API style where practical while implementing on v2 architecture.

Do not over-engineer this feature.

This is still a progressive maturity phase:
- no auto-generated tasks
- no dependency engine
- no SLA logic
- no workflow stage gating
- no automatic stage progression
- no heavy rules engine

==================================================
HIGH-LEVEL FEATURE SUMMARY
==================================================

Build a guided task creation layer on top of FLOW-001.

A. Task Templates
- reusable tenant-defined templates for common work
- can apply by case context, lien context, and optionally workflow stage context

B. Contextual Task Creation
- users can create tasks from template or from scratch
- when using a template, fields are pre-filled
- task still remains editable before save

C. Workflow Awareness
- current workflow stage should be visible where relevant
- task creation can optionally pre-fill workflowStageId
- workflow should inform task creation but not enforce behavior

D. Shared Governance
- task templates editable from:
  - Synq Liens → Product Settings → Task Templates
  - Control Center → Tenant → Synq Liens → Task Templates
- both surfaces manage the same underlying template set

==================================================
PHASE 2 PRODUCT PRINCIPLES
==================================================

1. Assist, do not enforce
2. Context should reduce user effort
3. Templates improve consistency but remain optional
4. Workflow informs task creation, but does not control it
5. Keep domain separation clean
6. Build toward future Flow automation without hard-wiring it now

STEP 4 — IMPLEMENT CONTEXTUAL TASK CREATION
==================================================

Enhance task creation UX so users can create from:
- template
- scratch

--------------------------------------------------
4.1 Entry Points
--------------------------------------------------

Update task creation from:
- My Tasks page
- Case page
- Lien page

When user clicks Create Task:
- show template-aware creation flow
- allow Create from Scratch

--------------------------------------------------
4.2 Required UX Behavior
--------------------------------------------------

Support this flow:

1. User clicks Create Task
2. User can:
   - choose a template
   - or choose Start from Scratch
3. If a template is selected:
   - pre-fill title
   - pre-fill description
   - pre-fill priority
   - pre-fill due date based on offset
   - pre-fill workflow stage where applicable
   - suggest assignee based on role mapping if possible
4. User can modify fields before saving
5. Save creates a normal task using existing FLOW-001 task APIs/services, extended as needed

--------------------------------------------------
4.3 Context Filtering for Templates
--------------------------------------------------

Template picker should filter/sort templates based on context.

Examples:
- on Case page: prefer CASE + GENERAL + matching STAGE templates
- on Lien page: prefer LIEN + GENERAL + matching STAGE templates
- on My Tasks page: show GENERAL and optionally all allowed templates, with context selection if needed

Do not hide templates so aggressively that users lose flexibility.
Prefer relevance-first ordering/filtering.

STEP 5 — WORKFLOW STAGE AWARENESS
==================================================

Introduce lightweight workflow stage awareness.

--------------------------------------------------
5.1 Display Stage Context
--------------------------------------------------

Where workflow stage already exists for the current case/lien context, surface it in the UI where practical:
- case task entry
- lien task entry
- task form

--------------------------------------------------
5.2 Template to Stage Relationship
--------------------------------------------------

Templates may optionally target a workflow stage.

If user is in a context that has a current workflow stage:
- matching stage templates should be surfaced first
- task form should pre-fill workflowStageId when appropriate

--------------------------------------------------
5.3 No Enforcement
--------------------------------------------------

Do NOT implement:
- stage transition gating
- mandatory templates per stage
- task completion changing stage
- stage changing task automatically

Workflow awareness is assistive only.

STEP 6 — DEFAULT ASSIGNMENT INTELLIGENCE
==================================================

Implement lightweight default assignment suggestion behavior.

--------------------------------------------------
6.1 Template Role Mapping
--------------------------------------------------

A template may optionally define a default role reference.

When a template with a role is selected:
- use the Identity service / existing user-role lookup pattern
- attempt to identify eligible users for that role in the tenant context

--------------------------------------------------
6.2 Assignment Behavior
--------------------------------------------------

Preferred behavior:
- suggest a user if one clear candidate exists
- if multiple candidates exist, show reasonable selection options if existing UI supports it
- if no candidate exists, leave unassigned and allow manual user selection

Fallback behavior must be safe and explicit.

Do NOT hardcode role names in application logic.

--------------------------------------------------
6.3 Scope
--------------------------------------------------

This is suggestion / prefill behavior, not auto-routing logic.
Do not introduce workload balancing or queue routing yet.
