Implement CareConnect Step 25: notifications + reminder hooks for referrals and appointments.

Goal:
Add the first notification layer for CareConnect by implementing:
1. notification event records
2. referral status change notification hooks
3. appointment reminder scheduling hooks
4. a clean path for future SynqNotify integration

Context:
- Gateway, Identity, and CareConnect are already working
- MySQL + EF Core are configured
- CareConnect already supports:
  - Providers
  - Referrals
  - Referral status history
  - Scheduling foundation
  - AppointmentSlots
  - Appointments
  - appointment lifecycle
  - ProviderAvailabilityExceptions
  - Referral notes
  - Appointment notes
  - Attachment metadata
- Tenant context, role-based auth, audit fields, and exception handling already exist

Important:
This step should prepare and implement the first notification workflow only.
Do NOT integrate external email/SMS providers yet.
Do NOT implement actual background job infrastructure yet unless a minimal in-process approach is needed.
Do NOT implement SynqNotify integration yet.
Keep it minimal, clean, and production-minded.

==================================================
PART 2 — TRIGGER POINTS
==================================================

3. Add referral notification hooks
When referral status changes, create a CareConnectNotification row for:
- NotificationType = ReferralStatusChanged
- RelatedEntityType = Referral
- RelatedEntityId = Referral.Id
- Status = Pending

Do not send anything externally yet.
Just record notification intent cleanly.

4. Add appointment notification hooks
When appointment is:
- Scheduled
- Confirmed
- Cancelled

Create CareConnectNotification rows:
- AppointmentScheduled
- AppointmentConfirmed
- AppointmentCancelled

Use:
- RelatedEntityType = Appointment
- RelatedEntityId = Appointment.Id
- Status = Pending

5. Add reminder generation hook
When an appointment is first scheduled or confirmed, create a reminder notification row:
- NotificationType = AppointmentReminder
- RelatedEntityType = Appointment
- RelatedEntityId = Appointment.Id
- Status = Pending
- ScheduledForUtc = a reasonable reminder time before appointment, such as 24 hours before, if possible and future-dated

If the appointment is too close for that timing, use a simple fallback rule.
Keep the logic straightforward and documented.
