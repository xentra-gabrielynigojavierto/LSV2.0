# LS-COR-AUT-004 â€” Groups + Inherited Access

## Objective

Build the group-based access assignment and inherited access layer for LS-COR-AUT.

This feature must extend the direct-access model from LS-COR-AUT-002 and the effective-access/JWT projection model from LS-COR-AUT-003 by introducing:

* groups
* group membership
* group-based product access
* group-based role assignment
* inherited effective access computation
* JWT claim projection that includes inherited access

At completion, users must be able to receive access through:

* direct assignment
* group inheritance
* or both

This feature must remain backward-compatible with LS-COR-AUT-001, LS-COR-AUT-002, and LS-COR-AUT-003.

---

## Architectural Constraints

### Identity

* Do NOT implement authentication changes
* Do NOT add new login flows
* JWT issuance remains in Identity
* This feature extends access computation only

### Synq Audit

* All mutation paths introduced here must publish audit events through the centralized audit abstraction
* Do NOT create local audit tables

### Compatibility

* Must preserve compatibility with LS-COR-AUT-001 runtime filters
* Must preserve LS-COR-AUT-003 claim projection pattern
* Must not break direct-access behavior already implemented

---

## In Scope

### 1. Group model

Add the source-of-truth entities needed to support reusable access assignment through groups.

### 2. Group membership

Allow users to belong to one or more groups within a tenant.

### 3. Group product access

Allow groups to grant product access to members.

### 4. Group role assignment

Allow groups to grant roles to members.

### 5. Effective access inheritance

Update effective access computation so inherited access is merged with direct access.

### 6. JWT claim projection

Ensure inherited access is included in JWT-generated claims without breaking existing downstream enforcement.

### 7. Audit integration

All group mutations must publish Synq Audit events.

## Required Data Model

## 1. Group

Represents a reusable access-assignment container within a tenant.

Suggested fields:

* `Id`
* `TenantId`
* `Name`
* `Description` nullable
* `Status` (`Active`, `Archived`)
* `ScopeType` (`Tenant`, `Product`, `Organization`)
* `ProductCode` nullable
* `OrganizationId` nullable
* `CreatedAtUtc`
* `UpdatedAtUtc`
* `CreatedBy`
* `UpdatedBy`

Rules:

* group belongs to exactly one tenant
* name uniqueness should be enforced per tenant where practical
* `ProductCode` only allowed for product-scoped groups
* `OrganizationId` only allowed for organization-scoped groups
* archived groups must not contribute inherited access

## Effective Access Engine Changes

Extend `IEffectiveAccessService` so effective access includes both direct and inherited access.

### Required resolution model

For each user, effective access must merge:

#### Direct sources

* `UserProductAccess`
* `UserRoleAssignment`

#### Inherited sources

* `GroupMembership`
* `GroupProductAccess`
* `GroupRoleAssignment`

#### Tenant gate

All product access remains gated by active `TenantProductEntitlement`

---

## Access Resolution Rules

### Effective product access

A user has effective access to a product if:

* tenant entitlement is active
  AND
* direct user access is granted
  OR
* inherited group product access is granted through at least one active group membership

### Effective roles

A role is effective if:

* direct role assignment is active
  OR
* inherited group role assignment is active through at least one active group membership
  AND
* the related product is effectively accessible when role is product-scoped

### De-duplication

* merged product lists must be distinct
* merged role lists must be distinct
* group overlap must not produce duplicate JWT claims

### Explainability

Enhance the effective access result and/or snapshot output to show source attribution where practical, for example:

* direct
* inherited via group X

This will help with debugging and future admin UI work.

## AccessVersion / Token Freshness Requirements

Any mutation introduced in this feature that changes inherited access must increment `AccessVersion` for affected users.

This includes:

* group created only if it does not affect users yet: no increment required
* user added to group: increment that user
* user removed from group: increment that user
* group product granted/revoked: increment all active members of that group
* group role assigned/removed: increment all active members of that group
* group archived: increment all active members who lose inherited access

Implement this carefully and tenant-safely.

If bulk updates are needed, keep them correct first. Optimization can come later.

## Acceptance Criteria

This feature is complete only if all of the following are true:

1. The system has durable source-of-truth entities for:

   * groups
   * group memberships
   * group product access
   * group role assignments

2. Users can inherit product access and roles through active group membership.

3. Effective access computation merges direct and inherited access correctly.

4. JWT claims include inherited access without breaking LS-COR-AUT-001 compatibility.

5. All group-related mutations increment `AccessVersion` for affected users.

6. All group-related mutations publish Synq Audit events through the centralized abstraction.

7. Archived groups stop contributing inherited access.

8. Cross-tenant group and membership operations are rejected.

9. Existing direct access behavior remains operational and non-broken.

10. The new code builds cleanly and fits existing architecture patterns.
