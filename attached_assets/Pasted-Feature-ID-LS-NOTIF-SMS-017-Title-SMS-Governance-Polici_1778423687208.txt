Feature ID: LS-NOTIF-SMS-017

Title:
SMS Governance Policies, Compliance Controls, and Operational Guardrails

Objective:
Add enterprise governance controls to the SMS platform including tenant-level governance policies, quiet-hour enforcement, geographic delivery restrictions, provider governance rules, operational throttling, escalation guardrails, and governance audit visibility.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- SMS governance policy evaluation
- SMS governance policy persistence
- quiet-hour enforcement
- geographic delivery restrictions
- SMS rate limiting/throttling
- provider governance rules
- governance decision auditability
- governance analytics APIs

Control Center must only consume Notification Service APIs and render/administer governance visibility. Do not duplicate governance evaluation, throttling, or compliance logic in Control Center.

Requirements:
- Implement SMS governance policy evaluation inside Notification Service.
- Evaluate governance before outbound SMS execution and before automated SMS retry execution.
- Preserve existing compliance/preference enforcement.
- Preserve existing recipient intelligence/suppression behavior.
- Preserve existing provider routing/adaptive routing behavior.
- Governance must be additive and must not rewrite the delivery pipeline.
- Do not store raw phone numbers in governance persistence.
- Do not expose provider credentials, webhook URLs, SettingsJson, CredentialsJson, auth tokens, raw provider payloads, or recipient phone numbers.
- Keep Control Center as admin/visibility UX only.
- Preserve all LS-NOTIF-SMS-001 through LS-NOTIF-SMS-016 behavior.

Implementation Requirements:
1. Add SMS governance policy persistence.
2. Add governance decision persistence.
3. Add governance policy evaluation service.
4. Add quiet-hour enforcement.
5. Add geographic allow/block restrictions.
6. Add rate limiting/throttling controls.
7. Add provider allow/block governance.
8. Add retry governance controls.
9. Add escalation guardrails if safe and consistent with existing escalation architecture.
10. Add governance admin APIs.
11. Add governance analytics/audit APIs.
12. Add Control Center governance UI if feasible.
13. Ensure governance decisions degrade safely when telemetry or policy data is unavailable.

Critical Constraints:
- Reuse existing NotificationAttempt, Notification, TenantId, ProviderRoutingService, SmsRoutingEngine, SmsRecipientIntelligenceService, SmsRetrySuppressionService, SmsRegionalInferenceService, SmsOperationalAlert, SmsAlertEscalationService, audit/logging, endpoint, migration, worker, and Control Center patterns where appropriate.
- Do NOT rewrite the Notification Service delivery pipeline.
- Do NOT bypass existing contact enforcement, preference checks, recipient suppression intelligence, routing decision persistence, audit, retry/dead-letter handling, or cost recording.
- Do NOT implement AI/ML compliance scoring.
- Do NOT call external compliance, telecom, phone validation, or enrichment APIs.
- Do NOT persist raw phone numbers in governance tables.
- Do NOT expose credentials or secrets.
- Do NOT add conversational inbound SMS.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 6 — Implement Quiet-Hour Enforcement
Implement quiet-hour policy type.

Suggested PolicyJson:
{
  "timezone": "America/New_York",
  "quietStart": "21:00",
  "quietEnd": "08:00",
  "daysOfWeek": ["Monday","Tuesday","Wednesday","Thursday","Friday","Saturday","Sunday"],
  "action": "delay",
  "nextAllowedWindow": true
}

Behavior:
- If current local time is within quiet hours:
  - action = delay or block based on policy.
  - if delay, compute next allowed time.
- If outside quiet hours:
  - allow.

Requirements:
- Use tenant/global timezone from policy or default.
- Handle overnight quiet windows.
- Handle invalid timezone safely.
- Persist decision when delay/block occurs.
- Do not send SMS during delay/block.
- For delay, set NextRetryAt or equivalent deferred scheduling field where existing Notification model supports it.
- If no scheduling field exists for initial sends, document and safely block/review rather than creating unsafe delivery behavior.

STEP 7 — Implement Geographic Restrictions
Implement geographic restriction policy type.

Suggested PolicyJson:
{
  "allowedCountries": ["US", "CA"],
  "blockedCountries": ["RU", "CN"],
  "action": "block"
}

Behavior:
- Infer country using SmsRegionalInferenceService.
- If country is blocked: block.
- If allowedCountries exists and country is not listed: block or review_required.
- If country unknown: allow/review based on policy.

Requirements:
- Do not persist raw phone.
- Country inference must be transient.
- If country unavailable, degrade according to policy.
- Persist decision for block/review outcomes.

STEP 8 — Implement Rate Limiting / Throttling
Implement tenant/platform SMS rate limit policy.

Suggested PolicyJson:
{
  "windowMinutes": 60,
  "maxMessages": 500,
  "scope": "tenant",
  "action": "throttle"
}

Supported scopes:
- platform
- tenant
- provider
- tenant_provider

Behavior:
- Count SMS attempts/notifications in window using local operational data.
- If limit exceeded:
  - throttle or delay/block based on action.
- If under limit:
  - allow.

Requirements:
- Use bounded queries.
- Do not store raw phone.
- Rate evaluation must be efficient.
- Persist decisions when threshold exceeded.
- For throttle/delay, use existing scheduling fields where practical.
- If exact throttling queue not available, defer retry/send safely or document limitation.

Requirements:
