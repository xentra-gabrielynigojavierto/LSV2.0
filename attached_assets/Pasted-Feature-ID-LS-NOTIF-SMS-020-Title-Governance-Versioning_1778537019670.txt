Feature ID: LS-NOTIF-SMS-020

Title:
Governance Versioning, Bulk Rule Import, and Rule Effectiveness Analytics

Objective:
Extend the dynamic SMS governance platform with immutable governance rule versioning, bulk rule import/export, governance rollout safety controls, rule effectiveness analytics, governance rollback capability, and operational visibility into governance impact and enforcement quality.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- governance rule versioning
- immutable rule snapshots
- rule-pack versioning
- rollback orchestration
- bulk import/export validation
- governance match/effectiveness analytics
- rule lifecycle auditability
- governance analytics APIs

Control Center must only consume Notification Service APIs and render/administer governance lifecycle UX. Do not duplicate import validation, rollback, versioning, or analytics logic in Control Center.

Requirements:
- Implement governance rule/version lifecycle management inside Notification Service.
- Preserve LS-NOTIF-SMS-017 governance policies.
- Preserve LS-NOTIF-SMS-018 template governance.
- Preserve LS-NOTIF-SMS-019 dynamic rule execution.
- Versioning and analytics must extend, not replace, existing governance behavior.
- Rule changes must create immutable snapshots.
- Bulk imports must validate fully before persistence.
- Invalid imports must not partially corrupt governance state.
- Rollback must be explicit, auditable, and safe.
- Analytics aggregation must not impact delivery pipeline stability.
- Do not store raw phone numbers in analytics/versioning tables.
- Do not expose provider credentials, webhook URLs, SettingsJson, CredentialsJson, auth tokens, raw provider payloads, or recipient phone numbers.
- Keep Control Center as admin/visibility UX only.
- Preserve all LS-NOTIF-SMS-001 through LS-NOTIF-SMS-019 behavior.

Implementation Requirements:
1. Add immutable rule version persistence.
2. Add immutable rule-pack version persistence.
3. Add rule snapshot creation on create/update/disable.
4. Add rule-pack snapshot creation on create/update/disable.
5. Add explicit rollback support for rules and rule packs.
6. Add bulk CSV/JSON import validation.
7. Add bulk export support.
8. Add rule effectiveness analytics.
9. Add rule match analytics.
10. Add analytics persistence or query-time aggregation, based on current codebase fit.
11. Add admin APIs for version history, rollback, import/export, and analytics.
12. Add Control Center governance lifecycle UI if feasible.
13. Ensure delivery-time rule evaluation remains fast and stable.
14. Ensure all failures degrade safely.

Critical Constraints:
- Reuse existing SmsGovernanceRulePack, SmsGovernanceRule, SmsComplianceProfile, SmsComplianceProfileAssignment, SmsGovernanceRuleEngine, SmsGovernanceRuleResolver, SmsGovernanceSimulationService, SmsTemplateGovernanceService, SmsGovernanceDecision, SmsTemplateGovernanceDecision, admin endpoint, EF/migration, audit/logging, and Control Center dynamic-rules UI patterns where appropriate.
- Do NOT rewrite the delivery pipeline.
- Do NOT rewrite the dynamic rule engine unless required for instrumentation.
- Do NOT replace LS-017, LS-018, or LS-019 behavior.
- Do NOT implement AI/ML governance tuning.
- Do NOT call external analytics, compliance, moderation, telecom, or fraud APIs.
- Do NOT persist raw phone numbers in versioning or analytics tables.
- Do NOT expose credentials or secrets.
- Do NOT add conversational inbound SMS.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 2 — Define Rule Version Entity
Add immutable rule version snapshots.

Suggested entity:
SmsGovernanceRuleVersion

Suggested fields:
- Id Guid
- RuleId Guid
- RulePackId Guid? nullable
- VersionNumber int
- RuleSnapshotJson string
- ChangeType string
- ChangeReason string? nullable
- CreatedAt DateTime
- CreatedBy string? nullable

Suggested ChangeType values:
- created
- updated
- disabled
- rollback
- imported

Requirements:
- Snapshot must include all rule fields needed to restore the rule.
- Snapshot must not contain secrets.
- VersionNumber increments per RuleId.
- Versions are immutable.
- Use EF/migration conventions.
- Add indexes:
  - RuleId, VersionNumber
  - RulePackId, CreatedAt
  - CreatedAt

STEP 5 — Add Versioning Service
Add service to create snapshots and perform rollback.

Suggested interface:
ISmsGovernanceVersioningService

Suggested methods:
- SnapshotRuleAsync(ruleId, changeType, changeReason, requestedBy, ct)
- SnapshotRulePackAsync(rulePackId, changeType, changeReason, requestedBy, includeRules, ct)
- GetRuleVersionsAsync(ruleId, ct)
- GetRulePackVersionsAsync(rulePackId, ct)
- RollbackRuleAsync(ruleId, versionNumber, requestedBy, reason, ct)
- RollbackRulePackAsync(rulePackId, versionNumber, requestedBy, reason, ct)

Requirements:
- Snapshot before or immediately after mutations consistently; document chosen approach.
- Rollback must create a new version with ChangeType = rollback.
- Rollback must not delete version history.
- Rollback must validate target version exists.
- Rollback must restore safe fields only.
- Use transactions where available.
- Audit/log rollback.
- Failure to snapshot during mutation should fail the mutation unless existing patterns require best-effort; prefer fail-safe.

STEP 7 — Add Bulk Import Service
Add import validation and persistence service.

Suggested interface:
ISmsGovernanceImportService

Suggested methods:
- ValidateImportAsync(request, ct)
- ImportAsync(request, requestedBy, ct)
- ExportAsync(query, ct)

Suggested import formats:
- JSON
- CSV

Suggested JSON structure:
{
  "rulePack": {
    "tenantId": null,
    "name": "Default prohibited phrases",
    "inheritanceMode": "merge",
    "priority": 100
  },
  "rules": [
    {
      "ruleType": "prohibited_phrase",
      "name": "No gambling terms",
      "pattern": "casino",
      "severity": "block",
      "priority": 10,
      "metadata": {}
    }
  ]
}

Suggested CSV columns:
- rulePackName
- tenantId
- inheritanceMode
- ruleType
- name
- pattern
- severity
- priority
- enabled
- metadataJson

Requirements:
- Validate entire import before writing anything.
- Use transaction for persistence.
- Reject invalid rule types.
- Reject invalid severities.
- Reject unsafe regex patterns.
- Reject oversized patterns.
- Reject invalid metadata JSON.
- Return row-level validation errors.
- Do not partially import invalid files.
- Do not store secrets.
- Import should support dry-run validation mode.
- Import should create versions/snapshots.
