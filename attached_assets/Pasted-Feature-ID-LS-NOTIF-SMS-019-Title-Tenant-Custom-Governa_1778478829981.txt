Feature ID: LS-NOTIF-SMS-019

Title:
Tenant Custom Governance Rules, Dynamic Policy Packs, and Compliance Rule Management

Objective:
Extend the SMS governance platform with configurable tenant-specific governance rule packs, prohibited phrase dictionaries, compliance profiles, dynamic enforcement severity controls, governance simulation/testing, rule versioning, policy inheritance, and governance rule analytics without requiring code changes.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- dynamic SMS governance rule evaluation
- rule-pack persistence
- compliance profile assignment
- rule inheritance resolution
- governance simulation/testing
- prohibited phrase dictionaries
- dynamic enforcement severity
- rule analytics APIs

Control Center must only consume Notification Service APIs and render/administer governance rule management UX. Do not duplicate rule evaluation, inheritance, classification, or simulation logic in Control Center.

Requirements:
- Implement dynamic governance rules inside Notification Service.
- Preserve LS-NOTIF-SMS-017 governance policies.
- Preserve LS-NOTIF-SMS-018 template governance and content classification.
- Dynamic rules must extend, not replace, existing governance.
- Support global/platform rule packs and tenant-specific rule packs.
- Support rule inheritance: global baseline plus tenant override/merge.
- Support configurable enforcement severity: allow, warn, review_required, block, override_allowed.
- Support governance simulation/testing without sending SMS.
- Do not store raw phone numbers in new governance-rule tables.
- Do not expose provider credentials, webhook URLs, SettingsJson, CredentialsJson, auth tokens, raw provider payloads, or recipient phone numbers.
- Keep Control Center as admin/visibility UX only.
- Preserve all LS-NOTIF-SMS-001 through LS-NOTIF-SMS-018 behavior.

Implementation Requirements:
1. Add SMS governance rule-pack persistence.
2. Add SMS governance rule persistence.
3. Add SMS compliance profile persistence.
4. Add tenant/profile assignment support.
5. Add dynamic rule inheritance resolver.
6. Add dynamic governance rule engine.
7. Add prohibited phrase dictionary support.
8. Add restricted regex/pattern support with safety controls.
9. Add variable-rule support for template payloads.
10. Add URL/domain rule support where feasible.
11. Add governance simulation/dry-run service.
12. Add admin APIs for rule packs, rules, profiles, assignments, and simulation.
13. Add Control Center governance rule management UI if feasible.
14. Add governance rule analytics/audit visibility.
15. Ensure dynamic rule failures degrade safely.

Critical Constraints:
- Reuse existing SmsGovernancePolicy, SmsGovernanceDecision, SmsTemplateGovernanceService, SmsTemplateGovernanceDecision, Notification, NotificationAttempt, routing/governance pipeline, admin endpoint, EF/migration, audit/logging, and Control Center governance UI patterns where appropriate.
- Do NOT rewrite the Notification Service delivery pipeline.
- Do NOT bypass existing preference/compliance enforcement.
- Do NOT bypass LS-017 governance policies.
- Do NOT bypass LS-018 template governance.
- Do NOT implement AI/LLM rule generation or moderation.
- Do NOT call external compliance, telecom, phone validation, moderation, or fraud APIs.
- Do NOT persist raw phone numbers in rule tables or decision tables.
- Do NOT expose credentials or secrets.
- Do NOT add conversational inbound SMS.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 6 — Add Rule Inheritance Resolver
Add rule-pack/profile resolver.

Suggested interface:
ISmsGovernanceRuleResolver

Suggested methods:
- ResolveRulesAsync(Guid? tenantId, string? context, CancellationToken ct)
- ResolveRulePacksAsync(Guid? tenantId, CancellationToken ct)

Behavior:
- Load active global packs.
- Load active tenant packs.
- Apply InheritanceMode:
  - merge = global + tenant
  - override = tenant replaces global for matching rule types or pack context
  - append_only = global evaluated first, tenant evaluated after
- Order by Priority.
- Exclude disabled/inactive/expired packs/rules.
- Respect compliance profile assignment where implemented.

Requirements:
- Deterministic resolution order.
- Safe fallback to global packs.
- If resolver fails, rule engine should fail open or according to config.
- Document inheritance behavior.

STEP 8 — Implement Rule Types
Implement initial rule types.

prohibited_phrase:
- Case-insensitive phrase matching.
- Whole-word option where MetadataJson specifies it.
- block/review/warn based on Severity.

restricted_pattern:
- Safe regex only.
- Enforce max pattern length.
- Enforce timeout if regex engine supports it.
- Reject catastrophic patterns where practical.
- If regex unsafe, disable/reject rule at creation.

classification_override:
- If content matches metadata criteria, override classification to configured value.
- Useful for tenant-specific category handling.

variable_rule:
- Validate variable names, disallowed variables, required variables, or value patterns.
- Should integrate with LS-018 variable validation.

link_rule:
- Detect URLs/domains.
- Block/warn/review based on allowedDomains/blockedDomains metadata.

delivery_restriction:
- Allow dynamic content/context restrictions without changing LS-017 policy schema.
- Example: block after-hours escalation text or specific category.

escalation_rule:
- Apply dynamic governance to escalation payloads if low-risk.
- If integration too risky, implement rule type and document escalation integration as future.
