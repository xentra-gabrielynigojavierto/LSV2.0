==================================================
OBJECTIVE
==================================================

Implement the minimum viable CareConnect referral experience with email-based provider routing and basic dashboard analytics.

This feature must allow a law firm user to:
1. search providers on a map or list
2. click Refer
3. complete a referral entry screen that shows provider context in the header
4. submit the referral in Pending status
5. trigger a provider email notification with a View Referral button
6. route the provider to the correct acceptance experience depending on whether the provider is:
   - a pending provider without activated tenant access
   - an active tenant provider with portal access
7. send confirmation emails when the referral is accepted
8. show basic dashboard analytics for referral activity

This is a deliberately minimized feature.
Do NOT expand into full workflow management, appointment lifecycle redesign, advanced analytics, or broad onboarding implementation.

==================================================
SCOPE (STRICT)
==================================================

You are ONLY implementing:

A. Law firm provider search initiation from map/list
B. Refer button behavior from provider search results
C. Referral entry UI with provider info in header
D. Referral creation with Pending status
E. Provider email notification with View Referral CTA
F. Secure referral-view routing based on provider tenant state
G. Pending-provider acceptance page with activation-benefits content
H. Active-tenant provider deep link into authenticated portal referral page
I. Acceptance confirmation emails to provider + referrer
J. Basic dashboard analytics cards
K. Report

Do NOT implement:
- appointment booking
- referral timeline
- advanced provider inbox workflows
- full onboarding flow
- advanced analytics
- charts/funnels/provider rankings
- exports
- messaging/chat
- case integration
- billing/lien integrations

==================================================
ARCHITECTURE RULES (MANDATORY)
==================================================

1. Reuse the existing CareConnect architecture and routes where possible
2. Preserve backend authorization and org scoping rules already implemented
3. Separate:
   - UI components
   - API / service layer
   - email / routing logic
   - analytics computation
4. Do NOT duplicate business rules in multiple places
5. Do NOT guess provider-to-tenant routing; use deterministic logic only
6. Do NOT silently fail email operations
7. Keep implementation minimal, production-safe, and compile-friendly
8. If a fallback is required, it must be explicit and documented

==================================================
REQUIREMENTS
==================================================

--------------------------------------------------
A. LAW FIRM PROVIDER SEARCH → REFER ENTRY POINT
--------------------------------------------------

Law firm users must be able to start a referral from both:

1. provider map results
2. provider list results

Requirements:
- the Refer action must behave consistently regardless of entry point
- if provider search already exists, reuse it
- if current map/list items do not surface a Refer button, add it minimally

Behavior:
- clicking Refer opens the referral entry experience
- provider context must be preserved and visible

If provider email is missing or invalid:
- do NOT allow referral submission
- show a clear error or disabled state explaining the provider cannot currently receive referrals

--------------------------------------------------
B. REFERRAL ENTRY SCREEN / MODAL
--------------------------------------------------

Implement a referral entry screen, modal, or drawer using the lightest approach that fits the current architecture.

Requirements:
1. Header must show provider information
   Minimum:
   - provider name
   - provider email if useful
   - optional location/basic metadata if already available

2. Referral form must collect the minimum required information for the referred party and referrer context.

Minimum recommended fields:
- referred party full name
- referred party phone
- referred party email
- referral notes / reason for referral

Referrer context:
- law firm name
- referrer name
- referrer email

Rules:
- derive referrer/law firm information from current authenticated context where possible
- only ask the user for fields that are truly required and not already known

Validation:
- required fields must validate inline
- invalid submission must not destroy entered data
- submission errors must be user-visible

--------------------------------------------------
C. REFERRAL CREATION
--------------------------------------------------

On successful referral submission:

1. create the referral record
2. set referral status = Pending
3. persist provider link and referrer information
4. trigger provider notification email workflow

Status model for this feature:
- Pending
- Accepted

Rules:
- all new referrals created by this flow must begin in Pending
- do not auto-accept
- do not auto-book appointments
- if existing backend status models differ, add the narrowest compatibility layer required and document it

Success UX:
- show clear success state/message
- redirect or land on a meaningful next page after submit

Fallback behavior:
- if referral save fails, email must not be sent
- if referral save succeeds but email send fails, referral remains created and the email failure must be logged/surfaced for retry handling per existing conventions

--------------------------------------------------
D. PROVIDER NOTIFICATION EMAIL
--------------------------------------------------

When a referral is successfully created, send an email to the provider email address.

Email must include:
- provider name
- referred party information
- referrer / law firm information
- referral summary / notes
- View Referral button

Subject line should clearly indicate a new referral has been received.

The View Referral button must use a secure link strategy.
Do NOT expose an insecure raw record link without protection.

Preferred strategies:
- signed tokenized link
- time-bound secure token
- equivalent secure access mechanism consistent with existing platform patterns

Document the chosen approach in the report.

--------------------------------------------------
E. PROVIDER STATE DETECTION & ROUTING
--------------------------------------------------

When the provider uses the View Referral link, backend logic must determine whether the provider is:

1. Pending Provider
   - exists in provider data
   - does not have activated tenant access

2. Active Tenant Provider
   - has active tenant account / authenticated portal path

This classification must be deterministic.

Rules:
- do NOT guess
- if provider state is ambiguous, route to a safe failure/manual handling state and document it
- do NOT widen access for unresolved state

--------------------------------------------------
F. PENDING PROVIDER EXPERIENCE
--------------------------------------------------

If the provider is pending, show a lightweight web page for referral review and acceptance.

This page must include:
- referral summary
- referred party information
- referrer / law firm information
- Accept Referral button
- a section describing the benefits of activating tenant access

The benefits section should be lightweight, not a full onboarding implementation.

Examples of acceptable benefit messaging:
- manage referrals in a portal
- track status updates
- centralize referral operations
- improve workflow visibility

Acceptance behavior:
- if provider clicks Accept Referral:
  1. referral status becomes Accepted
  2. send confirmation email to provider
  3. send confirmation email to referrer

Edge cases:
- if referral is already accepted, show accepted state and do not allow duplicate acceptance
- if secure token is invalid or expired, show safe error state
- if pending provider acceptance requires no login, secure token access must be strong enough to avoid insecure referral exposure

--------------------------------------------------
G. ACTIVE TENANT PROVIDER EXPERIENCE
--------------------------------------------------

If the provider is an active tenant provider:

1. View Referral must deep link into the provider portal
2. if provider is not logged in, require login first
3. after login, redirect to the referral page inside the portal
4. portal page must show referral information and Accept Referral action

Requirements:
- preserve destination across login
- use existing authenticated routing patterns where possible
- do not create a second duplicate acceptance flow if one already exists in the provider portal; adapt or minimally extend the existing referral detail flow

Acceptance behavior:
- accepting from the portal sets status = Accepted
- provider confirmation email sent
- referrer confirmation email sent

Edge cases:
- already accepted referral shows accepted state
- unauthorized access must not reveal referral details

--------------------------------------------------
H. ACCEPTANCE CONFIRMATION EMAILS
--------------------------------------------------

When a referral is accepted, send confirmation emails to:

1. provider
2. referrer / law firm email

Email content must clearly confirm:
- referral accepted
- provider identity
- referred party identity/summary
- law firm/referrer identity
- relevant next-step language if appropriate

Rules:
- avoid duplicate confirmation sends on repeated clicks or duplicate acceptance attempts
- if acceptance succeeds and email fails, keep Accepted status and log/surface the email issue for retry handling

--------------------------------------------------
I. BASIC DASHBOARD ANALYTICS
--------------------------------------------------

Add a lightweight dashboard section for referral activity only.

Route:
- /careconnect/dashboard

Section title example:
- Referral Activity Overview

Minimum metrics:
1. Total Referrals
2. Pending Referrals
3. Accepted Referrals
4. Acceptance Rate = Accepted / Total

Scope rules:
- law firm sees only own org-scoped referrals
- provider sees only received/inbound referrals
- tenant admin sees tenant-wide numbers per current conventions

Date behavior:
- default to Last 30 Days only
- do NOT add custom date picker in this feature
- do NOT add advanced filters
- if a simple 7d / 30d toggle already fits cleanly, you may include it only if low-risk and documented; otherwise keep fixed 30d

Display:
- simple metric cards only
- no charts
- no funnel graphics
- no provider rankings
- no advanced appointment analytics

Fallback behavior:
- if analytics fetch fails, dashboard still loads and analytics cards show safe fallback state
- if no data, show clear empty state such as "No referral activity yet"

Data strategy:
- prefer existing referral endpoints
- aggregate in frontend or dashboard server layer
- avoid heavy query patterns
- do not add a broad analytics backend unless clearly necessary
