FEATURE ID
TENANT-B12 — Full Lifecycle Ownership
================================================================

OBJECTIVE
Make the Tenant service the true lifecycle owner of tenant management while reducing Identity to an authentication, session, admin-user, role-assignment, and provisioning support service.

This block must:
1. move canonical tenant creation ownership to the Tenant service,
2. make Tenant the primary owner of tenant entitlements at the platform level,
3. introduce controlled Tenant → Identity orchestration for downstream auth/provisioning work,
4. reduce reliance on Identity-first create/update flows,
5. preserve rollback compatibility and staged cutover safety.

This block must NOT:
- drop Identity tenant tables/columns,
- remove Identity compatibility endpoints,
- move session/auth enforcement out of Identity,
- perform destructive cleanup.

SCOPE
================================================================

A) Make Tenant service the canonical owner of CreateTenant

Introduce a Tenant-first tenant creation flow.

Preferred pattern:
1. Control Center / BFF calls Tenant service admin create endpoint
2. Tenant service persists the canonical tenant record first
3. Tenant service calls Identity adapter to perform downstream work:
   - create admin user / tenant admin context
   - assign roles / scoped permissions
   - initialize required auth-side associations
   - trigger DNS/product provisioning hooks if applicable
4. Tenant service returns an explicit lifecycle/provisioning result

Recommended new/updated endpoint:
- POST /api/v1/admin/tenants

This endpoint must become the canonical create path for tenant lifecycle ownership.

Rules:
- Tenant record must be created in Tenant service first
- Identity must no longer be the authoritative creator of tenant master data
- downstream Identity/provisioning work must be explicit and controlled

B) Define create-tenant transactional model explicitly

You MUST choose and implement one clear failure model.

Recommended safe model:
- staged / compensating model, NOT distributed transaction

Example:
1. Tenant record created with status `Pending`
2. Identity provisioning starts
3. If Identity succeeds:
   - Tenant record transitions to `Active` (or remains current valid status)
4. If Identity fails:
   - Tenant record remains `Pending` or `ProvisioningFailed`
   - failure is returned with enough detail for operator remediation
   - no silent partial success

Do NOT attempt a fragile distributed transaction across services.

You may need to extend Tenant status handling slightly if a more explicit provisioning failure state is needed.
If you add a new status such as `ProvisioningFailed`, document it carefully and ensure backward compatibility.

C) Introduce Tenant → Identity provisioning adapter for lifecycle creation

Build a real Tenant-side adapter for Identity lifecycle dependencies.

Recommended interfaces:
- IIdentityProvisioningAdapter
- IIdentityTenantLifecycleAdapter
or similar

Possible operations:
- CreateTenantAdminContextAsync(...)
- ProvisionTenantAuthContextAsync(...)
- AssignTenantAdminRolesAsync(...)
- TriggerTenantProvisioningAsync(...)
- MirrorAuthRelevantTenantStateAsync(...)

Implementation:
- HTTP/internal service API calls only
- no direct DB access
- clear timeout handling
- structured error handling
- logs with correlation/context

Rules:
- adapter must be feature-flagged or safe by default
- failures must be visible and auditable
- do not hide provisioning failure behind silent success

D) Move platform-level entitlement ownership to Tenant

Tenant should own platform-level product entitlements as the source of truth.

In this block:
- entitlement write APIs used by Control Center should target Tenant service first
- Identity may still consume mirrored entitlement state if required for auth/product access checks
- if Identity still requires local entitlement knowledge, update via adapter/sync rather than remaining canonical owner

Implement/update Tenant endpoints if needed:
- POST /api/v1/admin/tenants/{id}/entitlements
- PUT /api/v1/admin/tenants/{id}/entitlements/{entitlementId}
- DELETE /api/v1/admin/tenants/{id}/entitlements/{entitlementId}
or reuse/normalize existing endpoints already built in earlier blocks

Control Center BFF entitlement routes should be switched to Tenant where appropriate.

Rules:
- Tenant = canonical owner
- Identity = consumer or mirrored dependency
- do not break current product access checks
- if a compatibility sync to Identity is required, implement it explicitly

E) Preserve Identity ownership only where still truly appropriate

The following are expected to remain in Identity in B12 unless proven otherwise:
- session timeout enforcement
- auth/login gating
- admin-user creation mechanics
- role assignment mechanics
- provisioning retry state machine
- DNS / tenant verification if still Identity-owned

Where these remain:
- Tenant should orchestrate them via adapter calls
- not duplicate their full ownership logic
- not create direct DB coupling

Transition BFF/server routes so they now target Tenant for canonical tenant lifecycle writes where applicable.

At minimum review and switch if safe:

Keep in Identity only what still belongs there:

Rules:

H) Keep Identity compatibility endpoints and rollback path intact

Do NOT remove:
- Identity compatibility endpoints
- Identity branding fallback endpoints

But:
- mark old create/entitlement paths as compatibility-only if they are no longer primary
- add deprecation warnings/headers where appropriate
