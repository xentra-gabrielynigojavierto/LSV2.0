==================================================
PROJECT: NOTIF-UI-008 — Tenant Template Override
==================================================

GOAL:
Allow tenant users to create and manage tenant-specific template overrides for product-scoped notification templates, while preserving the platform’s global template and branding architecture.

This phase adds:
- tenant override list/view
- create tenant override flow
- edit tenant override draft flow
- branded preview for tenant override
- publish tenant override
- fallback visibility to global template when no override exists

This phase is tenant-facing, but still controlled.

Do NOT overbuild.
Do NOT expose Control Center capabilities.
Do NOT allow tenants to edit global templates directly.
Do NOT break product scoping.
Do NOT redesign the tenant portal shell.

==================================================
CRITICAL DOMAIN RULES (MANDATORY)
==================================================

1. Templates are ALWAYS product-scoped
2. Tenant overrides are ALWAYS product-scoped
3. A tenant override never edits the global template directly
4. Tenant overrides layer on top of the global template model
5. Branding tokens remain system-controlled and cannot be overridden by tenant content
6. Tenant users must never see or affect other tenants' overrides
7. Product selection must happen BEFORE templates are displayed

Required conceptual model:

Global Template (product-scoped)
    ↓
Tenant Override (same product + same template key)
    ↓
Tenant Branding
    ↓
Render

The UI must reflect this clearly.

  - no override

If no tenant override exists:
- show "Using Global Template"
- show a clear CTA:
  - Create Override

==================================================
4. TENANT OVERRIDE CREATE FLOW
==================================================

Allow tenants to create an override draft for a specific global template.

Expected backend capability:
- create tenant-scoped override template/version associated with:
  - templateKey
  - productType
  - channel
  - tenantId

Preferred endpoint patterns (adapt if backend differs):
- POST /notifications/v1/templates/tenant-overrides
or
- POST /notifications/v1/templates/global/:id/tenant-overrides

Requirements:
- create flow must start from a specific global template
- productType comes from route/context, not user freeform input
- tenantId comes from session, never user input
- initial content should be pre-populated from the current global template version
- tenant can edit allowed content fields:
  - subject
  - body
  - text
  - editorJson if WYSIWYG-backed
- branding tokens remain reserved/system-owned

On success:
- override draft exists
- UI reflects override status
- user is taken to override edit/manage view

==================================================
5. TENANT OVERRIDE EDIT FLOW
==================================================

Allow editing of tenant override drafts only.

Expected backend capability:
- PATCH tenant override draft/version

Requirements:
- only editable if override is still a draft
- published overrides should not be edited directly unless backend lifecycle explicitly supports draft-after-publish
- if backend uses immutable version lifecycle, preserve that truthfully:
  - editing published override should require creating a new draft version

The UI must present lifecycle honestly.
Do NOT imply mutable published content if that is false.

==================================================
6. WYSIWYG-COMPATIBLE OVERRIDE EDITOR
==================================================

Tenant override editing must support the same constrained content model as the admin/global template system where appropriate.

Requirements:
- if the template/editorType is WYSIWYG:
  - use a constrained WYSIWYG-compatible editor
  - preserve editorJson + compiled HTML/body + text fallback
- if the template/editorType is html:
  - use structured textareas for HTML/text
- do NOT expose arbitrary script or unsafe content editing

Tenant users may customize content, but:
- must not override brand tokens themselves
- must not alter product scope
- must not change template identity fields like templateKey/productType/channel in unsafe ways

==================================================
7. TENANT OVERRIDE PREVIEW
==================================================

Provide preview for tenant overrides using real backend rendering.

Expected backend capability:
- preview endpoint for tenant override render
- if not available, use a compatible preview route that resolves tenant override + branding correctly

Requirements:
- preview must use:
  - tenantId from session
  - productType from route
  - override content
  - template variables/sample data
- preview result must show:
  - subject
  - rendered HTML
  - text version
  - branding indicator

If no tenant-specific branding exists:
- show that default branding is applied

Do NOT simulate preview if the backend already supports real rendering.

==================================================
8. TENANT OVERRIDE PUBLISH FLOW
==================================================

Allow tenants to publish an override draft.

Expected backend capability:
- publish tenant override version

Requirements:
- explicit publish action
- confirmation required
- clearly indicate after success:
  - tenant override is now active
  - tenant is no longer using global-only for that template
- if prior tenant override version exists, reflect lifecycle truthfully

Do NOT let tenants publish accidentally.

==================================================

Requirements:

- if tenant override is active:
  - label: Using Tenant Override
- if backend supports revert/deactivate override safely, you may expose it
- if not supported, do NOT fake it

At minimum, the UI must make fallback state visible.
