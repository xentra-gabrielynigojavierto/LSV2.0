- Internal monitoring is feature-complete for operator use
- `/api/monitoring/summary` already exists and is public-safe
- Internal monitoring page includes banner, component list, alert detail, resolve flow, and history
- This feature is public presentation only, not admin capability

- `/status` is accessible without authentication

STEP 3 — DESIGN THE PUBLIC STATUS PAGE

Create a simple, external-facing page at:

- /status

The page must include:

A. Header section
- title: "System Status"
- short subtitle such as "Current platform availability"
- last updated timestamp from `summary.system.lastCheckedAtUtc`

B. Overall status section
- prominently display `summary.system.status`
- color map:
  - Healthy → green
  - Degraded → amber/yellow
  - Down → red

C. Component status list
- all components from `summary.integrations`
- sort order:
  - Down
  - Degraded
  - Healthy
  - then alphabetical by name
- show only safe fields:
  - component name
  - status
  - optionally last checked
  - optionally latency if it is already shown publicly and is not judged too internal

D. Active incidents section
- show only if `summary.alerts.length > 0`
- for each alert show:
  - component name if derivable safely
  - message
  - severity
  - triggered timestamp
- do NOT show internal IDs or operator-only metadata

Design constraints:
- no authenticated shell / no internal navigation
- no admin controls
- no resolve buttons
- no filter controls required
- clean, calm, readable layout
- external-facing tone and presentation

Document the chosen layout and why it is appropriate.
