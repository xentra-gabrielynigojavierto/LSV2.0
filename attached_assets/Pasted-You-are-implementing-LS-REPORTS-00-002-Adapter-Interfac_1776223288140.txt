# STORY ID
**LS-REPORTS-00-002 — Adapter Interface Hardening**

---

# OBJECTIVE
Evolve the current adapter layer from mock-friendly scaffolding into production-ready integration contracts that define how the Reports Service interacts with external platform services while remaining fully portable.

This story must establish:
- clear, typed adapter contracts
- context models (tenant, user, product, request)
- standard request/response patterns
- structured adapter result contracts
- error handling model
- correlation propagation through the adapter layer
- updated mock implementations that match the stronger contracts

This is a boundary-definition story, not a persistence story.

---

# SCOPE

## In Scope
- adapter interface redesign with strong typing
- shared context models
- standard adapter result wrapper
- adapter error model
- correlation propagation model
- update mock adapter implementations
- update dependency injection wiring
- preserve health/readiness behavior

## Out of Scope
- real adapter integrations
- database/MySQL integration
- report templates
- report execution
- scheduling
- UI
- authentication/authorization middleware

---

# REQUIRED IMPLEMENTATION

## A. Context Models
Create standardized context models to be reused across adapters.

### Required models
At minimum:

#### `RequestContext`
Fields:
- `CorrelationId`
- `RequestId`
- `Timestamp`
- optional metadata dictionary

#### `UserContext`
Fields:
- `UserId`
- `Email` (optional)
- `Roles` (collection)
- `IsPlatformAdmin`

#### `TenantContext`
Fields:
- `TenantId`
- `TenantName` (optional)
- `OrganizationType`
- `IsActive`

#### `ProductContext`
Fields:
- `ProductCode`
- `ProductName` (optional)

### Rules
- place these in a reusable Contracts or Shared location
- they must be portable and platform-agnostic
- adapter methods should explicitly accept them where applicable

---

## B. Standard Adapter Result Model
Introduce a uniform typed result wrapper for all adapter responses.

### Required type
Create something equivalent to:

- `AdapterResult<T>`

Fields:
- `Success`
- `Data`
- `ErrorCode`
- `ErrorMessage`
- `IsRetryable`
- optional metadata dictionary

### Rules
- all adapter methods must return `AdapterResult<T>`
- do not return raw objects or primitives directly from adapter interfaces
- keep the wrapper lightweight and clear

---

## C. Error Model
Define a structured adapter error model.

This can be:
- constants
- enum
- typed error object
- or a combination

It must support at least:
- not found
- unauthorized
- forbidden
- unavailable
- invalid request
- timeout
- unknown error

Also support:
- retryable vs non-retryable distinction

Keep this general and reusable.

---

## D. Adapter Interface Redesign
Refactor all existing adapter interfaces to use:
- strongly typed inputs
- explicit context objects
- `AdapterResult<T>` return types

### Required adapters
- `IIdentityAdapter`
- `ITenantAdapter`
- `IEntitlementAdapter`
- `IAuditAdapter`
- `IDocumentAdapter`
- `INotificationAdapter`
- `IProductDataAdapter`

### Requirements
- keep methods minimal but sensible
- do not overbuild
- avoid vague `object` parameters
- use typed request/response models where useful
- support future real implementations without redesign

---

## E. Correlation & Traceability
Ensure the adapter layer supports traceability.

### Requirements
- `CorrelationId` must flow through adapter calls
- mock adapters should log the correlation id
- request context should be part of method signatures where appropriate

Do not build a full tracing system; just establish the contract properly.

---

## F. Mock Adapter Upgrade
Update all mock adapters to:
- match the new interface signatures
- return structured `AdapterResult<T>` responses
- log correlation id
- simulate clean success responses
- optionally simulate simple structured failure paths if easy and low-risk

Do not create complex fake behavior.

---

## G. Dependency Injection Updates
Update DI and service registration so:
- all adapter implementations are registered correctly
