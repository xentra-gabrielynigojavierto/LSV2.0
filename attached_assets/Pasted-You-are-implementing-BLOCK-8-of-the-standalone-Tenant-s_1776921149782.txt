SCOPE
================================================================

A) Runtime observability hardening

Upgrade observability from log-only to operationally useful metrics/diagnostics.

At minimum add one or more of the following where practical:
- per-source read counters:
  - Tenant reads attempted
  - Tenant reads succeeded
  - Identity fallbacks triggered
  - Tenant read failures
- per-failure counters:
  - timeout
  - transport error
  - 404 not found
  - incomplete payload
- dual-write counters:
  - sync attempts
  - sync success
  - sync failure
  - strict-mode failures

Implement these in the safest way that fits current architecture:
- in-memory counters are acceptable if persistent metrics infra does not exist yet
- admin/internal diagnostics endpoints are acceptable
- structured log summaries are acceptable as a supplement, not a replacement

Recommended admin/internal diagnostics endpoint(s), protected:
- GET /api/v1/admin/read-source-metrics
- GET /api/v1/admin/dual-write-metrics
or a consolidated:
- GET /api/v1/admin/runtime-metrics

Requirements:
- do NOT expose these publicly
- counters/metrics should be safe and non-sensitive
- document reset behavior if counters are process-memory only

B) Caching / performance hardening

Add safe caching for high-frequency read paths.

Target paths:
- public branding lookup
- public tenant resolution lookup
- web BFF `/api/tenant-branding`
- public logo route metadata lookup if appropriate

Requirements:
- cache only safe-to-cache read results
- include TTLs
- define cache key strategy clearly
- preserve correctness with fallback behavior

Recommended minimal approach:
- in-memory cache per service/process if no distributed cache exists
- short TTLs (for example 30–120 seconds depending on path)
- negative caching optional but must be conservative
- ability to disable caching via config if needed

Possible config examples:
- Features:TenantReadCachingEnabled = true/false
- Features:TenantReadCacheTtlSeconds = 60

Rules:
- cache must not make rollback impossible
- avoid caching admin-write responses
- avoid stale poisoning when dual-write updates occur; keep TTL modest
- document invalidation approach (TTL-based is acceptable in this block)

This block must produce a concrete and code-aware retirement plan for tenant fields still stored in Identity.

3. where safe, mark code paths as deprecated / compatibility only

Examples of candidate fields/concerns in Identity:
- Name
- Code

- Subdomain

Expected result:
