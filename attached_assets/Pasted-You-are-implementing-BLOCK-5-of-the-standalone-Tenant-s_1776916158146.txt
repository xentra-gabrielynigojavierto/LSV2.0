A) Upgrade migration utility from dry-run to explicit write mode

The migration utility currently supports dry-run reconciliation.
In this block, extend it to support explicit write-capable migration execution.

Requirements:
- Migration execution must be EXPLICIT, never automatic on startup
- Dry-run must remain the default and safest mode
- Write mode must require an explicit parameter or endpoint/action
- Write mode must be idempotent and re-runnable

Recommended interface options:
- GET /api/admin/migration/dry-run
- POST /api/admin/migration/execute
or
- POST /api/admin/migration/run?dryRun=false

Choose the cleanest API shape consistent with the existing Tenant admin endpoint style.

Write mode must:
1. Read Identity tenant data via IdentityDb read-only connection
2. Upsert tenants into Tenant service persistence
3. Preserve TenantId exactly
4. Upsert/update core profile fields
5. Upsert/update branding basics where mapped
6. Upsert/update domain/subdomain compatibility data as appropriate
7. Avoid deleting Tenant-side data in this block unless deletion is explicitly safe and documented
8. Produce a structured execution report/result

B) Migration scope for this block

Implement migration for the fields/entities that already exist in Tenant service.

Minimum migration scope:
1. Tenant core/profile
- Id
- Code
- DisplayName/Name
- Status
- LegalName if derivable, otherwise leave as-is/null
- Subdomain
- Website/contact/profile fields only if available from Identity source
- timestamps where appropriate

2. Branding basics
- LogoDocumentId
- LogoWhiteDocumentId
- create/update TenantBranding row as needed
- primary color only if available from source; otherwise leave existing Tenant values intact

3. Domain compatibility
- For Identity tenants with Subdomain, ensure the Tenant service has a canonical TenantDomain record when appropriate:
  - Host derived from subdomain + current base domain strategy if determinable from existing configuration
  - OR create a migration-compatible domain record only when the host derivation is explicit and safe
- If host derivation cannot be safely determined in the current environment, document the limitation and at minimum preserve Tenant.Subdomain

Do NOT invent unverifiable domain hosts.
Do NOT migrate speculative custom domains.

4. Product entitlements / settings / capabilities
- Do NOT force-migrate these unless Identity already contains a direct equivalent that is clearly mapped and safe
- In most cases for this block, these remain untouched
- If there are safe defaults needed, keep them explicit, minimal, and documented

C) Idempotency rules

Migration execution must be safely repeatable.

Rules:
- If Tenant with same Id already exists, update it rather than duplicate it
- If Tenant with same Code exists but different Id, detect and report conflict explicitly
- Branding row should be upserted, not duplicated
- Domain rows created by migration should not duplicate on repeat runs
- Re-running migration should converge toward the same result, not diverge

D) Pre-migration validation guards

Before write mode runs, validate:
- IdentityDb connection available
- TenantDb connection available
- no duplicate Identity tenant codes in source
- no code conflicts in Tenant target that would break exact-id migration
- any invalid or missing required Identity tenant fields are surfaced in the result
- host/subdomain collisions are surfaced before writes where applicable
- migration can be scoped to a single tenant or all tenants

Recommended support:
- full migration run
- single-tenant migration by tenantId or code
- dry-run before execute strongly encouraged

If full implementation of both scopes is too large, prioritize:
1. all tenants
2. single tenant by id
and document the rest.
