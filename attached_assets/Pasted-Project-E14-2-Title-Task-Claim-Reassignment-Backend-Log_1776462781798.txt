OBJECTIVE

- claim tasks from role/org queues
- reassign tasks between users
- (optionally) release tasks back to a queue

while enforcing:
- strict assignment mode invariants (E14.1)
- tenant isolation
- safe concurrency behavior
- non-interference with workflow execution

This phase introduces **ownership transitions**, not workflow transitions.

IMPLEMENTATION REQUIREMENTS

1. CLAIM TASK

Endpoint:
POST /api/v1/workflow-tasks/{id}/claim

Behavior:
- user claims a task from a queue (RoleQueue or OrgQueue)

Requirements:
- allowed only if AssignmentMode is RoleQueue or OrgQueue
- task must be in actionable state (Open or InProgress)
- task must not already be owned by a user

On success:
- AssignmentMode → DirectUser
- AssignedUserId → current user
- AssignedAt → now
- AssignedBy → current user

Response:
- updated task state

Failure cases:
- already claimed → 409
- invalid mode → 422
- unauthorized → 403
- not found → 404

---

2. REASSIGN TASK

Endpoint:
POST /api/v1/workflow-tasks/{id}/reassign

Request body:
{
  "targetUserId": "...",
  "reason": "optional"
}

Behavior:
- transfer ownership from one user to another

Requirements:
- allowed only if AssignmentMode = DirectUser
- caller must be authorized (see Authorization section)
- cannot reassign to null

On success:
- AssignedUserId → targetUserId
- AssignedAt → now
- AssignedBy → current user
- AssignmentMode remains DirectUser

Optional:
- record reason if field exists

Failure cases:
- invalid state → 422
- unauthorized → 403
- not found → 404

---

3. RELEASE / UNCLAIM (OPTIONAL BUT RECOMMENDED)

Endpoint:
POST /api/v1/workflow-tasks/{id}/release

Behavior:
- move task from DirectUser back to queue

Requirements:
- must remember original queue (role/org)
- only allowed if task was previously from a queue
- AssignmentMode → RoleQueue or OrgQueue
- AssignedUserId → null

If original queue not stored:
- document fallback behavior or defer feature

---

4. ASSIGNMENT TRANSITION RULES

Define strict allowed transitions:

| From | To | Action |
|------|----|--------|
| RoleQueue | DirectUser | claim |
| OrgQueue | DirectUser | claim |
| DirectUser | DirectUser | reassign |
| DirectUser | RoleQueue/OrgQueue | release (optional) |

Disallowed:
- queue → queue direct reassignment
- multiple simultaneous owners
- assigning without proper mode

---

5. AUTHORIZATION

Define minimal safe rules:

### Claim
- user must belong to role/org associated with task
- or platform admin

### Reassign
- owner can reassign (optional)
- or manager/admin role
- document chosen rule clearly

### Release
- owner or admin

Requirements:
- enforce server-side
- do not trust client

---

6. CONCURRENCY HANDLING

Critical for claim:

Use compare-and-swap logic:

- update WHERE AssignmentMode IN (RoleQueue, OrgQueue)
- ensure only one claimant succeeds

For reassign:
- optional check WHERE AssignedUserId = expected

Requirements:
- no double-claim
- no silent overwrite
- return 409 on conflict
