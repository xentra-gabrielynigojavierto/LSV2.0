Feature ID: LS-NOTIF-SMS-024

Title:
Cross-Channel Governance Federation and Unified Communications Governance Topology

Objective:
Federate governance enforcement, rollout orchestration, tenant governance resolution, lifecycle/versioning, overlays, analytics, and deployment topology across SMS, Email, Push, Webhook, and future notification channels into a unified communications governance platform.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- cross-channel governance federation
- unified governance topology resolution
- channel-aware governance inheritance
- federated rule-pack assignment
- channel/tenant overlays
- cross-channel rollout orchestration
- federated governance analytics
- unified governance APIs

Control Center must only consume Notification Service APIs and render/administer governance federation visibility. Do not duplicate topology resolution, federation evaluation, rollout orchestration, inheritance, or analytics logic in Control Center.

Requirements:
- Implement cross-channel governance federation inside Notification Service.
- Preserve existing SMS governance behavior from LS-NOTIF-SMS-017 through LS-NOTIF-SMS-023.
- Preserve SMS-only behavior for all existing tenants.
- Federation must extend existing governance, not replace SMS governance.
- Governance federation must support at minimum SMS, Email, Push, and Webhook channel scopes.
- Federation resolution must be channel-aware and tenant-aware.
- Existing SMS tenant-scoped rule-pack resolution must remain backward compatible.
- Cross-channel overlays must never mutate base governance rules.
- Federated rollout orchestration must not bypass LS-NOTIF-SMS-021 release approval or LS-NOTIF-SMS-021-HARDENING activation locking.
- Federation failures must degrade safely.
- Federation resolution must not block delivery execution.
- Do not store raw phone numbers, message bodies, credentials, webhook URLs, SettingsJson, CredentialsJson, auth tokens, or provider payloads in federation tables.
- Keep Control Center as admin/visibility UX only.
- Preserve all LS-NOTIF-SMS-001 through LS-NOTIF-SMS-023 behavior.

Implementation Requirements:
1. Add governance channel-scope persistence.
2. Add federated rule-pack mapping persistence.
3. Add cross-channel federation overlay persistence.
4. Add governance federation audit persistence if existing audit tables are insufficient.
5. Add unified governance topology resolver.
6. Add governance federation service.
7. Add channel-aware governance resolution.
8. Extend existing tenant resolution to account for channel scope.
9. Extend rollout orchestration to support cross-channel rollout metadata where feasible.
10. Add federated governance analytics.
11. Add unified governance topology APIs.
12. Add federation management APIs.
13. Add Control Center federation/topology UI if feasible.
14. Ensure all SMS-only behavior remains unchanged.

Critical Constraints:
- Reuse existing SmsGovernanceRulePack, SmsGovernanceRule, SmsGovernanceTenantRulePackAssignment, SmsGovernanceTenantOverlay, SmsGovernanceTenantResolutionService, SmsGovernanceRuleResolver, SmsGovernanceRuleEngine, SmsGovernanceRolloutPlan, SmsGovernanceRolloutService, SmsGovernanceReleasePackage, SmsGovernanceVersioningService, SmsGovernanceRuleMatchMetric, admin endpoint, EF/migration, audit/logging, worker, and Control Center governance UI patterns where appropriate.
- Do NOT rewrite the delivery pipeline.
