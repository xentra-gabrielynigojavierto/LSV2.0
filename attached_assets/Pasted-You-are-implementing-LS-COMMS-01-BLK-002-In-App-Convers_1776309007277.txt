STEP 5 — EXTEND ENUM / RULE DEFINITIONS
--------------------------------------------------

Review and extend rules, not necessarily enums, for:

Visibility:
- INTERNAL_ONLY
- SHARED_EXTERNAL

Message/channel rules:
- IN_APP messages may be INTERNAL_ONLY or SHARED_EXTERNAL depending on sender permissions
- SYSTEM_NOTE must always be INTERNAL_ONLY
- external participants must never see INTERNAL_ONLY messages
- internal participants can see both INTERNAL_ONLY and SHARED_EXTERNAL messages subject to authorization

Participant/reply rules:
- only active participants may post
- INTERNAL_USER participants may reply if canReply = true
- EXTERNAL_CONTACT participants should not post in-app messages in this block unless there is already an authenticated external portal pattern in the repo
- SYSTEM participants cannot post interactive user messages

Conversation status rules:
- NEW -> OPEN
- OPEN -> PENDING_INTERNAL
- OPEN -> PENDING_EXTERNAL
- PENDING_INTERNAL -> OPEN
- PENDING_EXTERNAL -> OPEN
- OPEN -> RESOLVED
- RESOLVED -> CLOSED
- CLOSED -> OPEN (reopen allowed if explicitly supported)
- invalid transitions should be rejected

If the repo already has a pattern for domain validation, use it.

STEP 8 — IMPLEMENT SERVICE-LAYER OPERATIONS
--------------------------------------------------

Update or add services to support these operations:

A. Conversation access validation
Implement centralized checks to ensure:
- current tenant matches conversation tenant
- current user is allowed to access the conversation
- only active participants can interact
- INTERNAL_ONLY visibility is enforced correctly
- SHARED_EXTERNAL messages are not leaked to unauthorized users

B. Ordered thread retrieval
MessageService must return messages in deterministic ascending order using:
- SentAtUtc then Id
or
- explicit sequence/order field if implemented

C. Add message with operational rules
When adding a message:
- verify sender is an active participant
- verify sender has canReply = true where applicable
- reject SYSTEM_NOTE from user-driven external/shared contexts if invalid
- enforce visibility rules
- update conversation.lastActivityAt
- optionally auto-transition conversation status from NEW to OPEN on first real message if appropriate

D. Read tracking
Implement:
- MarkConversationRead(conversationId, currentUser)
- GetReadState(conversationId, currentUser)
- update lastReadMessageId and lastReadAtUtc to the latest visible message for that user

E. Unread calculations
Where feasible for this block:
- mark conversation unread if latest visible message is newer than current user’s lastReadAtUtc
- expose per-conversation unread state
A full unread-count dashboard is not required unless easy to support cleanly.

F. Conversation lifecycle behavior
Implement valid status transition enforcement.
If conversation is CLOSED and a new message is added by an authorized internal participant, optionally reopen to OPEN if consistent with the chosen rules. If you implement this, document it clearly.
