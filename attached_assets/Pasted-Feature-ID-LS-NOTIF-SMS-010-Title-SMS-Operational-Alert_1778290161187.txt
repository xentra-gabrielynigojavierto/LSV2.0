Feature ID: LS-NOTIF-SMS-010

Title:
SMS Operational Alerting and Threshold Rules

Objective:
Add operational monitoring and threshold-based alerting inside Notification Service so platform administrators can proactively detect SMS delivery degradation, reconciliation failures, provider outages, retry spikes, dead-letter anomalies, tenant/provider operational issues, and stale reconciliation conditions.

Critical Architectural Constraint:
Although this feature will be implemented inside the LegalSynq monorepo, all SMS operational alert evaluation, threshold rules, alert persistence, and alert APIs must remain inside the Notification Service boundary.

Requirements:
- Do not build Control Center UI in this feature.
- Do not add SMS alert evaluation logic to unrelated services.
- Do not create shared cross-service alerting abstractions unless an existing Notification Service shared/internal module already exists.
- Do not modify other services except where strictly necessary for build/config compatibility.
- Notification Service must own:
  - SMS operational threshold configuration
  - SMS metric evaluation
  - SMS alert persistence
  - SMS alert deduplication/cooldown
  - SMS alert resolution
  - SMS alert admin APIs
- Dashboard/UI consumers should consume Notification Service alert APIs in a future feature.
- Preserve Notification Service as the system of record for SMS delivery, tracking, reconciliation, dashboard reporting, and operational alerting.

Implementation Requirements:
1. Add SMS operational threshold evaluation inside Notification Service.
2. Detect high SMS failure rates.
3. Detect dead-letter spikes.
4. Detect retry spikes.
5. Detect reconciliation lookup/config failure spikes.
6. Detect provider-specific delivery degradation.
7. Detect tenant-specific delivery degradation.
8. Detect stale/unreconciled SMS attempts.
9. Persist generated SMS operational alerts.
10. Deduplicate repeated alerts within a configurable cooldown window.
11. Add admin APIs to list, summarize, resolve, and optionally manually evaluate alerts.
12. Preserve existing outbound SMS send, retry/dead-letter, webhook, compliance, tenant provider resolution, reconciliation, activity APIs, dashboard APIs, and Control Center UI behavior.
13. Do not add or expand conversational inbound SMS behavior.

Critical Constraints:
- Reuse existing NotificationAttempt, reconciliation tracking, SmsDashboardRepository/SmsActivityRepository patterns, audit, worker, endpoint, authorization, migration, and startup schema conventions where appropriate.
- Do NOT introduce large rewrites.
- Keep implementation incremental and production-oriented.
- Dashboard/alert APIs must be read-only except explicit alert resolution and manual evaluation endpoint.
- Alert evaluation must not trigger SMS sends, retries, reconciliation, provider calls, or provider configuration changes.
- Alert evaluation must use local Notification Service operational data only.
- Do NOT store or expose provider credentials, CredentialsJson, SettingsJson, auth tokens, raw provider payloads, or full phone numbers.
- Alert metadata must contain only safe operational fields.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 6 — Implement Alert Rule Logic
Implement the following rules.

Rule: High global failure rate
- Scope: all SMS attempts in window.
- Metric: failed/dead_letter attempts divided by total attempts.
- Alert types:
  - sms.failure_rate_high
- Respect MinimumAttemptCount.

Rule: Dead-letter spike
- Scope: all SMS attempts in window.
- Metric: dead_letter count.
- Alert type:
  - sms.dead_letter_spike

Rule: Retry spike
- Scope: all SMS attempts in window.
- Metric: retrying count or AttemptNumber > 1 count where available.
- Alert type:
  - sms.retry_spike

Rule: Reconciliation failure rate high
- Scope: reconciled SMS attempts in window.
- Metric: lookup/config/runtime reconciliation failures divided by reconciled attempts.
- Alert type:
  - sms.reconciliation_failure_rate_high
- Use LastReconciledAt where available.

Rule: Provider degraded
- Scope: group by provider and providerConfigId where available.
- Metric: failed/dead_letter attempts divided by total attempts for provider group.
- Alert type:
  - sms.provider_degraded
- Respect MinimumAttemptCount.

Rule: Provider config failure spike
- Scope: reconciliation/send failures with provider config error categories.
- Metric: count of provider_config_* / provider_runtime_resolution_failed / auth_config_failure where applicable.
- Alert type:
  - sms.provider_config_failure_spike

Rule: Tenant anomaly
- Scope: group by tenant.
- Metric: tenant failed/dead_letter attempts divided by total tenant attempts.
- Alert type:
  - sms.tenant_anomaly
- Respect MinimumAttemptCount.

Rule: Stale unreconciled attempts
- Scope: non-terminal or sent SMS attempts older than configured stale age where ReconciliationAttemptCount = 0.
- Metric: count.
- Alert type:
  - sms.reconciliation_stale

Requirements:
- Apply filters consistently.
- Keep rule code readable and independently testable.
- If exact status names differ in codebase, align with existing constants/patterns.
- Document any rule that cannot be fully implemented due to missing fields.
