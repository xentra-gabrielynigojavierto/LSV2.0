SCOPE
================================================================

A) Tenant product entitlement model

Create a generic, portable product entitlement model owned by Tenant service.

Do NOT hardcode LegalSynq-specific business logic into the domain model.
Product names/keys may exist as data, but the service design must remain generic.

Recommended entity:
- TenantProductEntitlement

Recommended minimum fields:
- Id
- TenantId
- ProductKey
- ProductDisplayName (optional snapshot for convenience)
- IsEnabled
- IsDefault
- Tier or PlanCode (nullable)
- EffectiveFromUtc (nullable)
- EffectiveToUtc (nullable)
- CreatedAtUtc
- UpdatedAtUtc

Rules:
- One tenant can have many product entitlements
- ProductKey should be normalized, stable, and generic
- At most one active/default product per tenant
- If a new default is assigned, explicitly handle prior default
- Keep the model portable to other systems, not LegalSynq-only

Suggested example product keys as data only (not hardcoded assumptions):
- liens
- careconnect
- fund
- control-center
- task

B) Capability flags model

Create a tenant capability model for fine-grained enablement.

Recommended entity:
- TenantCapability

Recommended minimum fields:
- Id
- TenantId
- ProductEntitlementId (nullable if tenant-global)
- CapabilityKey
- IsEnabled
- CreatedAtUtc
- UpdatedAtUtc

Rules:
- Capability can be product-scoped or tenant-global
- CapabilityKey must be normalized and portable
- Avoid LegalSynq-only naming in the domain design
- Capabilities should support future feature-flag style use without introducing a full external flag platform

Examples of capability keys as data only:
- branding.advanced
- domain.custom
- portal.readonly
- notifications.custom-reply-to
- product.case-management

C) Default product model / behavior

Add service logic to support:
- one default product per tenant
- default must belong to an enabled entitlement
- when a new default is set:
  - either auto-demote prior default, or
  - reject the change
Choose one explicit behavior and document it in the report.
Preferred behavior: auto-demote the prior default for operator convenience.

D) Tenant settings model

Create a flexible tenant settings store that supports both:
1. typed core settings, and
2. extensible namespaced settings

Recommended entity:
- TenantSetting

Recommended minimum fields:
- Id
- TenantId
- SettingKey
- SettingValue
- ValueType
- Scope (nullable or enum if needed)
- ProductKey (nullable, if settings can be product-scoped)
- CreatedAtUtc
- UpdatedAtUtc

Recommended ValueType enum:
- String
- Boolean
- Number
- Json

Rules:
- SettingKey must be normalized and namespaced-friendly
- Examples:
  - platform.default-product
  - portal.locale.default
  - portal.timezone.default
  - notifications.reply-to
  - branding.login.tagline
- Keep the design portable and reusable
- Do not overbuild a dynamic config platform; keep it minimal but solid

E) Typed core settings support

Implement explicit service support for a small core set of typed settings:
- platform.default-product
- portal.locale.default
- portal.timezone.default

These can be stored in the generic TenantSetting store, but service methods/validation must treat them as first-class supported settings.

Rules:
- locale should accept IETF/BCP47-like strings (reasonable validation)
- timezone should be validated reasonably (string format validation acceptable; full tzdb validation optional if too heavy)
- default product must correspond to an enabled entitlement

F) API contracts

Add admin/internal APIs for entitlements and settings.

Entitlements APIs:
- GET /api/v1/tenants/{tenantId}/products
- POST /api/v1/tenants/{tenantId}/products
- PUT /api/v1/tenants/{tenantId}/products/{entitlementId}
- optionally DELETE or soft-disable endpoint if aligned to current patterns

Capabilities APIs:
- GET /api/v1/tenants/{tenantId}/capabilities
- POST /api/v1/tenants/{tenantId}/capabilities
- PUT /api/v1/tenants/{tenantId}/capabilities/{capabilityId}

Settings APIs:
- GET /api/v1/tenants/{tenantId}/settings
- GET /api/v1/tenants/{tenantId}/settings/{settingKey}
- PUT /api/v1/tenants/{tenantId}/settings/{settingKey}
- optionally POST if your current API style prefers create/update separation

Convenience APIs (optional but recommended if clean):
- GET /api/v1/tenants/{tenantId}/configuration-summary
This may aggregate:
  - default product
  - enabled products
  - selected core settings
Only implement this if it fits naturally without bloat.

Rules:
- Follow current auth patterns (AdminOnly for management endpoints)
- DTOs must remain generic and portable
- Request/response shapes must be stable and API-consumer friendly

G) Validation rules

Add validation for:
- ProductKey required and normalized
- CapabilityKey required and normalized
- SettingKey required and normalized
- no duplicate enabled entitlement rows for same tenant/product unless explicitly allowed
- at most one default product per tenant
- default product must be enabled
- setting value format consistent with ValueType
- locale/timezone validation for typed core settings
- namespaced setting key format
- effective date range validity where used
- capability uniqueness (tenant + optional product + key)

H) Persistence / schema

Add EF Core persistence for:
- TenantProductEntitlement
- TenantCapability
- TenantSetting

Expected:
- new tables
- indexes on TenantId
- indexes on keys
- uniqueness constraints where practical
- EF migration for Block 4

Recommended indexes:
- TenantProductEntitlement: TenantId, ProductKey, IsDefault
- TenantCapability: TenantId, CapabilityKey, ProductEntitlementId
- TenantSetting: TenantId, SettingKey, ProductKey

If filtered uniqueness is not practical in MySQL, enforce with service-layer validation and document it clearly.

   - mismatched active/status interpretations
   - missing subdomain/domain relationships
   - missing logo refs
   - counts

Rules:
- Prefer dry-run mode in this block
- If you implement write mode, it must be explicit and disabled by default
- Do not trigger automatic migration during service startup
- Do not perform read switch/cutover
- Do not modify Identity ownership boundaries

Suggested output:
