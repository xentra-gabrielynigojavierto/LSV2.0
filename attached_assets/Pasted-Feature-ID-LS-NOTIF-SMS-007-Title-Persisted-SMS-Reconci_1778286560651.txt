Feature ID: LS-NOTIF-SMS-007

Title:
Persisted SMS Reconciliation Outcome Tracking

Objective:
Persist the latest SMS reconciliation outcome directly on NotificationAttempt so SMS activity logs, summary APIs, and future dashboards can filter/report reconciliation state without relying only on audit events or inferred delivery status.

Critical Architectural Constraint:
Although this feature will be implemented inside the LegalSynq monorepo, all SMS reconciliation outcome tracking work must remain isolated within the Notification Service boundary.

Requirements:
- Do not add SMS reconciliation tracking logic to unrelated services.
- Do not create shared cross-service SMS abstractions unless an existing Notification Service shared/internal module already exists.
- Do not modify other services except where strictly necessary for build/config compatibility.
- Notification Service must own:
  - persisted SMS reconciliation outcome fields
  - reconciliation metadata updates
  - activity API reconciliation filters
  - summary API reconciliation counts
- Preserve Notification Service as the only SMS delivery, tracking, reconciliation, and reporting boundary.

Implementation Requirements:
1. Add persisted reconciliation tracking fields to NotificationAttempt.
2. Update those fields during manual and batch SMS reconciliation.
3. Persist lookup failures, skipped outcomes, no-change outcomes, and updated outcomes.
4. Add reconciliation filters to SMS activity APIs from LS-NOTIF-SMS-006.
5. Add reconciliation fields to SMS activity response payloads.
6. Add reconciliation outcome counts to SMS activity summary responses.
7. Preserve existing audit events from LS-NOTIF-SMS-004.
8. Preserve outbound SMS send, retry/dead-letter, webhook, compliance, and tenant provider resolution behavior.
9. Do not add or expand conversational inbound SMS behavior.

Critical Constraints:
- Reuse existing NotificationAttempt, SmsReconciliationService, SmsActivityService, SmsActivityRepository, audit, migration, and endpoint patterns.
- Do NOT introduce large rewrites.
- Keep implementation incremental and production-oriented.
- Do NOT store provider credentials or secrets in reconciliation fields.
- Do NOT expose Twilio credentials, tenant provider credentials, CredentialsJson, SettingsJson, auth tokens, or raw provider payloads.
- Reconciliation tracking must never resend SMS.
- Reconciliation tracking must not change the semantic result of reconciliation.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 2 — Add Reconciliation Tracking Fields to NotificationAttempt
Add persisted fields to NotificationAttempt.

Recommended fields:
- LastReconciliationOutcome string? nullable
- LastReconciledAt DateTime? nullable
- LastReconciliationErrorCode string? nullable
- LastReconciliationProviderStatus string? nullable
- LastReconciliationNormalizedStatus string? nullable
- ReconciliationAttemptCount int default 0

Requirements:
- Use existing naming/style conventions.
- Add EF mapping/migration if required.
- Add startup safety-net schema updates if that pattern exists.
- Fields must be nullable except count.
- Do not store raw provider response bodies.
- Do not store provider credentials.
- Keep stored provider status to safe status strings only.

STEP 3 — Add Repository Update Support
Add a repository method or extend existing update logic to persist reconciliation tracking fields.

Suggested method:

Task UpdateReconciliationTrackingAsync(
    Guid attemptId,
    string outcome,
    string? errorCode,
    string? providerStatus,
    string? normalizedStatus,
    DateTime reconciledAt,
    CancellationToken cancellationToken = default);

Requirements:
- Increment ReconciliationAttemptCount.
- Set LastReconciledAt.
- Set LastReconciliationOutcome.
- Set LastReconciliationErrorCode.
- Set LastReconciliationProviderStatus.
- Set LastReconciliationNormalizedStatus.
- Avoid overwriting delivery status incorrectly.
- Do not interfere with DeliveryStatusService status updates.
- Preserve existing optimistic/concurrency behavior if present.

STEP 4 — Update SmsReconciliationService
Update manual and batch reconciliation paths so every reconciliation attempt persists tracking metadata.

Required cases:
- updated
- no_change
- vendor_lookup_failed
- skipped_missing_provider_message_id
- skipped_not_sms
- skipped_unsupported_provider
- attempt_not_found where no attempt exists, no attempt update required
- provider_message_not_found
- missing_provider_config_context
- provider_config_not_found
- provider_config_inactive
- provider_config_invalid
- provider_runtime_resolution_failed

Requirements:
- If an attempt exists, reconciliation tracking should be persisted even when no delivery status changes.
- If provider lookup succeeds, persist vendor provider status and normalized status where available.
- If provider lookup fails, persist error code and outcome.
- If an attempt is skipped, persist skip outcome when attempt exists.
- Do not persist anything for attempt_not_found because no target row exists.
- Tracking update failures should be logged and should not crash the reconciliation process unless existing repository patterns require failure propagation.
- Existing reconciliation audit events must remain unchanged.
