STEP 2 — Add Granular Approval Role Enforcement
Extend approval enforcement beyond generic PlatformAdmin.

Suggested enhancements:
- Add RequiredApproverRole validation during approval.
- Enforce stage-specific role matching where identity/claims context supports it.
- Add optional AllowedApproverRolesJson to approval requests if needed.
- Add safe fallback to PlatformAdmin if granular role unavailable and configuration allows fallback.
- Add explicit audit events for role-mismatch approval attempts.

Suggested config:
SmsGovernanceReleaseManagement:
- EnforceApprovalRoles bool default true
- AllowPlatformAdminApprovalFallback bool default true

Requirements:
- Approval must fail safely if role mismatch occurs and fallback is disabled.
- Existing approvals remain backward compatible.
- Do not hardcode tenant-specific roles.
- Do not require new identity service changes unless existing claims already expose roles.
- Document fallback behavior.

STEP 3 — Add Release Activation Concurrency Locking
Prevent duplicate activation races.

Suggested approaches:
- Add optimistic concurrency token to SmsGovernanceReleasePackage
OR
- Add activation lock fields on release package:
  - ActivationLockId Guid? nullable
  - ActivationLockAcquiredAt DateTime? nullable
  - ActivationLockExpiresAt DateTime? nullable
  - ActivationLockedBy string? nullable
OR
- Add transactional activation guard using existing DB transaction patterns

Requirements:
- Two concurrent activation attempts must not both succeed.
- Scheduled worker activation and manual activation must coordinate safely.
- Activation lock release must occur safely after success/failure.
- Activation failure must not deadlock future activations.
- Stale locks must expire or be safely recoverable.
- Use existing EF/concurrency patterns if available.
- Add safe audit event when activation lock acquisition fails.

STEP 4 — Harden Release State Transitions
Add stricter release-state integrity enforcement.

Suggested validations:
- Only draft can accept new items.
- Only draft can remove items.
- Only draft can submit review.
- Only pending_review can approve/reject.
- Only approved can activate/schedule.
- Scheduled can activate when due or be archived.
- Archived/superseded immutable.
- Rejected cannot activate.
- Active release cannot return to draft.
- Approval stages cannot skip order.
- Scheduled activation date must be future.

Requirements:
- Return safe validation errors.
- Add audit events for invalid transition attempts where useful.
- Preserve backward compatibility where possible.
- Keep state validation centralized in release service.

STEP 5 — Add Scheduler Retry/Backoff Protection
Harden SmsGovernanceReleaseActivationWorker.

Suggested behavior:
- Exponential or bounded linear backoff for repeated activation failures.
- Max retries before terminal activation_failed state.
- Retry metadata persisted safely if needed.

Suggested fields on SmsGovernanceReleasePackage if needed:
- ActivationAttemptCount int
- LastActivationAttemptAt DateTime? nullable
- NextActivationRetryAt DateTime? nullable
- LastActivationFailureReason string? nullable

Suggested config additions:
SmsGovernanceReleaseManagement:
- ActivationRetryLimit int default 3
- ActivationRetryBackoffMinutes int default 10
- ScheduledActivationBatchSize int default 10

Requirements:
- Worker failures must not crash the application.
- Retry behavior must be deterministic.
- Failed activation attempts must be auditable.
- No external queues required.
- No delivery pipeline impact.

Suggested service:

- ValidateAuditTrailAsync(Guid releaseId, CancellationToken ct)

Suggested checks:

- Every approval/rejection has an audit event.
- Every activation attempt has an audit event.
- Every activation failure has an audit event.
- Audit timeline ordering is consistent.
- Release final state aligns with audit history.

Requirements:

- No destructive audit cleanup.
- No secrets/raw payloads in audit metadata.
- Integrity validation is read-only.
- Validation failures are reported, not auto-mutated.
