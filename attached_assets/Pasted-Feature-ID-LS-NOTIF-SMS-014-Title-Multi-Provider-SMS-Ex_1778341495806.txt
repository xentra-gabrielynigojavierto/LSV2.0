Feature ID: LS-NOTIF-SMS-014

Title:
Multi-Provider SMS Expansion and Intelligent Routing

Objective:
Expand Notification Service beyond Twilio into a resilient multi-provider SMS platform with intelligent provider selection, dynamic failover, cost-aware routing, health-aware routing, tenant/provider policy control, and regional delivery optimization.

Critical Architectural Constraint:
Notification Service remains the source of truth for:
- SMS provider execution
- provider routing
- runtime provider resolution
- provider health awareness
- provider cost awareness
- routing decision persistence
- routing analytics

Control Center must only consume Notification Service APIs and render/administer routing policy, provider health, and routing decision visibility. Do not duplicate SMS routing logic in Control Center.

Requirements:
- Implement multi-provider SMS routing inside Notification Service.
- Preserve Twilio as the existing/default provider.
- Add additional provider capability incrementally.
- Do not introduce large rewrites.
- Do not break existing Twilio SMS send, reconciliation, cost analytics, alerting, escalation, dashboard, or incident-management behavior.
- Do not expose provider credentials, webhook URLs, CredentialsJson, SettingsJson, auth tokens, raw provider payloads, or full phone numbers.
- Preserve tenant-owned provider runtime resolution.
- Keep Control Center as UI/admin consumer only.

Implementation Requirements:
1. Extend Notification Service SMS provider architecture to support multiple SMS providers at runtime.
2. Add at least one additional SMS provider adapter if feasible within the current codebase, with preference for Vonage, Telnyx, or Plivo based on lowest-risk implementation.
3. Add provider capability metadata so providers can declare whether they support send, status lookup, health check, cost estimate, and regional routing.
4. Add routing policy support for provider selection.
5. Add routing modes:
   - priority
   - cost_optimized
   - health_optimized
   - hybrid
   - regional
6. Add routing decision persistence for audit/debug/reporting.
7. Integrate routing decisions into outbound SMS send flow.
8. Preserve existing SMS compliance/preference enforcement before routing/provider execution.
9. Preserve existing provider runtime resolution and tenant-owned config support.
10. Preserve vendor reconciliation where supported; unsupported providers must fail safely or skip pull-reconciliation with structured reason.
11. Add admin APIs for routing policies, provider capabilities, provider health, and routing decisions.
12. Add Control Center UI for routing policy management and routing visibility if feasible.
13. Preserve all existing SMS operational APIs and UI behavior.

Critical Constraints:
- Reuse existing ISmsProviderAdapter, ISmsProviderStatusLookup, ProviderRoutingService, TenantProviderConfig, SmsProviderRuntimeResolver, SmsCostAnalytics, SmsDashboard, SmsOperationalAlert, and Control Center patterns where appropriate.
- Do NOT rewrite the Notification Service delivery pipeline.
- Do NOT bypass existing contact enforcement, preference checks, audit, retry/dead-letter, or attempt persistence.
- Do NOT hardcode routing logic directly inside provider adapters.
- Do NOT make provider adapters aware of tenant routing policy.
- Do NOT call external providers from dashboard/routing read APIs.
- Do NOT introduce AI/ML routing in this feature.
- Do NOT implement FX conversion.
- Do NOT add conversational inbound SMS.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.

STEP 8 — Implement Routing Modes
Implement initial routing modes.

Priority:
- Use ordered candidate provider routes.
- Preserve current behavior as default.

Cost optimized:
- Select lowest estimated cost provider among candidates with available estimate.
- If no estimate exists, fall back to priority.
- Respect MaxEstimatedCostPerMessage if configured.

Health optimized:
- Avoid providers marked degraded/unhealthy by local health/alert state where available.
- If no health data exists, fall back to priority.
- Do not call provider health live in send path unless existing pattern already does.

Hybrid:
- Combine health gate first, then cost optimization, then priority fallback.

Regional:
- Prefer providers/policies matching recipient country/region where available.
- If recipient region cannot be derived safely, fall back to priority.

Requirements:
- Document any routing mode partially implemented due to missing health/country data.
- Do not overbuild geographic parsing.
- Keep default behavior backward compatible.

STEP 9 — Integrate Routing Engine Into SMS Send Flow
Update outbound SMS send flow.

Expected flow:
1. Validate/preference/contact enforcement.
2. Resolve candidate routes using existing ProviderRoutingService.
3. Pass candidates to SmsRoutingEngine.
4. Persist routing decision.
5. Resolve runtime provider adapter for selected route.
6. Send SMS through selected provider.
7. Persist provider/providerConfigId/ownership on attempt as existing behavior.
8. Update routing decision with attemptId if feasible.

Requirements:
- Preserve Twilio default behavior when no routing policy exists.
- Preserve retry/dead-letter.
- Preserve failover semantics.
- Preserve audit.
- Preserve cost recording.
- No duplicate sends.
- No routing API/dashboard API triggers sends.
