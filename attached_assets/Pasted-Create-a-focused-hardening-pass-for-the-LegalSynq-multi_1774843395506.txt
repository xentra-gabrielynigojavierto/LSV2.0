## Phase 1 — OrgType string hardening and de-legacying

Current state:
- OrganizationTypeId exists and is active
- OrgType string still exists for backward compatibility
- JWT org_type still comes from OrgType string
- Phase G identified OrgType string retirement as the next future step

Tasks:
- inspect all places where Organization.OrgType is still read or written
- inspect JwtTokenService, CurrentRequestContext, DTOs, response models, and any auth/bootstrap code
- change logic so OrganizationTypeId is the primary source of truth everywhere practical
- where org_type string is still required, derive it from OrganizationTypeId using OrgTypeMapper
- stop treating OrgType string as the authoritative write field
- keep the OrgType column for compatibility, but mark all remaining usage with:
  TODO [Phase H — remove OrgType string]
- add defensive handling for rows where OrganizationTypeId is missing but OrgType string exists
- if practical, add a consistency checker or startup warning for:
  - OrganizationTypeId null while OrgType not null
  - OrganizationTypeId and OrgType code mismatch

Important:
- do NOT drop the OrgType column yet
- do NOT break JWT consumers
- do NOT break existing API contracts

## Phase 2 — authorization hardening beyond global scope

Current state:
- ScopedRoleAssignments are now the sole role store
- most proven usage is for GLOBAL scope
- schema supports:
  - GLOBAL
  - TENANT
  - ORGANIZATION
  - PRODUCT
  - RELATIONSHIP

Tasks:
- inspect AuthService, UserRepository, policies, request context, and any authorization helpers
- identify where only GLOBAL scope is currently used in practice
- implement or strengthen resolution for at least these paths where practical:
  - TENANT
  - ORGANIZATION
  - PRODUCT
  - RELATIONSHIP
- if full enforcement cannot be implemented everywhere safely, then:
  - add explicit helper methods
  - add clear TODO markers
  - add diagnostics/reporting so these scopes can be verified
- ensure login/bootstrap response can surface enough scoped information for downstream services if needed
- document actual supported scope behavior after changes

Also:
- verify there is no lingering hidden assumption that user has only one org or only one active scope
- strengthen tests/build validation if the project already has a test pattern

## Phase 3 — CareConnect / Identity consistency hardening

Current state:
- Referral and Appointment use OrganizationRelationshipId
- Provider and Facility can link to OrganizationId
- linkage is soft cross-service alignment, not hard DB FK

Tasks:
- inspect:
  - ProviderService
  - FacilityService
  - ReferralService
  - AppointmentService
  - resolver/service clients
  - DTOs and repositories
- improve consistency guarantees for:
  - Provider.OrganizationId
  - Facility.OrganizationId
  - Referral.OrganizationRelationshipId
  - Appointment.OrganizationRelationshipId

Add practical hardening such as:
- validation when linking Provider or Facility to OrganizationId
- safe re-link/update behavior
- better logging or diagnostics when relationship resolution fails
- consistency checks when appointment relationship differs from referral relationship
- retry-safe or null-safe resolver handling where useful

If practical:
- add an admin/reporting endpoint or health query that surfaces unresolved linkage counts:
  - providers without OrganizationId
  - facilities without OrganizationId
  - referrals with org IDs but no OrganizationRelationshipId
  - appointments missing OrganizationRelationshipId when referral has one

Do NOT over-engineer.
Prefer lightweight integrity checks and operational visibility.
