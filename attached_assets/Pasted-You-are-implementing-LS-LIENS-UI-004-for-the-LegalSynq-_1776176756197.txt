You are implementing LS-LIENS-UI-004 for the LegalSynq Liens v2 migration.

Feature ID: LS-LIENS-UI-004
Feature Title: Servicing + Task Manager Integration

Objective:
Wire the Servicing and Task Manager modules to real backend APIs using the same enforced frontend service-layer architecture established in LS-LIENS-UI-002 and LS-LIENS-UI-003. This feature must preserve the current UI baseline while replacing mock/store-driven operational behavior with real API-backed integration and real cross-entity navigation to Cases and Liens.

This feature makes the system operational, not just viewable.

--------------------------------------------------
CRITICAL ARCHITECTURE RULES (NON-NEGOTIABLE)
--------------------------------------------------

1. Preserve the v2 shell, including top nav and side nav.
2. Preserve the current UI layout and v1-derived workflow feel unless minimal adaptation is required for real API integration.
3. Enforce strict layering:
   - API client layer
   - service layer
   - mapper / adapter layer
   - UI state layer
   - presentation layer
4. No direct HTTP calls in pages or presentation components.
5. No DTO-to-UI transformation logic inside page/components.
6. No production mock/static servicing or task data.
7. Reuse the exact architectural pattern used in LS-LIENS-UI-002 and LS-LIENS-UI-003.
8. Do not invent backend APIs. Use only what actually exists.
9. Preserve donor SynqLiens behavior where backend support exists.
10. Operational data, KPI values, overdue flags, and task status changes must come from real backend truth or clearly documented client-side derivation from backend responses.
11. If Zustand or another store is used, it must be UI state only, not the source of truth for servicing/task data.

--------------------------------------------------
REQUIRED FRONTEND ARCHITECTURE
--------------------------------------------------

You must implement equivalent module structures for both Servicing and Task Manager.

Servicing stack:
- servicing.types.ts
- servicing.api.ts
- servicing.mapper.ts
- servicing.service.ts

Task stack:
- tasks.types.ts or task-manager.types.ts
- tasks.api.ts or task-manager.api.ts
- tasks.mapper.ts or task-manager.mapper.ts
- tasks.service.ts or task-manager.service.ts

Responsibilities:

Layer 1 - API client
- raw HTTP only
- endpoint definitions
- typed DTOs
- no UI shaping
- no business logic

Layer 2 - Service layer
- business-facing methods only
- only approved data-access path for UI
- calls API client
- applies mappers
- isolates backend quirks

Layer 3 - Mapper layer
- DTO to UI model transformation
- safe defaults / null normalization
- status / priority / overdue display mapping
- no duplication across components

Layer 4 - Types layer
- backend DTOs
- query models
- update/action request models
- UI models
- minimal cross-entity reference models where useful

Layer 5 - UI state
- filters
- pagination
- sorting
- board/list mode
- selected row/card
- drawer/modal state
- loading/error flags if consistent with existing architecture
- currentRole / addToast if already part of current UI pattern

Layer 6 - Presentation
- rendering only
- receives typed UI models
- no raw HTTP
- no ad hoc response parsing
