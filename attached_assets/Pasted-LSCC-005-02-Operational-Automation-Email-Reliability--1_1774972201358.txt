REQUIREMENTS
==================================================

--------------------------------------------------
A. AUTOMATIC EMAIL RETRY POLICY
--------------------------------------------------

Implement automatic retry for failed referral-related emails.

Applicable email types in scope:
- new referral notification to provider
- referral accepted confirmation to provider
- referral accepted confirmation to referrer

Retry policy (source of truth):
- Attempt 1: immediate existing send attempt
- Attempt 2: retry after 5 minutes
- Attempt 3: retry after 30 minutes from previous failed attempt
- Maximum attempts total: 3

After the third failed attempt:
- do not retry further
- mark as retry exhausted

Rules:
- automatic retry must apply only to failed email notifications
- automatic retry must not create new duplicate notification records for the same failed notification unless the chosen implementation absolutely requires it and is clearly documented
- manual resend remains a separate operator action and should remain available where appropriate
- if referral state changes such that the original email is no longer appropriate, do not continue retrying blindly; either stop retrying or document the minimal chosen rule

Preferred implementation approach:
- use the lightest viable scheduling mechanism already compatible with the repo
- avoid broad infrastructure additions
- if no true scheduler exists, a lightweight background worker or app-activity-driven retry pass is acceptable, but the exact mechanism must be explicit and deterministic

--------------------------------------------------
B. RETRY STATUS TRACKING
--------------------------------------------------

Extend the delivery state model so operators can understand retry lifecycle.

Minimum statuses/derived states to support in UI and/or backend logic:
- Pending
- Sent
- Failed
- Retrying
- RetryExhausted

If the persistence model already stores only Pending/Sent/Failed, it is acceptable to derive Retrying and RetryExhausted from:
- AttemptCount
- LastAttemptAtUtc
- next scheduled retry time
- max-attempt rule

Requirements:
- operators must be able to distinguish "failed and will retry" from "failed permanently until manual intervention"
- attempt count progression must remain visible
- timestamps must be meaningful and updated consistently

--------------------------------------------------
C. RETRY VS RESEND DISTINCTION
--------------------------------------------------

Keep a clear semantic difference:

Retry:
- system-driven
- continues lifecycle of the same failed notification if practical

Resend:
- user-driven
- explicit manual action
- should create a distinct event/audit signal

Rules:
- UI must not confuse retry with resend
- audit trail must distinguish them
- if user clicks Resend while a failed notification is still eligible for retry, behavior must be defined explicitly

Recommended minimal behavior:
- manual resend is allowed
- resend creates a new notification event
- original failed notification keeps its own history
- future auto-retries for the original failed notification should be suppressed once a manual resend succeeds, or the chosen rule must be explicitly documented

--------------------------------------------------
D. RETRY EXECUTION MECHANISM
--------------------------------------------------

Implement the smallest reliable mechanism for automatic retries.

Two acceptable patterns:

Option 1 — Lightweight in-process/background worker
- if the service architecture already supports hosted services or periodic workers, use that
- periodically scan for retry-eligible failed notifications
- process them safely

Option 2 — Lazy/system-activity-triggered retry
- retry evaluation happens on selected application activity
- only if deterministic and documented
- must still respect the retry schedule windows

Preferred choice:
- use the smallest approach that is actually reliable in this codebase

Safety requirements:
- no duplicate parallel retry attempts for the same notification
- no retry of Sent notifications
- no retry past max attempts
- if concurrency is possible, guard against double-processing

Data source:

- if one consolidated endpoint/helper is needed, keep it narrow

--------------------------------------------------
G. RETRY EXHAUSTION HANDLING
--------------------------------------------------

When retries are exhausted:

1. mark the notification as exhausted (explicitly or by derivation)
2. surface this clearly in the UI
3. allow manual resend where appropriate
4. record an audit event if feasible

UI should communicate:
- automatic retries have stopped
- manual action may be required

Do NOT leave operators guessing whether the system is still retrying.

--------------------------------------------------
H. INVALID / EDGE EMAIL CASES
--------------------------------------------------

Handle these safely:

1. SMTP down for extended period
- retries happen until max attempts
- then stop
- exhaustion is visible

2. user manually resends during retry lifecycle
- behavior must be deterministic and documented

3. referral accepted before retry occurs
- do not keep retrying an outdated notification blindly
- either cancel future retry for that notification or document the minimal rule used

4. already-sent notification
- must never be retried again

5. partially failed provider/referrer confirmation pair
- track each notification independently
