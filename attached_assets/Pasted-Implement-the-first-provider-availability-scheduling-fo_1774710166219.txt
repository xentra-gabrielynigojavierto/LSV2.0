Implement the first provider availability + scheduling foundation for LegalSynq CareConnect.

Goal:
Add the provider availability and scheduling model foundation, starting with:
1. Facilities
2. ProviderFacilities
3. ServiceOfferings
4. ProviderServiceOfferings
5. ProviderAvailabilityTemplates

Context:
- CareConnect already exists and is working
- Gateway is already routing /careconnect/*
- Identity and JWT are already working
- MySQL + EF Core are already configured
- CareConnect already supports:
  - Providers
  - Categories
  - Referrals
  - Referral status history
  - Workflow rules
  - search/filtering patterns
- Tenant context, role-based auth, audit fields, and exception handling already exist

Important:
This step should create the scheduling foundation only.
Do NOT implement slot generation, appointments, booking, or scheduling exceptions yet.

2. Keep domain clean

- use enterprise-friendly naming
- use existing audit field patterns where appropriate

3. Add EF Core mappings

Create tables:
- Facilities
- ProviderFacilities
- ServiceOfferings
- ProviderServiceOfferings
- ProviderAvailabilityTemplates

Keys:
- Facilities: Id
- ServiceOfferings: Id
- ProviderAvailabilityTemplates: Id
- ProviderFacilities: composite key (ProviderId, FacilityId)

For ProviderServiceOfferings:
- use a clean key strategy
- preferred: composite key (ProviderId, ServiceOfferingId, FacilityId) if EF/MySQL handling is clean
- if nullable FacilityId makes composite key awkward, use a surrogate Id and add a unique index instead
- choose one clean strategy and stay consistent

Indexes:
- Facilities: index on (TenantId, Name)
- ServiceOfferings: unique on (TenantId, Code)
- ProviderAvailabilityTemplates: index on (TenantId, ProviderId, DayOfWeek)
- ProviderAvailabilityTemplates: index on (TenantId, FacilityId)
- ProviderAvailabilityTemplates: index on (TenantId, ServiceOfferingId) if ServiceOfferingId is used

Relationships:
- ProviderFacilities links Providers and Facilities
- ProviderServiceOfferings links Providers and ServiceOfferings, optionally per Facility
- ProviderAvailabilityTemplates belongs to Provider and Facility, optionally ServiceOffering
- all tenant-scoped entities must remain tenant-safe

9. Validation

Facility validation:
- Name required, max 200
- AddressLine1 required
- City required
- State required
- PostalCode required
- Phone max 50 if provided

ServiceOffering validation:
- Name required, max 200
- Code required, max 100
- DurationMinutes > 0

AvailabilityTemplate validation:
- valid DayOfWeek (choose a consistent range and document it)
- SlotDurationMinutes > 0
- Capacity > 0
- EndTimeLocal must be after StartTimeLocal
- EffectiveTo must be >= EffectiveFrom if both supplied
- Provider must exist in tenant
- Facility must exist in tenant
- ServiceOffering must exist in tenant if provided
