PART A — DEFINE GOVERNANCE CONTROL PATTERNS

--------------------------------------------------

Create or formalize a small set of explicit access-control patterns.

At minimum define these governance categories:

1. Platform-only
   - requires PlatformAdmin
   - no TenantAdmin access
   - examples: integrity diagnostics, platform-wide admin tools

2. Tenant-scoped admin
   - PlatformAdmin may act platform-wide only where explicitly supported
   - TenantAdmin must be scoped to own tenant
   - requires tenant context guard

3. Provider self-service
   - authenticated provider/user context
   - no tenant-admin privilege implied
   - ownership enforced through user/provider identity

4. Product-role operational access
   - normal users require product role
   - TenantAdmin/PlatformAdmin bypass must be explicit and observable

Document the final model in the report.

--------------------------------------------------

PART B — CENTRALIZE TENANT-SCOPED ADMIN GUARD

--------------------------------------------------

Where possible, create a reusable helper/extension for TenantAdmin vs PlatformAdmin scoping.

Purpose:
- avoid repeating fragile inline code everywhere
- enforce consistent behavior when:
  - PlatformAdmin has no tenant claim
  - TenantAdmin requires tenant_id claim
  - PlatformAdmin must provide explicit tenantId for single-tenant operations

Example behavior:

ResolveAdminTenantScope(ctx, optionalTenantId, mode)

Suggested modes:
- PlatformWideAllowed
  - PlatformAdmin may return null scope
  - TenantAdmin must return ctx.TenantId

- SingleTenantRequired
  - PlatformAdmin must provide explicit tenantId
  - TenantAdmin must return ctx.TenantId

- TenantAdminOnly
  - PlatformAdmin rejected unless explicitly allowed
  - TenantAdmin must return ctx.TenantId

Rules:
- use existing types/patterns if already present
- do not overbuild a full authorization framework
- helper should return safe results or IResult-style errors consistently
