==================================================
PROJECT: NOTIF-UI-010 — Tenant Delivery Controls (Retry / Resend / Suppression Awareness)
==================================================

GOAL:
Give tenant users limited, safe operational controls over failed or blocked notification outcomes, while preserving tenant isolation, suppression policy integrity, and backend truth.

This phase adds:
- controlled retry/resend actions for eligible notifications
- suppression-aware messaging and visibility
- contact health visibility linkage from notification detail
- safe action gating and confirmation UX
- post-action status refresh and audit-friendly feedback

This phase is NOT broad orchestration.
It is controlled, operational remediation.

Do NOT overbuild.
Do NOT expose Control Center capabilities.
Do NOT build bulk send/retry.
Do NOT let tenants bypass suppression or policy protections.
Do NOT redesign the tenant portal shell.

==================================================
CRITICAL DOMAIN RULES (MANDATORY)
==================================================

1. All actions must remain tenant-scoped
2. Retry/resend must only be available when backend state makes it safe
3. Suppressed or policy-blocked notifications must NEVER be resent in a way that bypasses backend protection
4. UI must represent backend truth honestly
5. No bulk actions in this phase
6. No manual arbitrary send flow in this phase
7. If backend does not support a safe action, do NOT fake it

==================================================
ASSUMPTIONS / BACKEND EXPECTATION
==================================================

This UI phase assumes the backend either already supports or will support safe retry/resend and suppression-aware read endpoints.

Expected backend capability patterns may include one or more of:
- POST /notifications/v1/notifications/:id/retry
- POST /notifications/v1/notifications/:id/resend
- GET /notifications/v1/contacts/health/:channel/:contactValue
- GET /notifications/v1/contacts/suppressions?contactValue=...&channel=...
- notification detail indicating retry eligibility or blocked reason
- action denial responses for ineligible notifications

If actual backend routes differ:
- inspect and use the real routes
- adapt carefully
- do NOT invent fake success flows
- document capability gaps clearly in the report

==================================================
TENANT PORTAL SCOPE

Audience:

Allowed capabilities:
- inspect failed / blocked notifications
- retry or resend a single eligible notification
- understand suppression and contact health reasons
- navigate from notification detail to contact suppression/health context where appropriate

Not allowed:

PRIMARY FEATURES TO BUILD
==================================================

--------------------------------------
1. ELIGIBLE ACTIONS ON NOTIFICATION DETAIL
--------------------------------------

Enhance the tenant notification detail page from NOTIF-UI-009.

Suggested existing route:
- /tenant/notifications/activity/[notificationId]
or the equivalent route already implemented

Required behavior:
- show action area on notification detail
- determine action eligibility from backend truth where available

Possible actions:
- Retry
- Resend

Action display rules:
- show only if notification is eligible
- if not eligible, show clear explanation instead of dead buttons
- if blocked due to suppression/policy, do NOT show retry/resend as available unless backend explicitly allows safe resend after state change

Examples:
- failed transient/provider error → retry may be allowed
- blocked by suppression → retry not allowed
- already delivered/sent final state → retry may not be allowed
- accepted/processing → no retry action

If backend does not expose explicit eligibility:
- derive cautiously from status + failure category only if safe
- otherwise do not expose action and document limitation

==================================================
2. RETRY / RESEND ACTION FLOW
==================================================

Implement single-notification retry/resend flow.

Expected backend patterns:
- POST /notifications/v1/notifications/:id/retry
- or POST /notifications/v1/notifications/:id/resend

Requirements:
- explicit confirmation required before action
- show what the action does in plain language
- disable action while pending
- show success state and resulting notification/attempt outcome if backend returns it
- refresh notification detail/activity view after success
- preserve tenant context and auth

If backend returns a new notification ID or new attempt:
- surface that clearly
- optionally link to the new notification detail if appropriate

Do NOT allow repeated rapid-fire clicking.
Do NOT silently auto-retry in UI.

==================================================
3. SUPPRESSION AWARENESS PANEL
==================================================

On blocked or suppression-related failures, show a dedicated suppression awareness section.

Display where possible:
- blocked reason code
- suppression reason
- contact channel
- contact value
- whether contact is currently suppressed
- whether issue appears to be unsubscribe / complaint / bounce / invalid / manual suppression

Requirements:
- distinguish clearly between:
  - failed delivery
  - blocked by policy
  - blocked by suppression
- use plain language for tenant users
- do not expose irrelevant internal-only debug data

If backend supports relevant endpoints:
- fetch contact suppression details
- fetch contact health details

If not:
- use notification detail fields only and document limitation

==================================================
4. CONTACT HEALTH LINKAGE
==================================================

Allow tenant users to understand the health state of the contact behind a failed/blocked notification.

Requirements:
- where channel + contact value are available, show:
  - health status
  - last event
  - suppression presence
- link to existing tenant contact health / suppressions pages if they exist and if routing is already in place
- or show a compact embedded summary in the detail page

Goal:
- help user answer:
  - is this address/number unhealthy?
  - is it blocked?
  - should I retry later or not?

Do NOT let users directly lift suppression in this phase unless that capability already exists and is explicitly safe and allowed.

==================================================
5. ACTION OUTCOME / POST-ACTION UX
==================================================

After retry/resend:
- show success or failure feedback clearly
- refresh the page state
- if a new notification or attempt was created, make that visible
- if backend denies the action because state changed, show the denial clearly

Examples:
- "Retry submitted successfully"
- "Resend created a new notification"
- "This notification can no longer be retried"
- "Blocked notifications cannot be resent while the contact remains suppressed"

Use platform-native feedback patterns where available.

==================================================
6. LIST-LEVEL ACTION VISIBILITY (OPTIONAL LIGHTWEIGHT)
==================================================

On the activity list page, you may add lightweight indicators such as:
- Retry available
- Blocked by suppression
- Failed — review details

However:
- do NOT put full retry buttons inline in the list unless the UX is very clean and clearly safe
- primary control flow should still be from the detail page

The detail page is the main operational action surface.

  - no backend calls
