STEP 3 — TASK STAGE / STATUS / GOVERNANCE ENGINE (TASK-005)
--------------------------------------------------

Add reusable execution-stage support and governance rules.

IMPORTANT DOMAIN RULE:
- Task stages are execution stages owned by Task
- They are NOT Flow workflow steps
- Do NOT merge or redesign Flow concepts here

A. Status model
Retain and harden lifecycle behavior:
- OPEN
- IN_PROGRESS
- COMPLETED
- CANCELLED

Validate allowed transitions.
Preserve terminal-state protections.

B. Task stage model
Introduce a generic stage concept for task execution.

Create a new entity/table for task stages configuration, using platform-safe naming.
Recommended shape:
- id
- tenant_id
- source_product_code (nullable for global/default stages)
- code
- name
- display_order
- is_active
- created_at
- updated_at

Stage behavior:
- support generic/default stages for all products
- optionally support product-specific stage configuration
- a task may reference a current stage
- current stage must be valid for the tenant and, when relevant, the product context
- product tasks can use product-specific or default stages
- general tasks can use only general/default stage sets unless your codebase supports a better generic model

Add a stage reference to tasks if needed:
- current_stage_id (nullable initially if existing tasks do not yet use stages)

C. Governance settings
Introduce generic governance settings that control task behavior per tenant and optionally per product.

Create a reusable settings entity/table for rules such as:
- require_assignee
- require_due_date
- require_stage
- allow_unassign
- allow_cancel
- allow_complete_without_stage
- allow_notes_on_closed_tasks
- default_priority
- default_task_scope

Rules:
- support tenant-wide defaults
- support optional product-level overrides
- product overrides should not require Liens assumptions
- do not implement a massive rules engine; keep it simple, explicit, and code-driven

Enforcement:
- enforce governance during create/update/assign/status-change operations
- use a deterministic resolution order:
  1. product-level settings if matching source_product_code exists
  2. tenant-level default settings
  3. hard-coded fallback defaults only if nothing configured

APIs:
Add CRUD or at minimum admin-manageable endpoints for:
- stage config
- governance settings

You may use minimal APIs consistent with current platform patterns.

STEP 5 — REMINDER ENGINE (TASK-006 PART 2)
--------------------------------------------------

Add reminder support to Task without introducing a full scheduling platform rewrite.

Scope for this block:
- store reminder configuration
- support reminder eligibility evaluation
- support reminder dispatch trigger through application logic
- no need to build a heavyweight distributed scheduler if the platform does not already have one

Create a reminder entity/table, for example:
- id
- task_id
- tenant_id
- reminder_type
- remind_at
- status
- last_attempt_at
- sent_at
- failure_reason
- created_at
- updated_at

Recommended enums:
ReminderType:
- DUE_SOON
- OVERDUE
- CUSTOM

ReminderStatus:
- PENDING
- SENT
- FAILED
- CANCELLED

Behavior:
- reminders should be created/updated when due dates are added or changed
- closed/cancelled/completed tasks should not continue generating active reminders
- if due date removed, pending reminders should be cancelled or invalidated
- reminders must remain tenant-safe
- support at least one concrete system behavior:
  - create due-soon reminder before due date
  - create overdue reminder at/after due date

Implementation guidance:
- if a background worker pattern already exists in codebase, follow it
- if not, implement the minimum safe reminder processing pathway and document the operational trigger clearly
- do not invent a complex scheduler if the platform does not have one yet

APIs:
- optional admin/debug endpoint to evaluate or trigger pending reminders for validation
- do not expose unsafe public endpoints
