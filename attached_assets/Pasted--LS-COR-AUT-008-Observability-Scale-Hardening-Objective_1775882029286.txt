# LS-COR-AUT-008 — Observability + Scale Hardening

## Objective

Enhance LS-COR-AUT with **observability, performance optimization, and operational tooling** to support production-scale usage.

This feature focuses on:

* visibility into authorization decisions
* performance improvements for effective access computation
* operational debugging capabilities
* audit visibility for administrators

At completion, the system must be:

* observable
* diagnosable
* scalable under load
* supportable in production environments

---

## Architectural Constraints

### Identity

* Do NOT change authentication flow
* JWT structure remains:

  * `role`
  * `product_roles`
  * `access_version`

### Synq Audit

* Continue using Synq Audit as the source of truth for mutation events
* Do NOT duplicate audit storage
* UI must consume audit APIs, not recreate logs

### Compatibility

* No breaking changes to LS-COR-AUT-001 through LS-COR-AUT-007
* Admin UI must remain functional

## In Scope

---

## 1. Effective Access Caching

### Problem

Effective access is recomputed at login and potentially in validation paths.

---

### Required Implementation

Introduce caching layer:

### Cache Key

```text
tenantId:userId:accessVersion
```

### Cache Value

```json
EffectiveAccessResult
```

### TTL

* 2 to 5 minutes (configurable)

---

### Behavior

* On access request:

  * check cache
  * if hit → return cached result
  * if miss → compute + cache

* On access change:

  * AccessVersion increments → cache key changes → automatic invalidation

---

### Implementation Options

* In-memory cache (initial)
* Redis (optional, future-ready)

---

## 2. AccessVersion Optimization

### Problem

Group changes trigger updates across many users.

---

### Required Implementation

#### Batch Updates

* update users in batches instead of row-by-row

#### Optional Async Processing

* queue-based update (background worker)

---

### Requirements

* correctness > optimization
* ensure no stale access

---

## 3. Authorization Observability (Logging)

### Objective

Provide visibility into **why access was granted or denied**

---

### Required Logging

On each protected endpoint:

Log:

```json
{
  "userId": "...",
  "tenantId": "...",
  "endpoint": "...",
  "product": "SYNQ_FUND",
  "requiredRole": "SYNQFUND_REFERRER",
  "result": "ALLOW | DENY",
  "source": "Direct | Group:GroupName",
  "accessVersion": "..."
}
```

---

### Placement

* inside:

  * `RequireProductAccessFilter`
  * `RequireProductRoleFilter`

---

### Requirements

* structured logging (JSON)
* avoid sensitive data leakage
* log only at INFO level (or configurable)

---

## 4. Authorization Debug Endpoint

### Endpoint

```http
GET /api/tenants/{tenantId}/users/{userId}/access-debug
```

---

### Response

```json
{
  "products": [...],
  "roles": [...],
  "sources": {
    "products": {...},
    "roles": {...}
  },
  "groups": [...],
  "entitlements": [...],
  "accessVersion": ...
}
```

---

### Purpose

* explain "why user has access"
* support debugging
* used by UI and support teams
