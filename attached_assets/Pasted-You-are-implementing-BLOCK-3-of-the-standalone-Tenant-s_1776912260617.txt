SCOPE

A) Create a dedicated tenant domain ownership model

Add a dedicated domain entity owned by Tenant service.

The model must support:
- one tenant owning multiple domains/subdomains over time
- one primary active subdomain per tenant (business rule can be enforced in service logic if needed)
- public/internal resolution by:
  - exact host/domain
  - subdomain
  - tenant code
- room for future custom domain verification without implementing it now

Recommended entity name:
- TenantDomain

Recommended minimum fields:
- Id
- TenantId
- Host
- DomainType
- Status
- IsPrimary
- CreatedAtUtc
- UpdatedAtUtc

Where:
- Host = exact host string to match against requests
  - examples:
    - acme.legalsynq.net
    - careconnect.legalsynq.net
    - portal.clientdomain.com
- DomainType = enum or equivalent, minimum:
  - Subdomain
  - CustomDomain
- Status = enum or equivalent, minimum:
  - Pending
  - Active
  - Inactive
  - VerificationRequired
  - VerificationFailed
- IsPrimary = boolean

Rules:
- Keep this generic and portable
- Do not hardcode LegalSynq-only product semantics
- Do not implement DNS verification yet
- Normalize host matching behavior (case-insensitive, trimmed)
- Prevent duplicate active host ownership across tenants

B) Preserve current compatibility with Block 1/2 Tenant model

Current Tenant entity already has a `Subdomain` field from migration compatibility with Identity.
In this block:
- do NOT remove Tenant.Subdomain
- do NOT break existing APIs
- TenantDomain should become the canonical future model
- Tenant.Subdomain can remain as a compatibility field
- document the relationship between:
  - Tenant.Subdomain
  - TenantDomain.Host
  - IsPrimary = true
  - DomainType = Subdomain

C) Resolution service

Add a domain resolution service that can resolve a tenant by:
1. exact host
2. subdomain
3. tenant code

Resolution behavior should be explicit and safe.

Required behaviors:
- Exact host match:
  - input: full host like `acme.legalsynq.net`
  - output: resolved tenant summary
- Subdomain match:
  - input: subdomain like `acme`
  - output: resolved tenant summary based on primary/active subdomain
- Tenant code match:
  - input: tenant code
  - output: resolved tenant summary

Resolution result should include, at minimum:
- tenantId
- code
- displayName
- status
- matchedHost if applicable
- matchedBy (Host / Subdomain / Code)
- branding summary if easily available and consistent with existing service patterns
- primaryColor/logo fields only if appropriate for consumer convenience

Keep the response safe for public lookup where needed.

D) Public and internal API contracts

Add public and internal resolution endpoints.

Internal/admin endpoints:
- GET /api/v1/tenants/{tenantId}/domains
- POST /api/v1/tenants/{tenantId}/domains
- PUT /api/v1/tenants/{tenantId}/domains/{domainId}
- optionally PATCH or dedicated endpoint to set primary if needed
- optionally DELETE or soft deactivate domain if aligned to current conventions

Public/internal resolution endpoints:
- GET /api/v1/public/resolve/by-host/{host}
- GET /api/v1/public/resolve/by-subdomain/{subdomain}
- GET /api/v1/public/resolve/by-code/{code}

If route-safe encoding is awkward for full hosts, you may instead use:
- GET /api/v1/public/resolve/by-host?host=acme.legalsynq.net

Use the most robust approach for ASP.NET Core routing and gateway compatibility.

Rules:
- Public endpoints must be anonymous/read-only
- Internal domain management endpoints must follow current auth patterns
- Public resolution responses must not expose sensitive internal data
- Keep DTOs generic and portable

E) Domain management rules

Implement domain management rules at the service layer:

1. Host uniqueness
- Active hosts must be globally unique
- Prevent two tenants from owning the same active host

2. Subdomain uniqueness
- A primary active subdomain-style host must be unique

3. Primary domain rules
- A tenant can have at most one primary active subdomain domain record
- if a new subdomain primary is assigned, ensure prior primary is demoted or validation fails
- choose one explicit behavior and document it

4. Status-aware resolution
- Only active domains should resolve publicly
- pending/verification-required/failed/inactive should not resolve publicly

5. Host normalization
- lower-case
- trim whitespace
- strip protocol if mistakenly supplied
- do not store path/query fragments
- validate host format

F) Validation

Add validation for:
- valid hostname/domain format
- subdomain-safe values
- no protocol in stored hosts after normalization
- uniqueness constraints
- allowed status transitions if you implement transition rules
- allowed domain type values

G) Persistence / schema

Add EF Core persistence for the domain model in tenant_db.

Expected:
- new domain table
- indexes for host uniqueness and tenant lookups
- migration for Block 3

Recommended indexes:
- unique index on normalized active host if feasible
- index on TenantId
- index on Host
- index on Status

If filtered indexes are not practical in MySQL/EF for active-only uniqueness, enforce with service-layer validation and document it clearly.

H) Consumer readiness

Design resolution contracts so they can later support:
- API Gateway tenant resolution
- Next.js login/public bootstrap flows
- future replacement of Identity tenant lookup logic

Do not modify the frontend in this block unless absolutely necessary.
This block is backend-first and consumer-ready.
