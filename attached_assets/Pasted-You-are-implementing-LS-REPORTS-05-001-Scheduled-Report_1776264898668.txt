# STORY ID
**LS-REPORTS-05-001 — Scheduled Report Execution & Delivery**

---

# OBJECTIVE
Enable tenants to create routine scheduled report jobs that:
- execute a report on a defined cycle
- export it in a chosen format
- deliver it through a configured channel
- persist run history and status

This story must establish:
- scheduling model
- delivery model
- background execution flow
- run tracking
- operational guardrails

---

# REQUIRED IMPLEMENTATION

## A. Scheduling Domain / Persistence Model

Add persistence for scheduled report jobs and run history.

### 1. ReportSchedule
Suggested fields:
- `Id`
- `TenantId`
- `ReportTemplateId`
- `ProductCode`
- `OrganizationType`
- `Name`
- `Description`
- `IsActive`
- `FrequencyType` (`Daily`, `Weekly`, `Monthly`)
- `FrequencyConfigJson` (for day-of-week/day-of-month/time)
- `Timezone`
- `NextRunAtUtc`
- `LastRunAtUtc` (nullable)
- `UseOverride` (bool)
- `ExportFormat` (`CSV`, `XLSX`, `PDF`)
- `DeliveryMethod` (`OnScreen`, `Email`, `SFTP`)
- `DeliveryConfigJson`
- `ParametersJson` (nullable)
- `RequiredFeatureCode` (nullable)
- `MinimumTierCode` (nullable)
- `CreatedAtUtc`
- `CreatedByUserId`
- `UpdatedAtUtc`
- `UpdatedByUserId`

### 2. ReportScheduleRun
Suggested fields:
- `Id`
- `ReportScheduleId`
- `ExecutionId` (nullable until created)
- `ExportId` (nullable until created)
- `Status` (`Pending`, `Running`, `Completed`, `Failed`, `Delivered`, `DeliveryFailed`)
- `ScheduledForUtc`
- `StartedAtUtc` (nullable)
- `CompletedAtUtc` (nullable)
- `DeliveredAtUtc` (nullable)
- `FailureReason` (nullable)
- `DeliveryResultJson` (nullable)
- `GeneratedFileName` (nullable)
- `GeneratedFileSize` (nullable)
- `CreatedAtUtc`

### Rules
- schedules are tenant-scoped
- schedules are product-scoped
- schedules must point to a valid template
- schedules may reference tenant overrides implicitly through `UseOverride=true`

## D. Delivery Abstraction

Create a delivery abstraction.

### Interface
- `IReportDeliveryAdapter`

### Responsibilities
- deliver generated export through configured channel
- return structured delivery result

### First implementations
1. `OnScreenReportDeliveryAdapter`
   - stores the generated file content in-memory only for the run response or marks it as available through existing export response metadata
   - for this story, acceptable behavior is to treat on-screen as “generated and ready” without external delivery

2. `EmailReportDeliveryAdapter`
   - initial implementation can use the existing notification adapter pathway or a mock email delivery adapter if the real email path is not ready
   - must accept recipients, subject, body template if provided in config

3. `SftpReportDeliveryAdapter`
   - initial implementation can be stub/mock-safe if real SFTP is not yet available
   - must keep contract real and delivery result structured

### Important
Keep delivery behind adapter abstraction.
Do NOT hardcode delivery logic into the schedule service.

## E. Schedule Service Layer

Create:
- `IReportScheduleService`
- `ReportScheduleService`

### Responsibilities
- validate schedule creation/update
- enforce schedule eligibility
- create/update/deactivate schedules
- calculate next run time
- list schedules
- get schedule by id
- trigger manual run if included
- create run records
- orchestrate execution → export → delivery
- update run status
- emit audit events

The API layer must not call repositories or adapters directly.

---

## F. Scheduler / Worker Processing

Implement a background processing path using the existing worker foundation.

### Requirements
- periodically scan active schedules due for execution
- create/update `ReportScheduleRun`
- invoke execution service
- invoke export service
- invoke delivery adapter
- update schedule next-run time
- persist success/failure state

### First-pass acceptable model
- polling background worker every N seconds
- claim due schedules carefully
- no elaborate distributed lock manager required yet, but avoid obvious double-processing

### Guardrails
- max schedules processed per poll (document chosen cap)
- safe error handling per schedule
- one failed schedule must not stop the worker

## G. Frequency Model

Support at minimum:
- `Daily`
- `Weekly`
- `Monthly`

### FrequencyConfigJson examples
- Daily: `{ "hour": 9, "minute": 0 }`
- Weekly: `{ "dayOfWeek": "Monday", "hour": 9, "minute": 0 }`
- Monthly: `{ "dayOfMonth": 1, "hour": 9, "minute": 0 }`

### Rules
- use timezone-aware calculation
- persist `NextRunAtUtc`
- recompute after successful or failed attempt

Do NOT implement complex cron parsing in this story.

## J. Validation Rules

At minimum enforce:

### Schedule creation
- `TenantId` required
- `ReportTemplateId` required
- `ProductCode` required
- `OrganizationType` required
- `Name` required
- `FrequencyType` required and supported
- `Timezone` required
- `ExportFormat` required and supported
- `DeliveryMethod` required and supported
- `CreatedByUserId` required

### Eligibility
- template must exist
- template must be assigned to tenant
- published version must exist
- product code must align with template
- if `UseOverride=true`, override path must remain valid if one exists

### Delivery validation
- Email requires recipient config
- SFTP requires minimum connection/path config
- OnScreen requires no external config

### Conflict / safety
- reject unsupported combinations
- reject invalid frequency config
- reject inactive/unassigned template scheduling
