You are implementing LS-LIENS-UI-006 for the LegalSynq Liens v2 migration.

Feature ID: LS-LIENS-UI-006
Feature Title: Contacts & Participants Integration

Objective:
Wire the Contacts module and participant relationships to real backend APIs so Cases, Liens, Servicing, and related workflows reference real people and organizations instead of mock records, static labels, or flattened display strings. Use the same enforced frontend service-layer architecture established in LS-LIENS-UI-002 through LS-LIENS-UI-005.

This feature gives the system real participant identity and relationship integrity.

--------------------------------------------------
CRITICAL ARCHITECTURE RULES (NON-NEGOTIABLE)
--------------------------------------------------

1. Preserve the v2 shell (top nav + side nav).
2. Preserve the current UI layout and workflow feel unless minimal changes are required for real API integration.
3. Enforce strict layering:
   - API client layer
   - service layer
   - mapper / adapter layer
   - UI state layer
   - presentation layer
4. No direct HTTP calls in pages/components.
5. No DTO-to-UI transformation logic inside components.
6. No production mock/static contact or participant data.
7. Follow the same architecture pattern used in LS-LIENS-UI-002, UI-003, UI-004, and UI-005.
8. Do not invent backend APIs. Use only what actually exists.
9. Preserve donor SynqLiens participant semantics where backend support exists.
10. Participant relationships shown in Cases, Liens, and Servicing must come from real backend-linked records, not hardcoded labels.
11. If Zustand or another store is used, it must be UI state only, not the source of truth for contact or participant data.
