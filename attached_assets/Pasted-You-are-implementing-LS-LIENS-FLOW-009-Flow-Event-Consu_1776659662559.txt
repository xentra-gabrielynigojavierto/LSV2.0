==================================================
OBJECTIVE
==================================================

Enable real-time synchronization between:

- Flow workflow instance step changes
AND
- Synq Liens tasks linked to those workflow instances

This feature must ensure:

1. when a Flow workflow step changes, related tasks update automatically
2. tasks remain continuously aligned with Flow, not only on read
3. system becomes event-driven instead of purely reactive
4. orchestration behavior becomes near-real-time and reliable

This is the next step after:
- FLOW-007 task ↔ Flow instance linking
- FLOW-008 task ↔ Flow stage synchronization

==================================================
CRITICAL ARCHITECTURE RULES
==================================================

1. Flow remains the source of truth for workflow execution
2. Liens consumes Flow events and updates task state accordingly
3. Event handling must be:
   - idempotent
   - safe
   - non-blocking
4. Duplicate or delayed events must not corrupt task state
5. Do NOT introduce polling
6. Do NOT redesign the full platform messaging architecture
7. Prefer internal-first event ingestion if no production bus/webhook pipeline already exists
8. Reuse FLOW-008 synchronization logic instead of duplicating it

{
  "eventType": "workflow.step.changed",
  "tenantId": "<guid>",
  "productCode": "SYNQ_LIENS",
  "workflowInstanceId": "<guid>",
  "previousStepKey": "<string|null>",
  "currentStepKey": "<string>",
  "timestamp": "<iso8601>"
}

Requirements:
- validate required fields
- enforce tenant isolation
- include only the minimal fields needed for task synchronization
- use stable naming consistent with platform conventions where possible

If platform event naming differs, align to existing conventions and document the final contract clearly.

==================================================
STEP 4 — IMPLEMENT EVENT HANDLER SERVICE
==================================================

Create a dedicated event handler, for example:

- `FlowEventHandler`
or
- `TaskFlowSyncEventHandler`

When receiving:
- workflow.step.changed

Behavior:
1. validate event
2. find all tasks where:
   - `WorkflowInstanceId == event.workflowInstanceId`
3. for each relevant task:
   - resolve mapped task stage from `currentStepKey`
   - reuse FLOW-008 mapping rules
4. if mapped stage differs from task's current stage:
   - update task stage
   - update `WorkflowStepKey`
   - persist changes
   - emit audit event
5. if already aligned:
   - no-op
6. if no valid stage mapping exists:
   - no-op or mark unmapped behavior according to FLOW-008 conventions
   - document behavior clearly

STEP 6 — BULK UPDATE STRATEGY
==================================================

Implement efficient lookup/update behavior.

Requirements:
- batch query tasks by `WorkflowInstanceId`
- avoid N+1 queries where possible
- update only tasks that actually need changes
- do not rewrite all matching tasks blindly
- keep writes minimal

If multiple tasks are linked to the same workflow instance:
- process all relevant tasks deterministically
- emit audit only for actual changes

==================================================
STEP 7 — IDEMPOTENCY AND SAFETY
==================================================

Event handling must be idempotent.

Rules:
- duplicate identical events must not cause duplicate state changes
- already-aligned tasks must not be rewritten unnecessarily
- stale events must not regress state if a newer step is already stored, when determinable

Implement the safest available strategy supported by the current codebase.

If the payload timestamp can be trusted:
- use it to ignore older events if useful

If not:
- rely on current Flow state lookup or current persisted state for safe no-op behavior

Document the chosen strategy clearly.

STEP 9 — ERROR HANDLING
==================================================

Implement safe failure behavior.

Requirements:
- invalid payload → reject safely with clear internal error response
- unknown `workflowInstanceId` → no-op
- no matching tasks → no-op
- duplicate event → no-op
- mapping failure → no-op or unsynced behavior per FLOW-008 conventions
- task update failure for one task should not corrupt unrelated tasks

Do NOT:
- crash request pipeline
- block other operations
- make task system dependent on event delivery for correctness
