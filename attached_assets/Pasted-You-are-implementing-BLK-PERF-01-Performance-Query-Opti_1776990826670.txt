OBJECTIVE

Improve production performance and scalability for tenant-scoped LegalSynq flows.

At the end of this block:

- high-traffic tenant-scoped queries are indexed or optimized
- pagination is enforced where needed
- admin/reporting queries avoid unnecessary full scans
- tenant-aware cache/query patterns are safe
- no security, governance, or tenant-isolation regressions are introduced

--------------------------------------------------

SCOPE

This block includes:

- query hotspot audit
- database index review
- pagination enforcement review
- no-tracking read optimization
- tenant-aware query-shape improvements
- safe performance improvements for public/admin/tenant portal endpoints

This block does NOT include:

- new analytics products
- UI redesign
- broad schema redesign
- replacing EF Core
- distributed caching unless a safe existing pattern already exists

--------------------------------------------------

PART A — PERFORMANCE HOTSPOT AUDIT

--------------------------------------------------

Audit likely high-traffic or heavy endpoints across CareConnect first.

At minimum inspect:

1. Public routes
   - public `/network`
   - public network providers
   - public provider markers
   - public referral submission

2. Tenant portal routes
   - referral list/detail
   - network list/detail
   - provider map/list/search
   - appointments
   - notes

3. Admin routes
   - dashboard
   - analytics funnel
   - performance metrics
   - blocked-provider logs
   - activation queue
   - admin referrals monitor

4. Provider/common portal
   - onboarding status
   - provider dashboard/referral detail if present

For each, document:
- query shape
- tenant filter presence
- pagination status
- potential full table scans
- whether `AsNoTracking()` is appropriate
- indexes likely needed

--------------------------------------------------

PART B — DATABASE INDEX REVIEW

--------------------------------------------------

Review EF Core entity configurations and migrations for critical tables.

At minimum inspect tables/entities related to:

- Referrals
- Providers
- Networks
- NetworkProviders / join tables
- ActivationRequests
- BlockedProviderAccessLogs
- Appointments
- ReferralNotes
- ProviderFacilities / Facilities if applicable

Add or update indexes where justified.

Priority index patterns:

1. Tenant-scoped lookups
   - `(TenantId, Id)` where applicable
   - `(TenantId, CreatedAtUtc)`
   - `(TenantId, Status)`
   - `(TenantId, Status, CreatedAtUtc)`

2. Public network/provider lookups
   - `(TenantId, NetworkId)`
   - provider marker/map queries
   - active/accepting-referrals filters

3. Admin dashboards / analytics
   - rolling-window queries by created date
   - status + tenant filters
   - blocked logs by tenant/date/user/reason

4. Provider/common portal
   - `IdentityUserId`
   - provider onboarding/status lookup
   - pending tenant onboarding fields if queried

Rules:
- avoid adding redundant indexes
- prefer composite indexes that match actual WHERE/order patterns
- document why each new index exists
- add EF migration if schema changes

--------------------------------------------------

PART C — PAGINATION ENFORCEMENT

--------------------------------------------------

Audit list endpoints for pagination.

Requirements:

- high-cardinality lists must be paginated
- page size must be clamped
- default page size must be safe
- unbounded list endpoints must be justified or fixed

Focus on:

- referrals
- providers
- networks
- activation requests
- blocked-provider logs
- notes/history/audit-style lists
- admin monitor endpoints

If an endpoint intentionally returns all rows, document why the cardinality is bounded.

--------------------------------------------------

PART D — QUERY SHAPE OPTIMIZATION

--------------------------------------------------

Improve query shapes where safe.

Examples:

- add `AsNoTracking()` to read-only queries
- project only needed fields instead of loading full entities
- avoid unnecessary `Include()` on list endpoints
- avoid N+1 query patterns
- apply tenant filters before grouping / aggregation
- apply date/status filters before materialization
- avoid post-fetch filtering when query-level filtering is possible

Rules:
- preserve response shapes
- preserve tenant scoping
- preserve governance checks
- do not optimize by removing necessary authorization/ownership validation

--------------------------------------------------

PART E — PUBLIC ROUTE PERFORMANCE

--------------------------------------------------

Optimize public network/provider queries safely.

Requirements:

- public routes must remain trust-boundary validated from BLK-SEC-02-02
- public queries must remain tenant-scoped
- provider marker/list endpoints should avoid loading unnecessary large fields
- public referral submission should not do unnecessary broad provider lookup

No public endpoint may trust client-supplied tenant context.
