--------------------------------------
1. PROVIDER CONFIG CREATE FLOW
--------------------------------------

Enhance:
- /notifications/providers

Add a create flow for provider configs.

Expected backend endpoint:
- POST /notifications/v1/providers/configs

UI requirements:
- add "New Provider Config" action
- open a create form (modal or drawer preferred)
- form must support at minimum:
  - provider type
  - channel
  - ownership mode if required by backend
  - display name
  - credentials / endpoint fields appropriate to selected provider
  - sender config fields where appropriate

Supported provider types for this phase:
- SendGrid
- SMTP
- Twilio

Behavior:
- provider-specific fields should change based on selected provider type
- secret fields must be entered but never echoed back after save
- on success:
  - close form
  - revalidate providers data
  - show success feedback

Validation:
- required fields by provider type
- do not allow incomplete or ambiguous provider configs
- surface backend validation errors clearly

Do NOT build a multi-step onboarding wizard unless absolutely necessary.
A structured single-form or dynamic form is preferred.

2. PROVIDER CONFIG EDIT FLOW
==================================================

Enhance:
- /notifications/providers

Add an edit flow for existing provider configs.

Expected backend endpoint:
- PATCH /notifications/v1/providers/configs/:id

UI requirements:
- add Edit action per provider config row
- open edit form (modal/drawer/page)
- prefill editable non-secret fields
- for secret fields:
  - show configured/not configured state
  - allow replacement/rotation
  - never show plaintext secret values

Behavior:
- on save:
  - revalidate providers data
  - show success state
- support credential rotation cleanly if backend allows it

Do NOT mix edit flow with validate/test/activate/deactivate buttons in a confusing way.
Keep actions and edit clearly separated.
