==================================================
METRIC DEFINITIONS
==================================================

--------------------------------------------------
A. TIME TO ACCEPT (TTA)
--------------------------------------------------

Definition:
- accepted timestamp minus created timestamp

Formula:
- TTA = AcceptedAtUtc - CreatedAtUtc

Requirements:
- only compute for referrals that have actually been accepted
- if accepted timestamp is unavailable in current schema, derive it from the earliest referral status history entry where status transitioned to Accepted
- if no reliable accepted timestamp exists, document the fallback approach clearly in the report
- corrupted or missing timestamps must be excluded from TTA calculations

Output:
- average time to accept (hours, or a consistent unit documented in report)

--------------------------------------------------
B. ACCEPTANCE RATE
--------------------------------------------------

Definition:
- accepted referrals divided by total referrals in the selected range

Formula:
- Acceptance Rate = Accepted Referrals / Total Referrals

Requirements:
- selected time window must be clearly defined
- divide-by-zero must never produce NaN or Infinity
- return 0 or null consistently when denominator is zero; document chosen behavior

--------------------------------------------------
C. REFERRAL AGING (UNACCEPTED)
--------------------------------------------------

Definition:
- how long referrals in New have been waiting since creation

Formula:
- Aging = NowUtc - CreatedAtUtc for referrals currently in New

Required buckets:
- < 1 hour
- 1–24 hours
- 1–3 days
- 3+ days

Requirements:
- only count referrals currently in New
- bucket boundaries must be deterministic and documented
- use UTC only

--------------------------------------------------
D. PROVIDER RESPONSIVENESS
--------------------------------------------------

For each provider in the selected range, compute:
- Total Referrals
- Accepted Referrals
- Acceptance Rate
- Average Time to Accept

Requirements:
- provider rows with zero accepted referrals may show avg TTA as null/blank/—
- provider rows must still show totals and acceptance rate
- provider display should be sortable on key fields if low-risk

REQUIREMENTS

--------------------------------------------------
A. PERFORMANCE API
--------------------------------------------------

Add a minimal admin endpoint.

Suggested route:
- GET /api/admin/performance

Suggested query params:
- since=<ISO datetime>   (optional)
- or simple preset params if preferred (e.g. days=7)

Recommended default:
- last 7 days

Response should include:

1. Summary metrics
   - totalReferrals
   - acceptedReferrals
   - acceptanceRate
   - avgTimeToAcceptHours (or documented unit)

2. Aging distribution
   - lt1h
   - h1to24
   - d1to3
   - gt3d

3. Provider performance rows
   - providerId
   - providerName
   - totalReferrals
   - acceptedReferrals
   - acceptanceRate
   - avgTimeToAcceptHours

Rules:
- admin-only
- deterministic
- reasonably fast
- no full BI-style query engine

--------------------------------------------------
B. DATA SOURCE STRATEGY
--------------------------------------------------

Preferred sources:
- referrals
- referral status histories / audit history if needed for AcceptedAt derivation

Rules:

- document any approximations clearly

G. EMPTY / PARTIAL DATA HANDLING
--------------------------------------------------

Must handle safely:

1. no referrals in range
- show empty state
- summary cards should display safe values

2. no accepted referrals
- acceptance rate = 0 or documented safe value
- avg TTA = null/blank/— or documented safe value

3. provider with zero accepted
- still show row if there are referrals
- avg TTA handled safely

4. missing timestamps
- exclude affected records from TTA calculation
- do not crash or misreport silently; document fallback

- performance page: admin-only

--------------------------------------------------
I. PERFORMANCE / IMPLEMENTATION CONSTRAINTS
--------------------------------------------------

Requirements:
- typical admin load should be reasonably fast
- avoid scanning unnecessary data where filtering can be applied early
- use indexes and filtered queries where practical
- in-memory aggregation is acceptable only after a bounded/refined dataset is queried
- do not introduce expensive joins unless justified

If there is a tradeoff between precision and speed:
- choose the smallest correct approach
- document it clearly in the report

--------------------------------------------------
J. EDGE CASES
--------------------------------------------------

Handle safely:

1. accepted referral with no accepted timestamp in primary record
- derive from status history or documented fallback

2. duplicate/odd history entries
- choose consistent derivation rule (e.g. earliest Accepted transition)

3. referrals created outside the range but accepted within the range
- define whether selection is based on created date, accepted date, or another anchor
- document the chosen rule clearly

Recommended default:
- base the cohort on referral created date for total/acceptance-rate consistency
- if another anchor is used, document it explicitly

4. old referrals still New
- they must appear in aging buckets correctly if currently New and within query logic assumptions
