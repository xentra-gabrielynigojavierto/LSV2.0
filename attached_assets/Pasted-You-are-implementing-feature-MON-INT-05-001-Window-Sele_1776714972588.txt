You are implementing feature MON-INT-05-001 — Window Selector (7d / 30d View).

OBJECTIVE
Extend the monitoring UI to support selectable time windows so operators can switch the availability bars and latency trend views from the current fixed 24h window to additional historical windows, starting with 7d and 30d.

This is a visualization and UX enhancement built on top of the existing uptime aggregation engine, availability bars, and response time charts.

ARCHITECTURE RULE (NON-NEGOTIABLE)
- Monitoring Service = source of truth
- Control Center = read-only consumer
- Windowed views must be driven by real Monitoring uptime/history data
- Do NOT fabricate longer-history metrics
- Do NOT change uptime aggregation formulas unless an actual blocker is proven
- Do NOT expose internal-only metrics on the public page beyond what is already intentionally public
- Prefer using existing BFF patterns and extend them minimally

MANDATORY EXECUTION RULES
- Create the report file FIRST at: /analysis/MON-INT-05-001-report.md
- Update the report incrementally after each meaningful step
- Do NOT wait until the end to write the report
- Prefer minimal, reversible changes
- Do NOT redesign the monitoring UI or public status page
- Keep internal and public concerns clearly separated
- Maintain existing 24h behavior as the default/fallback when no window is selected

CURRENT CONTEXT
- Public `/status` page exists
- Internal `/monitoring` page exists
- Uptime aggregation engine supports 24h / 7d / 30d / 90d windows
- Availability timeline bars exist on the public status page for 24h
- Response time charts exist on the internal monitoring page for 24h
- The next requirement is selectable windowing for historical views

GOAL OF THIS FEATURE
By the end of this feature:
- users can select a larger historical window where appropriate
- availability bars and/or latency views react to the selected window using real data
- the UI remains stable and understandable
- no fake interpolation or placeholder history is introduced
