OBJECTIVE

Evolve Flow into a true execution engine by shifting execution state to WorkflowInstance, enabling full end-to-end workflow execution, introducing machine-to-machine authentication, and improving observability and scalability, while preserving the bounded-service architecture.

This phase focuses on execution correctness, autonomy, and real operational validation.

SCOPE INCLUDES

- shifting execution responsibility to WorkflowInstance
- implementing a lightweight step/state progression model
- enabling full end-to-end workflow execution
- introducing machine-to-machine authentication
- updating APIs for execution behavior
- updating product integrations to support execution
- improving observability and logging
- documentation updates

DO NOT BUILD

- full BPM engine or DSL
- large-scale UI redesign
- full event bus rollout
- deep analytics/reporting system
- shared database consolidation
- external ecosystem integrations beyond immediate needs

CRITICAL REQUIREMENTS

You MUST:
- make WorkflowInstance the execution authority
- ensure workflows can be executed end-to-end
- maintain tenant and authorization integrity
- preserve Flow as a bounded service
- avoid breaking existing integrations unless clearly documented
- validate real execution flows across products

CURRENT CONTEXT FROM PHASE 4

Flow already has:
- WorkflowInstance entity (basic)
- Product integration across SynqLien, CareConnect, SynqFund
- API-based integration via IFlowClient
- centralized permission codes
- operational hardening (retry, logging, error handling)
- gateway routing and auth enforcement

Known limitations:
- execution logic still partially tied to TaskItem
- WorkflowInstance is not yet the execution driver
- no formal step/state engine
- no machine-to-machine authentication
- no full positive end-to-end validation completed

IMPLEMENTATION REQUIREMENTS

1. WORKFLOW EXECUTION MODEL SHIFT

Requirements:
- extend WorkflowInstance to include:
  - CurrentStep
  - Status (Active, Completed, Failed, Cancelled)
  - StartedAt
  - CompletedAt
  - optional AssignedTo
- move execution control from TaskItem to WorkflowInstance
- ensure TaskItem becomes step-level artifact only
- update services (WorkflowService, TaskService) to use WorkflowInstance for execution decisions
- maintain backward compatibility or clearly document breaking changes

2. STEP / STATE ENGINE (LIGHTWEIGHT)

Requirements:
- define step structure (id, name, order)
- support:
  - next step transition
  - completion
  - optional simple branching (basic conditional logic only)
- ensure:
  - valid transitions only
  - invalid transitions are rejected
- store step metadata in workflow definition or lightweight config
- DO NOT implement full BPM engine

3. API ENHANCEMENTS

Requirements:
- extend Flow APIs to support:
  - get workflow instance details
  - get current step
  - progress workflow step
  - complete workflow
- enforce:
  - tenant validation
  - authorization
  - valid state transitions
- avoid unnecessary breaking changes

4. MACHINE-TO-MACHINE AUTHENTICATION

Requirements:
- introduce service-to-service authentication model
- support:
  - service tokens OR
  - on-behalf-of (OBO) pattern if feasible
- update IFlowClient to:
  - optionally attach service credentials
  - preserve user context where needed
- ensure Flow can distinguish:
  - user-initiated calls
  - system/service calls
- do not break existing user-based flows

5. PRODUCT INTEGRATION UPDATE

Requirements:
- update product integrations (Liens, CareConnect, Fund) to:
  - use WorkflowInstanceId
  - support step progression
- ensure:
  - products can start workflows
  - retrieve instance status
  - progress workflow steps
- keep product logic thin (orchestration only)

6. FULL END-TO-END VALIDATION

- implement and validate real execution flows for each product:

SynqLien:

- complete workflow
