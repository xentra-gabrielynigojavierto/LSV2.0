FEATURE ID
TENANT-B01 — Standalone Tenant Service Foundation
================================================================

OBJECTIVE
Create the foundation for a brand new standalone Tenant service that will become the canonical owner of tenant/company master data, separate from Identity. This first block must establish:
1. migration-aware field mapping from Identity to Tenant,
2. a new .NET Tenant service scaffold,
3. a separate MySQL database configuration for tenant_db,
4. EF Core persistence and initial schema,
5. health/diagnostic endpoints,
6. base API contract,
7. a canonical core Tenant entity.

This block does NOT perform full migration or cutover yet. It prepares for that safely.

================================================================
PLATFORM / STACK RULES
================================================================

Frontend stack context:
- Next.js (React)
- TypeScript
- Tailwind CSS
- Radix UI / shadcn/ui
- pnpm monorepo

Backend stack rules for this feature:
- C# / .NET only
- ASP.NET Core only
- Entity Framework Core only
- MySQL only
- Follow existing microservice architecture and service layering patterns
- Reuse current solution/project structure conventions from existing services
- Must fit current monorepo and Replit shell-script orchestration approach

Service placement:
- Create a new Tenant service as a backend microservice
- Recommended port: 5005
- This service must be standalone and portable, not hardcoded to LegalSynq-only assumptions

Database rule:
- Use a separate MySQL database for this service
- Database name: tenant_db
- Do NOT reuse identity_db
- Do NOT require direct cross-service DB writes from other services

================================================================
ARCHITECTURAL RULES
================================================================

Tenant service WILL own in the future:
- tenant/company profile
- branding metadata
- domain/subdomain ownership
- product entitlements
- tenant settings
- external references

Tenant service will NOT own:
- users
- authentication
- passwords
- sessions
- invitations
- roles / permissions / RBAC logic
- document binary storage

Identity will eventually keep:
- users
- groups
- roles
- permissions
- access control
- tenant references only where needed

Documents service remains the file/blob owner.
Tenant service stores asset references only.

Critical ID rule:
- The future migration must preserve TenantId exactly from Identity.
- In this block, design the schema and contracts to preserve the same TenantId values.
- Do NOT generate a new tenant identity model that breaks cross-service references.
