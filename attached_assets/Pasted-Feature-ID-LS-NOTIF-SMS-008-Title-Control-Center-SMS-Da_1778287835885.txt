Feature ID: LS-NOTIF-SMS-008

Title:
Control Center SMS Dashboard APIs

Objective:
Add Control Center-ready SMS dashboard capabilities inside Notification Service so platform/admin users can view SMS delivery activity, reconciliation health, tenant/provider activity, and failure trends through Notification Service APIs.

Critical Architectural Constraint:
Although this feature will be implemented inside the LegalSynq monorepo, all SMS dashboard aggregation, reporting, and Control Center-facing SMS data APIs must remain inside the Notification Service boundary.

Requirements:
- Do not build Control Center UI in this feature.
- Do not add SMS dashboard aggregation logic to unrelated services.
- Do not create shared cross-service SMS dashboard abstractions unless an existing Notification Service shared/internal module already exists.
- Do not modify other services except where strictly necessary for build/config compatibility.
- Notification Service must own:
  - SMS dashboard aggregation
  - SMS delivery KPI calculations
  - SMS reconciliation KPI calculations
  - SMS tenant activity grouping
  - SMS provider/provider-config grouping
  - SMS failure-category grouping
  - Control Center-facing SMS dashboard APIs
- Control Center should consume Notification Service APIs in a future UI feature.
- Preserve Notification Service as the system of record for SMS delivery, tracking, reconciliation, and dashboard reporting.

Implementation Requirements:
1. Add platform/admin SMS dashboard APIs inside Notification Service.
2. Provide high-level SMS delivery KPIs.
3. Provide SMS reconciliation KPIs.
4. Provide failure category/error breakdowns.
5. Provide tenant activity breakdowns.
6. Provide provider/provider-config/ownership breakdowns.
7. Provide time-series trend data for delivery and reconciliation status.
8. Support date range, tenant, provider, providerConfigId, providerOwnershipMode, and status filters where appropriate.
9. Require existing platform/admin authorization.
10. Preserve existing SMS activity APIs from LS-NOTIF-SMS-006 and reconciliation tracking from LS-NOTIF-SMS-007.
11. Preserve outbound SMS send, retry/dead-letter, webhook, compliance, tenant provider resolution, and reconciliation behavior.
12. Do not add or expand conversational inbound SMS behavior.

Critical Constraints:
- Reuse existing NotificationAttempt, SmsActivityRepository, SmsActivityService, SmsActivityDtos, admin endpoint, authorization, and query patterns where appropriate.
- Do NOT introduce large rewrites.
- Keep implementation incremental and production-oriented.
- Do NOT store or expose provider credentials, CredentialsJson, SettingsJson, auth tokens, or raw provider payloads.
- Do NOT expose full phone numbers in dashboard responses.
- Dashboard APIs must be read-only.
- Dashboard APIs must not trigger SMS sends, retries, reconciliation, or provider calls.
- Validate against the current codebase before implementing.
- Create the report FIRST and continuously update it during implementation.
