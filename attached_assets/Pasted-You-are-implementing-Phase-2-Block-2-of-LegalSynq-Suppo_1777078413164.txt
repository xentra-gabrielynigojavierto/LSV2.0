STEP 3 — PRODUCT REFERENCE MODEL ALIGNMENT:

Goal: Normalize how Support references external product records.

Ensure Support uses a **generic reference model**, such as:

- productKey (string) → e.g. "liens", "fund", "careconnect"
- entityType (string) → e.g. "case", "account", "patient"
- entityId (string) → ID of record in that service
- entityNumber (optional display identifier)
- metadata (optional JSON)

If already present:
- validate and standardize naming
- do NOT redesign unnecessarily

If minimal extension required:
- add only non-breaking fields

DO NOT:
- add foreign keys
- add cross-service joins
- replicate product schemas

STEP 4 — DEEP LINK STRATEGY:

Define and implement a **pure URL-based deep link model**

Each product reference must be convertible to a URL.

Example pattern:

- Liens:
  /liens/cases/{id}

- Fund:
  /fund/accounts/{id}

- CareConnect:
  /care/patients/{id}

Implementation:
- Add a lightweight mapping layer:
  productKey + entityType → route template

Example:
{
  "liens.case": "/liens/cases/{id}",
  "fund.account": "/fund/accounts/{id}",
  "careconnect.patient": "/care/patients/{id}"
}

Replace {id} with entityId.

This mapping must:
- live in UI layer OR shared config
- NOT be embedded in Support backend logic (except optional metadata passthrough)

STEP 5 — BACKEND ADJUSTMENTS (MINIMAL):

Support backend should:
- accept product reference data when creating/updating tickets
- store references only (no validation against product service)
- return references in ticket response

Optional:
- validate presence of required fields (productKey, entityId)
- validate format only (not existence)

DO NOT:
- call Liens/Fund/CareConnect APIs
- enforce referential integrity
