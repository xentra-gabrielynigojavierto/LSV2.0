PART B — REMOVE TRUST IN CLIENT-SUPPLIED TENANT INPUTS

--------------------------------------------------

Where tenant context should be server-derived, remove or guard trust in:

- `tenantId` from request body
- `tenantId` from query string
- `tenantId` from route parameter
- `tenantCode` from client when resolved tenant context already exists
- `X-Tenant-Id` or similar headers if they can originate from untrusted clients directly

Rules:

1. Public flows:
   - tenant context must come from host/subdomain or approved server-side resolution
   - not from arbitrary caller-provided tenant IDs

2. Authenticated tenant portal flows:
   - tenant context must come from authenticated request context / JWT claims / server-side context
   - not from caller-supplied tenantId parameters

3. Common Portal flows:
   - must not accidentally inherit arbitrary tenant portal context
   - only use tenant context where explicitly and correctly derived

Implement defensive rejection or ignore behavior where needed.

--------------------------------------------------

PART C — PUBLIC ROUTE HARDENING

--------------------------------------------------

Audit and fix public routes so they cannot be forced into another tenant/network context.

Requirements:

1. Public `/network`
   - tenant/network context must resolve from host/subdomain or approved dev fallback only
   - no body/query tampering should redirect data access into another tenant

2. Public provider directory / lookup
   - results must be constrained to the resolved public tenant/network context only

3. Public referral submission
   - referral must be created only for the resolved tenant/network context
   - no client-supplied tenantId override should be honored

4. Any public map/detail endpoints
   - must return only the resolved tenant/network-scoped data

If a trusted server-side header such as `X-Tenant-Id` is used behind a BFF, ensure:
- it is only consumed where the request path is known to come from that trusted server layer
- untrusted direct callers cannot spoof it in production deployment assumptions
Document the trust boundary explicitly.
