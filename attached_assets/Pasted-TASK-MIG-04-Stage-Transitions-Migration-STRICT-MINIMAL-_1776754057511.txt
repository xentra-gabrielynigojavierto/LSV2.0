TASK-MIG-04 — Stage Transitions Migration (STRICT MINIMAL MODEL)

OBJECTIVE:
Introduce and migrate the Liens task-board stage transition model into the Task service using the **smallest possible schema and logic** so Task can own allowed task-stage movement rules without recreating workflow/orchestration behavior.

This step is ONLY about simple task-stage transitions.

NON-NEGOTIABLE RULES:
- DO NOT migrate Flow orchestration transitions
- DO NOT add rules engine
- DO NOT add conditions JSON
- DO NOT add branching logic
- DO NOT add automation hooks/actions
- DO NOT redesign workflow behavior
- DO NOT delete Liens transition tables in this step
- ZERO UI behavior regression
- MUST preserve existing Liens task-board movement behavior
- MUST support rollback
- MUST create report FIRST and update incrementally
- MUST inspect actual code/schema before implementation
- NO placeholders

STRICT MODEL:
The ONLY transition model allowed in Task in this prompt is:

- `from_stage_id`
- `to_stage_id`
- tenant/product scope
- active/inactive flag
- ordering metadata only if needed

Nothing beyond that.

CRITICAL BOUNDARY:
- `liens_WorkflowTransitions` = task-board allowed moves → can move to Task
- `flow_workflow_transitions` = orchestration graph transitions → must stay in Flow
- Do NOT blur these concepts anywhere in implementation
