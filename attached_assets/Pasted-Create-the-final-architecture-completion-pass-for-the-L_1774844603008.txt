## Phase 1 — finalize OrganizationType as the only true source of truth

Current state:
- OrganizationTypeId is primary in practice
- OrgType string still exists for compatibility and fallback
- TODO markers indicate OrgType string removal is the next phase

Tasks:
- inspect all remaining OrgType reads/writes across:
  - Identity.Domain
  - AuthService
  - UserService
  - JwtTokenService
  - DTOs
  - CurrentRequestContext
  - Admin endpoints
  - response mappers
- make OrganizationTypeId the only authoritative source for business logic
- where org_type string is still needed for compatibility, derive it from OrgTypeMapper using OrganizationTypeId
- remove all places where OrgType string is treated as the write authority
- add a backfill migration if needed to ensure all existing Organizations have OrganizationTypeId populated from OrgType
- add defensive validation so new/updated Organizations cannot drift into mismatched OrgType / OrganizationTypeId state

Important:
- do NOT break existing API contracts
- do NOT break JWT consumers
- you may keep the OrgType column temporarily if needed for compatibility, but it must no longer be authoritative
- if safe and practical, prepare the codebase so dropping OrgType later becomes a trivial migration

Optional if safe:
- add `org_type_id` claim while keeping `org_type`
- keep `org_type` derived, not sourced from the column
