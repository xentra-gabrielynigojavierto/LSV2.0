==================================================
OBJECTIVE
==================================================

Implement task creation governance for Synq Liens using the Flow service as the execution engine.

This feature must ensure that new tasks created under My Tasks are governed by Flow-backed workflow rules, while Synq Liens remains the product-facing configuration and business-context surface.

For governed tenants, new tasks must:

1. be created through / aligned with the Flow service
2. be assigned to a responsible user
3. be linked to a case
4. enter the My Tasks workflow in a valid starting stage
5. fire email notifications on create / assign / reassign

This upgrades task intake from:
- Liens-local flexible task entry
to
- Flow-governed operational work intake for Synq Liens

This is NOT:
- assignment intelligence
- workload balancing
- approval routing
- SLA enforcement
- a full intake rules engine
- full case/lien workflow unification

==================================================

Use v2 shared services only:
- Identity

- Flow service (as the workflow/task execution engine)

Use existing patterns:
- current Synq Liens product settings governance patterns
- current Flow service patterns where task/workflow runtime ownership already exists

- audit and notification publishing patterns

CRITICAL ARCHITECTURE RULE:

- Flow service must be treated as the source of truth for task/workflow runtime behavior in this feature area
- Synq Liens must remain the product-facing configuration + business-context layer

Do NOT:
- redesign the full platform architecture
- add a heavy policy engine
- introduce SLA/timer logic
- add role-based routing intelligence

- add required-task gating
- add due-date governance unless already necessary for compatibility
- fake multi-assignee if runtime model does not support it

Keep this feature focused on:
- required assignee
- required case link
- valid workflow entry stage
- Flow-aligned task create / assign / reassign email notifications

STEP 4 — ENFORCE FLOW-ALIGNED TASK CREATION GOVERNANCE
==================================================

Backend must be the source of truth.

This feature must govern NEW task creation aligned with Flow-backed runtime behavior.

--------------------------------------------------
4.1 Creation Validation
--------------------------------------------------

Update the task creation flow so the following are enforced for NEW task creation:

If `requireAssigneeOnCreate = true`
- task creation must fail if no valid assignee is provided

If `requireCaseLinkOnCreate = true`
- task creation must fail if caseId is missing
- case must belong to same tenant

If `requireWorkflowStageOnCreate = true`
- task creation must result in a valid workflow stage in the My Tasks workflow
- if not supplied, derive it from start-stage rules
- stage must belong to the active My Tasks workflow used by the Flow-backed runtime

--------------------------------------------------
4.2 Start Stage Behavior
--------------------------------------------------

Implement start-stage behavior:

Mode: FIRST_ACTIVE_STAGE
- choose first active stage ordered by stage order from the My Tasks workflow

Mode: EXPLICIT_STAGE
- use configured explicitStartStageId
- validate it belongs to the active My Tasks workflow and is active

Creation rules:
- initial stage placement is always allowed
- subsequent movement remains governed by transition rules
- do NOT require a prior transition for initial workflow entry

--------------------------------------------------
4.3 Assignee Model Reality
--------------------------------------------------

IMPORTANT:
Respect the real runtime assignment model.

If the current Flow/task runtime supports only one assignee:
- enforce exactly one assignee when required
- do NOT fake multi-assignee in UI/API

If multiple assignees are already truly supported cleanly end-to-end:
- allow one or more only when `allowMultipleAssignees = true`

Document clearly:
- what the codebase actually supports
- what you implemented
- what remains future work

--------------------------------------------------
4.4 Transitional Architecture Rule
--------------------------------------------------

If task creation is still currently executed in Liens service rather than directly in Flow:
- align validation and stage resolution to Flow-backed workflow rules
- avoid introducing more long-term divergence
- document exactly what remains transitional

Do NOT create a separate permanent Liens-only governance path.

--------------------------------------------------
4.5 Error Behavior
--------------------------------------------------

Return clear validation messages such as:
- "Task assignee is required."
- "Task must be linked to a case."
- "A valid My Tasks workflow stage is required for task creation."
- "Selected assignee is not valid for this tenant."

Do not return vague errors.

STEP 6 — IMPLEMENT EMAIL NOTIFICATION RULES
==================================================

Use the v2 Notifications service and existing platform notification patterns.

--------------------------------------------------
6.1 Required Email Events
--------------------------------------------------

Email must fire for:
1. task created with assignee already present
2. task assigned after creation
3. task reassigned to a new assignee

--------------------------------------------------
6.2 Anti-Duplicate Rule
--------------------------------------------------

If task is created already assigned:
- send ONE combined email only
- do NOT send both:
  - task created
  - task assigned

Preferred combined semantic:
- TASK_CREATED_ASSIGNED

If task is created without assignee and later assigned:
- send TASK_ASSIGNED

If task is reassigned:
- send TASK_REASSIGNED to the new assignee

--------------------------------------------------
6.3 Minimum Email Payload
--------------------------------------------------

Include at minimum:
- tenantId
- taskId
- task title
- caseId or case reference
- lien context if available
- assigned user
- created by / assigned by
- priority
- due date
- workflow stage
- direct task link if current URL conventions support it

--------------------------------------------------
6.4 Event Taxonomy
--------------------------------------------------

Use the cleanest existing notification taxonomy consistent with platform patterns.

Recommended logical events:
- liens.task.created_assigned
- liens.task.assigned
- liens.task.reassigned

If platform standard differs, align to it and document clearly.

--------------------------------------------------
6.5 Triggering Rules
--------------------------------------------------

Create path:
- if task created with assignee present → send created-assigned email

Assignment path:
- if task previously had no assignee and now has one → send assigned email

Reassignment path:
- if assignee changed from A to B → send reassigned email to B

Do NOT emit duplicate emails for the same user action.

--------------------------------------------------
6.6 Failure Handling
--------------------------------------------------

Follow existing notification publishing pattern.

Preferred behavior:
- task create/update succeeds
- notification publish is fire-and-observe / non-blocking if that matches platform behavior
- notification failures are logged/audited according to current platform pattern

Document exact behavior.
