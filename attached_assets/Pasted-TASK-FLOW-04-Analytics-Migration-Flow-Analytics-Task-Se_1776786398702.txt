TASK-FLOW-04 — Analytics Migration (Flow Analytics → Task Service)

OBJECTIVE:
Remove Flow analytics dependency on the deprecated `flow_workflow_tasks` shadow model by moving task analytics sourcing to the Task service. This closes the final remaining read dependency identified after TASK-FLOW-03, where `FlowAnalyticsService` still relied on complex aggregates over `flow_workflow_tasks`. :contentReference[oaicite:0]{index=0}

This step must deliver Task-backed analytics for:
- SLA analytics
- queue analytics
- assignment analytics
- dashboard summary composition

CONTEXT:
After TASK-FLOW-03:
- Flow no longer owns task execution state
- `flow_workflow_tasks` is dropped or being dropped
- Flow runtime reads already come from Task service
- the only remaining shadow dependency is analytics, explicitly deferred to TASK-FLOW-04 because the existing analytics queries are complex aggregates and require Task-native analytics support. :contentReference[oaicite:1]{index=1}

NON-NEGOTIABLE RULES:
- DO NOT recreate `flow_workflow_tasks`
- DO NOT move Flow orchestration tables
- DO NOT merge Flow and Task databases
- DO NOT redesign workflow orchestration
- DO NOT break existing analytics endpoint contracts unless absolutely necessary
- MUST create report FIRST and update incrementally
- MUST validate against the current codebase before implementing
- MUST preserve tenant isolation and auth behavior
- MUST keep Task service generic enough to support future product analytics, not just Flow hacks
- NO placeholders

CRITICAL GOAL:
After this step:
- `FlowAnalyticsService` no longer queries `flow_workflow_tasks`
- Task service exposes analytics/query endpoints sufficient for Flow analytics needs
- Flow becomes a pure consumer of Task analytics
- the Flow → Task decoupling lane is fully complete for runtime + analytics
