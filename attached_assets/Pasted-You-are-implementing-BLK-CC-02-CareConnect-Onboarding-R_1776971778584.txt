--------------------------------------------------

PART C — MAKE ONBOARDING RESUMABLE

--------------------------------------------------

Refactor ProviderOnboardingService so onboarding becomes resumable.

Target behavior:

### Case 1 — brand new onboarding
1. Validate provider is eligible
2. Call Tenant service to provision tenant
3. Persist returned tenant info into pending fields
4. Attempt Identity membership assignment
5. On success:
   - provider.MarkTenantProvisioned(tenantId)
   - clear pending fields
   - mark onboarding completed

### Case 2 — retry after partial failure
If provider has pending tenant info already stored:
- DO NOT call Tenant service provision again immediately
- reuse existing PendingTenantId / PendingTenantCode / PendingTenantSubdomain
- retry Identity membership assignment
- on success:
  - complete onboarding
  - clear pending fields

### Case 3 — duplicate code / pre-existing tenant known from prior attempt
If tenant already exists and matches the stored pending tenant state:
- treat as resumable
- do NOT fail permanently just because code is taken

PART E — FAILURE HANDLING MODEL

--------------------------------------------------

Implement explicit handling for:

1. Tenant service succeeds, Identity fails
   - store pending tenant state
   - provider remains non-TENANT until membership succeeds
   - retry resumes from pending state

2. Tenant service returns duplicate code
   - if no pending state exists, surface conflict
   - if pending state exists and code/subdomain match prior attempt, allow recovery path

3. Identity service unavailable
   - preserve pending state
   - mark onboarding status appropriately
   - return retryable failure

4. Provider already TENANT
   - reject cleanly / idempotent success as appropriate

5. Provider not COMMON_PORTAL
   - reject cleanly

6. Missing IdentityUserId
   - reject cleanly
