================================================================================
OBJECTIVE
================================================================================

Standardize the Notifications producer contract so the Notifications service
is a standalone, reusable, product-agnostic shared platform service.

The service must be usable by LegalSynq products and by future unrelated
platforms without embedding LegalSynq-specific business logic in the contract.

Products must adapt to Notifications.
Notifications must not adapt itself to each product.

================================================================================
STANDING ARCHITECTURE RULE
================================================================================

Do NOT embed LegalSynq-specific business workflows, terminology, or product
rules into the Notifications domain, API contracts, routing logic, or template
selection beyond configuration, metadata, and adapters.

================================================================================
CANONICAL CONTRACT REQUIREMENTS
================================================================================

The canonical producer request should support at minimum:

- tenantId
- productKey
- eventKey
- templateKey (optional/recommended)
- channel or channels
- recipient or recipients
- payload
- metadata
- correlationId
- idempotencyKey
- sourceSystem
- requestedBy
- priority (optional if practical)

Rules:

1. productKey
   - stable product/module identifier
   - lowercase slug/kebab style
   - no UI labels
   - no LegalSynq-only assumption

2. eventKey
   - stable business event identifier
   - not channel-specific
   - not template-specific

3. templateKey
   - explicit template identifier when needed
   - separate concern from eventKey

4. payload
   - render data only
   - concise and deliberate
   - no giant domain dumps

5. metadata
   - operational/context data only
   - not a dumping ground
   - not for secrets
   - not the main rendering model

6. recipient model
   - generic
   - explicit type-based recipient specification
   - no product-specific recipient assumptions

================================================================================
IMPLEMENTATION GUIDELINES
================================================================================

- Reuse existing layers where practical
- Prefer additive, backward-compatible changes
- If legacy contracts exist, add compatibility mapping rather than breaking them
- Introduce contract versioning only if necessary
- Keep endpoint handlers thin
- Put mapping/validation logic in service/application layer
- Shared client/helper should reduce producer-side drift
