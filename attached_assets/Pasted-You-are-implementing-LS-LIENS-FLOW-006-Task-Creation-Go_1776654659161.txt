==================================================
OBJECTIVE
==================================================

Implement governed task creation for Synq Liens so new tasks follow minimum operational requirements defined by the tenant.

This feature must ensure, for governed tenants:

1. a task must be assigned
2. a task must be linked to a case
3. a task must enter the workflow in a valid starting stage
4. task creation and assignment must send email notifications through the v2 Notifications service

This upgrades task creation from:
- generic / flexible entry
to
- accountable operational work intake

This is NOT:
- assignment intelligence
- workload balancing
- approval routing
- SLA enforcement
- a full intake rules engine
- required-task gating for transitions

STEP 6 — IMPLEMENT EMAIL NOTIFICATION RULES
==================================================

Use the v2 Notifications service and existing task notification patterns.

--------------------------------------------------
6.1 Required Email Events
--------------------------------------------------

Email must fire for:

1. task created with assignee already present
2. task assigned after creation
3. task reassigned to a new assignee

--------------------------------------------------
6.2 Anti-Duplicate Rule
--------------------------------------------------

If task is created already assigned:
- send ONE combined email only
- do NOT send both:
  - task created
  - task assigned

Preferred template/event meaning:
- TASK_CREATED_ASSIGNED

If task is created without assignee and later assigned:
- send TASK_ASSIGNED

If task is reassigned:
- send TASK_REASSIGNED to the new assignee
- optionally document whether previous assignee is not notified in this phase

--------------------------------------------------
6.3 Minimum Email Payload
--------------------------------------------------

Include at minimum:
- tenantId
- taskId
- task title
- caseId or case reference
- linked lien context if relevant and available
- assigned user
- created by / assigned by
- priority
- due date
- workflow stage
- direct task link if existing URL conventions support it

--------------------------------------------------
6.4 Email Templates / Event Taxonomy
--------------------------------------------------

Use the cleanest available notification event taxonomy consistent with existing patterns.

Recommended events:
- liens.task.created_assigned
- liens.task.assigned
- liens.task.reassigned

If existing notification event naming conventions differ, align to the platform standard and document it.

--------------------------------------------------
6.5 Triggering Rules
--------------------------------------------------

Create path:
- if task created with assignee present → send created-assigned email

Assignment path:
- if task previously had no assignee and now has one → send assigned email

Reassignment path:
- if assignee changed from A to B → send reassigned email to B

Do NOT emit duplicate emails for the same user action.

--------------------------------------------------
6.6 Failure Handling
--------------------------------------------------

Task creation/update should not silently fail due to email notification issues if existing platform pattern is fire-and-observe.

Follow existing notification publishing pattern:
- create/update task succeeds
- notification publish failure is logged/audited per platform pattern if already supported

Document exact behavior.
