OBJECTIVE

Take Flow from a product-consumable workflow service to an operationally hardened, production-grade, multi-product platform capability by completing real product-side integrations, introducing a dedicated workflow-instance model, centralizing authorization constants/policies, executing DB-backed validation, and tightening operational behavior without breaking the bounded-service architecture established in earlier phases.

This phase is operational hardening and real product integration, not shared-DB consolidation or wholesale workflow-engine redesign.

SCOPE INCLUDES

- product-side integration from SynqLien, CareConnect, and SynqFund into Flow APIs
- dedicated WorkflowInstance model introduction
- migration away from TaskItem as workflow-instance surrogate
- safe migration/data transition updates
- permission/capability constant centralization
- DB-backed and end-to-end validation
- operational hardening for logging/config/adapter safety
- minimal product UI validation needed to prove real usage
- documentation updates

DO NOT BUILD

- shared DB merge
- direct DB access from product services into Flow
- full enterprise event bus rollout
- broad workflow redesign
- broad UI redesign across all products
- deep analytics/reporting suite
- generalized external integrations unless required for chosen product paths
- full platform authorization redesign

CRITICAL REQUIREMENTS

You MUST:
- preserve Flow as a bounded shared service under /apps/services/flow
- keep Flow API-first and DB-isolated
- introduce a real WorkflowInstance model
- migrate product mapping away from TaskItem-as-instance
- centralize new permission/capability strings into shared platform constants
- wire real product-side integrations through APIs/gateway, not DB access
- validate real end-to-end product-linked workflow paths
- harden operational behavior enough for supportable multi-product usage
- document all migration/data/runtime assumptions clearly

CURRENT CONTEXT FROM PHASE 3

Flow already has:
- real event emission from WorkflowService and TaskService
- Audit + Notifications triggered via dispatcher
- product-aware authorization policies
- ProductWorkflowMapping entity and APIs
- product-facing Flow endpoints for synqlien / careconnect / synqfund
- safe tenant enforcement and gateway exposure

Known Phase 3 findings to address now:
- product-side adapters/callers are not yet built
- WorkflowInstanceTaskId is still a TaskItem surrogate
- permission claim strings are inline and should be centralized
- DB-backed validation was limited by local environment

IMPLEMENTATION REQUIREMENTS

1. DEDICATED WORKFLOW INSTANCE MODEL

Requirements:
- introduce a dedicated WorkflowInstance entity/model
- clearly separate workflow definition/template vs workflow instance vs tasks
- update Flow services/APIs/mapping model to use the new instance concept
- migrate ProductWorkflowMapping to reference WorkflowInstanceId instead of WorkflowInstanceTaskId
- keep backward compatibility where practical, or document any controlled contract change clearly

2. MIGRATION / DATA TRANSITION

Requirements:
- create EF/Core migrations for the new WorkflowInstance model and mapping updates
- safely handle existing data, including:
  - legacy "default" tenant rows
  - existing ProductWorkflowMappings
  - existing workflow/task records that need instance correlation
- prefer explicit, idempotent, well-documented migration logic
- do not perform silent destructive migration

3. AUTHORIZATION / CAPABILITY HARDENING

Requirements:
- centralize Flow-related permission/capability constants into the shared platform location (for example BuildingBlocks.PermissionCodes)
- replace inline strings in Flow and product integration code with centralized constants
- validate policy behavior for platform admins, tenant admins, product-scoped users, and unauthorized users
- keep any dev-only fallback explicitly environment-gated and justified

4. REAL PRODUCT-SIDE INTEGRATION

Requirements:
- implement the maximum safe subset of real product-side integration for:
  - SynqLien
  - CareConnect
  - SynqFund
- product services should call Flow through API/gateway patterns
- products should be able to create/correlate workflows from real product context
- products should be able to retrieve workflow status and progress relevant tasks where appropriate
- do not introduce direct DB joins/shared-table access

5. END-TO-END VALIDATION

Requirements:
- validate at least one real product-linked path for each chosen target subset
- validation should cover:
  - auth
  - tenant enforcement
  - product-to-flow correlation
  - audit path
  - notification path where applicable
  - runtime startup/gateway/API behavior
- use real DB-backed validation where available
- if environment limitations prevent full validation, perform the maximum safe subset and document the exact gap

6. OPERATIONAL HARDENING

Requirements:
- improve logging around workflow execution and integration points as needed
- ensure adapter failures remain safe and diagnosable
- review retry/timeout/config behavior for HTTP integrations
- improve health/info/status/config clarity as needed
- document operational run commands and config expectations clearly

7. PRODUCT UX VALIDATION

Requirements:
- add the minimum safe UI integration needed in product surfaces to prove real usage
- examples:
  - launch/start workflow from product record
  - view correlated workflow status from product screen
  - navigate to Flow task/workflow detail from product context
- keep UI changes narrow and validation-oriented
- do not perform broad redesign
