OBJECTIVE
==================================================

Complete the core CareConnect referral acceptance flow so a law firm can send a referral to a provider, the provider can securely receive and review it through authenticated access, accept it, and trigger the correct confirmation notifications.

This feature must provide:

1. law firm provider search and provider selection for referral creation
2. referral submission with canonical initial state
3. provider notification email with secure referral link
4. login-required referral access from email
5. post-login redirect to the correct referral
6. provider acceptance action
7. acceptance confirmation emails to:
   - law firm
   - client
8. law firm UI reflecting accepted state

Do NOT expand into booking or appointments.
Do NOT expand into scheduling or availability management.
Do NOT redesign unrelated CareConnect features.

This feature finishes the provider intake handshake end to end.

==================================================
REQUIREMENTS
==================================================

--------------------------------------------------
A. LAW FIRM SEARCH / SELECT PROVIDER
--------------------------------------------------

Use the current CareConnect provider search experience.

Requirements:
- law firm can search providers using the existing provider search flow
- law firm can select exactly one provider for a referral
- selected provider context carries into referral submission flow
- do not redesign provider search unless needed to complete this feature safely

Outcome:
- referral submission uses the selected provider deterministically

--------------------------------------------------
B. REFERRAL SUBMISSION
--------------------------------------------------

Law firm submits a referral for a client.

Requirements:
- referral record is created successfully
- selected provider is assigned correctly
- client/referral information is persisted
- initial referral state is canonical:
  - New

Rules:
- preserve existing referral creation security and validation
- do not introduce appointment coupling
- no regression to existing referral creation behavior

On successful creation:
- trigger provider referral email
- return a meaningful success state to law firm

--------------------------------------------------
C. PROVIDER REFERRAL EMAIL
--------------------------------------------------

When referral is created successfully, send a provider-facing referral email.

Email must include:
- provider-facing referral notice
- concise referral summary
- sufficient context to know what the referral is about
- CTA link/button:
  - View Referral

Security rule:
- the email link must NOT directly reveal referral details without authentication
- it must support secure token/context validation plus login-required routing

Use existing email infrastructure where possible.

--------------------------------------------------
D. SECURE EMAIL LINK → LOGIN → RETURN FLOW
--------------------------------------------------

When provider clicks the referral email link:

1. validate token / referral context safely
2. if not authenticated:
   - redirect to login
   - preserve safe returnTo
3. after successful login:
   - redirect provider to the correct referral detail page

Requirements:
- no open redirect vulnerability
- invalid, expired, or revoked token fails safely
- wrong authenticated account must not gain access
- authenticated access must still enforce provider participation/authorization

Do NOT rely on public unauthenticated referral detail access for this feature.
If a public flow still exists for backward compatibility, it must not be the primary intended path here.

--------------------------------------------------
E. AUTHENTICATED PROVIDER REFERRAL VIEW
--------------------------------------------------

After login, provider lands on the referral detail page.

Requirements:
- provider must be authorized to view the referral
- provider must be on the receiving side / valid participant
- invalid provider access must fail safely
- referral detail page must expose Accept action only when valid

If referral is already accepted:
- show accepted state
- do not show duplicate accept action

--------------------------------------------------
F. PROVIDER ACCEPTANCE
--------------------------------------------------

Provider must be able to accept the referral from the authenticated referral detail page.

Required transition:
- New → Accepted

Preconditions:
- provider is authenticated
- provider is authorized
- referral is currently New
- correct receiving provider/participant context is present

Requirements:
- backend-enforced transition only
- duplicate acceptance blocked safely
- transition history recorded
- no duplicate side effects on repeated click/retry
- use existing corrected referral state machine rules from LSCC-01-001-01

This feature does NOT need to expand rejection UX.
If reject already exists in backend, do not broaden or redesign it here.

--------------------------------------------------
G. ACCEPTANCE CONFIRMATION EMAILS
--------------------------------------------------

On successful provider acceptance, send:

1. Law firm acceptance email
2. Client acceptance email

Law firm email should include:
- provider accepted referral
- client/referral summary
- provider identity
- any concise next-step language if appropriate

Client email should include:
- provider accepted referral
- provider name
- simple acknowledgement / next-step message

Rules:
- send only on first successful acceptance
- repeated accept attempts must not duplicate sends
- use existing notification/email infrastructure where possible

If client email is missing:
- do NOT block acceptance
- send what is possible
- log/document the missing client email behavior clearly

--------------------------------------------------
H. LAW FIRM ACCEPTED-STATE VISIBILITY
--------------------------------------------------

After provider accepts:

- law firm referral list must show Accepted
- law firm referral detail page must show Accepted
- refresh/reload must show persisted accepted state
- no stale New state after backend success

Use canonical status display consistent with LSCC-01-001-01.

--------------------------------------------------
I. BACKEND ENFORCEMENT
--------------------------------------------------

Backend must remain the final authority for:

- token validation
- authenticated referral access
- provider participation checks
- acceptance transition validity
- duplicate acceptance prevention

No client should be able to:
- set Accepted arbitrarily
- bypass login
- bypass provider authorization
- bypass state machine rules

--------------------------------------------------
J. DATA INTEGRITY / SECURITY RULES
--------------------------------------------------

Must enforce:

1. no public unauthenticated referral detail exposure
2. no invalid provider access to referral
3. no duplicate acceptance side effects
4. no stale/revoked token bypass
5. no cross-tenant / cross-org leakage
6. no open redirect behavior

If the wrong provider account logs in:
- deny access safely
- do not leak additional information

--------------------------------------------------
K. EDGE CASES
--------------------------------------------------

Handle safely:

1. invalid email link
- safe invalid page/message

2. expired or revoked link
- safe invalid page/message

3. provider clicks link after referral already accepted
- show accepted state
- no duplicate accept

4. provider logs in with wrong account
- block access safely

5. client email missing
- acceptance still succeeds
- client email skipped/logged
- law firm email still sent if possible

6. stale UI/retry/double click
- no duplicate transition/history/email side effects

--------------------------------------------------
L. TESTING (MINIMUM)
--------------------------------------------------

Add focused tests for:

Backend:
1. referral created in New
2. provider email link resolves into authenticated flow correctly
3. invalid/expired/revoked token blocked safely
4. wrong provider access blocked safely
5. provider acceptance allowed only for correct actor/state
6. duplicate acceptance blocked
7. law firm acceptance email triggered on first successful acceptance
8. client acceptance email triggered on first successful acceptance when email exists
9. missing client email does not block acceptance

Frontend:
1. provider redirected to login with preserved returnTo
2. authenticated provider lands on correct referral
3. accept action visible only in valid provider/state combinations
4. law firm sees accepted status after acceptance

If full frontend auth-flow tests are difficult:
- test URL builder / redirect logic
- test backend acceptance logic thoroughly
- test component action visibility separately
- document any UI testing limitation in the report
