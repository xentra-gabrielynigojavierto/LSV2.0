IMPLEMENTATION REQUIREMENTS

1. ATOMIC OWNERSHIP ENFORCEMENT IN FLOW

Goal:
Move ownership and parent-correlation validation fully into Flow so it is checked atomically in the same request that reads or mutates workflow execution state.

Requirements:
- add new Flow endpoints that include parent/product context in the route and validate ownership server-side in Flow
- the endpoint must validate in one request:
  - tenant match
  - product match
  - sourceEntityType match
  - sourceEntityId / parent match
  - workflow instance ownership
  - caller authorization/capability
- support at minimum:
  - get workflow instance by parent context
  - advance workflow instance by parent context
  - complete workflow instance by parent context
- the legacy direct instance execution endpoints may remain, but product services should be updated to use the new atomic ownership-aware endpoints
- return a clear deny response such as 404 or a documented equivalent error code when the workflow instance does not belong to the parent context

Recommended endpoint shape:
- GET  /api/v1/product-workflows/{product}/{sourceEntityType}/{sourceEntityId}/{workflowInstanceId}
- POST /api/v1/product-workflows/{product}/{sourceEntityType}/{sourceEntityId}/{workflowInstanceId}/advance
- POST /api/v1/product-workflows/{product}/{sourceEntityType}/{sourceEntityId}/{workflowInstanceId}/complete

Rules:
- do not rely on product services to do the authoritative ownership check
- do not introduce shared DB access from product services

2. PRODUCT PASSTHROUGH UPDATE

Requirements:
- update Liens, CareConnect, and Fund product endpoints to call the new atomic ownership-aware Flow endpoints
- remove the old pre-check plus second-call pattern from product passthrough execution paths
- keep product logic thin and orchestration-only
- preserve existing auth and tenant context propagation
- ensure product endpoints still map correct product slug + source entity type values

3. INTEGRATION TEST SUITE

Add automated integration tests for the highest-risk paths.

Minimum required coverage:

A. Ownership / IDOR
- advancing a workflow through the wrong parent route must fail closed
- getting a workflow through the wrong parent route must fail closed
- completing a workflow through the wrong parent route must fail closed

B. Tenant isolation
- tenant A cannot read or mutate tenant B workflow instance
- service token with wrong tenant cannot operate on another tenant's workflow

C. Capability enforcement
- user lacking required product capability cannot start, list, advance, or complete
- product-scoped user for one product cannot operate another product's workflow

D. Multi-auth behavior
- valid user JWT works where expected
- valid service token works where expected
- invalid issuer/audience fails
- missing tenant claims fail
- missing secret / invalid signing key behavior is explicit

E. Transition integrity
- stale expectedCurrentStepKey returns 409
- inactive instance advance/complete returns 409
- concurrent writers result in 409 concurrent_state_change

F. Product correlation
- SynqLien route cannot mutate CareConnect-owned workflow
- CareConnect route cannot mutate Fund-owned workflow
- Fund route cannot mutate Liens-owned workflow

Suggested test layers:
- Flow API integration tests
- Product API passthrough integration tests
- Auth scheme tests
- Concurrency tests where feasible

5. AUTHENTICATION / SECURITY HARDENING

Requirements:
- tighten service-token validation:
  - issuer must match
  - audience must match
  - subject must be present and valid
  - tenant claim must be present and consistent
- make caller type explicit in request context:
  - user
  - service
  - service-on-behalf-of-user if actor is present
- ensure service tokens do not bypass:
  - tenant checks
  - parent ownership checks
  - product capability checks if those still apply to execution
- clearly document auth modes:
  - user bearer
  - service token
  - fallback behavior if any
- in non-development environments, startup or configuration validation should fail clearly if service-token auth is expected but the required secret/config is missing
- preserve existing user-based flows

6. DIAGNOSTICS / ERROR HANDLING CONSISTENCY

Requirements:
- standardize error codes/messages for execution and ownership paths
- ensure at minimum the following codes are used consistently where applicable:
  - workflow_instance_not_owned
  - expected_step_mismatch
  - instance_not_active
  - concurrent_state_change
  - flow_unavailable
  - flow_upstream_error
  - invalid_service_token
  - missing_tenant_context
- ensure logs for execution endpoints always include:
  - workflowInstanceId
  - tenantId
  - productKey

- ensure operationally useful logs exist for:
  - ownership-denied requests
