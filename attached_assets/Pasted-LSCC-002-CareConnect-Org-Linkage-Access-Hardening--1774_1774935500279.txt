REQUIREMENTS
==================================================

--------------------------------------------------
A. PROVIDER-TO-ORGANIZATION LINKAGE HARDENING
--------------------------------------------------

Implement a reliable path for linking providers to organizations so org-based scoping can function in production data.

Requirements:

1. Identify how provider-to-organization linkage is currently represented in the data model.
2. Make provider organization linkage enforceable going forward.
3. Provide a practical backfill path for existing providers with null organization linkage.

Acceptable implementation approaches:
- migration script
- admin-only API endpoint/service
- seed/backfill job
- a combination of the above

Minimum required outcome:
- there is a clear, deterministic way to populate `Provider.OrganizationId`
- new or updated providers cannot remain invalid if project conventions allow enforcing this now
- existing null-linked providers are addressed through an explicit compatibility/backfill strategy

Rules:
- preserve current tenant boundaries
- do not guess organization mappings without a deterministic rule
- if deterministic auto-linking is impossible, implement an explicit admin mapping path and document manual steps

Fallback behavior:
- a provider with no valid organization linkage must NOT widen access
- such providers should be excluded from org-sensitive access flows where necessary
- log or surface the condition in a way consistent with project conventions

--------------------------------------------------
B. ROW-LEVEL REFERRAL ACCESS CONTROL
--------------------------------------------------

Harden referral detail access for:

GET /api/referrals/{id}

Expected behavior:
- allow if caller is PlatformAdmin
- allow if caller is TenantAdmin and current platform conventions allow broader tenant visibility
- allow if caller org is the referring org participant
- allow if caller org is the receiving org participant
- otherwise return 404, not 403

Requirements:
1. Enforce participant access at the row/detail level
2. Reuse shared participant-check logic where possible
3. Do not rely only on list-level filtering
4. Preserve tenant boundary checks

Fallback behavior:
- if org context cannot be resolved for a non-admin user, deny access
- if referral lacks required org relationship data, treat as inaccessible to non-admin users

--------------------------------------------------
C. APPOINTMENT ORG SCOPING
--------------------------------------------------

Implement org-based visibility for appointment reads.

At minimum, apply org-scoping to:
- appointment list/query endpoints
- appointment detail/read endpoints if they exist
- any appointment retrieval path used by CareConnect workflows

Visibility rule:
A caller may read an appointment only if:
- they are admin per platform conventions, OR
- their org participates in the linked referral / organization relationship that owns the appointment context

Use the most reliable existing relationship chain in the domain model, for example:
- appointment -> referral -> referring/receiving orgs
or
- appointment -> organization relationship
depending on actual code structure

Requirements:
1. Enforce appointment scoping in backend logic, not UI
2. Centralize scoping logic where feasible
3. Do not widen appointment access when org linkage is missing
4. Keep behavior consistent with referral visibility rules

Fallback behavior:
- if appointment cannot be tied to a valid participant context, restrict non-admin access
- missing org context for caller must deny access

--------------------------------------------------
D. SHARED PARTICIPANT-SCOPING LOGIC
--------------------------------------------------

Create or extend a reusable helper/service for participant-based access checks.

Goal:
One consistent definition of whether a caller's org is a participant in a CareConnect record.

This shared logic should support:
- referral list scoping
- referral detail access
- appointment list scoping
- appointment detail access

Preferred behaviors:
- explicit method names
- testable inputs/outputs
- no hidden fallback that broadens access

Example intent only (adapt to actual architecture):
- IsReferralParticipant(referral, callerOrgId)
- ApplyReferralOrgScope(query, callerContext)
- IsAppointmentParticipant(appointment, callerOrgId)

Do not duplicate the same org-check logic in multiple endpoints unless unavoidable.
