IN-SCOPE FUNCTIONAL REQUIREMENTS
==================================================

--------------------------------------
1. PRODUCT-TYPE-AWARE GLOBAL TEMPLATES
--------------------------------------

Extend the template system so global templates are grouped by product type.

Required initial product types:
- careconnect

Design for future expansion such as:
- synqlien
- synqfund
- synqrx
- synqpayout

A global template must now be identifiable by:
- productType
- channel
- templateKey
- templateScope

Required uniqueness rule:
- productType + channel + templateKey + templateScope must be unique

For this phase, templateScope must support at minimum:
- global

Design must not block future:
- tenant scope
- tenant-specific overrides of global templates

==================================================
2. TEMPLATE METADATA EXTENSIONS
==================================================

Extend the Template model with at minimum:
- productType
- templateScope
- editorType
- category nullable
- isBrandable boolean

Definitions:
- templateScope: global for now, tenant later
- editorType: wysiwyg, html, text
- isBrandable: whether tenant branding applies at render time

These fields must integrate with existing template APIs and persistence safely.

==================================================
3. WYSIWYG-COMPATIBLE TEMPLATE VERSION STORAGE
==================================================

Extend TemplateVersion so email templates can store both:
- structured editor source
- compiled renderable output

Add/update fields at minimum:
- editorJson nullable
- htmlTemplate
- textTemplate
- designTokensJson nullable
- layoutType nullable

Required behavior:
- for WYSIWYG email templates:
  - editorJson is the editing source of truth
  - htmlTemplate is the compiled renderable HTML
  - textTemplate is the plain-text fallback

Important:
- do NOT store only HTML for WYSIWYG templates
- preserve structured editor representation for future editing

==================================================
4. TENANT BRANDING MODEL
==================================================

Create a new model:

TenantBranding

Fields must include at minimum:
- id
- tenantId
- productType
- brandName
- logoUrl nullable
- primaryColor nullable
- secondaryColor nullable
- accentColor nullable
- textColor nullable
- backgroundColor nullable
- buttonRadius nullable
- fontFamily nullable
- emailHeaderHtml nullable
- emailFooterHtml nullable
- supportEmail nullable
- supportPhone nullable
- websiteUrl nullable
- createdAt
- updatedAt

Required behavior:
- branding is tenant-scoped and product-type-scoped
- one tenant may have different branding for different products

==================================================
5. PRODUCT DEFAULT BRANDING FALLBACK
==================================================

Rendering must not fail if tenant branding is missing.

Required behavior:
- if no TenantBranding exists for tenantId + productType
- use a platform default branding fallback for that productType

Acceptable implementation for this phase:
- DB-backed default branding
OR
- centralized config/code-backed default branding resolver

Behavior must be explicit, centralized, and replaceable later.

==================================================
6. RESERVED BRANDING TOKENS
==================================================

Introduce reserved branding tokens injected automatically at render time.

Required examples:
- {{brand.name}}
- {{brand.logoUrl}}
- {{brand.primaryColor}}
- {{brand.secondaryColor}}
- {{brand.accentColor}}
- {{brand.textColor}}
- {{brand.backgroundColor}}
- {{brand.buttonRadius}}
- {{brand.fontFamily}}
- {{brand.supportEmail}}
- {{brand.supportPhone}}
- {{brand.websiteUrl}}

Important rule:
- these tokens come from branding resolution
- they must NOT be overridden unsafely by caller template data

==================================================
7. RENDER PIPELINE UPDATE
==================================================

Extend the template render pipeline to support:
- templateKey
- productType
- tenantId
- channel
- templateData

Required render order:
1. resolve template by:
   - productType
   - channel
   - templateKey
   - templateScope = global
   - published version
2. resolve branding by:
   - tenantId
   - productType
3. merge:
   - system branding tokens
   - caller template data
4. render:
   - subject
   - html
   - text

Important rule:
- branding must be applied at render time
- do NOT hardcode branding into stored template content at create time

==================================================
8. BRANDED PREVIEW ENDPOINT

Implement:

POST /v1/templates/global/:id/versions/:versionId/preview

Request must support at minimum:
- tenantId
- productType
- templateData

Response must include:
- rendered subject
- rendered html
- rendered text
- branding metadata used optional but helpful

Required behavior:
- preview must simulate real rendering using:
  - selected tenant branding
  - selected product type
  - selected version
  - supplied template variables
