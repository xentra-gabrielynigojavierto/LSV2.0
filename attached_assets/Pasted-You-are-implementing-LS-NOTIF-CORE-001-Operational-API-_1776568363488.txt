FUNCTIONAL REQUIREMENTS:

1. Stats endpoint
   Add:
   GET /v1/notifications/stats

   Must return useful admin metrics such as:
   - total notifications
   - queued count
   - sent count
   - delivered count where provider/webhook data supports it
   - failed count
   - suppressed count
   - channel breakdown
   - provider breakdown when practical
   - recent trend summary
   - status distribution summary

   Support filters where practical:
   - date range
   - status
   - channel
   - product
   - tenant
   - provider

2. Events endpoint
   Add:
   GET /v1/notifications/{id}/events

   Must return chronological event history such as:
   - created
   - queued
   - recipients resolved
   - provider selected
   - attempted
   - sent
   - delivered
   - suppressed
   - retry scheduled
   - retry attempted
   - failed

3. Issues endpoint
   Add:
   GET /v1/notifications/{id}/issues

   Must return actionable issue/failure information such as:
   - validation failures
   - render/template failures
   - suppression issues
   - routing/provider failures
   - delivery failures
   - exhausted route or retry failures

4. Retry endpoint
   Add:
   POST /v1/notifications/{id}/retry

   Rules:
   - only allow retry for retryable states
   - must not duplicate successful deliveries
   - must use existing delivery logic where possible
   - must create audit/log trace
   - must return updated notification state/result

5. Resend endpoint
   Add:
   POST /v1/notifications/{id}/resend

   Rules:
   - resend is distinct from retry
   - resend creates a new linked notification
   - preserve traceability to the original
   - do not silently mutate the original history
   - only support overrides if already safe and practical; otherwise skip overrides in this feature

   GET /v1/notifications

   - page

   - provider
   - recipient

   - to
