OBJECTIVE

Integrate the structurally merged Flow service into the LegalSynq platform as a first-class shared service by introducing Identity, strict tenant enforcement, controlled gateway exposure, and foundational platform adapters for Audit and Notifications, while preserving Flow's independent service boundary and DB ownership.

This phase is platform integration, not product workflow implementation.

SCOPE INCLUDES

- Identity v2 integration for Flow API
- strict tenant resolution and propagation
- removal of unsafe tenant fallback behavior
- environment-safe CORS/platform exposure hardening
- gateway exposure for Flow endpoints
- foundational Audit adapter integration
- foundational Notifications adapter integration
- local/dev run alignment as needed
- minimal frontend updates required for secured/platform-routed access
- fixing the known frontend prerender issue if required for stable validation

DO NOT BUILD

- SynqLien-specific workflow mappings
- CareConnect-specific workflow mappings
- SynqFund-specific workflow mappings
- shared database merge
- direct DB access from other services into Flow
- full event bus implementation
- broad UI redesign
- workflow engine redesign
- broad API redesign unrelated to platform integration
- major frontend workspace consolidation unless trivial and clearly safe

CRITICAL REQUIREMENTS

You MUST:
- preserve Flow as a bounded service under /apps/services/flow
- keep Flow API-first and DB-isolated
- integrate with LegalSynq Identity patterns rather than inventing a parallel auth model
- remove unsafe default tenant fallback behavior
- harden CORS enough for controlled platform exposure
- expose Flow through gateway only after auth and tenant enforcement are correctly wired
- add clean adapter seams for Audit and Notifications
- avoid product-specific logic injection
- keep the result ready for Phase 3 product consumption work

IMPLEMENTATION REQUIREMENTS

1. IDENTITY INTEGRATION

Integrate Flow API with LegalSynq Identity v2/shared auth conventions already used by platform services where feasible.

Requirements:
- add JWT/authentication wiring to Flow API
- validate authenticated user identity on protected endpoints
- make user context available in Flow backend
- make tenant context derivable from authenticated claims/platform conventions
- reuse existing LegalSynq auth patterns if available
- do not redesign Identity itself

2. STRICT TENANT RESOLUTION

Replace unsafe tenant fallback behavior.

Requirements:
- remove any fallback that silently assigns a default tenant when tenant context is missing
- derive tenant context from authenticated identity/platform-approved request context
- reject invalid/missing tenant context where appropriate
- ensure tenant context is available consistently for workflow/task operations and integration events
- do not introduce product-specific tenant rules

3. SECURITY / CORS HARDENING

Requirements:
- replace permissive CORS settings with environment-driven allowed origins
- remove unsafe wildcard-equivalent credential behavior
- preserve explicit local dev viability through config
- ensure middleware ordering is correct for auth, tenant resolution, and authorization

4. INTERNAL EVENT ABSTRACTION

Requirements:
- introduce a lightweight internal event abstraction for Flow actions
- support event types sufficient for audit and notification fan-out:
  - workflow created/started
  - workflow updated/state changed
  - task assigned
  - task completed
  - workflow completed
- keep this local and lightweight
- do not implement a platform-wide event bus

5. AUDIT INTEGRATION

Requirements:
- add an Audit adapter/client seam
- emit audit events for core workflow/task actions
- include entity/action/user/tenant context where available
- use platform-friendly patterns
- avoid direct DB coupling to Audit service
- if full real wiring is unsafe, create the seam and implement the maximum safe subset, documenting any remaining gaps

6. NOTIFICATIONS INTEGRATION

Requirements:
- add a Notifications adapter/client seam
- support initial triggers such as task assignment, approval request, workflow completion
- keep this to system notifications only
- do not build threaded/two-way communications
- if full real wiring is unsafe, implement the maximum safe subset and document the remainder

7. GATEWAY INTEGRATION

Requirements:
- add gateway routing for Flow
- support at minimum:
  - /flow/health
  - /flow/info if applicable
  - /flow/{**catch-all} or equivalent
- preserve auth header/bearer token flow through gateway according to platform conventions
- avoid route collisions
- do not expose insecure anonymous routes unintentionally

8. DEV / RUN ALIGNMENT

Requirements:
- update local run/dev scripts only as needed so Flow can be started consistently with the platform
- document startup commands and dependencies clearly
- adding Flow to LegalSynq.sln is optional and only allowed if clearly low-risk and beneficial
- if not added, document the explicit independent build/start commands

9. FRONTEND ALIGNMENT

Requirements:
- update frontend API configuration as needed for secured/platform-routed backend access
- preserve frontend isolation unless deeper consolidation is trivial and safe
- ensure authenticated requests compile/work structurally
- fix the known `/tasks` prerender issue if needed for stable validation/build success
- do not perform broad UI redesign/refactor work

10. DOCUMENTATION

Update docs under /apps/services/flow/docs to cover:
- service purpose
- authentication model
- tenant model
- gateway model
- audit model
- notifications model
- local run instructions
- explicit Phase 3+ deferrals

Suggested files:

- merge-phase-2-notes.md

11. EXPLICIT DEFERRALS

Document that these remain deferred:
- product-specific workflow templates/mappings
- DB consolidation
- full event bus
- major UI redesign
- workflow engine redesign
- broad workspace consolidation unless trivial
- advanced authorization redesign beyond current platform Identity patterns
